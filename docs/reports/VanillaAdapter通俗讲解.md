# VanillaAdapter适配层通俗讲解

## 一、问题是什么？（用生活例子解释）

### 1.1 vanilla游戏的规则

想象RimWorld的子弹系统就像一个**快递配送系统**：

1. **发货点（origin）**：快递从哪里发出
2. **目的地（destination）**：快递要送到哪里
3. **收件人（usedTarget）**：快递单上写的收件人

vanilla的规则：
- **墙壁拦截**：快递车离发货点越远，越容易被路上的障碍物拦住
- **优先送达**：如果快递单上有收件人，不管距离多远，都会优先送给他
- **一次性配送**：快递发出后，目的地就固定了，不能改

### 1.2 BDP追踪弹的需求

BDP的追踪弹就像**智能无人机配送**：

- **实时追踪**：目标移动了，无人机会调整路线
- **绕过障碍**：遇到墙壁，无人机会绕过去
- **切换目标**：原目标消失了，无人机会找新目标

### 1.3 冲突在哪里？

**冲突1：墙壁穿透问题**

vanilla规则：快递车离发货点越远，越容易被拦住

BDP的做法：每秒都重新"发货"（重定向），把当前位置当作新的发货点

结果：发货点永远贴着无人机，距离永远很近 → 墙壁永远拦不住 → **无人机穿墙而过**

**类比**：
- 就像你每走一步，就把起点改成当前位置
- 那么你永远"刚出发"，路上的障碍物就失效了

**冲突2：幽灵命中问题**

vanilla规则：快递单上的收件人，不管多远都会送达

BDP的做法：无人机飞行途中切换了目标，但快递单没改

结果：无人机飞到地图另一端，突然"隔空送达"给最初的收件人 → **幽灵命中**

**类比**：
- 就像你改了收货地址，但快递单没更新
- 快递员到了新地址，却把包裹送回了旧地址

**冲突3：目标未就绪问题**

vanilla规则：快递发出时，目的地就固定了

BDP的做法：引导弹的真实目标要在飞行途中才确定

结果：快递单上写的是错误的地址，需要飞行途中修改

**类比**：
- 就像你下单时还不知道收货地址
- 快递发出后，才告诉快递员真实地址

## 二、解决方案是什么？（适配层的作用）

### 2.1 核心思想

**不改变vanilla的规则，而是"欺骗"vanilla**

就像给快递系统装一个**智能翻译器**：
- 无人机说"我要去A点"
- 翻译器告诉快递系统"他要去B点"
- 快递系统按B点的规则处理
- 无人机实际飞向A点

### 2.2 三个"欺骗"策略

**策略1：假装发货点很远（解决墙壁穿透）**

vanilla看到的：发货点在6格外（墙壁可以拦截）
实际情况：发货点就在当前位置（无人机实时调整）

**类比**：
- 无人机每次调整路线时，告诉快递系统"我是从6格外发出的"
- 快递系统就会正常检查墙壁拦截
- 但无人机实际上是从当前位置出发的

**策略2：偷偷修改快递单（解决幽灵命中）**

vanilla看到的：快递单上的收件人是当前目标
实际情况：无人机中途切换了目标，适配层自动更新快递单

**类比**：
- 无人机切换目标时，适配层偷偷把快递单上的收件人改成新目标
- vanilla检查快递单时，看到的就是正确的收件人
- 不会出现"送错人"的情况

**策略3：记住真实发货点（解决目标未就绪）**

vanilla看到的：发货点可能会变化（因为重定向）
适配层记住的：真实的发货点永远不变

**类比**：
- 适配层像一个记事本，记住了最初的发货点
- 即使快递系统看到的发货点在变化
- 适配层始终知道真实的起点在哪里

## 三、适配层怎么工作？（工作流程）

### 3.1 初始化阶段

**步骤1：记录真实发货点**

```
无人机发射 → 适配层记录origin → 永久保存
```

**类比**：
- 就像快递员出发前，在本子上记下仓库地址
- 即使后面改了很多次路线，仓库地址永远不变

**步骤2：配置适配策略**

```
检测弹道类型 → 决定启用哪些适配
- 追踪弹：启用origin偏移 + usedTarget同步
- 引导弹：只启用origin偏移
- 普通弹：不启用任何适配
```

**类比**：
- 就像快递员根据包裹类型选择配送方式
- 生鲜快递需要冷链，普通快递不需要

### 3.2 飞行阶段

**每次重定向时**：

```
1. 无人机计算新方向
2. 适配层决定是否"欺骗"origin
   - 如果是追踪/引导Phase → origin后退6格
   - 如果是普通Phase → origin = 当前位置
3. vanilla用"欺骗"后的origin检查墙壁拦截
4. 无人机继续飞行
```

**类比**：
- 无人机每次调整路线，适配层就"翻译"一次
- vanilla看到的是"翻译"后的信息
- 无人机按真实路线飞行

### 3.3 命中阶段

**到达目标时**：

```
1. 适配层检查Phase
   - 如果是TrackingLost/Free → 强制打地面
   - 如果是Tracking/FinalApproach → 同步usedTarget
2. 适配层修改usedTarget（如果需要）
3. vanilla用修改后的usedTarget判断命中
4. 命中正确的目标
```

**类比**：
- 快递员到达目的地前，适配层检查快递单
- 如果收件人地址变了，适配层偷偷改快递单
- 快递员按新快递单送货，不会送错

## 四、为什么这个方案好？（优势）

### 4.1 职责清晰（各司其职）

**之前**：
- 追踪模块：计算追踪方向 + 处理vanilla冲突 + 修改命中判定
- 职责混乱，代码难懂

**现在**：
- 追踪模块：只计算追踪方向（纯粹的弹道逻辑）
- 适配层：只处理vanilla冲突（纯粹的兼容逻辑）
- 各司其职，代码清晰

**类比**：
- 就像餐厅分工：厨师做菜，服务员上菜
- 不会让厨师既做菜又上菜，职责混乱

### 4.2 易于理解（新人友好）

**之前**：
- 看追踪模块代码，到处是hack和补丁
- 新人：为什么要后退origin？为什么要修改usedTarget？
- 需要理解vanilla机制才能看懂代码

**现在**：
- 追踪模块：纯粹的追踪逻辑，一目了然
- 适配层：详细注释说明每个hack的原因
- 新人：看追踪模块不会被困扰，看适配层能理解所有冲突

**类比**：
- 就像说明书分成"使用指南"和"故障排除"
- 正常使用时只看使用指南
- 遇到问题时再看故障排除

### 4.3 易于维护（集中管理）

**之前**：
- hack分散在各个模块
- 修改一个hack，要找遍所有模块
- 容易遗漏，容易出错

**现在**：
- 所有hack集中在适配层
- 修改一个hack，只需改适配层
- 不会影响模块代码

**类比**：
- 就像把所有工具放在工具箱里
- 需要工具时，只去工具箱找
- 不会到处乱放，找不到

### 4.4 易于扩展（未来友好）

**场景1：RimWorld更新，拦截机制变化**
- 只需修改适配层的origin计算方法
- 模块代码完全不受影响

**场景2：添加新的弹道类型（如螺旋弹）**
- 实现新模块，产出螺旋方向
- 调用适配层配置策略
- 适配层自动处理vanilla兼容

**场景3：需要自定义拦截逻辑**
- 在适配层添加新方法
- 不影响现有代码

**类比**：
- 就像手机的充电接口
- 换了新充电器，只需换接口适配器
- 不需要换整个手机

### 4.5 零性能开销（高效）

**适配层只是计算逻辑**：
- 没有额外的数据结构
- 没有额外的内存占用
- 只是简单的条件判断和赋值

**相比原来的实现**：
- 性能完全相同
- 只是代码重新组织
- 不会变慢，也不会变快

**类比**：
- 就像整理房间
- 东西还是那些东西
- 只是摆放位置变了，更整齐了

## 五、具体怎么做？（实施步骤）

### 5.1 第一步：创建适配层

**做什么**：
- 创建`VanillaAdapter.cs`文件
- 实现三个核心方法：
  1. `ComputeAdaptedOrigin` — 计算"欺骗"后的origin
  2. `TrySyncUsedTarget` — 同步usedTarget
  3. `CheckBeforeImpact` — 命中前检查

**类比**：
- 就像给快递系统装一个翻译器
- 翻译器有三个功能：翻译地址、修改快递单、检查收件人

### 5.2 第二步：集成到子弹系统

**做什么**：
- 在`Bullet_BDP`中添加适配层实例
- 在初始化时配置适配层策略
- 在重定向时调用`ComputeAdaptedOrigin`
- 在命中前调用`CheckBeforeImpact`

**类比**：
- 就像把翻译器安装到快递车上
- 快递车每次调整路线，都通过翻译器
- 快递车到达目的地前，都通过翻译器检查

### 5.3 第三步：简化追踪模块

**做什么**：
- 删除`TrackingModule`中的`ResolveHit`方法
- 删除`IBDPHitResolver`接口实现
- 保留纯粹的追踪逻辑

**类比**：
- 就像把快递员的"翻译"工作交给翻译器
- 快递员只负责开车送货
- 不再需要自己"翻译"地址

### 5.4 第四步：测试验证

**测试什么**：
1. 墙壁拦截：子弹会被墙壁拦住（不穿墙）
2. 目标切换：子弹命中正确的目标（不幽灵命中）
3. 引导飞行：子弹绕过障碍物（正常工作）
4. 丢锁处理：子弹打地面（不命中尸体）

**类比**：
- 就像测试翻译器是否工作正常
- 发几个快递，看看是否送对地方
- 遇到障碍物，看看是否正常绕过

## 六、常见疑问解答

### Q1：为什么不直接修改vanilla的代码？

**A**：
- vanilla是游戏引擎，我们不能修改
- 只能通过继承和组合来"欺骗"vanilla
- 适配层就是这个"欺骗"的工具

**类比**：
- 就像你不能改快递公司的规则
- 但你可以在包裹上贴标签，让快递员按你的方式送

### Q2：为什么origin要后退6格？

**A**：
- vanilla的拦截规则：origin到墙壁 < 5格 → 拦截概率0%
- 后退6格可以让墙壁进入有效拦截范围
- 6格是一个经验值，覆盖大部分场景

**类比**：
- 就像快递车要离仓库6米远，才会检查路上的障碍物
- 如果太近，就认为"还没出发"，不检查障碍物

### Q3：适配层会让游戏变慢吗？

**A**：
- 不会，适配层只是简单的计算
- 没有额外的数据结构，没有额外的内存
- 性能和原来完全一样

**类比**：
- 就像给快递车装一个GPS导航
- GPS只是计算路线，不会让车变慢
- 反而让路线更清晰，更高效

### Q4：如果vanilla更新了怎么办？

**A**：
- 只需修改适配层的代码
- 模块代码完全不受影响
- 这正是适配层的价值所在

**类比**：
- 就像快递公司改了规则
- 只需更新翻译器的"翻译规则"
- 快递车和快递员不需要改

### Q5：为什么不用Harmony Patch？

**A**：
- Harmony Patch会影响所有投射物，不只是BDP
- 后退origin的方案更简单，不需要额外的Patch
- 保留了未来使用Harmony的可能性

**类比**：
- Harmony Patch就像改快递公司的规则（影响所有快递）
- 适配层就像给自己的快递贴标签（只影响自己的快递）
- 贴标签更简单，更安全

## 七、总结

### 7.1 核心思想

**不改变vanilla的规则，而是"欺骗"vanilla**

- 追踪模块：只负责计算追踪方向（纯粹的弹道逻辑）
- 适配层：负责"翻译"给vanilla（纯粹的兼容逻辑）
- vanilla：按"翻译"后的信息处理（不知道被"欺骗"了）

### 7.2 三个"欺骗"

1. **假装发货点很远**：解决墙壁穿透问题
2. **偷偷修改快递单**：解决幽灵命中问题
3. **记住真实发货点**：解决目标未就绪问题

### 7.3 五大优势

1. **职责清晰**：各司其职，代码清晰
2. **易于理解**：新人友好，不被hack困扰
3. **易于维护**：集中管理，修改方便
4. **易于扩展**：未来友好，不影响模块
5. **零性能开销**：高效，不变慢

### 7.4 实施步骤

1. 创建适配层（VanillaAdapter.cs）
2. 集成到子弹系统（Bullet_BDP.cs）
3. 简化追踪模块（TrackingModule.cs）
4. 测试验证（游戏内测试）

## 八、类比总结

**整个方案就像给快递系统装一个智能翻译器**：

- **无人机**（追踪模块）：只负责飞行，不管快递规则
- **翻译器**（适配层）：负责"翻译"给快递系统
- **快递系统**（vanilla）：按"翻译"后的信息处理

**三个"翻译"功能**：

1. **翻译地址**：把"当前位置"翻译成"6格外"
2. **修改快递单**：把"旧收件人"改成"新收件人"
3. **记住仓库**：记住真实的发货点

**结果**：

- 无人机可以自由飞行（实时追踪）
- 快递系统正常工作（墙壁拦截）
- 快递送对地方（不幽灵命中）
