# BDP战斗体延时破裂阶段 - 测试指南

## 实现概述

已成功实现战斗体"延时破裂阶段"（Collapsing Phase），核心特性：
- 破裂延时1.5秒（90 ticks）
- 延时期间无敌、禁止攻击、允许倒地
- 武器不掉落
- 关键部位破坏时显示视觉标记
- **进入破裂和破裂结束时自动打断当前动作**

## 测试场景

### 场景1：基础延时破裂
1. 激活战斗体
2. 受到致命伤害（Trion耗尽或关键部位破坏）
3. **预期结果**：
   - 状态转换：Active → Collapsing
   - 添加Hediff：`BDP_CombatBodyCollapsing`
   - 日志显示："战斗体触发破裂！进入延时破裂阶段 (90 ticks)"
   - 等待90 ticks后自动解除

### 场景2：延时期间无敌
1. 触发破裂（进入Collapsing状态）
2. 延时期间受到多次攻击
3. **预期结果**：
   - 所有伤害被拦截
   - 日志显示："延时破裂阶段无敌，忽略伤害"
   - Trion和影子HP不变

### 场景3：延时期间禁止攻击
1. 触发破裂（进入Collapsing状态）
2. 尝试攻击敌人
3. **预期结果**：
   - 无法执行攻击（Manipulation=0）
   - 移动速度降低50%

### 场景4：延时期间允许倒地
1. 触发破裂（进入Collapsing状态）
2. **预期结果**：
   - Pawn可以倒地
   - 触发体不掉落
   - 日志显示："拦截触发体掉落 (战斗体延时破裂中)"
   - 延时破裂继续进行

### 场景5：关键部位破坏视觉标记
1. 关键部位（头、心脏等）被破坏触发破裂
2. **预期结果**：
   - 添加Hediff：`BDP_CombatBodyPartPending`
   - 健康面板显示："[部位名]结构失效"
   - 日志显示："添加部位待失效标记: [部位名]"

### 场景6：存档/读档
1. 触发破裂（进入Collapsing状态）
2. 延时期间保存游戏
3. 读取存档
4. **预期结果**：
   - Collapsing状态恢复
   - 倒计时继续
   - 到时间后正常解除

### 场景7：动作打断测试
1. 小人正在攻击敌人时触发破裂
2. **预期结果**：
   - 立即停止攻击动作
   - 日志显示："打断当前动作: [小人名] (原因: 进入延时破裂)"
3. 等待90 ticks破裂完成
4. **预期结果**：
   - 日志显示："打断当前动作: [小人名] (原因: 破裂流程结束)"
   - 小人不会继续破裂前的攻击动作

### 场景8：边界情况
1. 延时期间触发体被卸下 → 验证拦截掉落
2. 延时期间地图切换 → 验证倒计时继续
3. 延时期间Pawn死亡（其他原因） → 验证`preventsDeath`生效

## 架构优势

1. **事件驱动倒计时**：使用Hediff的Tick()方法，无需全局轮询
2. **状态机扩展**：保持FSM转换合法性，无循环调用风险
3. **Pipeline原子性**：伤害处理入口检查无敌，不影响原版逻辑
4. **模块化设计**：清晰的职责分离，易于维护和扩展
