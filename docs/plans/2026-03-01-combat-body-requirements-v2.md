# 战斗体系统需求设计文档 v2

## 前言

本文档是BDP模组战斗体系统的第2版需求设计文档。在v1的25条需求基础上，经过查漏补缺和歧义点探究，对需求进行了扩充、细化和修正。

**设计原则**（沿用v1）：有选择性的机制移植，为游戏服务。不是100%还原原作，而是根据RimWorld的游戏框架和玩家体验做出合理的取舍。

**关键技术发现**：RimWorld的`HediffStage`原生字段可以覆盖大部分战斗体特性（疼痛免疫、出血禁止、自愈禁止、需求禁用、精神崩溃阻止等），这意味着"战斗体Hediff"可以通过XML配置实现大量功能，极大减少C#代码量。

**文档状态**：本版本仍有6个开放问题待继续讨论（标记为⚠️），至少再做1版迭代。

**变更标记说明**：
- 🆕 = 本版本新增的需求
- ✏️ = 本版本修改的需求（相比v1有实质变化）
- ✅ = v1原有需求，本版本未变

## 一、核心价值定位

### NR-001: 完全伤害隔离 ✅
- 战斗体受伤完全不影响真身
- 解除后真身回到激活时的状态
- 这是战斗体系统的根本价值，不可妥协

---

## 二、激活机制

### NR-002: 激活前置条件 ✅
- 装备触发体（触发器装备）
- 当前Trion ≥ 激活所需的占用量
- 不在战斗体状态下

### NR-003: 激活过程 ✅
- 引导时间：1-3秒
- 引导期间可被打断（受到伤害/被击倒/失去意识等）
- 引导完成后立即进入战斗体状态

### NR-004: 激活效果 ✏️（重大扩充）
激活完成时以下操作作为**一套原子操作**同时执行：
1. **真身状态快照**（🆕）：记录真身当前的完整生理状态（详见NR-026）
2. **Hediff替换**（🆕）：移除所有现有Hediff，添加"战斗体Hediff"（详见NR-027）
3. **衣物隐藏**（🆕）：隐藏当前穿戴的衣物，替换为战斗体外观（详见NR-028）
4. **需求冻结**（🆕）：禁用生理类需求（详见NR-029）
5. 芯片注册：已装载的触发芯片向Trion资源池注册，锁定占用量
6. 状态切换：进入战斗体状态
7. Trion冻结：设置CompTrion.frozen = true，停止Trion自然恢复

**关键约束**：
- 激活被打断 → 上述操作均不执行（快照不生成）
- 引导过程中受伤 → 伤口和状态在快照时一起被记录（因为快照在激活完成瞬间生成）

## 三、解除机制

### 3.1 主动解除（Trigger Off）

#### NR-005: 主动解除触发条件 ✅
- 玩家主动发出解除指令

#### NR-006: 主动解除过程 ✏️（扩充）
1. 战斗体状态结束
2. 芯片注销：调用CompTrion.Release(allocated)，占用的Trion返还
3. Trion解冻：设置CompTrion.frozen = false，恢复Trion自然恢复
4. **Hediff回滚**（🆕）：移除所有战斗体期间的Hediff（包括战斗体Hediff），从快照恢复原有Hediff
5. **需求恢复**（🆕）：重新添加被禁用的需求，恢复到快照记录的值
6. **衣物恢复**（🆕）：恢复隐藏的衣物，移除战斗体外观
7. 真身状态回滚：上述4-6的综合效果

#### NR-007: 主动解除Trion结算 ✅
- 占用量返还到可用量
- 已消耗的Trion不返还
- 玩家可以较快重新激活

### 3.2 被动破裂

#### NR-008: 被动破裂触发条件 ✅（任一满足）
1. Trion可用量耗尽（Available ≤ 0）
2. 头部被击毁（拦截原版死亡结果）
3. 心脏被击毁（拦截原版死亡结果）
4. 玩家主动触发紧急脱离（战术撤退）

#### NR-009: 被动破裂过程 ✏️（扩充）
1. 战斗体破裂（视觉效果：裂纹扩散→消散）
2. 芯片强制注销：调用CompTrion.ForceDeplete()，占用量随战斗体流失
3. 紧急脱离判定：
   - 如果装备了紧急脱离芯片：传送真身到安全位置（自定义建筑）
   - 如果没有装备：真身在原地释放
4. Trion解冻：设置CompTrion.frozen = false
5. **Hediff回滚**（🆕）：同NR-006步骤4
6. **需求恢复**（🆕）：同NR-006步骤5
7. **衣物恢复**（🆕）：同NR-006步骤6
8. 真身状态回滚：上述5-7的综合效果

#### NR-010: 被动破裂Trion结算 ✅
- 所有Trion清零（cur = 0, allocated = 0）
- 需要长时间恢复才能重新激活

## 四、弱点系统

### NR-011: 弱点定义 ✅
- 头部和心脏是战斗体的弱点
- 世界观解释：头部=Trion中继中心，心脏=Trion供给系统（仅概念层面）

### NR-012: 致命伤拦截 ✅
- 当头部或心脏受到致命伤害时，拦截原版的死亡结果
- 将死亡结果改为触发被动破裂流程（NR-009）
- 真身不会死亡，而是安全释放

---

## 五、紧急脱离系统

### NR-013: 紧急脱离芯片 ✅
- 类型：触发芯片（特殊类型）
- 槽位：只能插在触发器的"特殊槽位"，不能插在"左右槽位"
- 效果：战斗体破裂时自动传送真身到安全位置

### NR-014: 传送目标建筑 ✏️（修正歧义）
- 新增自定义建筑（如"紧急脱离信标"或"战略传送点"）
- 玩家需要在基地建造此建筑作为传送坐标
- ⚠️ **待确认**：如果没有建造传送目标建筑，紧急脱离芯片的行为是"无法生效（原地释放）"还是"传送到地图边缘"

### NR-015: 紧急脱离触发 ✅
- 触发条件：战斗体被动破裂时自动触发
- 传送效果：
  - 真身传送到最近的传送目标建筑
  - 传送过程中真身无敌（不可被攻击）
  - 传送完成后真身出现在建筑旁边

### NR-016: 紧急脱离Trion消耗 ✅
- 紧急脱离触发时Trion已经清零（被动破裂路径）
- 不需要额外消耗Trion

## 六、战斗体特性

### NR-017: 耐久度统一 ✏️（待细化）
- 所有战斗体的耐久度相同，不受真身属性影响
- ⚠️ **待确认**：具体指HP相同、护甲值相同、还是两者都是？具体数值待定。
- 技术方向：通过战斗体Hediff的`capMods`和`statOffsets`实现

### NR-018: 无限体力 ✏️（修正歧义）
- 战斗体状态下不消耗体力
- 技术实现：`restFallFactor = 0f`（HediffStage原生字段）
- ~~"或体力恢复速度极快"~~（删除备选方案，直接使用不消耗方案）

### NR-019: 疼痛免疫 ✏️（明确实现）
- 战斗体状态下免疫疼痛效果
- 技术实现：`painFactor = 0f`（HediffStage原生字段）

### NR-020: 无法自愈 ✏️（明确实现）
- 战斗体受伤后不会自然愈合
- 只能通过解除重建来"治疗"
- 技术实现：`naturalHealingFactor = 0f`（HediffStage原生字段）

### NR-021: 不实现非Trion免疫 ✅
- 战斗体对所有武器都是正常受伤的
- 出于游戏平衡考虑

### NR-022: 不实现缺陷修复 ✅
- 战斗体状态下不会移除真身的负面健康状态
- 出于游戏平衡考虑

---

## 七、战斗体状态下的限制

### NR-023: Trion无法恢复 ✅
- 战斗体状态下CompTrion.frozen = true
- Trion只减不增，一切消耗不可逆

### NR-024: 无法使用原版治疗 ✅
- 战斗体状态下无法使用医疗床/药物治疗
- 因为伤害不影响真身，治疗无意义

### NR-025: 生理状态冻结 ✏️（重大细化）
- **原则**：只冻结生理层面，不冻结心理层面
- 战斗体状态下没有进食/睡眠等**生理**需求
- 心理层面（心情、思想、技能、人际关系）正常运行
- 解除时**生理**状态回滚到激活时的快照
- 心理状态不回滚（记忆/经验保留）
- 详细分类见NR-029~NR-033

## 八、真身状态快照系统 🆕

### NR-026: 快照内容 🆕
快照记录以下数据：

| 记录项 | 说明 | 回滚方式 |
|---|---|---|
| **Hediff列表** | 所有伤口、疾病（含阶段和进度）、永久伤残、植入物等 | 解除时从快照恢复 |
| **需求值** | 被冻结的生理需求的当前数值（饥饿值、休息值、舒适度等） | 解除时恢复到快照值 |
| **衣物列表** | 当前穿戴的所有衣物及其状态 | 解除时恢复穿戴 |

**不记录的数据**（心理层面，正常运行不回滚）：
- 心情/思想/记忆
- 技能经验
- 人际关系
- 工作设置/时间表

**快照归属**：属于Pawn

### NR-027: Hediff替换策略 🆕
- **激活时**：记录快照 → 移除所有现有Hediff → 添加"战斗体Hediff"
- **解除时**：移除所有战斗体期间的Hediff（包括战斗体Hediff本身）→ 从快照恢复原有Hediff
- 两者无缝衔接
- "战斗体Hediff"是一个特殊Hediff，通过HediffStage原生字段实现大部分战斗体特性

### NR-028: 衣物与外观 🆕
- **激活时**：隐藏当前穿戴的衣物，替换为战斗体视觉外观
- **解除时**：恢复隐藏的衣物，移除战斗体外观
- 衣物列表记录在快照中
- 武器不是问题：激活战斗体的前提是手持触发体
- ⚠️ **待确认**：携带物品（Inventory/背包）的处理方式

## 九、生理/心理分类详细需求 🆕

### NR-029: 禁用的生理需求 🆕
以下需求在战斗体状态下通过战斗体Hediff的`disablesNeeds`禁用：
- 饥饿（Food）
- 休息（Rest）
- 舒适度（Comfort）

**效果**：需求从Pawn的需求列表中移除，相关ThoughtWorker返回Inactive（不产生正面或负面心情影响）

### NR-030: 不禁用的心理需求 🆕
以下需求在战斗体状态下正常运行：
- 心情（Mood）：正常计算
- 思想/记忆（Thoughts）：战斗经历正常产生思想
- 技能经验（Skills）：战斗经验正常增长
- 人际关系（Relations）：战斗互动正常影响关系
- ⚠️ **待最终确认**：娱乐（Joy）、美观（Beauty）、户外（Outdoors）——建议归为心理层面不冻结

### NR-031: 精神崩溃阻止 🆕
- 战斗体状态下**不冻结心情下降**
- 但**阻止触发精神崩溃**
- ⚠️ **待确认**：建议通过HediffStage的`blocksMentalBreaks = true`实现

### NR-032: 伤害机制映射 🆕
战斗体的伤害不走原版的出血/疼痛逻辑，而是映射为Trion消耗：

| 原版概念 | 战斗体映射 | 技术实现 |
|---|---|---|
| 掉血（HP减少） | 扣Trion可用值 | CompTrion.Consume() |
| 流血（持续出血） | Trion泄漏（维持消耗速率增加） | CompTrion.RegisterDrain() |
| 部位被毁 | 扣Trion可用值 + 部位被毁 | Consume() + Hediff |
| 出血（视觉） | 禁止 | totalBleedFactor = 0 |
| 疼痛 | 禁止 | painFactor = 0 |

### NR-033: 环境免疫 🆕
战斗体状态下免疫以下环境效果：
- 体温影响（不受极端温度伤害）
- 中毒（免疫毒素）
- 疾病（不会感染新疾病，已有疾病进度冻结于快照）
- ⚠️ **待确认**：疾病进度冻结是否接受为"战斗体附带好处"（玩家可能利用此机制避免疾病恶化）

## 十、MVP范围

### MVP必须实现的功能

**优先级P0**（核心机制，MVP必须）：

1. **激活/解除机制** (NR-002 ~ NR-010)
   - 短暂引导激活（原子操作）
   - 主动解除和被动破裂两条路径
   - Trion结算差异

2. **完全伤害隔离** (NR-001)
   - 战斗体受伤不影响真身
   - 解除后真身100%完好

3. **真身状态快照** (NR-026 ~ NR-028) 🆕
   - Hediff快照与替换
   - 需求快照与冻结/恢复
   - 衣物快照与隐藏/恢复

4. **弱点系统** (NR-011 ~ NR-012)
   - 头部/心脏致命伤触发战斗体破裂
   - 拦截死亡结果

5. **紧急脱离** (NR-013 ~ NR-016)
   - 破裂时传送到安全位置
   - 需要新建筑和芯片系统

6. **战斗体特性** (NR-017 ~ NR-022)
   - 耐久度统一、无限体力、疼痛免疫、无法自愈

7. **生理/心理分类** (NR-029 ~ NR-033) 🆕
   - 生理需求禁用
   - 伤害机制映射
   - 环境免疫

8. **Trion消耗与恢复限制** (NR-023 ~ NR-025)
   - 战斗体状态下Trion无法恢复
   - 无法使用原版治疗

### MVP不实现的功能

**优先级P1**（后续版本）：
- **战斗体视觉表现**：特殊的外观/特效，MVP可先用简单UI指示器

## 十一、与其他模块的依赖关系

### 依赖的模块

1. **微内核(BDP.Core)**
   - CompTrion的API：Allocate/Release/ForceDeplete/SetFrozen/Consume/RegisterDrain/UnregisterDrain
   - OnAvailableDepleted事件：用于Trion耗尽触发被动破裂

2. **触发器模块(BDP.Trigger)**
   - CompTriggerBody：检查是否装备触发体
   - 芯片系统：紧急脱离芯片的实现

### 被依赖的模块
- 无（战斗模块是L2层，不被其他模块依赖）

### 架构位置
- 层级：L2（第二层）
- 命名空间：BDP.Combat
- 状态：需求分析中

### 设计原则（沿用v1 + 新增）

1. **有选择性的机制移植**：不是100%还原原作，而是为游戏服务
2. **利用现有机制**：如弱点系统利用原版头部/心脏致命伤，战斗体特性利用HediffStage原生字段
3. **增加战术深度**：如激活引导时间、两条解除路径的Trion结算差异
4. **保持游戏平衡**：如不实现非Trion免疫、缺陷修复
5. 🆕 **生理/心理分离**：生理冻结回滚，心理正常运行保留
6. 🆕 **XML优先**：能通过HediffStage XML配置实现的特性，优先使用XML而非C#

### 后续工作

1. **解决开放问题**：6个待确认项
2. **概念验证**：验证Hediff替换策略和快照机制的可行性
3. **需求迭代**：至少再做1版需求分析（v3）
