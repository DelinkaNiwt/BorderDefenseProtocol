## 问题1：Gene.GetGizmos() NullReferenceException

### 错误现象
```
System.NullReferenceException: Object reference not set to an instance of an object
  at BDP.Gene_TrionGland.GetGizmos().MoveNext()
```

### 根本原因
`base.GetGizmos()` 在某些游戏初始化状态下可能返回null，直接在foreach中使用导致异常。

### 错误代码
```csharp
public override IEnumerable<Gizmo> GetGizmos()
{
    foreach (var gizmo in base.GetGizmos()) // ❌ base.GetGizmos()可能返回null
    {
        yield return gizmo;
    }
}
```

### 正确修复
```csharp
public override IEnumerable<Gizmo> GetGizmos()
{
    if (pawn == null) yield break;

    var baseGizmos = base.GetGizmos();
    if (baseGizmos != null) // ✅ 必须检查null
    {
        foreach (var gizmo in baseGizmos)
        {
            if (gizmo != null)
                yield return gizmo;
        }
    }

    // 分步创建Command_Action，避免对象初始化器中的异常
    var cmd = new Command_Action();
    cmd.defaultLabel = combatBodyActive ? "解除战斗体" : "激活战斗体";
    cmd.icon = BaseContent.BadTex; // 确保不为null
    cmd.action = () => { /* ... */ };
    yield return cmd;
}
```

### 教训
- RimWorld的基类方法在特殊状态下可能返回null而非空集合
- 必须对`base.GetGizmos()`进行null检查
- 避免在对象初始化器中使用复杂表达式

## 问题2：战斗体装甲纹理路径错误

### 错误现象
```
Failed to find any textures at Things/Pawn/Humanlike/Apparel/ShieldBelt/ShieldBelt_Male
```

### 根本原因
ThingDef中`texPath`和`wornGraphicPath`引用了不存在的纹理文件。

### 错误配置
```xml
<graphicData>
  <texPath>Things/Pawn/Humanlike/Apparel/ShieldBelt/ShieldBelt</texPath>
</graphicData>
<apparel>
  <wornGraphicPath>Things/Pawn/Humanlike/Apparel/ShieldBelt/ShieldBelt</wornGraphicPath>
</apparel>
```

### 正确修复
```xml
<graphicData>
  <texPath>Things/Pawn/Humanlike/Apparel/Duster/Duster</texPath>
  <graphicClass>Graphic_Single</graphicClass>
</graphicData>
<apparel>
  <wornGraphicPath>Things/Pawn/Humanlike/Apparel/Duster/Duster</wornGraphicPath>
</apparel>
```

### 教训
- 装备ThingDef必须使用存在的纹理路径
- 临时可使用原版纹理作为占位
- 后续需要制作专用的战斗体装甲纹理

## 问题3：CompRottable腐烂阻止

### 错误尝试
最初尝试patch `CompTick`方法，但`CompRottable`没有`CompTick`方法。

### 根本原因
`CompRottable`实际是通过`CompTickRare` → `TickInterval` → 检查`Active`属性来控制腐烂。`Active`属性返回`false`时，腐烂逻辑不会执行。

### 正确修复
```csharp
[HarmonyPatch(typeof(CompRottable), nameof(CompRottable.Active), MethodType.Getter)]
public static class Patch_CompRottable_Active
{
    [HarmonyPostfix]
    public static void Postfix(CompRottable __instance, ref bool __result)
    {
        if (!__result) return;

        var thing = __instance.parent;
        if (thing?.holdingOwner?.Owner is CombatBodySnapshot)
        {
            __result = false; // 阻止腐烂
        }
    }
}
```

### 教训
- 优先patch属性而非方法，利用游戏原有逻辑
- 不要阻断执行流，而是修改返回值
- 通过`holdingOwner?.Owner`检查容器所有权

## 总结

这3个问题的共同特点：
1. **防御性编程**：必须对RimWorld API返回值进行null检查
2. **利用原有逻辑**：优先patch属性/返回值，而非阻断执行流
3. **资源路径验证**：确保所有引用的资源文件存在

这些经验对后续M2-M5里程碑的开发同样适用。
