# 科研解锁机制与模组扩展

**总览**：科研完成后的解锁通过"反向引用"机制实现——不是ResearchProjectDef声明"我解锁什么"，而是各种Def（ThingDef、RecipeDef、TerrainDef等）通过`researchPrerequisite`/`researchPrerequisites`字段声明"我需要什么科研"。`ResearchProjectDef.UnlockedDefs`属性在运行时反向搜索所有引用自己的Def，用于UI显示。

## 1. 科研解锁的6种内容类型

`ResearchProjectDef.UnlockedDefs`属性搜索以下6种Def类型：

| # | Def类型 | 搜索字段 | 典型实例 |
|---|--------|---------|---------|
| 1 | **RecipeDef产品** | `researchPrerequisite` / `researchPrerequisites` → `products` | 研究Smithing解锁长剑制造配方的产品 |
| 2 | **ThingDef（建筑/物品）** | `researchPrerequisites` | 研究Electricity解锁电力建筑 |
| 3 | **植物** | `plant.sowResearchPrerequisites` | 研究TreeSowing解锁种植树木 |
| 4 | **TerrainDef（地板）** | `researchPrerequisites` | 研究StonecutterTable解锁石质地板 |
| 5 | **手术RecipeDef** | `researchPrerequisites`（且`IsSurgery=true`） | 研究Prosthetics解锁安装假肢手术 |
| 6 | **PsychicRitualDef** | `researchPrerequisite` | Anomaly DLC心灵仪式 |

> **关键认知**：解锁是"被动声明"模式——ResearchProjectDef本身不存储解锁列表，而是各Def声明自己的前置科研。`UnlockedDefs`是运行时缓存的反向查询结果。

## 2. researchPrerequisite在各Def上的分布

| Def类型 | 字段名 | 类型 | 说明 |
|---------|--------|------|------|
| **ThingDef** | `researchPrerequisites` | List\<ResearchProjectDef\> | 建筑/物品的科研前置（复数，需全部完成） |
| **RecipeDef** | `researchPrerequisite` | ResearchProjectDef | 配方的单个科研前置 |
| **RecipeDef** | `researchPrerequisites` | List\<ResearchProjectDef\> | 配方的多个科研前置（需全部完成） |
| **TerrainDef** | `researchPrerequisites` | List\<ResearchProjectDef\> | 地板的科研前置 |
| **ThingDef（植物）** | `plant.sowResearchPrerequisites` | List\<ResearchProjectDef\> | 种植的科研前置 |
| **DesignationCategoryDef** | `researchPrerequisites` | List\<ResearchProjectDef\> | 建筑师菜单分类的科研前置 |
| **PsychicRitualDef** | `researchPrerequisite` | ResearchProjectDef | 心灵仪式的科研前置 |

**DesignationCategoryDef科研解锁**：整个建筑师菜单分类可以被科研锁定——原版中Power分类需要Electricity研究，Biotech分类需要BasicMechtech研究。这意味着研究完成前，玩家甚至看不到该分类下的所有建筑。

## 3. ResearchMod（科研完成副作用）

`ResearchMod`是一个抽象基类，只有一个`Apply()`方法。通过`ResearchProjectDef.researchMods`字段配置，在科研完成时执行。

```csharp
public abstract class ResearchMod
{
    public abstract void Apply();
}
```

**原版使用情况**：原版代码中没有ResearchMod的子类实现。`researchMods`字段在原版XML中也未被使用。这是一个预留的扩展点。

**ReapplyAllMods机制**：`ResearchManager.ReapplyAllMods()`在科研完成时遍历所有已完成项目的researchMods并调用Apply()。这意味着ResearchMod的效果是幂等的——每次有新科研完成时，所有已完成科研的Mod都会重新应用。

> **关键认知**：ResearchMod是模组实现"科研完成时执行代码逻辑"的标准扩展点——继承ResearchMod重写Apply()，在XML中配置到researchMods字段即可，无需Harmony补丁。

## 4. 其他完成效果

| 效果 | 字段 | 说明 |
|------|------|------|
| **重算电力** | `recalculatePower=true` | 完成后重算所有电力设备的PowerVars |
| **发现信件** | `discoveredLetterTitle` + `discoveredLetterText` | 完成后发送信件（如Electricity研究完成后的说明信） |
| **教学概念** | `teachConcept` | 完成后触发教学提示 |
| **全局信号** | — | 发送`"ResearchCompleted"`全局信号 |
| **通知ThingDef** | — | 遍历UnlockedDefs调用`Notify_UnlockedByResearch()` |

## 5. 模组自定义科研的4种扩展模式

| # | 模式 | 需要C# | 适用场景 | 典型实例 |
|---|------|--------|---------|---------|
| 1 | **纯XML定义ResearchProjectDef** | 否 | 标准科研项目（前置链+解锁建筑/配方） | 大多数模组 |
| 2 | **自定义ResearchTabDef** | 否 | 独立科研标签页 | UFHeavyIndustries（联合重工） |
| 3 | **自定义ResearchMod子类** | 是 | 科研完成时执行代码逻辑 | 理论支持，原版未使用 |
| 4 | **Harmony补丁** | 是 | 修改科研速度/条件/UI | 修改ResearchPerformed或FinishProject |

### 5.1 纯XML定义（最常用）

标准流程：
1. 定义`ResearchProjectDef`（设置baseCost、techLevel、prerequisites、researchViewX/Y）
2. 在需要解锁的ThingDef/RecipeDef上设置`researchPrerequisites`
3. 可选：设置`requiredResearchBuilding`和`requiredResearchFacilities`

**XML继承模板模式**（推荐）：

```xml
<!-- 抽象模板：统一科技等级和标签页 -->
<ResearchProjectDef Name="MyMod_ResearchBase" Abstract="true">
    <techLevel>Spacer</techLevel>
    <tab>MyMod_Tab</tab>
</ResearchProjectDef>

<!-- 高级模板：需要HiTech研究台+MultiAnalyzer -->
<ResearchProjectDef Name="MyMod_AdvancedBase" Abstract="true">
    <techLevel>Spacer</techLevel>
    <tab>MyMod_Tab</tab>
    <requiredResearchBuilding>HiTechResearchBench</requiredResearchBuilding>
    <requiredResearchFacilities>
        <li>MultiAnalyzer</li>
    </requiredResearchFacilities>
</ResearchProjectDef>

<!-- 具体项目继承模板 -->
<ResearchProjectDef ParentName="MyMod_ResearchBase">
    <defName>MyMod_BasicTech</defName>
    <label>basic tech</label>
    <baseCost>1000</baseCost>
    <researchViewX>0</researchViewX>
    <researchViewY>0</researchViewY>
</ResearchProjectDef>
```

### 5.2 自定义ResearchTabDef

```xml
<ResearchTabDef>
    <defName>MyMod_Tab</defName>
    <label>My Mod Research</label>
</ResearchTabDef>
```

> **关键认知**：自定义标签页会在科研UI中显示为独立的标签，与Main和Anomaly并列。模组科研项目通过`<tab>MyMod_Tab</tab>`归入该标签页。

### 5.3 UFHeavyIndustries（联合重工）代表性实例

UFHeavyIndustries展示了模组自定义科研的标准模式：

**架构特点**：
1. **独立标签页**：`KT_Research` ResearchTabDef
2. **双层抽象模板**：`KT_ResearchProject_low_Base`（Spacer，无特殊要求）和`KT_ResearchProject_high_Base`（Spacer，需HiTech+MultiAnalyzer）
3. **分级科技树**：T0→T1→T2（分支：火炮/能量/磁轨/火箭）→T3→T4
4. **成本递增**：T0=1000 → T1=1500 → T2=2000~3000 → T3=3000~4000 → T4=5000+
5. **坐标布局**：researchViewX按层级递增（0→1→2→3→4），researchViewY按分支偏移

**设计模式总结**：
- 用抽象模板统一`techLevel`和`tab`，避免每个项目重复声明
- 高级项目模板额外设置`requiredResearchBuilding`和`requiredResearchFacilities`
- 科研树从左到右按层级排列，分支向上下展开
- 每个科研项目的baseCost随层级递增，体现技术难度梯度

## 6. 科研系统与其他系统的交互

| 交互系统 | 交互方式 | 说明 |
|---------|---------|------|
| **建筑系统** | ThingDef.researchPrerequisites | 建筑需要科研解锁才能建造 |
| **配方系统** | RecipeDef.researchPrerequisite(s) | 配方需要科研解锁才能使用 |
| **建筑师菜单** | DesignationCategoryDef.researchPrerequisites | 整个菜单分类需要科研解锁 |
| **派系系统** | FactionDef.techLevel | 派系科技等级影响研究速度CostFactor |
| **技能系统** | SkillDefOf.Intellectual | 研究时获得0.1/tick智力经验 |
| **Storyteller** | tags（如TribalStart） | 部落开局预完成带TribalStart标签的科研 |
| **任务系统** | Signal "ResearchCompleted" | 科研完成发送全局信号，任务可监听 |

## 7. 开发者要点

1. **科研解锁是"被动声明"**——在需要解锁的Def上设置researchPrerequisites，而非在ResearchProjectDef上声明解锁列表
2. **ResearchMod是预留扩展点**——原版未使用，模组可继承实现"科研完成时执行代码"
3. **自定义标签页只需1行XML**——定义ResearchTabDef，项目通过tab字段归入
4. **抽象模板是标准做法**——用Abstract模板统一techLevel/tab/requiredResearchBuilding，具体项目只需设置defName/label/baseCost/坐标/前置
5. **researchViewX/Y是UI坐标**——X轴从左到右表示科技层级，Y轴从上到下表示分支，坐标间距建议≥0.7避免重叠
