# 模组扩展战斗系统实例

**总览**：社区模组通过**5种技术路径**扩展RimWorld战斗系统——XML继承DamageDef（最简单）、自定义DamageWorker（特殊伤害逻辑）、自定义Verb（新攻击方式）、ThingComp/HediffComp（新护盾/拦截/附加效果）、Harmony补丁（修改原版管线）。优先级原则：**Comp > 继承 > Harmony补丁**——Comp无侵入性最安全，继承扩展性好但需理解基类，Harmony补丁最灵活但有兼容风险。

## 1. 自定义DamageWorker（新伤害类型）

### 1.1 实例总览

| # | 模组 | 类 | 继承自 | 核心行为 |
|---|------|-----|--------|---------|
| 1 | VEF | `DamageWorker_OPToxic` | DamageWorker | 检查防毒面具→绕过护甲直接扣HP |
| 2 | VEF | `DamageWorker_ExtraInfecter` | DamageWorker_Cut | 攻击后按概率附加伤口感染 |
| 3 | GD3(闪耀毁灭) | `DamageWorker_PsychicStrike` | DamageWorker | 按PsychicSensitivity分4级效果 |
| 4 | VPE(灵能扩展) | `DamageWorker_NeuralHeatBlast` | DamageWorker | 神经热爆发伤害 |
| 5 | AncotLibrary | `DamageWorker_StunExceptInstigatorFaction` | DamageWorker | 眩晕但豁免攻击者同派系 |

### 1.2 ★ VEF DamageWorker_ExtraInfecter（攻击附加效果模式）

**模组**：Vanilla Expanded Framework
**核心思路**：继承`DamageWorker_Cut`，在`ApplySpecialEffectsToPart()`中附加感染效果。

```csharp
public class DamageWorker_ExtraInfecter : DamageWorker_Cut
{
    protected override void ApplySpecialEffectsToPart(Pawn pawn, float totalDamage,
        DamageInfo dinfo, DamageWorker.DamageResult result)
    {
        base.ApplySpecialEffectsToPart(pawn, totalDamage, dinfo, result);  // 先执行原版Cut伤害
        if (!pawn.IsGhoul) {
            CompInfecter comp = dinfo.Instigator?.TryGetComp<CompInfecter>();
            Hediff hediff = pawn?.health?.hediffSet?.GetFirstHediffOfDef(HediffDefOf.WoundInfection);

            if (hediff != null && comp?.Props.worsenExistingInfection == true)
                hediff.Severity += comp.Props.severityToAdd;           // 已有感染→加重
            else if (pawn.GetStatValue(StatDefOf.ToxicResistance) < 1f
                     && random.NextDouble() > ((float)(100 - comp.GetChance) / 100))
                pawn.health.AddHediff(HediffDefOf.WoundInfection, dinfo.HitPart);  // 新增感染
        }
    }
}
```

**设计模式**：DamageWorker + CompInfecter协作——DamageWorker负责伤害逻辑，CompInfecter（附加在攻击者身上）提供感染概率和严重度参数。这种"Worker读取攻击者Comp配置"的模式可复用于任何"攻击附加效果"场景。

### 1.3 ★ GD3 DamageWorker_PsychicStrike（按属性分级效果模式）

**模组**：闪耀毁灭5 (Glitterworld Destroyer 5)
**核心思路**：继承`DamageWorker`，在`Apply()`中按目标`PsychicSensitivity`分4级效果。

```csharp
public class DamageWorker_PsychicStrike : DamageWorker
{
    public override DamageResult Apply(DamageInfo dinfo, Thing victim)
    {
        DamageResult damageResult = base.Apply(dinfo, victim);
        Pawn vict = victim as Pawn;

        if (vict.RaceProps.IsMechanoid)
        {   // 机械体：固定300 ticks眩晕
            AddHediff(vict, GDDefOf.PsychicVertigo, 300);
            return damageResult;
        }

        float psy = vict.GetStatValue(StatDefOf.PsychicSensitivity);
        if (psy == 0f) return damageResult;                    // 心灵免疫→无效果
        else if (psy > 0f && psy < 2f)
            AddHediff(vict, GDDefOf.PsychicVertigo, (int)(900 * psy));  // 低敏感：眩晕
        else if (psy >= 2f && psy < 3f)
        {   AddHediff(vict, GDDefOf.PsychicVertigo, (int)(900 * psy));
            DoEffectOn(vict);                                   // 中敏感：眩晕+心灵休克
        }
        else if (psy >= 3f && psy < 4f)
        {   AddHediff(vict, GDDefOf.PsychicVertigo, (int)(900 * psy));
            DoEffectOn(vict);
            DamageVictim(attacker, vict, dinfo.Weapon, 0.4f);  // 高敏感：+脑部伤害
        }
        else
            DamageVictim(attacker, vict, dinfo.Weapon, 30f);   // 极高敏感：重度脑伤

        return damageResult;
    }
}
```

**设计模式**：按目标Stat值分级施加不同效果——低值轻效果，高值重效果。`AddHediff()`使用`HediffComp_Disappears`控制持续时间，`DamageVictim()`直接对脑部造成Burn伤害。

### 1.4 自定义DamageWorker实现模式总结

| 模式 | 重写方法 | 适用场景 | 代表 |
|------|---------|---------|------|
| **完全自定义** | `Apply()` | 需要完全控制伤害流程 | OPToxic, PsychicStrike |
| **附加效果** | `ApplySpecialEffectsToPart()` | 在原版伤害基础上附加效果 | ExtraInfecter |
| **条件豁免** | `Apply()` + 派系/属性检查 | 友军豁免、属性免疫 | StunExceptInstigatorFaction |

> **要点**：
> 1. 继承`DamageWorker_Cut`/`DamageWorker_Blunt`等子类比直接继承`DamageWorker`更安全——保留原版部位选择和护甲逻辑
> 2. "Worker读取攻击者Comp"模式（ExtraInfecter）是最灵活的——同一DamageWorker可被不同Comp配置复用

### 2.2 ★ VEF HediffComp_Shield（Hediff级护盾）

**模组**：Vanilla Expanded Framework
**核心思路**：`HediffComp_Shield : HediffComp_Draw`，通过Hediff附加护盾效果，支持高度可配置的吸收/反射行为。

**CompProperties配置字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `maxEnergy` | float | 最大能量（0=无限护盾） |
| `energyLossPerDamage` | float | 每点伤害消耗能量 |
| `energyPerTick` | float | 每tick恢复能量 |
| `energyPctOnReset` | float | 重置后能量百分比 |
| `rechargeDelay` | int | 击破后重充延迟(ticks) |
| `fullOnAdd` | bool | 添加时是否满能量 |
| `absorbAttackType` | AttackType | 吸收类型：Melee/Ranged/Both |
| `damageOnAttack` | AttackType | 反射伤害触发类型 |
| `damageType` | DamageDef | 反射伤害类型 |
| `damageAmount` | float | 反射伤害量 |
| `armorPenetration` | float | 反射穿甲值 |
| `cannotUseAttackType` | AttackType | 护盾激活时禁用的攻击类型 |
| `breakOn` | List | 直接击破护盾的伤害类型（如EMP） |
| `absorbedFleck` | FleckDef | 吸收时视觉特效 |
| `brokenFleck` | FleckDef | 击破时视觉特效 |
| `soundBroken` | SoundDef | 击破音效 |
| `soundRecharge` | SoundDef | 重充音效 |
| `sustainer` | SoundDef | 持续音效 |

**核心拦截逻辑**：

```csharp
public virtual void PreApplyDamage(ref DamageInfo dinfo, ref bool absorbed)
{
    if (absorbed) return;
    if (ShieldActive)
    {
        if (Props.breakOn.Contains(dinfo.Def)) { Break(); return; }  // EMP等直接击破

        // 1. 判断是否吸收（按攻击类型过滤）
        var absorb = Props.absorbAttackType switch {
            AttackType.Melee  => !dinfo.Def.isRanged,
            AttackType.Ranged => dinfo.Def.isRanged || dinfo.Def.isExplosive,
            AttackType.Both   => true,
        };

        if (absorb && Props.Absorbs(dinfo.Def) && AbsorbDamage(ref dinfo))
        {
            absorbed = true;
            // 视觉反馈：FleckMaker + 灰尘粒子
        }

        // 2. 判断是否反射伤害（独立于吸收判定）
        var damage = Props.damageOnAttack switch { /* 同上逻辑 */ };
        if (damage && dinfo.Instigator != null)
            dinfo.Instigator.TakeDamage(new DamageInfo(Props.damageType, Props.damageAmount, Props.armorPenetration));
    }
}
```

**能量系统**：

```csharp
protected virtual bool AbsorbDamage(ref DamageInfo dinfo)
{
    if (useEnergy)
    {
        var loss = dinfo.Amount * Props.energyLossPerDamage;
        if (loss < energy)
        {
            energy -= loss;
            dinfo.SetAmount(0f);    // 完全吸收
            return true;
        }
        Break();                     // 能量不足→击破
        dinfo.SetAmount(dinfo.Amount - energy / Props.energyLossPerDamage);  // 溢出伤害穿透
        return false;
    }
    dinfo.SetAmount(0f);             // 无限护盾→总是完全吸收
    return true;
}
```

**与原版CompShield的对比**：

| 维度 | 原版CompShield | VEF HediffComp_Shield |
|------|---------------|----------------------|
| **基于** | ThingComp（装备/Pawn） | HediffComp（Hediff） |
| **附加方式** | 穿戴装备/内置 | 任何Hediff来源（基因/药物/能力） |
| **吸收类型** | 仅远程+爆炸 | 可配置：近战/远程/全部 |
| **伤害反射** | 无 | 支持（可配置伤害类型和量） |
| **溢出伤害** | 无（总是完全吸收） | 支持（能量不足时穿透） |
| **武器限制** | 阻止远程武器 | 可配置：近战/远程/全部/无 |
| **EMP弱点** | 硬编码 | 可配置breakOn列表 |


### 2.3 VEF ArmorUtility Harmony补丁（护盾纳入护甲管线）

**模组**：Vanilla Expanded Framework
**补丁目标**：`ArmorUtility.GetPostArmorDamage` (Prefix)
**核心思路**：在原版护甲计算之前，先让VEF自定义护盾（`CompShield`，注意这是VEF的ThingComp版护盾，非上面的HediffComp版）参与护甲判定。

```csharp
[HarmonyPatch(typeof(ArmorUtility), nameof(ArmorUtility.GetPostArmorDamage))]
public static class VanillaExpandedFramework_ArmorUtility_GetPostArmorDamage
{
    public static bool Prefix(Pawn pawn, ref float amount, ref float armorPenetration,
        BodyPartRecord part, ref DamageDef damageDef, ...)
    {
        if (damageDef.armorCategory != null)
        {
            var armourRating = damageDef.armorCategory.armorRatingStat;
            var equipmentList = pawn.apparel.WornApparel;
            for (int i = 0; i < equipmentList.Count; i++)
            {
                var curEq = equipmentList[i];
                if (curEq.IsShield(out CompShield sC) && sC.UsableNow && sC.CoversBodyPart(part))
                {
                    float prevAmount = amount;
                    // 调用原版ArmorUtility.ApplyArmor()，将护盾作为"额外护甲层"
                    NonPublicMethods.ArmorUtility_ApplyArmor(ref amount, armorPenetration,
                        curEq.GetStatValue(armourRating), curEq, ref damageDef, pawn, out bool metalArmour);

                    if (amount < 0.001f) { __result = 0; return false; }  // 完全偏转
                }
            }
        }
        return true;  // 继续原版护甲计算
    }
}
```

**设计模式**：Harmony Prefix + 调用原版内部方法——不替换原版逻辑，而是在原版之前插入额外护甲层。`CoversBodyPart(part)`实现部位级护盾覆盖。

### 2.4 护盾实现模式总结

| 模式 | 基于 | 拦截位置 | 适用场景 |
|------|------|---------|---------|
| **ThingComp护盾** | ThingComp | 阶段4 PostPreApplyDamage | 装备/内置护盾（原版模式） |
| **HediffComp护盾** | HediffComp | 阶段4 PreApplyDamage | 状态/能力/基因附加护盾（VEF模式） |
| **护甲管线护盾** | Harmony补丁 | 阶段5 护甲计算 | 部位级护甲增强（VEF模式） |
| **区域护盾** | ThingComp | 投射物飞行阶段 | 区域防护（原版+VEF模式） |

> **要点**：
> 1. HediffComp_Shield填补了原版空白——原版没有基于Hediff的护盾，VEF的实现证明这条路可行且灵活
> 2. 护盾纳入护甲管线需要Harmony补丁——原版`GetPostArmorDamage`不检查自定义护盾Comp

## 3. Harmony补丁修改战斗机制

### 3.1 实例总览

| # | 模组 | 补丁目标 | 补丁类型 | 效果 |
|---|------|---------|---------|------|
| 1 | VEF | `ArmorUtility.GetPostArmorDamage` | Prefix | 护盾参与护甲计算（见2.3） |
| 2 | AncotLibrary | `Pawn_HealthTracker.PreApplyDamage` | Prefix | 武器Comp也能拦截伤害 |
| 3 | VEF | `Verb_MeleeAttackDamage.DamageInfosToApply` | Postfix | 修改近战伤害生成（附加效果） |
| 4 | VEF | `Verb.TryCastNextBurstShot` | Postfix | 修改连发行为（弹药消耗等） |
| 5 | VEF | `Thing.TakeDamage` | Prefix | 拦截特定Thing的伤害 |

### 3.2 ★ AncotLibrary PreApplyDamage补丁（武器Comp拦截伤害）

**模组**：AncotLibrary
**核心思路**：原版`PreApplyDamage`只检查Pawn自身Comp和服装Comp，**不检查武器Comp**。此补丁填补这个盲点。

```csharp
[HarmonyPatch(typeof(Pawn_HealthTracker), "PreApplyDamage")]
public static class Ancot_Pawn_HealthTracker_PreApplyDamage_Patch
{
    [HarmonyPrefix]
    public static bool Prefix(Pawn_HealthTracker __instance, DamageInfo dinfo, ref bool absorbed)
    {
        Pawn pawn = GetPawn(__instance);  // 通过Expression编译的高性能字段访问
        ThingWithComps weapon = pawn.equipment?.Primary;
        if (weapon != null)
        {
            List<ThingComp> allComps = weapon.AllComps;
            for (int i = 0; i < allComps.Count; i++)
            {
                allComps[i].PostPreApplyDamage(ref dinfo, out absorbed);
                if (absorbed) return false;  // 武器Comp吸收了伤害→跳过原版逻辑
            }
        }
        return true;  // 继续原版PreApplyDamage
    }
}
```

**揭示的原版盲点**：原版伤害管线阶段4的拦截链是：基因damageFactors → Hediff damageFactors → ThingComp.PostPreApplyDamage → Apparel.CheckPreAbsorbDamage。注意**武器（equipment.Primary）的Comp不在此链中**。AncotLibrary通过Harmony Prefix补全了这个缺失。

**技术细节**：使用`Expression.Lambda`编译的委托访问`Pawn_HealthTracker`的私有`pawn`字段，比反射快10倍以上。

### 3.3 补丁位置与风险分析

| 补丁位置 | 管线阶段 | 风险等级 | 兼容性影响 |
|---------|---------|---------|-----------|
| `PreApplyDamage` Prefix | 阶段4 | ★★☆ 中 | 可能与其他Prefix冲突 |
| `ArmorUtility.GetPostArmorDamage` Prefix | 阶段5 | ★★☆ 中 | 影响所有护甲计算 |
| `Thing.TakeDamage` Prefix | 阶段3 | ★★★ 高 | 影响所有伤害入口 |
| `Verb.TryCastNextBurstShot` Postfix | 阶段1 | ★☆☆ 低 | 仅影响连发行为 |
| `DamageInfosToApply` Postfix | 阶段1 | ★☆☆ 低 | 仅影响近战伤害生成 |

> **要点**：
> 1. Postfix比Prefix安全——Postfix不阻止原版执行，多个Postfix可共存；Prefix可能互相覆盖
> 2. 补丁越早（阶段3-4）影响越大——`Thing.TakeDamage`的Prefix影响所有伤害，应极度谨慎

### 4.2 ★ AncotLibrary Verb_MeleeAttackDamage_Charged（蓄力近战）

**模组**：AncotLibrary
**核心思路**：继承`Verb_MeleeAttack`，与`CompWeaponCharge`协作实现蓄力机制——蓄力完成后下一次攻击伤害和穿甲倍增。

```csharp
public class Verb_MeleeAttackDamage_Charged : Verb_MeleeAttack
{
    public CompWeaponCharge compCharge => EquipmentSource.TryGetComp<CompWeaponCharge>();
    public CompMeleeCombo compMeleeCombo => EquipmentSource.TryGetComp<CompMeleeCombo>();

    // 蓄力伤害倍率
    public virtual float ChargedDamageFactor()
        => compCharge.CanBeUsed ? compCharge.MeleeDamageFactorCharged : 1f;

    // 蓄力穿甲倍率
    public virtual float ChargedArmorPenetrationFactor()
        => compCharge.CanBeUsed ? compCharge.MeleeArmorPenetrationFactorCharged : 1f;

    private IEnumerable<DamageInfo> DamageInfosToApply(LocalTargetInfo target)
    {
        float num = verbProps.AdjustedMeleeDamageAmount(this, CasterPawn);
        float armorPenetration = verbProps.AdjustedArmorPenetration(this, CasterPawn)
                                 * ChargedArmorPenetrationFactor();  // ★ 穿甲倍增
        num = Rand.Range(num * 0.8f, num * 1.2f)
              * ChargedDamageFactor();                               // ★ 伤害倍增
        EffecterCharged(target);                                     // ★ 蓄力消耗+视觉特效
        // ... 构建DamageInfo并yield return
    }
}
```

**设计模式**：Verb + Comp协作——Verb负责攻击流程和伤害计算，Comp负责蓄力状态管理和参数配置。`CompMeleeCombo`还支持连击系统。

### 4.3 自定义Verb实现模式总结

| 模式 | 继承自 | 重写方法 | 适用场景 |
|------|--------|---------|---------|
| **即时命中** | Verb | `TryCastShot()` | 激光/射线（无投射物） |
| **持续伤害** | Verb | `TryCastShot()` + Tick逻辑 | 持续激光束 |
| **增强近战** | Verb_MeleeAttack | `DamageInfosToApply()` | 蓄力/连击/附加效果 |
| **位移攻击** | Verb | `TryCastShot()` + PawnFlyer | 跳跃攻击/冲锋 |
| **模式切换** | Verb | `TryCastShot()` + Comp状态 | 多模式武器 |

> **要点**：
> 1. 自定义Verb的核心是重写`TryCastShot()`——原版投射物逻辑在此方法中，替换它可实现任何攻击方式
> 2. Verb + Comp协作是最佳实践——Verb处理攻击逻辑，Comp管理状态和配置，XML可调参数

## 5. 5种技术路径对比与选型指南

| 维度 | XML继承 | 自定义DamageWorker | 自定义Verb | Comp | Harmony补丁 |
|------|--------|-------------------|-----------|------|------------|
| **需要C#** | 否 | 是 | 是 | 是 | 是 |
| **侵入性** | 无 | 低 | 低 | 低 | 高 |
| **兼容风险** | 无 | 低 | 低 | 低 | 中~高 |
| **灵活性** | 低 | 中 | 高 | 高 | 最高 |
| **适用场景** | 改数值/标志位 | 特殊伤害逻辑 | 新攻击方式 | 新护盾/拦截 | 修改原版管线 |
| **典型实例** | 新弹药类型 | 毒性/心灵伤害 | 激光/蓄力 | Hediff护盾 | 护甲计算修改 |

**选型决策树**：

```
需要新伤害类型？
  ├── 只改数值/标志位 → XML继承DamageDef
  └── 需要特殊逻辑 → 自定义DamageWorker
需要新攻击方式？
  ├── 投射物变体 → 自定义Projectile子类
  └── 非投射物（射线/跳跃） → 自定义Verb
需要新防护机制？
  ├── 装备级 → ThingComp（原版模式）
  ├── 状态级 → HediffComp（VEF模式）
  └── 需要修改护甲计算 → Harmony补丁
需要修改原版行为？
  └── Harmony补丁（最后手段）
```
