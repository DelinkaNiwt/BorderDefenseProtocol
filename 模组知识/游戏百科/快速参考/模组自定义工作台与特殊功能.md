# 模组自定义工作台与特殊功能

**总览**：社区模组通过**5种扩展模式**自定义工作台——纯XML继承（标准制造，零C#）、Comp扩展标准工作台（新资源条件/环境副作用/自动产出）、自定义Bill子类（操作特定物品而非通用材料）、Building_Enterable替代（Pawn进入建筑被处理）、完全自定义工作流（绕过Bill系统的独立交互逻辑）。本文分析3个代表性模组：VEF（框架级工具）、AlteredCarbon（复杂工作台体系）、Milira（环境副作用工作台）。

```
┌─────────────────────────────────────────────────────────────────┐
│         模组自定义工作台的5种扩展模式（按复杂度排列）            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  模式1：纯XML继承BenchBase ← 最简单，零C#                      │
│    └── ParentName继承BenchBase，recipeUsers/recipes绑定配方    │
│                                                                 │
│  模式2：Comp扩展标准工作台 ← 新资源/环境条件                   │
│    └── ThingComp附加管道资源消耗/热量推送/自动产出等机制       │
│                                                                 │
│  模式3：自定义Bill子类 ← 操作特定物品                          │
│    └── 继承Bill_Production，targetThing绑定操作目标            │
│        + Gizmo选择UI替代ITab_Bills                             │

│    └── 继承Building_Enterable，Pawn进入→自动处理→退出         │

│  模式5：完全自定义工作流 ← 绕过Bill系统                        │
│    └── 自定义WorkGiver+Job+Comp，完全独立的交互逻辑           │
│                                                                 │
```

## 1. VEF（原版扩展框架）——框架级工作台工具

### 1.1 Building_ResourceWorkTable（管道资源消耗型工作台）

**核心思路**：继承`Building_WorkTable`，增加管道资源（水/气/化学品等）作为工作前置条件。工作台必须连接到PipeSystem管道网络并有足够资源才能使用。

```csharp
// Building_ResourceWorkTable — 需要管道资源的工作台
public class Building_ResourceWorkTable : Building_WorkTable, IBillGiver, IBillGiverWithTickAction
{
    private List<CompResourceTrader> traders = new List<CompResourceTrader>();

    // 是否可以无资源工作（unpoweredWorkTableWorkSpeedFactor > 0 时允许）
    public bool CanWorkWithoutResource =>
        traders.NullOrEmpty() || def.building.unpoweredWorkTableWorkSpeedFactor > 0f;

    // 所有管道资源是否就绪
    public bool AllResourceTraderOn =>
        traders.All(comp => comp.ResourceOn);

    public override void UsedThisTick()
    {
        base.UsedThisTick();
        // ★ 每tick通知所有资源交易器——消耗管道资源
        for (int i = 0; i < tradersCount; i++)
            traders[i].Notify_UsedThisTick();
    }

    // ★ 可用性判定：原版条件 + 管道资源就绪
    public new bool CurrentlyUsableForBills() =>
        base.CurrentlyUsableForBills() && (CanWorkWithoutResource || AllResourceTraderOn);
}
```

**设计模式**：**继承Building_WorkTable + CompResourceTrader条件检查**——最小侵入式扩展，仅重写`CurrentlyUsableForBills()`和`UsedThisTick()`两个方法。`CompResourceTrader`是PipeSystem的通用管道资源消费组件，与`CompPowerTrader`设计对称。支持`unpoweredWorkTableWorkSpeedFactor`降级工作（类似断电降速）。

### 1.2 RecipeInheritanceExtension（配方继承扩展）

**核心思路**：DefModExtension让工作台从其他工作台继承配方，支持产品白名单/黑名单和配方白名单/黑名单过滤。

```csharp
// RecipeInheritanceExtension — 配方继承+过滤
public class RecipeInheritanceExtension : DefModExtension
{
    public List<ThingDef> inheritRecipesFrom;      // 从哪些工作台继承配方
    public ThingFilter allowedProductFilter;        // 产品白名单
    public ThingFilter disallowedProductFilter;     // 产品黑名单
    public List<RecipeDef> allowedRecipes;          // 配方白名单
    public List<RecipeDef> disallowedRecipes;       // 配方黑名单

    public bool Allows(RecipeDef recipe)
    {
        var producedThingDef = recipe.ProducedThingDef;
        return (producedThingDef == null
                || ((allowedProductFilter == null || allowedProductFilter.Allows(producedThingDef))
                    && (disallowedProductFilter == null || !disallowedProductFilter.Allows(producedThingDef))))
               && (allowedRecipes == null || allowedRecipes.Contains(recipe))
               && (disallowedRecipes == null || !disallowedRecipes.Contains(recipe));
    }
}
```

**设计模式**：**DefModExtension + 双重过滤**——模组新增"升级版工作台"只需XML配置`inheritRecipesFrom`+过滤条件，无需手动复制配方列表。VEF在启动时自动将继承的配方注入目标工作台的`AllRecipes`列表。

### 1.3 CompConfigurableSpawner（可配置自动产出）

**核心思路**：Comp附加在建筑上，周期性自动生成物品，无需Pawn操作。通过Gizmo让玩家选择产出类型（`ConfigurableSpawnerDef`列表）。

```csharp
// CompConfigurableSpawner — 自动产出（无需Pawn）
public class CompConfigurableSpawner : ThingComp
{
    public ConfigurableSpawnerDef currentThingList = null;  // 当前选择的产出配置
    private int ticksUntilSpawn;

    private void TickInterval(int interval)
    {
        if (currentThingList == null || parent.MapHeld is null) return;
        // 前置条件检查：电力+燃料+非休眠
        if (PropsSpawner.requiresPower && !PowerOn) return;
        if (PropsSpawner.requiresFuel && !FuelOn) return;
        ticksUntilSpawn -= interval;
        if (ticksUntilSpawn <= 0)
        {
            TryDoSpawn();       // 在相邻格生成物品
            ResetCountdown();   // 重置倒计时
        }
    }

    // Gizmo：选择产出类型
    public override IEnumerable<Gizmo> CompGetGizmosExtra()
    {
        yield return ConfigurableSpawnerSettableUtility.SetItemsToSpawnCommand(this);
    }
}
```

**设计模式**：**Comp + 倒计时 + Gizmo选品**——不使用Bill系统，完全独立的自动产出逻辑。`ConfigurableSpawnerDef`定义可选产出列表+间隔时间，玩家通过Gizmo切换。适合"资源采集器"类建筑（如矿井、水井、气体收集器）。

## 2. AlteredCarbon（副本2重生）——复杂工作台体系

### 2.1 Building_NeuralEditor（Gizmo驱动的特殊操作工作台）

**核心思路**：继承`Building_WorkTable`，但不使用标准ITab_Bills添加配方，而是通过5种自定义Gizmo按钮让玩家选择目标物品，然后创建自定义Bill子类实例。支持连接到`Building_NeuralMatrix`设施获取额外功能。

**5种Gizmo操作**：

| # | 操作 | Gizmo类型 | Bill类型 | 目标 |
|---|------|----------|---------|------|
| 1 | 擦除神经堆栈 | `Command_ActionOnStack` | `Bill_OperateOnStack` | 神经堆栈物品 |
| 2 | 复制神经堆栈 | `Command_ActionOnStack` | `Bill_OperateOnStack` | 神经堆栈/Pawn |
| 3 | 编辑神经堆栈 | `Command_ActionOnStack` | `Bill_EditStack` | 神经堆栈（打开编辑窗口） |
| 4 | 重置生物编码 | `Command_ActionOnBiocoding` | `Bill_OperateOnThing` | 生物编码物品 |
| 5 | 解绑人格武器 | `Command_ActionOnPersona` | `Bill_OperateOnThing` | 人格武器 |

```csharp
// Building_NeuralEditor — Gizmo驱动的特殊操作工作台
public class Building_NeuralEditor : Building_WorkTable, IMatrixConnectable
{
    // ★ 通过CompAffectedByFacilities连接到NeuralMatrix
    public Building_NeuralMatrix ConnectedMatrix =>
        CompAffectedByFacilities.LinkedFacilitiesListForReading
            .OfType<Building_NeuralMatrix>().FirstOrDefault();

    public override IEnumerable<Gizmo> GetGizmos()
    {
        foreach (Gizmo gizmo in base.GetGizmos()) yield return gizmo;

        // 5种操作Gizmo（每种含操作按钮+取消按钮）
        foreach (Gizmo cmd in GetCommands<Command_ActionOnStack>(new CommandInfo
        {
            icon = "UI/Gizmos/WipeStack",
            action = InstallWipeStackBill,
            lockedProjects = new List<ResearchProjectDef> { AC_DefOf.AC_NeuralEditing },
            recipe = AC_DefOf.AC_WipeActiveNeuralStack,
            // ...
        })) yield return cmd;
        // ... 其他4种操作类似

        // ★ 连接到NeuralMatrix时，显示矩阵管理Gizmo
        if (ConnectedMatrix != null)
            foreach (Gizmo item in ConnectedMatrix.GetManageMatrix())
                yield return item;
    }

    // Gizmo点击后：验证目标→友方警告对话框→创建自定义Bill
    public void InstallWipeStackBill(LocalTargetInfo x)
    {
        HandleStackBill(x, "AC.WipingFriendlyStackWarning", delegate
        {
            billStack.AddBill(new Bill_OperateOnStack(x.Thing, AC_DefOf.AC_WipeActiveNeuralStack));
        });
    }

    // 防重复操作：检查BillStack中是否已有针对同一物品的Bill
    public bool CanAddOperationOn(Thing thing)
    {
        Bill_OperateOnThing existing = billStack.Bills.OfType<Bill_OperateOnThing>()
            .FirstOrDefault(x => x.targetThing == thing);
        if (existing != null) { Messages.Message(...); return false; }
        return true;
    }
}
```

**自定义Bill继承体系**：

```
Bill_Production（原版）
  └── Bill_OperateOnThing（操作特定物品）
        ├── Bill_OperateOnStack（操作神经堆栈）
        │     └── Bill_EditStack（编辑神经堆栈，含编辑数据）
        └── （直接用于生物编码/人格武器）
```

**设计模式**：**Gizmo选目标 + 自定义Bill + 设施连接**——用Gizmo替代ITab_Bills作为操作入口（因为操作对象是特定物品而非通用材料），用自定义Bill子类的`targetThing`字段绑定操作目标，用`CompAffectedByFacilities`连接到NeuralMatrix获取额外功能。每种操作都有配对的取消Gizmo。

### 2.2 Building_SleeveGestator（Pawn身体孵化器）

**核心思路**：继承`Building_Enterable`（非Building_WorkTable），实现"Pawn身体培育"功能。玩家通过Gizmo打开自定义Window配置身体参数，然后建筑自动孵化，完成后产出新Pawn身体。

**完整工作流**：

```
Gizmo按钮 → Window_SleeveCustomization（配置身体参数）
  → 设置孵化任务 → WorkGiver_StartIncubatingProcess扫描
    → JobDriver_StartIncubatingProcess（殖民者启动孵化）
      → 建筑自动处理（Tick中推进进度）
        → 完成后产出Pawn身体
```

**协作类**：

| 类 | 职责 |
|----|------|
| `Building_SleeveGestator` | 孵化器主体（继承Building_Enterable→Building_Incubator） |
| `Window_SleeveCustomization` | 自定义Window：配置身体外观/基因/性别等参数 |
| `WorkGiver_StartIncubatingProcess` | 扫描需要启动的孵化器 |
| `JobDriver_StartIncubatingProcess` | 殖民者前往孵化器执行启动操作 |
| `WorkGiver_HaulCorpseToSleeveGrower` | 搬运尸体到孵化器（用于身体回收） |
| `JobDriver_HaulCorpseToGestation` | 搬运尸体的Job执行逻辑 |

**设计模式**：**Building_Enterable + 自定义Window + WorkGiver/Job协作**——Building_Enterable提供"Pawn进入→处理→退出"的标准框架，自定义Window提供复杂配置UI（远超ITab_Bills的能力），WorkGiver/Job负责殖民者交互。这是原版GrowthVat/GeneExtractor/MechGestator的模组扩展版。

## 3. Milira（米莉拉）——环境副作用工作台

### 3.1 Building_SunBlastFurnace（太阳高炉）

**核心思路**：继承`Building_WorkTable`，通过3个自定义Comp在使用时产生环境副作用——推送热量、室外故障、非法使用检测。

```csharp
// Building_SunBlastFurnace — 有环境副作用的工作台
public class Building_SunBlastFurnace : Building_WorkTable
{
    private CompSunBlastFurnaceOutdoorBreakdown furnaceOutdoorBreakdownComp;
    private CompSunBlasterFurnaceHeatPusher sunBlasterFurnaceHeatPusherComp;
    private CompSunBlastFurnaceIllegalUse sunBlastFurnaceIllegalUseComp;

    public override void UsedThisTick()
    {
        base.UsedThisTick();
        // ★ 使用时推送热量（类似熔炉但更强）
        if (sunBlasterFurnaceHeatPusherComp != null && Map != null)
            sunBlasterFurnaceHeatPusherComp.HeatPush();

        // ★ 室外使用时累积故障（有燃料时才检查）
        if (furnaceOutdoorBreakdownComp != null
            && Position.UsesOutdoorTemperature(Map)
            && refuelComp.Fuel != 0f)
            furnaceOutdoorBreakdownComp.Notify_UsedThisTick();

        // ★ 非法使用检测（可能触发派系惩罚等）
        if (sunBlastFurnaceIllegalUseComp != null && Map != null)
            sunBlastFurnaceIllegalUseComp.Notify_UsedThisTick();
    }
}
```

**设计模式**：**Building_WorkTable + 多Comp副作用**——工作台本身保持标准Bill系统不变，所有特殊行为通过Comp在`UsedThisTick()`中触发。3个Comp各自独立：热量推送（影响房间温度）、室外故障（限制放置位置）、非法使用（叙事/派系后果）。这种"标准工作台+副作用Comp"模式可复用于任何需要"使用时产生环境影响"的工作台。

## 4. 其他扩展工具

### 4.1 AncotLibrary——机械体Bill限制

**核心思路**：扩展原版Bill对话框，增加"限制机械体执行"的下拉选项。让模组工作台可以指定特定机械体执行配方。

```csharp
// AncotBillDialogUtility — 机械体Bill限制
public static class AncotBillDialogUtility
{
    public static IEnumerable<Widgets.DropdownMenuElement<Pawn>>
        GetMechRestrictionOptionsForBill(Bill bill, Func<Pawn, bool> pawnValidator = null)
    {
        // 列出地图上所有殖民地机械体
        IEnumerable<Pawn> mechs = bill.Map.mapPawns.SpawnedColonyMechs
            .Where(p => pawnValidator == null || pawnValidator(p))
            .OrderBy(p => p.LabelShortCap);

        // 按技能等级排序（如果配方有workSkill要求）
        if (bill.recipe.workSkill != null)
            mechs = mechs.OrderByDescending(p => p.RaceProps.mechFixedSkillLevel);

        // 生成下拉菜单项：显示机械体名称+技能等级+工作分配状态
        foreach (Pawn pawn in mechs)
            yield return new Widgets.DropdownMenuElement<Pawn>
            {
                option = new FloatMenuOption($"{pawn.LabelShortCap} ({pawn.RaceProps.mechFixedSkillLevel} {bill.recipe.workSkill.label})",
                    () => bill.SetPawnRestriction(pawn)),
                payload = pawn
            };
    }
}
```

**设计模式**：**静态工具类 + Harmony补丁注入Bill对话框**——原版Bill对话框只支持殖民者限制，AncotLibrary通过Harmony补丁在`Dialog_BillConfig`中注入机械体选项。模组工作台如果需要机械体执行配方，可直接使用此工具类。

### 4.2 GD5——完全自定义工作流（CompCommunicationStation）

**核心思路**：通讯站不使用Bill系统，而是通过自定义`CompCommunicationStation`+`WorkGiver_CommunicationStation`+`JobDriver`实现完全独立的交互逻辑。殖民者前往通讯站执行固定时长的操作Job。

```csharp
// WorkGiver_CommunicationStation — 自定义工作流（绕过Bill系统）
public class WorkGiver_CommunicationStation : WorkGiver_Scanner
{
    public override ThingRequest PotentialWorkThingRequest =>
        ThingRequest.ForDef(GDDefOf.GD_CommunicationStation);

    public override bool HasJobOnThing(Pawn pawn, Thing t, bool forced = false)
    {
        // 条件检查：同派系+可预约+Comp就绪+未拆除+未着火
        return t.Faction == pawn.Faction
            && pawn.CanReserve(t, 1, -1, null, forced)
            && t.TryGetComp<CompCommunicationStation>().CanOperateNow()
            && t.Map.designationManager.DesignationOn(t, DesignationDefOf.Uninstall) == null
            && !t.IsBurning();
    }

    public override Job JobOnThing(Pawn pawn, Thing t, bool forced = false)
    {
        // 固定1500 ticks的操作Job
        return JobMaker.MakeJob(GDDefOf.GD_OperateStation, t, 1500, checkOverrideOnExpiry: true);
    }
}
```

**设计模式**：**Comp + WorkGiver + Job（无Bill）**——适合"不需要材料、不需要配方选择、只需要殖民者操作一段时间"的建筑。比Bill系统轻量得多，但失去了配方选择/材料管理/重复模式等功能。

## 5. 5种扩展模式对比与选型指南

| 维度 | 纯XML继承 | Comp扩展 | 自定义Bill | Building_Enterable | 完全自定义 |
|------|----------|---------|-----------|-------------------|-----------|
| **需要C#** | 否 | 是 | 是 | 是 | 是 |
| **复杂度** | 低 | 中 | 中~高 | 高 | 中 |
| **保留Bill系统** | ✅ | ✅ | ✅（扩展） | ❌ | ❌ |
| **配方选择** | ✅ | ✅ | Gizmo替代 | 自定义Window | 无 |
| **材料消耗** | ✅ | ✅ | 可选 | 可选 | 无 |
| **兼容风险** | 无 | 低 | 低 | 低 | 低 |
| **适用场景** | 标准制造 | 新资源条件/副作用 | 操作特定物品 | Pawn进入处理 | 简单定时操作 |
| **典型实例** | 大多数模组 | VEF/Milira | AlteredCarbon | AlteredCarbon/原版 | GD5/VEF |

**选型决策树**：

```
需要自定义工作台？
  ├── 标准制造（材料→产品） → 纯XML继承BenchBase
  │     └── 需要额外资源条件？ → Comp扩展（CompResourceTrader等）
  │     └── 需要环境副作用？ → Comp扩展（UsedThisTick中触发）
  ├── 操作特定物品（非通用材料） → 自定义Bill子类 + Gizmo选目标
  ├── Pawn进入建筑被处理 → Building_Enterable + 自定义Window
  ├── 无需材料的定时操作 → 自定义WorkGiver + Job（无Bill）
  └── 无需Pawn的自动产出 → CompConfigurableSpawner模式
```
