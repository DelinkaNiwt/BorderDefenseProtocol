# 建筑供电系统

**总览**：RimWorld的电力系统采用四层架构——`CompPower`组件层（附加在建筑上，定义电力角色）→ `PowerNet`电网层（管理一组连通的电力组件，执行每tick电力平衡）→ `PowerNetManager`管理层（管理地图上所有电网的创建/销毁/合并/分裂）→ `PowerNetGrid`网格层（IntVec3→CompPower的空间索引）。电力以"瓦特日"（Wd）为存储单位，以"瓦特"（W）为功率单位，负值basePowerConsumption表示产出。

## 1. 类继承体系

```
CompPower (抽象基类, RimWorld)
│  核心字段: transNet, connectParent, connectChildren
│  核心属性: TransmitsPowerNow, PowerNet
│
├── CompPowerTransmitter  — 电力传输（空类，仅标记身份）
│     使用者: 电线(PowerConduit)、墙内电线、电源开关
│
├── CompPowerBattery      — 电力存储
│     核心: storedEnergy, AmountCanAccept, AddEnergy(), DrawPower()
│     使用者: 电池
│
└── CompPowerTrader        — 电力交易（消费或生产）
      核心: PowerOn, PowerOutput, EnergyOutputPerTick
      使用者: 所有用电建筑 + 所有发电建筑
      │
      └── CompPowerPlant    — 发电厂基类
            核心: DesiredPowerOutput, UpdateDesiredPowerOutput()
            条件检查链: 故障? → 无燃料? → 开关关闭? → 自动关闭? → 毒化器不可用?
            │
            ├── CompPowerPlantSolar          — 太阳能（日照因子）
            ├── CompPowerPlantWind           — 风力（风速因子×遮挡检测）
            ├── CompPowerPlantSteam          — 地热（需间歇泉）
            ├── CompPowerPlantWater          — 水力（需河流地形）
            ├── CompPowerPlantGravcore       — 重力核心(Odyssey)
            ├── CompPowerPlantElectroharvester — 电力收割器(Anomaly)
            └── CompPowerPlantAncientReactor — 远古反应堆(Odyssey)
```

## 2. PowerNet电网系统

每个`PowerNet`实例代表一组物理连通的电力组件，维护4个列表：`transmitters`（传输器）、`connectors`（连接器）、`powerComps`（CompPowerTrader列表）、`batteryComps`（CompPowerBattery列表）。

### 2.1 电网生命周期

| 事件 | 触发方法 | 结果 |
|------|---------|------|
| **传输器放置** | `Notify_TransmitterSpawned()` | 在该格创建新电网 / 合并相邻电网 |
| **传输器移除** | `Notify_TransmitterDespawned()` | 销毁旧电网 → 对相邻格重建电网（可能分裂为多个） |
| **开关切换** | `Notfiy_TransmitterTransmitsPowerNowChanged()` | 等同于移除+重新放置（电网分裂或合并） |
| **连接器放置** | `Notify_ConnectorWantsConnect()` | 搜索最近传输器 → 注册到其所属电网 |
| **连接器移除** | `Notify_ConnectorDespawned()` | 从电网注销 |

### 2.2 电网创建算法（BFS连通）

`TryCreateNetAt()`从一个传输器格子出发，BFS遍历所有相邻且`TransmitsPowerNow=true`的传输器，形成一个连通分量作为新电网。然后遍历所有传输器的相邻格，将范围内的连接器（CompPowerTrader/CompPowerBattery）注册到该电网。

### 2.3 延迟动作机制

所有电网拓扑变更通过`DelayedAction`队列延迟到`UpdatePowerNetsAndConnections_First()`统一执行，避免在Tick中途修改电网结构导致迭代器失效。

## 3. 传输器 vs 连接器

| 维度 | 传输器（Transmitter） | 连接器（Connector） |
|------|---------------------|-------------------|
| **CompPower子类** | CompPowerTransmitter | CompPowerTrader / CompPowerBattery |
| **ThingDef标志** | `transmitsPower=true` | `ConnectToPower=true`（CompProperties_Power） |
| **电网角色** | 构成电网骨架（BFS连通） | 附着在电网上 |
| **连接方式** | 相邻格自动连通 | 搜索最近传输器（≤6格），通过`connectParent`连接 |
| **连接距离** | 相邻格（1格） | 最大6.9格（`MaxDistanceToLinkToTransmitter`） |
| **典型建筑** | 电线、墙内电线、电源开关 | 工作台、炮塔、灯、发电机、电池 |

**关键设计**：发电机和电池虽然是电力"源"，但在拓扑上是连接器——它们连接到最近的电线，而非自身构成电网骨架。

## 4. 电力平衡算法（PowerNetTick）

`PowerNetTick()`每tick执行一次，是电力系统的核心调度逻辑。

### 4.1 完整流程

```
PowerNetTick()
  ① 计算 energyGainRate = Σ 所有PowerOn的CompPowerTrader.EnergyOutputPerTick
     （发电机贡献正值，用电建筑贡献负值）
  ② 计算 storedEnergy = Σ 所有电池.StoredEnergy
  ③ 检查 storedEnergy + energyGainRate ≥ 0 且 电力未被游戏条件禁用?
  │
  ├── YES（电力充足分支）：
  │   ④ 计算有效存储 = 有电池且存储≥0.1 ? storedEnergy - 5 : storedEnergy
  │     （预留5Wd缓冲，防止电池自放电导致误判）
  │   ⑤ 有效存储 + energyGainRate ≥ 0 ?
  │   │
  │   ├── YES：渐进式开启关闭的设备
  │   │   收集所有 !PowerOn && WantsToBeOn && !BrokenDown 的设备
  │   │   每次开启 max(1, count×5%) 个（随机选取）
  │   │   开启间隔 = max(30, 200/count) ticks
  │   │   开启前检查: energyGainRate + storedEnergy ≥ -设备消耗
  │   │
  │   └── NO：电力不足，进入关闭分支
  │
  │   ⑥ ChangeStoredEnergy(energyGainRate)
  │      正值 → DistributeEnergyAmongBatteries（均匀充电）
  │      负值 → 从有电电池均匀放电
  │
  └── NO（电力不足分支）：
      ⑦ 每20 ticks执行一次渐进式关闭
      收集所有 PowerOn && EnergyOutputPerTick < 0 的消费设备
      每次关闭 max(1, count×5%) 个（随机选取）
```

### 4.2 渐进式开关的设计意图

- **不是瞬间全开/全关**——每次只操作5%的设备，避免电力系统震荡
- **开启有能量检查**——只在剩余能量足够承担新设备消耗时才开启
- **关闭无优先级**——随机选择关闭哪个设备，玩家无法控制关闭顺序
- **开启间隔自适应**——设备越多间隔越短（最快30 ticks），设备少则间隔长（最慢200 ticks）

## 6. 电池系统

### 6.1 核心参数

| 参数 | 值 | 来源 |
|------|-----|------|
| **最大存储** | 600 Wd | `CompProperties_Battery.storedEnergyMax` |
| **充电效率** | 50% | `CompProperties_Battery.efficiency = 0.5` |
| **自放电** | 5W | `CompPowerBattery.CompTick()`: `DrawPower(5 × WattsToWattDaysPerTick)` |
| **EMP禁用** | 是 | `StunnedByEMP` → `AmountCanAccept=0` + `AddEnergy`拒绝 |
| **故障清空** | 是 | `ReceiveCompSignal("Breakdown")` → `DrawPower(StoredEnergy)` |

### 6.2 充电算法（DistributeEnergyAmongBatteries）

当电网有多余能量时，`DistributeEnergyAmongBatteries()`执行均匀充电：

1. 随机打乱电池列表（避免固定顺序偏向）
2. 找到所有电池中`AmountCanAccept`的最小值
3. 如果多余能量 ≥ 最小值 × 电池数：每个电池充入最小值，移除已满电池
4. 如果多余能量 < 最小值 × 电池数：平均分配给所有电池
5. 重复直到能量耗尽或所有电池满

**AmountCanAccept计算**：`(storedEnergyMax - storedEnergy) / efficiency`——因为效率50%，实际需要输入2倍能量才能充满，所以AmountCanAccept返回的是"需要输入的能量"而非"实际存储增量"。

**AddEnergy效率损耗**：`storedEnergy += amount × efficiency`——输入100Wd只存储50Wd。

### 6.3 放电算法（ChangeStoredEnergy负值分支）

当电网能量不足时，从有电电池均匀放电：

1. 收集所有`StoredEnergy > 0`的电池
2. 计算每个电池应放电量 = 总需求 / 电池数
3. 每个电池放电`min(应放电量, 实际存储)`
4. 最多迭代10次（处理部分电池提前耗尽的情况）

### 6.4 自放电机制

`CompPowerBattery.CompTick()`每tick执行：`DrawPower(min(5W × WattsToWattDaysPerTick, storedEnergy))`。5W自放电意味着一个满电池（600Wd）在无充电情况下约120天耗尽。

## 7. 短路机制

### 7.1 触发条件（两种路径）

| 路径 | 触发条件 | 方法 |
|------|---------|------|
| **雨天短路** | 电力建筑淋雨 + (`CompPowerTrader.PowerOn && shortCircuitInRain`) 或 (`CompPowerBattery.StoredEnergy > 100`) | `TryShortCircuitInRain()` |
| **事件短路** | Storyteller调度`IncidentWorker_ShortCircuit`事件 → 选择电网中的电线 | `DoShortCircuit()` |

### 7.2 短路效果（按电池存储量分级）

| 条件 | 效果 | 详情 |
|------|------|------|
| **任意电池StoredEnergy > 20Wd** | 爆炸 + 排空所有电池 | 爆炸半径 = `clamp(sqrt(totalEnergy) × 0.05, 1.5, 14.9)`；>3.5格追加Bomb爆炸 |
| **所有电池StoredEnergy ≤ 20Wd** | 起火 | 在短路点附近随机起火 |
| **雨天短路** | 小爆炸 | 固定1.9格火焰爆炸，不排空电池 |

### 7.3 爆炸半径速查

| 总存储能量 | 爆炸半径 | 追加Bomb爆炸 |
|-----------|---------|-------------|
| 100 Wd | 1.5格（最小值） | 否 |
| 900 Wd | 1.5格 | 否 |
| 5000 Wd | 3.5格 | 否 |
| 10000 Wd | 5.0格 | 是（1.5格Bomb） |
| 50000 Wd | 11.2格 | 是（3.4格Bomb） |
| 90000 Wd | 14.9格（最大值） | 是（4.5格Bomb） |

## 8. 电源开关（Building_PowerSwitch）

`Building_PowerSwitch`是唯一能动态分割/合并电网的建筑。

### 8.1 核心机制

```
Building_PowerSwitch
  ├── TransmitsPowerNow → FlickUtility.WantsToBeOn(this)
  │     开关打开 = true → 电力可通过
  │     开关关闭 = false → 电力不可通过
  │
  ├── CompFlickable联动
  │     接收信号: FlickedOff / FlickedOn / ScheduledOn / ScheduledOff
  │     → UpdatePowerGrid()
  │
  └── UpdatePowerGrid()
        检测 WantsToBeOn 是否变化
        → Map.powerNetManager.Notfiy_TransmitterTransmitsPowerNowChanged()
        → 销毁旧电网 → 重建电网（分裂为两个或合并为一个）
```

### 8.2 电网分割/合并

- **关闭开关**：开关的`TransmitsPowerNow`变为false → BFS无法穿过该格 → 两侧形成独立电网
- **打开开关**：开关的`TransmitsPowerNow`变为true → BFS可穿过 → 两侧合并为一个电网
- **延迟执行**：所有拓扑变更通过DelayedAction队列延迟到下一帧统一处理

## 9. 电力消费建筑（CompPowerTrader）

### 9.1 PowerOn状态机

| 状态 | 条件 | 行为 |
|------|------|------|
| **PowerOn=true** | 有电网 + 电力充足 + WantsToBeOn + 未故障 | 正常工作，消耗`basePowerConsumption`瓦特 |
| **PowerOn=false（无电）** | 电网电力不足 | PowerNetTick渐进关闭；等待电力恢复后渐进开启 |
| **PowerOn=false（开关关闭）** | `FlickUtility.WantsToBeOn()=false` | 不消耗电力，不参与PowerNetTick |
| **PowerOn=false（故障）** | `IsBrokenDown()=true` | 不消耗电力，需修复 |
| **PowerOn=false（EMP）** | `StunnedByEMP=true` | 不消耗电力，等待眩晕结束 |

### 9.2 关键信号响应

`CompPowerTrader.ReceiveCompSignal()`响应以下信号：

| 信号 | 来源 | 效果 |
|------|------|------|
| `FlickedOn` / `FlickedOff` | CompFlickable | 重置电力变量 |
| `RanOutOfFuel` | CompRefuelable | PowerOutput = 0 |
| `Refueled` | CompRefuelable | 恢复PowerOutput |
| `ScheduledOn` / `ScheduledOff` | CompSchedule | 重置电力变量 |
| `Breakdown` | CompBreakdownable | PowerOutput = 0 |
| `FixedBreakdown` | CompBreakdownable | 恢复PowerOutput |

### 9.3 EnergyOutputPerTick

`EnergyOutputPerTick = PowerOutput × WattsToWattDaysPerTick`

`WattsToWattDaysPerTick = 1.0 / 60000.0`（1天=60000 ticks，1W=1Wd/天）

- 消费建筑：`basePowerConsumption`为正值（如200W），`PowerOutput`设为负值（-200W），`EnergyOutputPerTick`为负
- 发电建筑：`basePowerConsumption`为负值（如-1000W），`PowerOutput`设为正值（1000W），`EnergyOutputPerTick`为正
