## 一、代码组织最佳实践

### 1.1 文件和文件夹结构

**最佳实践：** 按照功能/模块组织，而非按照类型

**✓ 推荐结构：**

```
MyFramework/
├─ 01_Core/                     # 核心系统
│  ├─ FrameworkInit.cs          # 初始化入口
│  ├─ FrameworkUtility.cs       # 工具函数
│  └─ FrameworkDefOf.cs         # DefOf池
│
├─ 02_Components/               # 组件实现
│  ├─ CompFrameworkXXX.cs
│  ├─ CompProperties_XXX.cs
│  └─ ...
│
├─ 03_Patches/                  # Harmony补丁
│  ├─ Patch_Building.cs
│  ├─ Patch_Pawn.cs
│  └─ ...
│
├─ 04_UI/                        # UI系统
│  ├─ MainTabWindow_Framework.cs
│  └─ Dialog_Framework.cs
│
├─ 05_Defs/                      # XML配置
│  ├─ FrameworkDefs.xml
│  └─ Content/
│
└─ 06_Helper/                    # 辅助类
   └─ FrameworkHelper.cs
```

**✗ 避免的结构：**

```
MyFramework/
├─ Classes/
│  ├─ Comp_Framework.cs
│  ├─ Patch_Framework.cs
│  ├─ UI_Framework.cs
│  └─ Helper_Framework.cs
└─ Defs/
   └─ All.xml

问题：
- 功能关联的代码分散
- 难以快速定位代码
- 不清楚各模块边界
```

### 1.2 命名规范

**最佳实践：** 使用清晰的命名约定，便于代码搜索和理解

**命名规范表：**

| 对象类型 | 命名规范 | 示例 |
|---------|--------|------|
| 工具类 | Framework + Utility | WF_Utility, CMCT_Utility |
| 配置Def | FrameworkConfigDef | MyFrameworkConfigDef |
| DefOf池 | FrameworkDefOf | WF_DefOf, CMCT_DefOf |
| 组件属性 | CompProperties_Name | CompProperties_WeaponFitting |
| 组件类 | Comp_Name | Comp_WeaponFitting |
| 初始化类 | FrameworkInit | WF_Init, CMCT_Init |
| Harmony补丁 | Patch_TargetClass | Patch_Building_Turret |
| UI窗口 | MainTabWindow_Name | MainTabWindow_Turret |
| 私有方法 | PascalCase或 _SnakeCase | CalculateDamage, _applyEffect |
| 常量 | UPPER_CASE | MAX_LEVEL, DEFAULT_VALUE |

**命名示例：**

```csharp
// ✓ 好的命名
public static class WF_Utility        // 工具类：框架缩写 + Utility
{
    public static void ApplyFitting() // 公开方法：动词+名词

    private static void _validateInput()  // 私有方法：_驼峰

    private const int MAX_FITTING_SLOTS = 5;  // 常量：全大写
}

[DefOf]
public static class WF_DefOf         // DefOf：框架缩写 + DefOf
{
    public static ThingCategoryDef WF_Fitting;
}

public class CompProperties_WeaponFitting  // 组件属性：Comp + Properties
{
    public int slots = 3;
}

// ✗ 不好的命名
public static class Utils            // 太通用
{
    public static void Do()          // 含义不清
}
```

### 1.3 类和方法组织

**最佳实践：** 按逻辑关系组织成员，便于阅读

**成员组织顺序：**

```csharp
public class MyClass
{
    // 1. 常量
    public const int CONSTANT = 5;
    private const string DEFAULT = "value";

    // 2. 静态字段
    private static MyClass instance;

    // 3. 静态属性
    public static MyClass Instance => instance ??= new MyClass();

    // 4. 字段
    [SerializeField] private int value;
    [NonSerialized] private Dictionary<string, int> cache;

    // 5. 属性
    public int Value => value;

    // 6. 构造函数
    public MyClass() { }

    // 7. 静态方法
    public static void StaticMethod() { }

    // 8. 公开方法
    public void PublicMethod() { }

    // 9. 私有方法
    private void PrivateMethod() { }

    // 10. 重写方法 (ToString, GetHashCode等)
    public override string ToString() => nameof(MyClass);
}
```

## 二、性能优化最佳实践

### 2.1 缓存策略

**最佳实践：** 预计算昂贵操作，避免每帧重复计算

**缓存原则：**

```
┌──────────────────────────────────┐
│ 缓存决策流程                       │
├──────────────────────────────────┤
│ 1. 这个操作多久执行一次？
│    ├─ 每帧 → 必须缓存
│    ├─ 每秒 → 考虑缓存
│    └─ 每分钟+ → 无需缓存
│
│ 2. 这个操作的计算成本？
│    ├─ 高 (遍历100+项) → 必须缓存
│    ├─ 中 (遍历10-100项) → 考虑缓存
│    └─ 低 (简单计算) → 无需缓存
│
│ 3. 缓存数据会不会改变？
│    ├─ 很少改变 → 易于缓存
│    ├─ 经常改变 → 需要失效策略
│    └─ 总是改变 → 不适合缓存
└──────────────────────────────────┘
```

**缓存实现示例：**

```csharp
// 类型1：静态初始化缓存（最优）
[StaticConstructorOnStartup]
public static class WF_Utility
{
    private static Dictionary<ThingDef, List<ConfigDef>> cache;

    static WF_Utility()
    {
        RebuildCache();
    }

    static void RebuildCache()
    {
        cache = new Dictionary<ThingDef, List<ConfigDef>>();
        foreach (var config in DefDatabase<ConfigDef>.AllDefs)
        {
            foreach (var thing in config.thingDefs)
            {
                if (!cache.ContainsKey(thing))
                    cache[thing] = new List<ConfigDef>();
                cache[thing].Add(config);
            }
        }
    }

    // O(1) 查询
    public static List<ConfigDef> GetConfigsFor(ThingDef thing)
        => cache.TryGetValue(thing, out var result) ? result : new List<ConfigDef>();
}

// 类型2：延迟初始化缓存
public class MapComponent_Framework : MapComponent
{
    private Dictionary<IntVec3, CellData> cache;

    public CellData GetCellData(IntVec3 pos)
    {
        if (cache == null)
            RebuildCache();

        return cache.TryGetValue(pos, out var data) ? data : null;
    }

    void RebuildCache()
    {
        cache = new Dictionary<IntVec3, CellData>();
        for (int i = 0; i < map.cellIndices.NumGridCells; i++)
        {
            var cell = map.cellIndices.IndexToCell(i);
            cache[cell] = new CellData(cell);
        }
    }
}

// 类型3：脏标志缓存（检测变化）
public class FrameworkComponent
{
    private Dictionary<ThingDef, int> cache;
    private bool dirty = true;

    public void InvalidateCache() => dirty = true;

    public int GetValue(ThingDef def)
    {
        if (dirty)
            RebuildCache();

        return cache.TryGetValue(def, out var value) ? value : 0;
    }

    void RebuildCache()
    {
        cache = new Dictionary<ThingDef, int>();
        // ... 构建缓存 ...
        dirty = false;
    }
}
```

**缓存性能对比：**

```
操作：在100个Def中查询1个特定Def的配置

不缓存版本（每次）:
  foreach (def in AllDefs) { if (def == target) return config; }
  → O(n) 最坏100次比较
  → 时间: 0.5ms

缓存版本（后续访问）:
  cache[target]
  → O(1) 1次字典查询
  → 时间: 0.01ms

性能提升: 50x
```

### 2.2 避免GC分配

**最佳实践：** 减少每帧的内存分配

**优化技巧：**

```csharp
// ❌ 不好：每帧分配新数组
public void Tick()
{
    for (int i = 0; i < 100; i++)
    {
        var result = new List<Thing>();  // 每帧分配!
        DoSomething(result);
    }
}

// ✓ 好：复用对象
private List<Thing> cachedList = new List<Thing>();

public void Tick()
{
    for (int i = 0; i < 100; i++)
    {
        cachedList.Clear();  // 复用，无分配
        DoSomething(cachedList);
    }
}

// ✓ 更好：参数传递
public void Tick()
{
    for (int i = 0; i < 100; i++)
    {
        DoSomethingWithoutAllocation(i);
    }
}

private void DoSomethingWithoutAllocation(int index)
{
    // 使用值类型或引用传递，无新分配
}
```

**常见GC分配陷阱：**

```
❌ 避免：
- new List<T>() 在循环中
- new Dictionary<K,V>() 在循环中
- string.Format() 在循环中（如果频繁）
- LINQ 在性能关键路径中
- Coroutine 在频繁Tick中

✓ 使用：
- List<T>.Clear() 复用
- Dictionary<K,V> 初始化后复用
- StringBuilder 而非 string concat
- for循环 而非 LINQ
- 同步方法 而非 Coroutine
```

### 2.3 更新频率管理

**最佳实践：** 使用 IsHashIntervalTick 管理更新频率

```csharp
public class MapComponent_Framework : MapComponent
{
    private const int UPDATE_INTERVAL = 60;  // 每60帧更新一次

    public override void MapComponentTick()
    {
        // ✗ 不好：每帧都更新
        // UpdateCache();

        // ✓ 好：定期更新
        if (map.IsHashIntervalTick(UPDATE_INTERVAL))
        {
            UpdateCache();
        }
    }
}

// 性能对比：
// 每帧更新：100% CPU占用
// IsHashIntervalTick(60)：~1.7% CPU占用 (60倍优化)
```

### 2.4 集合选择

**最佳实践：** 根据使用模式选择合适的集合类型

```csharp
// ✓ List<T> - 有序、随机访问
var list = new List<Thing>();
list.Add(thing);
var item = list[0];

// ✓ Dictionary<K,V> - 快速查询
var dict = new Dictionary<ThingDef, int>();
dict[thingDef] = 5;

// ✓ HashSet<T> - 快速成员检查
var set = new HashSet<IntVec3>();
if (set.Contains(cell)) { }

// ✗ 避免 - LinkedList (除非需要频繁中间插入)
var linked = new LinkedList<Thing>();  // 不推荐

// ✓ 特殊优化 - FishSet<T> (AncotLibrary)
// 高性能集合，推荐用于性能关键代码
```

## 三、版本管理与兼容性

### 3.1 语义化版本

**最佳实践：** 遵循语义化版本，清晰沟通变化

```
版本号: MAJOR.MINOR.PATCH
  ├─ MAJOR: 不兼容的更改
  │  例：0.2.0 → 1.0.0 (重构框架)
  │
  ├─ MINOR: 向后兼容的新功能
  │  例：1.0.0 → 1.1.0 (新增功能)
  │
  └─ PATCH: bug修复
     例：1.0.0 → 1.0.1 (修复bug)

更新策略：
✓ PATCH版本：直接更新（向后兼容）
~ MINOR版本：建议更新（向后兼容，可选功能）
✗ MAJOR版本：谨慎更新（不兼容，需要迁移）
```

### 3.2 向后兼容设计

**最佳实践：** 设计前向和后向兼容

```csharp
// ✓ 前向兼容：新代码能用旧配置
public class ConfigDef : Def
{
    public int newField = 10;  // 新字段有默认值
}

// ✓ 后向兼容：旧代码能用新配置
public static void Apply(Thing thing)
{
    if (thing.HasModExtension<ConfigDef>())
    {
        var config = thing.GetModExtension<ConfigDef>();
        // 只用通用字段，不用新字段
    }
}

// ✗ 不兼容
public class ConfigDef : Def
{
    public int requiredField;  // 必需字段，无默认值
}
```

### 3.3 API弃用策略

**最佳实践：** 逐步弃用过时API，给用户迁移时间

```csharp
// 步骤1：标记为过时
[Obsolete("使用 NewMethod() 替代", false)]
public static void OldMethod()
{
    // 保留实现，重定向到新方法
    NewMethod();
}

// 步骤2：文档说明
/// <summary>
/// [已弃用] 使用 NewMethod() 替代
/// 此方法将在 v2.0 中删除
/// </summary>
[Obsolete("使用 NewMethod() 替代", false)]
public static void OldMethod() { }

// 步骤3：最终删除
// 在主版本更新时删除标记为[Obsolete]的方法
```

```csharp
    }

    private void MigrateFromVersion1()
    {
        Log.Message("框架数据从v1迁移到v2...");

    }
```

### 4.2 依赖版本检查

**最佳实践：** 在初始化时检查依赖

```csharp
[StaticConstructorOnStartup]
public static class FrameworkInit
{
    static FrameworkInit()
    {
        CheckDependencies();
    }

    static void CheckDependencies()
    {
        // 检查所需的MOD是否安装
        var requiredMods = new[] { "ludeon.rimworld", "aaa.corlib" };
        var loadedMods = ModsConfig.ActiveModsInLoadOrder.Select(m => m.PackageId).ToList();

        foreach (var mod in requiredMods)
        {
            if (!loadedMods.Contains(mod))
            {
                Log.Error($"框架需要{mod}，但未找到");
                return;
            }
        }

        Log.Message("所有依赖已满足，框架初始化成功");
    }
}
```

## 五、框架实现中的常见陷阱

### 5.1 实现阶段陷阱

**陷阱1：频繁反射**

```
❌ 不好
public void Tick()
{
    foreach (var comp in components)
    {
        var method = comp.GetType().GetMethod("Tick");  // 每帧反射!
        method?.Invoke(comp, null);
    }
}

✓ 好
foreach (var comp in components)
{
    comp.Tick();  // 直接调用
}
```

**陷阱2：GC压力过大**

```
❌ 不好
public void Tick()
{
    for (int i = 0; i < 100; i++)
    {
        var list = new List<Thing>();  // 每帧分配!
    }
}

✓ 好
private List<Thing> cachedList = new List<Thing>();

public void Tick()
{
    for (int i = 0; i < 100; i++)
    {
        cachedList.Clear();  // 复用，无分配
    }
}
```

### 5.2 维护阶段陷阱

**陷阱3：破坏向后兼容**

```
❌ 不好 - 直接删除方法
public class v2
{
    // OldMethod 消失了 - 使用旧版本的MOD崩溃!
}

✓ 好 - 逐步弃用
public class v1
{
    [Obsolete("使用 NewMethod()", false)]
    public void OldMethod() => NewMethod();
}

public class v2
{
    // 最新版本才删除
}
```

**陷阱4：版本冲突**

```
❌ 不好 - 版本限制太严格
[Requirement("1.0.0 exactly", true)]  // 只支持1.0.0

✓ 好 - 版本范围合理
[Requirement(">=1.0", true)]  // 1.0及更新版本
```

**陷阱5：文档缺失**

```
❌ 不好
public static void DoSomething() { }  // 没有文档，不知道干嘛的

✓ 好
/// <summary>对Thing应用框架效果</summary>
/// <param name="thing">目标Thing</param>
public static void DoSomething(Thing thing) { }
```

## 核心要点速记

### 实现阶段
1. ✓ 清晰的文件组织
