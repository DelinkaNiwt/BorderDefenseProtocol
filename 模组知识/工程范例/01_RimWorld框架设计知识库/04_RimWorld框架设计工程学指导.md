## 第一部分：框架设计的第一性原理

### 1.1 什么是"好的框架"

一个好的RimWorld框架应该满足：

| 维度 | 指标 | 为什么重要 |
|------|------|----------|
| **可复用性** | 被多个mod使用 | 单次开发，多次使用 |
| **学习曲线** | 新手能快速上手 | 降低使用门槛 |
| **扩展友好** | 用户容易定制 | 避免fork和修改 |
| **性能稳定** | 不引入帧数问题 | RimWorld是实时游戏 |
| **兼容性** | 与其他mod并存 | 社区mod数量众多 |
| **可维护性** | 代码清晰易理解 | 版本更新时好维护 |

### 1.2 框架 vs 功能模组

```
框架                           功能模组
────────────────────────────────────────────
可被多个mod使用              单一mod完整功能
定义接口和扩展点             自给自足的实现
希望有衍生品                 直接面向玩家
重视可配置性和灵活性         重视游戏平衡和体验
抽象度高                     具体度高
维护成本随使用量增加         维护成本相对固定
```

### 1.3 框架设计的三个层次

```
第1层：问题域定义
├─ 这个框架要解决什么问题？
├─ 覆盖范围是什么？
├─ 边界在哪里？
└─ 不做什么？

第2层：架构选择
├─ 采用什么设计哲学？（XML优先 vs 灵活性至上 vs 深度优化）
├─ 选择什么架构模式？（单层 vs 多层 vs 多框架）
├─ 如何管理复杂度？
└─ 性能约束是什么？

第3层：具体设计
├─ API设计
├─ 配置系统
├─ 扩展点设计
└─ 实现细节
```

## 第二部分：设计哲学的选择

### 2.1 三种主流设计哲学

#### 哲学1：XML优先（HAR）
**适用场景**：
- 美术/策划可参与配置
- 无编码复杂度要求
- 高度个性化配置

**优势**：
- ✓ 学习曲线平缓
- ✓ 易于快速迭代
- ✓ 支持零编码扩展

**劣势**：
- ✗ 一些复杂逻辑难以表达
- ✗ 需要大量C#来处理XML

**适用框架规模**：中等（100-200个C#文件）

**标志性框架**：
```
HumanoidAlienRaces (113个C# + 大量XML)
特点：26个Condition类来支持XML表达复杂条件
```

#### 哲学2：灵活性至上（VEF）
**适用场景**：
- 开发者用户
- 需要高度定制
- 功能多样性

**优势**：
- ✓ 最大化灵活性
- ✓ 支持各种扩展
- ✓ 特性开关系统

**劣势**：
- ✗ 学习曲线陡峭
- ✗ 复杂度难以管理
- ✗ 维护成本高

**适用框架规模**：超大型（1000+个C#文件）

**标志性框架**：
```
VanillaExpandedFramework (1251个C# + 24个子系统)
特点：5个独立DLL、特性开关、模块选择
```

#### 哲学3：深度优化（ASF）
**适用场景**：
- 单一问题域
- 性能敏感
- 需要极限优化

**优势**：
- ✓ 性能极限
- ✓ 深入理解域问题
- ✓ 可复用优化库

**劣势**：
- ✗ 覆盖范围窄
- ✗ 难以跨域应用
- ✗ 需要高深的优化知识

**适用框架规模**：中等（150-200个C#文件）

**标志性框架**：
```
AdaptiveStorageFramework (168个C# + 100+Fishery库)
特点：Fishery库性能优化、4层策略模式、脏标志缓存
```

### 2.2 如何选择设计哲学

```
你的框架要做什么？
│
├─ 支持用户纯配置（无编码）
│  └─→ XML优先（HAR模式）
│
├─ 支持开发者高度定制（编码扩展）
│  └─→ 灵活性至上（VEF模式）
│
├─ 优化单一问题的极限性能
│  └─→ 深度优化（ASF模式）
│
└─ 介于两者之间
   └─→ 混合哲学（选择主导 + 辅助）
```

### 2.3 混合哲学示例

```
主要：XML优先 + 辅助：灵活性
├─ 大部分配置用XML
├─ 复杂逻辑用C#继承
└─ 用户可选择深度或广度

主要：灵活性 + 辅助：性能优化
├─ 提供高度灵活的扩展点
├─ 为关键路径做缓存优化
└─ 用脏标志避免重复计算
```

## 第三部分：架构模式选择

### 3.1 五种架构模式

#### 模式1：单层架构
```
Def配置
    ↓
Worker处理
    ↓
结果
```

**适用**：
- 简单框架（<50个C#文件）
- 逻辑直线性
- 无复杂协调

**例子**：AutoBlink（10个C#文件，纯XML+最少代码）

**优势**：简单清晰 | **劣势**：难以扩展

#### 模式2：分层架构
```
UI层
↓
业务逻辑层
↓
配置层
↓
API层
```

**适用**：
- 中等框架（100-300个C#文件）
- 多个关注点
- 需要清晰分离

**例子**：HAR（113个C#文件，清晰的分层）

**优势**：易维护，职责清晰 | **劣势**：层间通信复杂

#### 模式3：模块化架构
```
Module A    Module B    Module C
├─ Defs     ├─ Defs     ├─ Defs
├─ Code     ├─ Code     ├─ Code
└─ UI       └─ UI       └─ UI

统一协调层
```

**适用**：
- 大型框架（300-600个C#文件）
- 多个独立功能
- 需要部分使用

**例子**：AdaptiveStorage的多个模块（存储、颜色、标签）

**优势**：高内聚，低耦合 | **劣势**：模块间协调复杂

#### 模式4：多框架架构
```
Framework1 (DLL1)
├─ 独立API
└─ 可单独使用

Framework2 (DLL2)
├─ 独立API
└─ 可单独使用

Framework3 (DLL3)

统一整合框架 (DLL4)
└─ 组合上述框架
```

**适用**：
- 超大型框架（1000+个C#文件）
- 多个不同问题域
- 需要灵活组合

**例子**：VEF（MVCF、KCSG、PipeSystem四大框架）

**优势**：最大灵活性，可独立使用 | **劣势**：极高复杂度

#### 模式5：插件架构
```
Core API (定义接口)
    ↓
Plugin System (加载机制)
    ↓
Plugin1 Plugin2 Plugin3
```

**适用**：
- 需要用户添加自定义插件
- 动态加载/卸载
- 版本兼容性要求高

**例子**：不常见，RimWorld mod生态主要用前4种

**优势**：高度可扩展 | **劣势**：实现复杂度高

### 3.2 架构选择决策树

```
框架规模？
│
├─ <50个C#
│  └─ 单层架构
│
├─ 50-150个C#
│  ├─ 逻辑简单？ → 单层或分层
│  └─ 逻辑复杂？ → 分层
│
├─ 150-500个C#
│  ├─ 单一问题？ → 分层（+性能优化）
│  └─ 多个问题？ → 模块化
│
├─ 500-1000个C#
│  └─ 模块化 或 多框架的准备
│
└─ 1000+个C#
   └─ 多框架架构 （必须）
```

## 第四部分：复杂度管理策略

### 4.1 复杂度的来源

```
代码复杂度      功能复杂度    维护复杂度
├─ 行数          ├─ 问题域大   ├─ 文档量
├─ 文件数        ├─ 交互点多   ├─ API稳定性
├─ 依赖关系      ├─ 配置选项   ├─ 版本适配
└─ 循环复杂度    └─ 扩展点多   └─ 兼容性
```

### 4.2 复杂度管理的五个原则

#### 原则1：单一职责（SRP）
```
错误：一个类做所有事
class StorageSystem {
    void GenerateGraphics() { }
    void UpdateColors() { }
    void HandleUI() { }
    void OptimizePerformance() { }
    void ManageCache() { }
}

正确：职责分离
class StorageRenderer { void GenerateGraphics() {} }
class ColorManager { void UpdateColors() {} }
class StorageUI { void HandleUI() {} }
class CacheSystem { void ManageCache() {} }
```

#### 原则2：特性开关系统
```csharp
// VEF的特性开关
Feature_MultiVerb
Feature_ExtraEquipment
Feature_ApparelVerbs
// ...

// 好处：
// ✓ 只在需要时启用补丁
// ✓ 运行时可配置
// ✓ 减少默认开销
```

#### 原则3：模块隔离
```
不要：模块间循环依赖
A → B → C → A  ✗ 难以理解

要：单向依赖或独立
A → B
B → C
A 和 C 可独立    ✓ 清晰
```

#### 原则4：文件夹分类
```
明确的分类，易于导航
│
├─ Core/             核心系统
├─ Rendering/        渲染系统
├─ UI/               用户界面
├─ Harmony/          补丁系统
├─ Utils/            工具类
└─ ...
```

#### 原则5：文档和注释
```
关键点必注释：
// 为什么选这个设计
// 这里有什么陷阱
// 与其他系统的关系
```

### 4.3 复杂度的可视化度量

```
框架复杂度评分 (1-10)

代码复杂度  3个C#文件 → 10个 → 100个 → 500个 → 1000+
分数        1         2      4      7      10

功能复杂度  单一     混合    多层   跨域   极限
分数        1        3      5      7      10

维护复杂度  无依赖   低耦   中耦   高耦   极高耦
分数        1        2      4      7      10

总复杂度 = (代码 + 功能 + 维护) / 3
```

## 第五部分：性能优化策略

### 5.1 性能问题的三个层次

#### 层次1：架构级（最重要）
```
✓ 正确：架构设计避免性能问题
  ├─ 缓存热数据
  ├─ 延迟计算冷数据
  └─ 避免频繁对象创建

✗ 错误：热循环中复杂计算
  ├─ 每帧反射
  ├─ 每帧遍历大列表
  └─ 每帧创建新对象
```

#### 层次2：算法级（其次）
```
✓ 原生数据结构：O(1) 查询
  └─ 数组直接索引

✗ 次优算法：O(n) 或 O(n²)
  └─ 频繁遍历大列表
```

#### 层次3：微观优化（最后）
```
✓ 最后才考虑
  ├─ 局部变量快于字段访问
  ├─ 对象池避免GC
  └─ 委托缓存避免反射

✗ 不值得
  ├─ 循环计数from 0 vs 1
  ├─ i++ vs ++i
  └─ 其他编译器自动优化的事
```

### 5.2 缓存与脏标志模式

```csharp
// 模式
class CachedSystem {
    private CacheData? _cache = null;
    private bool _isDirty = true;

    public void SetDirty() => _isDirty = true;

    public CacheData GetData() {
        if (_isDirty) {
            _cache = ComputeExpensiveData();
            _isDirty = false;
        }
        return _cache!;
    }
}
```

**好处**：
- ✓ 避免重复计算
- ✓ 智能更新
- ✓ 性能可预测

### 5.3 对象池模式

```csharp
// 问题
List<T> GetItems() {
    return new List<T>();  // 每次创建GC压力
}

// 解决
private static PooledList<T> s_pool = new();

PooledList<T> GetItems() {
    var list = s_pool.Get();
    PopulateList(list);
    return list;
}
```

## 第六部分：扩展点设计

### 6.1 四种扩展点

#### 扩展点1：继承（C#）
```csharp
public abstract class MyBase {
    public abstract void DoSomething();
}

// 用户可继承
public class UserImpl : MyBase {
    public override void DoSomething() { }
}
```

**优势**：类型安全 | **劣势**：需要编码

#### 扩展点2：接口（C#）
```csharp
public interface IProcessor {
    void Process();
}

public class MyClass {
    public IProcessor processor;
    // 用户实现IProcessor即可
}
```

**优势**：灵活 | **劣势**：需要编码

#### 扩展点3：Def扩展（XML）
```xml
<modExtensions>
  <li Class="MyFramework.CustomExtension">
    <!-- 用户配置 -->
  </li>
</modExtensions>
```

**优势**：无编码 | **劣势**：仅能表达简单配置

#### 扩展点4：配置驱动（XML）
```xml
<MyDef>
  <workerClass>UserNamespace.UserWorker</workerClass>
  <!-- 用户在C#中实现UserWorker -->
</MyDef>
```

**优势**：灵活而清晰 | **劣势**：需要编码

### 6.2 扩展点的设计原则

```
原则1：最小权限
├─ 只暴露需要的接口
├─ 内部实现对用户隐藏
└─ 便于后期修改

原则2：清晰的契约
├─ 明确说明输入输出
├─ 明确说明调用时机
└─ 明确说明与其他系统的关系

原则3：版本兼容性
├─ 接口不随意改动
├─ 添加新方法用默认实现
└─ 标记过时API

原则4：文档完善
├─ 提供示例实现
├─ 列出常见陷阱
└─ 说明最佳实践
```

## 第七部分：兼容性与维护

### 7.1 版本兼容性

```csharp
// 1.4 和 1.5+ 代码差异
#if V1_4
    // 1.4专用代码
    var result = OldAPI();
#else
    // 1.5+代码
    var result = NewAPI();
#endif
```

**检查列表**：
- ✓ 标记版本变化的代码
- ✓ 提供条件编译
- ✓ 测试多个版本
- ✓ 文档说明最低版本要求

### 7.2 多mod兼容性

```csharp
// 检查mod是否存在
if (ModLister.HasActiveModWithName("modname")) {
    // 应用兼容性补丁
}

// 使用DefModExtension避免侵入式修改
var ext = def.GetModExtension<MyExtension>();
if (ext != null) {
    // 应用自定义逻辑
}
```

**检查列表**：
- ✓ 用Harmony而非直接修改
- ✓ 用DefModExtension扩展Def
- ✓ 避免单全局实例
- ✓ 补丁优先级合理

### 7.3 向后兼容性

```
✓ 能做的事
├─ 添加新接口方法（提供默认实现）
├─ 扩展已有类（用继承而非修改）
└─ 添加新配置选项

✗ 不能做的事
├─ 删除公开API
├─ 改变方法签名
├─ 改变Def序列化格式
└─ 改变配置格式
```

## 第八部分：风险规避

### 8.1 常见陷阱

#### 陷阱1：过度设计
```
❌ 做不到
为假设的需求设计复杂系统
成本：高复杂度、难维护、过度优化

✅ 正确
先实现核心功能，再根据需求扩展
```

#### 陷阱2：性能优化时机错误
```
❌ 做不到
在架构设计时就做微观优化

✅ 正确
1. 确保架构正确（不做无用功）
2. 测量识别瓶颈
3. 只优化热点
```

#### 陷阱3：忽视文档
```
❌ 做不到
认为代码可自解释

✅ 正确
├─ 关键决策写注释
├─ 扩展点提供示例
├─ API明确说明
└─ README提供快速入门
```

#### 陷阱4：Harmony补丁冲突
```
❌ 做不到
补丁无序应用，可能冲突

✅ 正确
├─ 补丁分组管理
├─ 明确优先级
├─ 测试多mod场景
└─ 提供补丁禁用选项
```

### 8.2 决策检查清单

在最终设计前，检查：

- [ ] 问题域是否清晰（我要解决什么？）
- [ ] 设计哲学是否明确（XML优先？灵活性？性能？）
- [ ] 架构模式是否合理（规模匹配？）
- [ ] 复杂度是否可控（有管理策略吗？）
- [ ] 性能瓶颈是否识别（哪些会热循环？）
- [ ] 扩展点是否充足（用户如何定制？）
- [ ] 兼容性是否考虑（多版本？多mod？）
- [ ] 文档是否完善（示例？API说明？）
- [ ] 是否有回退计划（出问题怎么办？）

```
第1步：快速浏览

   ├─ 主要特性？

   ├─ 如何分类的？

   ├─ 扩展点在哪？

第4步：配置示例
└─ 看使用示例
   ├─ 怎样配置？
   ├─ 怎样扩展？
   └─ 常见陷阱？

第5步：尝试扩展
└─ 自己写小扩展
   ├─ 按文档做扩展
   ├─ 遇到问题时深入
   └─ 记录经验
```

```

## 第十部分：总结与建议

### 10.1 框架设计的黄金法则

```
1. 先解决问题，再优化性能
   └─ 正确性 > 性能

2. 清晰的分离关注点
   └─ 一个类一个职责

3. 最小化公开API
   └─ 只暴露必要接口

4. 完善的文档和示例
   └─ 代码之外还要说人话

5. 向后兼容优先
   └─ 一旦发布就难以改动

6. 测试多个版本和mod组合
   └─ 集成问题很难发现

7. 倾听使用者的反馈
   └─ 框架最终服务社区
```

### 10.2 你的框架可能需要

```
如果你要设计新框架，可能需要：

  "参考哪个框架的设计哲学"

  "用户如何提出需求/问题"

### 10.3 成功框架的特征

基于分析的3个框架，成功框架往往：

- ✓ 解决了真实问题（不是过度工程化）
- ✓ 有清晰的设计哲学（XML优先 or 灵活性 or 性能）
- ✓ 管理了复杂度（特性开关、模块隔离等）
- ✓ 优化了关键路径（缓存、对象池等）
- ✓ 提供了丰富扩展点（继承、接口、配置）
- ✓ 有活跃的社区使用（衍生品多）
- ✓ 持续维护支持（版本更新跟进）

这份指导文档不是代码模板，而是框架设计的**工程学原理**。它基于成功框架（HAR、VEF、ASF）的经验提炼：

> 成功的框架不是最复杂的，而是在**问题规模、复杂度、性能、可维护性**之间找到最优平衡的框架。

当你设计自己的框架时，先问这些问题：
1. 我要解决什么问题？
2. 我的目标用户是谁？
3. 我采用什么设计哲学？
4. 我如何管理复杂度？
5. 我的性能约束是什么？

答好这些问题，你的框架就有了方向。
