### 1.2 框架分类维度

#### **按复杂度分类**
- **超大型** (1000+)：VEF
- **大型** (400-500)：HAR、Milira
- **中型** (150-200)：ASF、CeleTech
- **轻型** (50-100)：AncotLib、VoidUniverse
- **微型** (10-20)：AutoBlink

#### **按设计理念分类**
- **XML优先框架**：HAR（约定优于配置 + XML模板）
- **多框架组合型**：VEF（灵活性至上 + 模块选择）
- **单一职责框架**：ASF（深度优化 + 特定问题）
- **工具库型**：AncotLib（公用基础设施）

#### **按可复用性分类**
- **✓✓✓ 极高**：HAR、VEF核心、ASF（可独立使用）
- **✓✓ 高**：MVCF、KCSG、PipeSystem（VEF独立框架）
- **✓ 中**：WeaponFitting、Milira（依赖其他框架）
- **- 低**：AutoBlink（单纯功能模组）

## 二、3大核心框架详解

### 2.1 HumanoidAlienRaces（HAR）- XML模板框架

#### 核心特征
```
设计哲学：约定优于配置 + XML驱动 + 零编码扩展
框架目标：让美术和策划纯XML定义复杂种族，无需C#编程
复杂度：极高（113个C#文件）但对用户极友好
```

#### 关键创新点
1. **条件图形系统**（26个Condition类）
   - 基于性别、年龄、装备、伤害等动态显示附件
   - 完全由XML配置驱动

2. **策略化颜色生成**（IAlienChannelColorGenerator）
   - 12个独立颜色通道
   - 支持性别/Pawn特定的颜色生成

3. **无缝Harmony集成**（130+补丁）
   - 修补渲染、生成、思想等系统
   - 对原生API影响最小化

#### 使用场景
- ✓ 创建新种族（米莉拉、Kiiro等都基于HAR）
- ✓ 定制种族外观（身体部件、颜色、尺寸）
- ✓ 种族特定的思想/关系系统
- ✗ 修改核心游戏机制

#### 学习价值
- 如何用XML替代C#配置
- 如何设计灵活的条件系统
- 如何通过Harmony最小化修改
- 种族系统的完整参考实现

---

### 2.2 VanillaExpandedFramework（VEF）- 多框架组合

#### 核心特征
```
设计哲学：灵活性至上 + 模块选择 + 开发者友好
框架目标：提供多个独立的可复用系统供开发者选择
复杂度：最高（1251个C#文件 + 5个DLL）但模块独立
```

#### 4个独立编译单元
| 单元 | 文件数 | 用途 | 独立性 |
|------|--------|------|-------|
| **MVCF** | 92 | 多武器动词系统 | ✓ 完全独立 |
| **KCSG** | 66 | 参数化基地生成 | ✓ 完全独立 |
| **PipeSystem** | 87 | 流体资源网络 | ✓ 完全独立 |
| **Outposts** | 19 | 前哨管理系统 | ✓ 完全独立 |
| **VEF Core** | 968 | 内容扩展集合 | ✗ 依赖其他 |

#### 关键创新点
1. **MVCF（多动词框架）**
   - 单个角色多个武器
   - 武器可来自装备、服装、伤害、库存
   - VerbComp可链式扩展

2. **KCSG（基地生成）**
   - 参数化的符号替换系统
   - 18个SymbolResolver支持复杂布局
   - DefModExtension动态配置

3. **PipeSystem（管道系统）**
   - 自动化资源网络
   - 18个Comp系统
   - 动态流动计算与优先级分配

#### 使用场景
- ✓ 多武器装备系统
- ✓ 参数化地图生成（遗址、定居点）
- ✓ 流体自动化（水、油、魔法等）
- ✓ 复杂的游戏机制扩展
- ✗ 简单的单功能增强（过度设计）

#### 学习价值
- 如何组织超大型框架
- 特性开关模式降低耦合
- 多层架构的管理
- RimWorld 的各大核心系统（渲染、生成、补丁）

### 2.3 AdaptiveStorageFramework（ASF）- 单一职责框架

#### 核心特征
```
设计哲学：深度优化 + 单一问题域 + 极限性能
框架目标：完全自适应的、可着色的、可标签化的存储系统
复杂度：中等（168个C#文件 + 100+Fishery库）但极精巧
```

#### 关键创新点
1. **多层级级联配置**
   - Def → Graphic → GraphicData三层
   - 支持部分覆盖，无需全量重写
   - 复杂度与灵活性的完美平衡

2. **AllowedRequirement枚举**（11种）
   - Majority/Minority 百分比判定
   - 大幅简化配置复杂度
   - 涵盖大多数场景

3. **Fishery高效库**（100+文件）
   - FishSet、IntFishTable 自制集合
   - 相比.NET快10-100倍
   - 避免垃圾回收压力

4. **动态着色系统**（12通道）
   - 实时计算堆叠物品的主色调
   - 与建筑颜色结合
   - 视觉反馈即时更新

#### 使用场景
- ✓ 存储建筑美化（自适应图形）
- ✓ 内容可视化（动态着色）
- ✓ 性能敏感的系统（高频集合操作）
- ✗ 修改存储的核心逻辑（太特化）

#### 学习价值
- 如何深度优化性能（对象池、脏标志）
- 策略模式在实际中的应用
- 多层级配置的设计
- 自制集合库的必要性认识

## 四、设计模式与最佳实践

### 4.1 跨框架的通用模式

#### 模式1：策略模式
```
使用框架      关键类
─────────────────────────────
HAR       ChannelColorGenerator (颜色生成策略)
VEF       GenStep (地图生成策略)
          VerbComp (武器能力策略)
ASF       GraphicsDefSelector (图形选择策略)
          StorageGraphicWorker (渲染策略)
```

**应用指导**：
- 当有多种实现方式时，定义接口/基类
- 支持用户继承创建自定义策略
- 通过配置指定具体策略类名

#### 模式2：组件模式
```
RimWorld原生基类：
├─ ThingComp          (物品组件)
├─ HediffComp         (伤害组件)
├─ PawnRenderNode     (渲染节点)
└─ GenStep            (生成步骤)

框架扩展：
├─ HAR: CompAlienRace, AlienPartGenerator
├─ VEF: CompAbilityEffect*, CompResource*
└─ ASF: ContentsITab, StorageRenderer
```

**应用指导**：
- 优先使用RimWorld内置的Comp系统
- 组件间用事件/信号通信而非直接耦合
- 支持多组件堆叠（链式调用）

#### 模式3：工厂模式
```
创建方式              使用框架
─────────────────────────────
Def.workerClass   HAR、VEF、ASF (都用)
MakeThing()       VEF-KCSG
DefDatabase<>     VEF (全局获取)
```

**应用指导**：
- 复杂对象用工厂而非直接构造
- 支持通过Def指定具体工厂类
- 工厂方法要记录创建参数便于缓存

#### 模式4：对象池 & 脏标志优化
```
框架           应用位置
─────────────────────────────
ASF   Fishery库 (100+文件)
      PooledList, SimplePool
VEF   PipeSystem (缓存系统)
      CachedXXX 类
HAR   CachedData (反射缓存)
      委托缓存，避免反射开销
```

**应用指导**：
- 热循环代码必须缓存
- 脏标志标记：何时需要重新计算
- 定期检查性能瓶颈（使用Analysis工具）

### 4.2 性能最佳实践

#### 缓存策略（来自VEF、ASF、HAR）
```csharp
// ✓ 正确：缓存高频访问
private Dictionary<Key, Value> _cache;
public void SetCacheDirty() => _cacheDirty = true;
public Value GetValue(Key k) {
    if (_cacheDirty) RefreshCache();
    return _cache[k];
}

// ✗ 错误：每次都重新计算
public Value GetValue(Key k) {
    return ExpensiveComputation(k);  // 热循环中很慢
}
```

#### 集合选择（来自ASF-Fishery库）
```
使用场景              选择
────────────────────────────────
频繁添加/删除    FishSet<T> (ASF)
                IntFishTable<K,V> (ASF)
单次使用         List<T> (标准)
需要对象池      PooledList<T> (ASF)
需要拍扁查询    BoolGrid (RimWorld内置)
```

#### Harmony补丁优化
```
框架中的最佳实践：
✓ Prefix返回false阻止原方法 (比Postfix快)
✓ 补丁分组，只在需要时应用
✓ 缓存反射结果（使用AccessTools.Field缓存）
✓ 避免在补丁中创建List/Dictionary
✗ 在Harmony补丁中写热循环
```

### 4.3 兼容性最佳实践

#### 多mod协存
```
框架做法         好处
─────────────────────────────────────
使用DefModExt  (VEF推荐) 无侵入式扩展
使用Harmony    (通用)     支持多mod修改
避免全局单例   (都遵守)   防止状态污染
版本检查       (通用)     识别版本差异
```

#### 向后兼容性
```
HAR:    支持0.19-1.6 (超7年)
        ├─ 版本特定的Assemblies目录
        ├─ LoadFolders.xml条件加载
        └─ 代码中的#if V1_4条件编译

VEF:    支持1.0-1.6
        └─ 模块化设计易于版本适配

ASF:    支持1.4-1.6

```
