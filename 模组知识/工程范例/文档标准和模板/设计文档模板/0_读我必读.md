## 与"代码工程师文档"的区别

### 这套文档（架构师）vs 那套文档（代码工程师）

| 维度 | 代码工程师文档 | 架构师文档 |
|------|----------------|-----------|
| **用途** | 教代码工程师"怎么做" | 告诉代码工程师"做什么" |
| **内容** | 通用规范、规范、教程 | **本项目**的具体设计 |
| **编写者** | 技术管理员（可复用） | **项目设计师**（项目特定） |
| **变更频率** | 低（规范稳定） | 高（设计常变更） |
| **优先级** | 参考资料 | **核心交付物** |

### 两套文档的关系

```
【架构师文档】(本目录)
  ↓ 明确定义"做什么"
代码工程师读此了解需求
  ↓
【代码工程师编码时】
  ├─ 参考【代码工程师文档】(另一目录)学习"怎么做"
  ├─ 使用API参考、编码规范等技术文档
  └─ 参考【架构师文档】做出具体的技术决策
  ↓
【完成编码】
```

## 关键概念

### "做什么" vs "怎么做"

**做什么（架构师负责）**：
```
- 功能是什么、怎么工作、有什么步骤
- 需要什么数据、什么关系、什么约束
- 参数怎么配、范围是什么、影响什么
```

**怎么做（代码工程师负责）**：
```
- 选择什么RimWorld API来实现
- 代码怎样组织、怎样避免性能问题
- 遵循什么编码规范、什么设计模式
```

架构师：**定义问题**
代码工程师：**解决问题**

## 如何使用这套文档

### 场景1：代码工程师刚接手项目

```
步骤1：读【核心概念设计】
  理解：这个模组是什么、目标是什么、创新点是什么

步骤2：读【功能详细设计】
  理解：需要实现哪些功能、每个功能怎么工作、有什么交互

步骤3：读【数据结构设计】
  理解：需要什么数据、怎么存储、什么关系

步骤4：开始编码
  在编码时需要时，参考其他3个文档
```

### 场景2：代码工程师实现某个功能时卡住了

```
情况：不知道这个功能的数据怎么组织
  → 查看【功能详细设计】的具体功能说明

情况：不知道这个功能怎么与其他模块交互
  → 查看【系统交互流程设计】

情况：不知道某个参数应该怎么处理
  → 查看【配置参数定义】

情况：不确定设计决策的背景
  → 查看【技术方案选择与考量】
```

### 场景3：设计需要变更

```
设计师：想修改功能A的参数
  ↓
步骤1：更新【配置参数定义】
步骤2：通知代码工程师："参数X的范围改了"
步骤3：代码工程师修改验证规则

设计师：想修改功能A的工作流程
  ↓
步骤1：更新【功能详细设计】
步骤2：可能需要更新【数据结构设计】、【系统交互流程】
步骤3：通知代码工程师："功能A的步骤改了"
```

## 文档维护原则

这6份文档应该在以下情况更新：

| 触发条件 | 哪个文档更新 | 谁来更新 |
|---------|-------------|---------|
| 发现新的功能需求 | 核心概念、功能设计 | 设计师 |
| 需要调整参数 | 参数定义 | 设计师或平衡师 |
| 数据关系改变 | 数据结构、系统交互 | 设计师 |
| 发现技术风险 | 技术方案 | 架构师 |
| 发现设计漏洞 | 相关文档 | 设计师或工程师 |

**重要**：更新后应该**通知代码工程师**，让他们知道需要重新审视某些设计。

## 常见问题

### Q：文档怎样"足够详细"？

A：满足这些条件就足够了：
- [ ] 代码工程师能根据文档编码，无需频繁询问
- [ ] 文档明确说了"怎么验证功能是否正确"
- [ ] 参数的范围和含义都说清楚了
- [ ] 模块间的通信方式明确了

### Q：如果文档不够详细怎么办？

A：代码工程师应该**立即反馈**，而不是自己猜：
```
"【功能详细设计】中功能A的步骤2不清楚，
请补充说明：用户输入什么数据？系统怎么响应？"
```

设计师应该**立即补充或澄清**。

### Q：设计改了，是否需要更新代码工程师文档（那套）？

A：**一般不需要**。代码工程师的文档是通用的，只有以下情况需要更新：
- 编码规范改变了（很少）
- 某个API的用法改变了（很少）

大部分改变只需要更新架构师文档。

## 必须附带：RiMCP验证清单

### 什么是RiMCP验证清单

**设计文档必须附带一份RiMCP验证清单**，列出设计中使用的所有RimWorld API，并确认这些API在目标版本中确实存在且可用。

### 验证清单的格式

| 序号 | 功能点 | API引用 | 引用签名 | 所在文件路径 | 类名 | 方法名/常数 | 行号 | 验证结果 | 备注 |
|------|--------|--------|---------|------------|------|-----------|------|---------|------|
| 1 | [功能描述] | [如RimWorld.ThingComp] | [完整引用] | C:\RimWorld\Source\...cs | ClassName | MethodName | 123 | ✓通过 | [如有问题则说明] |
| 2 | [功能描述] | [如GenMath.LerpDoubleClamped] | [常数值] | C:\RimWorld\Source\Utilities\GenMath.cs | GenMath | EPSILON = 0.001f | 45 | ✓通过 | 浮点数比较精度 |
| 3 | [下一个API] | ... | ... | ... | ... | ... | ... | ✓通过 | ... |

### 验证清单必须包含

每一条API引用或常数的信息都要完整：
- **序号**：便于追踪
- **功能点**：说明这个API/常数在哪个功能中被使用
- **API引用**：如 `RimWorld.Pawn`、`Verse.Hediff` 或常数名称
- **引用签名**：
  - 对于方法：完整的方法签名，如 `void AddHediff(HediffDef def)`
  - 对于常数：常数的定义，如 `const float EPSILON = 0.001f`
- **所在文件路径**：RimWorld源代码中的绝对路径
- **类名和方法名/常数**：
  - 对于方法：具体的类和方法名
  - 对于常数：所在的类和常数名称
- **行号**：源代码中的具体行号
- **验证结果**：✓通过 / ❌不通过 / ⚠️需要注意
- **备注**：如果有版本限制或特殊约束，在这里说明

### 为什么需要这个清单

1. **确保可行性**：确认设计中使用的API和常数真的存在
2. **避免风险**：提前发现API不兼容或常数值变更的问题
3. **加快编码**：代码工程师不需要自己查证API和常数真实性
4. **可追踪性**：清晰记录设计依赖的技术基础（包括动态值）
5. **保证设计稳定性**：游戏源码中的常数可能在更新中改变，提前记录可避免后期风险

### 验证工具

使用 **RiMCP**（RimWorld代码查询工具）进行验证：
- 验证API的**存在性**（确认API真的存在）
- 验证API的**基本签名**（参数类型、返回值）
- 不需要深入理解API的实现逻辑

## 何时"完成"一个功能

根据这套文档，功能完成的标准是：

- [ ] 功能详细设计中的"完成标准"都通过
- [ ] 数据都能正确保存/读取（根据数据结构设计）
- [ ] 与其他模块的交互正确（根据系统交互流程）
- [ ] 所有参数都在配置参数定义的范围内
- [ ] 没有控制台报错
