## 核心架构决策

### 决策1：组件模式 (Comp) 而非继承

| 方案 | 优点 | 缺点 | **选择** |
|------|------|------|---------|
| **Comp 模式** | 灵活组合、易扩展、无侵入 | 性能稍低 | ✓ 选择 |
| 直接继承 | 性能好 | 难以组合、易产生冗余 | ✗ 不选 |

**决策理由**：
```
模组需要灵活地组合多个功能，Comp模式最合适。
性能开销可以通过缓存优化，不是问题。
```

**代码工程师需要**：
```
- 使用ThingComp作为基类
- 在XML中配置组合
- 不要尝试继承Thing或Building
```

---

### 决策2：数据持久化方案

| 方案 | 优点 | 缺点 | **选择** |
|------|------|------|---------|
| **Scribe 序列化** | RimWorld标准、支持版本迁移 | 需要手动实现 | ✓ 选择 |
| JSON序列化 | 易实现、易调试 | 不符合RimWorld规范 | ✗ 不选 |
| 数据库 | 性能好 | 过度设计、太复杂 | ✗ 不选 |

**决策理由**：
```
RimWorld官方都用Scribe，我们也应该遵循。
确保兼容性和未来维护的便利性。
```

**代码工程师需要**：
```
- 实现ExposeData()
- 使用Scribe_Values, Scribe_References, Scribe_Collections
- 注意版本迁移问题
```

---

### 决策3：UI 交互方式

| 方案 | 优点 | 缺点 | **选择** |
|------|------|------|---------|
| **Gizmo 按钮** | 原生、易理解 | 空间有限 | ✓ 选择 |
| 菜单系统 | 功能多 | 操作复杂 | ◐ 备选 |
| 窗口系统 | 直观 | 容易遮挡视图 | ✗ 不选 |

**决策理由**：
```
游戏内原生UI，玩家最熟悉。
功能简单，不需要复杂的菜单。
```

**代码工程师需要**：
```
- 实现CompGetGizmosExtra()返回Gizmo列表
- 使用Command_Action或Command_Toggle
- 保持UI简洁，不超过3个按钮
```

## 性能设计

### 性能指标

| 指标 | 目标值 | 为什么重要 |
|------|--------|-----------|
| **Tick消耗** | <1ms/Tick | 保持 60FPS |
| **内存占用** | <50MB | 不占用太多内存 |
| **加载时间** | <2秒 | 游戏体验 |

### 性能优化方案

#### 问题1：Tick操作太频繁

**现象**：如果在CompTick()中做重操作，会卡顿

**解决方案**：
```csharp
public override void CompTick()
{
    // 错误：每Tick都计算
    // var result = ExpensiveCalculation();

    // 正确：按间隔计算
    if (Find.TickManager.TicksGame % 60 == 0)  // 每秒一次
    {
        ExpensiveCalculation();
    }
}
```

**代码工程师需要**：
```
- 重操作使用tickInterval延迟
- 缓存频繁访问的数据
- 避免在Tick中创建新对象
```

#### 问题2：数据查询太慢

**现象**：每次都从大集合中查询，很慢

**解决方案**：
```csharp
// 缓存查询结果，而不是每次都查
private Dictionary<Thing, Value> _cache = new();

public Value GetValue(Thing thing)
{
    if (!_cache.ContainsKey(thing)) {
        _cache[thing] = Calculate(thing);
    }
    return _cache[thing];
}
```

**代码工程师需要**：
```
- 使用缓存，减少查询
- 及时清理缓存，避免内存泄漏
- 考虑缓存失效策略
```

## 扩展性设计

### 未来可能的扩展方向

如果要在这个模组基础上扩展，怎么设计才能容易扩展？

#### 扩展1：添加新的子功能

**设计考虑**：
```
当前设计支持：
  功能A + 功能B + 功能C

未来可能：
  功能A + 功能B + 功能D（替换C）

设计原则：
  - 功能应该是可插拔的
  - 用配置选择启用哪个功能，而不是代码
```

**代码工程师需要**：
```
- 功能之间不要硬编码耦合
- 使用策略模式或工厂模式
- 在XML中配置选择哪个功能
```

#### 扩展2：支持不同的对象类型

**设计考虑**：
```
当前：只支持Building
未来：可能支持Pawn、Item等

设计方案：
  - 使用通用的Comp接口
  - 避免Thing特定的代码
```

## 版本迁移策略

模组版本更新时，旧存档怎么处理？

### 迁移规则

```
版本1.0 → 2.0 时的迁移：

if (!HasLoadedVersion2) {
    // 从v1.0迁移数据到v2.0
    MigrateFromV1ToV2();
    HasLoadedVersion2 = true;
}
```

**代码工程师需要**：
```
- 在ExposeData()中检查版本
- 如果旧版本，执行迁移逻辑
- 迁移后设置标志，下次不再迁移
```

## 安全性考虑

### 用户输入验证

玩家输入的参数应该验证：

```csharp
public void SetParam(int value)
{
    // 验证范围
    if (value < MIN_VALUE || value > MAX_VALUE) {
        Log.Error($"参数超出范围：{value}");
        return;
    }

    // 验证一致性
    if (value < 0 && SomeOtherParam > 0) {
        Log.Warning("参数不一致");
    }

    _param = value;
}
```

**代码工程师需要**：
```
- 验证所有外部输入
- 不要信任玩家输入的参数
- 提供有意义的错误提示
```

## 文档和注释

### 代码注释原则

```csharp
// 复杂算法需要注释
public int CalculateOptimal(int a, int b)
{
    // 使用贪心算法，时间复杂度O(n)
    // 注意：假设a > 0
    ...
}

// 业务逻辑需要注释解释"为什么"
public void OnTick()
{
    // 每60Tick更新一次，而不是每Tick
    // 原因：频繁更新会卡顿，60Tick间隔平衡性能和响应性
    if (Find.TickManager.TicksGame % 60 == 0)
    {
        Update();
    }
}

// 显而易见的代码不需要注释
public int GetValue()
{
    return _value;  // 不需要注释"返回value"
}
```
