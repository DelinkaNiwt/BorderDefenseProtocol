## 1. 设计目标与约束

### 1.1 设计目标

| 目标 | 含义 |
|------|------|
| **独立开发与测试** | 各模块可独立开发、独立测试，不必等其他模块完成 |
| **可扩展性** | 新增子系统（新派系、新触发器类型）无需修改核心代码 |
| **可维护性** | 清晰的代码组织，新人或AI能快速理解各部分职责 |
| **实用主义(YAGNI)** | 架构复杂度与实际需求匹配，不为假想的未来需求增加复杂度 |

### 1.2 约束条件

- RimWorld模组通常编译为**单个DLL**，"模块"是逻辑划分（命名空间+文件夹），不是独立程序集
- 必须遵循RimWorld的Def/Comp/Worker/Tracker四大设计模式
- Trion能量系统的技术选型已确定：CompTrion(ThingComp)为核心的三层架构

### 1.3 核心架构思想

**微内核模式**：将最基础、最通用的能力放在内核中，其他功能作为可插拔的模块围绕内核组织。

- **内核**：Trion能量系统——所有模块共享的基础资源管理
- **模块**：触发器、角色、设施、战斗、世界——各自独立的功能域

## 4. 依赖关系矩阵

区分两种本质不同的耦合——C#代码依赖（紧耦合）和XML Def引用（松耦合）：

```
C#代码依赖：代码A直接import并调用代码B的类/方法
            → 改B可能导致A编译失败

XML Def引用：Def A通过defName字符串引用Def B
            → 仅改B的defName才影响A，改B的内部实现不影响
```

```
┌─────────────────────────────────────────────────────────────┐
│                    依赖关系矩阵                              │
│                                                             │
│  行 = 依赖方（谁依赖别人）                                    │
│  列 = 被依赖方（谁被别人依赖）                                │
│  C# = C#代码级依赖    XML = XML Def引用    — = 无依赖        │
├──────────┬──────────┬──────────┬──────────┬──────────┬──────┤
│ 依赖方→  │ 微内核   │ 触发器   │ 角色     │ 设施     │      │
│ ↓被依赖  │          │          │          │          │      │
├──────────┼──────────┼──────────┼──────────┼──────────┼──────┤
│ 触发器   │ C#       │    —     │    —     │    —     │ L1   │
├──────────┼──────────┼──────────┼──────────┼──────────┼──────┤
│ 角色     │ C#       │    —     │    —     │    —     │ L1   │
├──────────┼──────────┼──────────┼──────────┼──────────┼──────┤
│ 设施     │ C#       │    —     │    —     │    —     │ L1   │
├──────────┼──────────┼──────────┼──────────┼──────────┼──────┤
│ 战斗     │ C#       │ C# ★    │    —     │    —     │ L2   │
├──────────┼──────────┼──────────┼──────────┼──────────┼──────┤
│ 世界     │ C#       │    —     │ XML      │ XML      │ L2   │
└──────────┴──────────┴──────────┴──────────┴──────────┴──────┘

★ = 整个架构中唯一的跨模块C#硬依赖
    战斗体激活时需检查Pawn是否装备了触发体（CompTriggerBody）

L1 = 第一层模块    L2 = 第二层模块
```

**关键结论**：
- 第一层三个模块之间：零C#依赖，完全可并行开发
- 世界模块对角色/设施的依赖仅在XML层面，C#代码完全独立
- 战斗→触发器是唯一需要在C#层面import另一个模块的依赖

│  │         │  Gene_TrionGland                                   │
│  │ 产生源  │──statOffsets──→ StatDef:TrionCapacity               │
│  │         │                      │                             │
│  │ 触发角  │──statOffsets──→ StatDef:TrionOutputPower            │
│  └─────────┘                      │                             │
│                                   ↓                             │
│                    ┌──────────────────────────┐                  │
│                    │        微内核             │                  │
│                    │                          │                  │
│                    │  Stat聚合 → CompTrion.max │                  │
│                    │                          │                  │
│                    │  CompTrion               │                  │
│                    │  ┌──────────────────┐    │                  │
│                    │  │ max  (最大容量)   │    │                  │

│                    │  └──────┬───────────┘    │                  │
│                    │         │                │                  │
│                    │  Need_Trion              │                  │
│                    │  └─ 驱动恢复: cur ↑      │                  │
│                    └─────────┼────────────────┘                  │
│                              │                                  │
│            ┌─────────────────┼─────────────────┐                │
│            ↓                 ↓                 ↓                │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐          │
│  │  触发器模块   │  │  战斗模块     │  │  设施模块     │          │
│  │              │  │              │  │              │          │
│  │ 芯片使用     │  │ 战斗体维持   │  │ 建筑运作     │          │

│  │              │  │ Consume()   │  │              │          │
│  │              │  │ cur ↓       │  │              │          │
│  └──────────────┘  └──────────────┘  └──────────────┘          │

│  ┌──────────────┐                                               │
│  │  世界模块     │                                               │

│  └──────────────┘                                               │
│                                                                 │
│  图例: ↑ = 增加    ↓ = 减少    → = 数据流向                      │

**这张图回答的问题**：Trion从哪来、到哪去、谁在增加它、谁在消耗它。所有对Trion的操作都通过CompTrion的统一API（Consume/Recover/Allocate/Release），没有任何模块绕过内核直接修改Trion值。

## 7. 模块间通信：通过微内核的间接协作

模块之间不直接互相调用（除战斗→触发器外），而是通过微内核间接协作。以下是两个典型场景：

### 场景1：芯片使用导致Trion耗尽→战斗体破裂

```
  触发器模块              微内核(CompTrion)          战斗模块
  ──────────            ────────────────          ──────────
       │                      │                       │
  芯片使用                    │                       │
       │──── Consume(30) ───→│                       │
       │                      │ cur: 35→5             │
       │                      │                       │
  芯片使用                    │                       │
       │──── Consume(10) ───→│                       │
       │                      │ cur: 5→0              │
       │                      │                       │
       │                      │── 信号:Trion耗尽 ───→│
       │                      │                       │
       │                      │              HComp_TrionDrain
       │                      │              检测到cur==0
       │                      │              触发战斗体破裂
       │                      │                       │
       │                      │←── ForceDeplete() ───│
       │                      │    allocated→0        │
       │                      │                       │

  关键：触发器模块不知道战斗模块的存在
       战斗模块不知道是谁消耗了Trion
       它们通过CompTrion的状态变化间接协作
```

### 场景2：副作用能力消耗Trion（简单场景）

```
  角色模块                微内核(CompTrion)
  ──────────            ────────────────
       │                      │
  副作用Ability               │
  .Activate()                │
       │──── Consume(50) ───→│
       │                      │ cur减少
       │←── true(成功) ──────│
       │                      │

  角色模块直接调用微内核API，不涉及其他模块
```

## 8. 模块接口摘要 ⚠️ 部分待验证

> **置信度说明**：微内核(CompTrion)的接口 = 已确定（来源于技术选型报告）。触发器模块的接口 = 待验证（合理推测，需触发器模块深入分析后确认）。

每个模块对外暴露的核心API——其他模块或内核通过这些接口与之交互：

```
┌─────────────────────────────────────────────────────────────┐
│  微内核 (RimWT.Core) — 所有模块的共同依赖                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  CompTrion : ThingComp                                      │
│  ├─ 只读: Cur, Max, Allocated, Available, Percent           │
│  ├─ 操作: Consume(amount) → bool                            │
│  │        Recover(amount)                                   │
│  │        Allocate(amount) → bool                           │
│  │        Release(amount)                                   │
│  │        ForceDeplete()                                    │
│  └─ 访问: thing.GetComp<CompTrion>()                        │
│                                                             │
│  StatDef: TrionCapacity, TrionOutputPower, TrionRecoveryRate│
│  └─ 访问: pawn.GetStatValue(RimWT_StatDefOf.TrionCapacity)  │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  触发器模块 (RimWT.Trigger) — 被战斗模块C#依赖               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  CompTriggerBody : ThingComp                                │
│  ├─ 只读: IsActivated, LoadedChips                          │
│  ├─ 操作: Activate() / Deactivate()                         │
│  └─ 访问: pawn.equipment → GetComp<CompTriggerBody>()       │
│                                                             │
│  CompChipSlot : ThingComp                                   │
│  ├─ 操作: LoadChip(chip) / UnloadChip(slot)                 │
│  └─ 只读: SlotCount, OccupiedSlots                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘

  角色模块、设施模块、战斗模块、世界模块：
  不对外暴露被其他模块C#依赖的API

```

## 11. 关键设计决策记录

| # | 决策 | 选择 | 理由 | 否决的替代方案 |
|---|------|------|------|--------------|
| D1 | 架构模式 | 微内核+多模块 | Trion是所有系统的共同依赖，天然适合作为内核 | 单体架构（不利于独立开发测试） |
| D2 | 模块划分粒度 | 5模块2层 | 依赖层级清晰，开发顺序自然 | 3模块1层（依赖关系被隐藏在模块内部） |
| D3 | 角色模块与设施模块 | 保持分开 | 零依赖、不同RimWorld子系统、避免"杂物桶"模块 | 合并为"基础模块"（名称模糊，混杂不相关概念） |
| D4 | 模块的物理形态 | 命名空间+文件夹（单DLL） | RimWorld模组惯例，管理成本最低 | 多DLL（过度工程，RimWorld不需要） |
| D5 | 跨模块通信 | 通过CompTrion间接协作 | 解耦模块，触发器不需要知道战斗模块的存在 | 模块间直接调用（增加耦合） |
