# RimWT提示词模板系统·使用指南

## 📚 系统概览

### 这套系统包含什么

```
提示词模板系统
├── 00_元结构_通用提示词模板.md（最高层设计）
├── 01_场景1_阶段衔接模板.md（通用场景模板）
├── 02_场景2_阶段内继续模板.md（通用场景模板）
├── 03_阶段1_启动提示词.md（具体实例）
└── README_使用指南.md（本文档）
```

### 三层架构

```
【顶层】元结构（00）
  定义：7 大模块的通用框架
  用途：保证所有提示词的结构一致性

【中层】场景模板（01-02）
  定义：两种常见场景的模板
  用途：快速生成针对特定场景的提示词

【底层】具体实例（03+）
  定义：某个阶段的实际可用提示词
  用途：直接复制到新对话中使用
```

## 📖 详细使用流程

### 场景A：项目刚启动，进入第1阶段

**步骤**：
1. 打开 `03_阶段1_启动提示词.md`
2. 复制"🎬 提示词本体"部分的全部内容
3. 在新对话的第一条消息中粘贴
4. AI会立即理解项目背景和当前任务
5. 开始按第1阶段的要求工作

**示例消息**：
```
（复制03文档的提示词本体）

我已经准备好开始第1阶段的工作了。我会按照要求制作7份笔记。
```

---

### 场景B：第1阶段完成，准备进入第2阶段

**步骤**：
1. 打开 `01_场景1_阶段衔接模板.md`
2. 复制模板主体
3. 填充以下变量：
   - `<版本号>`：当前项目版本（如 v1.0）
   - `<日期>`：今天的日期
   - `<阶段号>`：第2阶段
   - `<阶段名>`：从蓝图复制（如"素材提炼"）
   - `<完成度>`：0/6（从蓝图复制）

4. **填充"前阶段成果总结"**：
   - 列出第1阶段完成的7份笔记
   - 总结第1阶段做出的关键决策
   - 说明第1阶段的限制条件

5. **填充"后阶段启动信息"**：
   - 从蓝图复制第2阶段的目标、关键决策点
   - 列出第2阶段要产出的文档清单
   - 定义第一个任务

6. **更新"必读资料"**：
   - 列出第1阶段产出的文档路径
   - 列出规范文档路径

7. 保存为 `RimWT_阶段2_启动提示词.md`

8. 在新对话中粘贴生成的提示词

**预计耗时**：5-10 分钟

---

### 场景C：第2阶段进行到一半，对话中断，需要新对话继续

**步骤**：
1. 打开 `02_场景2_阶段内继续模板.md`
2. 复制模板主体
3. 填充以下信息：

**基础信息**：
   - `<版本号>`、`<日期>`
   - `<阶段号>`：2
   - `<阶段名>`：素材提炼
   - `<完成度>`：3/6（假设完成了3个文档）
   - `<上次中断时间>`：上次对话的日期

**进度快照**：
   - 列出已完成的文档（如 2.1、2.2、2.3）
   - 记录每个文档的核心输出
   - 总结已做出的关键决策

**当前进行中的工作**：
   - 确定正在做哪个文档（如 2.4）
   - 列出已完成的子任务
   - 列出待完成的子任务
   - 说明当前的阻塞点（如有）

**未开始的文档**：
   - 列出还没做的文档（如 2.5、2.6）

**下一步行动**：
   - 明确第一优先级任务（继续文档2.4的下一个子任务）
   - 说明成功标准

4. 保存为 `RimWT_阶段2_继续_YYYYMMDD.md`

5. 在新对话中粘贴

**预计耗时**：10-15 分钟（需要回顾上次的进度）

---

### 场景D：为第3-14阶段创建启动提示词

**步骤**：
1. 打开 `03_阶段1_启动提示词.md`
2. 另存为 `RimWT_阶段X_启动提示词.md`
3. 修改以下部分：

**基础信息**：
   - 阶段号、阶段名
   - 进度（从蓝图复制）

**阶段目标**：
   - 从蓝图的对应表格中复制阶段目标
   - 说明为什么这个阶段重要

**关键决策点**：
   - 从蓝图表格的最后一列复制

**文档清单**：
   - 从蓝图的对应表格中复制
   - 填充文档类型、产出者、任务描述

**任务描述**：
   - **重新编写**：根据该阶段的特点重新写
   - 不能照搬阶段1的任务描述

**参考资料**：
   - 更新为该阶段需要的参考资料

4. 保存

**预计耗时**：15-20 分钟/阶段

## 📝 最佳实践

### ✅ 推荐做法

1. **为每个阶段提前准备启动提示词**
   - 在阶段开始前1天，制作好提示词
   - 保存为独立文件，便于快速使用

2. **保持提示词版本控制**
   - 每次对话中断后，更新提示词
   - 文件名加上日期后缀（如 `阶段2_继续_20260127.md`）

3. **简化重复信息**
   - 规范文档路径可以用变量代替
   - 创建一个"路径索引表"，提示词里引用

4. **定期更新元结构**
   - 如果发现某些信息总是缺失，修改元结构
   - 确保元结构反映实际需求

### ❌ 避免的错误

1. **不要过度自定义**
   - 保持 7 大模块的结构
   - 自定义内容应该在模块内部，不要改变模块顺序

2. **不要遗漏"前阶段成果"**
   - 场景1（阶段衔接）必须详细列出前阶段的输出
   - 这是新AI理解上下文的关键

3. **不要忽略"阻塞点"**
   - 场景2（阶段内继续）必须说明当前的障碍
   - 否则新AI会重复做已经失败的工作

4. **不要写太长的提示词**
   - 控制在 2000-3000 字以内
   - 超长的提示词容易被忽略关键信息

## 🔄 维护与更新

### 什么时候更新模板？

**元结构（00）**：
- 发现结构性问题时
- 增加新的通用模块时
- 优化信息组织逻辑时

**场景模板（01-02）**：
- 发现新的使用场景时
- 某个场景的需求变化时

**具体实例（03+）**：
- 阶段目标变化时
- 文档清单调整时
- 参考资料更新时

### 更新流程

```
1. 识别需要更新的部分
   ↓
2. 更新对应模板文件
   ↓
3. 更新版本号和历史记录
   ↓
4. 通知正在进行的对话（如有）
   ↓
5. 后续新对话使用新版本
```

## 🆘 常见问题

### Q1：我应该为每次对话都创建新提示词吗？
**A**：不需要。只有在以下情况需要：
- 进入新阶段
- 上次对话中断，新对话继续
- 进度有重大更新

### Q2：提示词太长，AI 会不会忽略？
**A**：元结构设计时已考虑这个问题：
- 使用层次化结构和清晰标题
- 最重要的信息放在前面
- 建议控制在 2000-3000 字

### Q3：如果蓝图文档更新了，提示词要全部重做吗？
**A**：不需要全部重做：
- 已完成阶段的提示词保持不变
- 只更新受影响的未开始阶段
- 优先更新元结构和场景模板

### Q4：能用这套系统管理其他项目吗？
**A**：完全可以！这套系统是通用的：
- 修改项目名和目标
- 调整阶段数量和文档清单
- 保持 7 大模块结构不变
