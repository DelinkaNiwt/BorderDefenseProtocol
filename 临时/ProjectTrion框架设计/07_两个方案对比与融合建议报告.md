## 第一部分：架构对比

### 1.1 您的v3.0方案：通用ECS框架

**核心思想：** 所有拥有Trion能力的实体（人、兵器、建筑）统一通过`CompTrion`组件，不同策略实现区分业务逻辑。

**架构图：**
```
CompTrion (通用入口)
  ├─ Strategy_HumanOverlay (人类覆写)
  ├─ Strategy_Mechanoid (自律兵器)
  └─ Strategy_Building (建筑)

TrionMountController (通用挂载)
  └─ TriggerComponent (触发器)
```

**优点：**
1. ✅ **架构广度极优**
   - 一个Comp处理所有实体类型
   - 新增实体类型只需新增一个Strategy
   - 避免复制冗余代码

2. ✅ **低耦合**
   - Strategy接口解耦，每种实体逻辑独立
   - 未来支持"黑触发"、"副作用"等新特性时，无需改动核心Comp

3. ✅ **RimWorld原生契合**
   - CompTrion就是一个普通ThingComp，高度兼容
   - TrionMountController的抽象符合RimWorld的模块化思想

4. ✅ **灵活性强**
   - 同一个Pawn可以在不同场景选择不同Strategy
   - 举例：NPC可能是"Strategy_Passive"（只做被动防御）

**缺点：**
1. ❌ **Trion能量管理不够细致**
   - 缺少"维持消耗 vs 动作消耗 vs 伤口泄漏"的细分
   - 缺少"组件激活引导"的完整流程
   - 缺少"预留占用 vs 可用余量"的资源管理

2. ❌ **实现复杂度高**
   - 需要定义ILifecycleStrategy接口，新手容易误解
   - TrionMountController的管理需要额外的初始化逻辑

3. ❌ **缺少参数配置指导**
   - Strategy层的细节参数（如伤口泄漏速率）没有明确说明
   - 一旦要调整游戏感，需要理解整个Strategy体系

---

### 1.2 我的v2.0方案：分层Comp专用架构

**核心思想：** 专注于人类Trion战斗体，分解为4个职能Comp，各司其职。

**架构图：**
```
TrionPawnComp (数据中心)
TriggerSystemComp (触发器系统)
CombatBodyComp (战斗体转换)
TrionConsumptionEngine (能量消耗)
```

**优点：**
1. ✅ **业务深度极优**
   - Trion能量的"维持、动作、伤口、激活"四维消耗清晰
   - 组件激活的"状态机"完整（Inactive → Activating → Active → Cooling）
   - 能量资源的"预留 vs 可用"明确分离

2. ✅ **参数配置完整**
   - 所有消耗参数外挂XML（维持0.5/Tick、激活5-50等）
   - 初级开发者无需理解复杂架构，只需调参数

3. ✅ **简洁易懂**
   - 4个Comp的职责明确，新手快速上手
   - 无需理解Strategy模式、工厂模式等设计模式

4. ✅ **快速原型**
   - 可立即开始编码，不需要搭建复杂的接口体系

**缺点：**
1. ❌ **扩展性差**
   - 如果要加Trion兵武器，需要为其新增一套Comp
   - 如果要加Trion炮台，又是一套Comp
   - 代码重复，维护困难

2. ❌ **耦合度高**
   - 4个Comp之间通过GetComp()强耦合
   - 每个Comp都假设宿主是Pawn，难以复用到Building

3. ❌ **不符合RimWorld哲学**
   - RimWorld的设计哲学是"万物皆是Thing，万物可挂Comp"
   - 我的方案绑死了"必须是Pawn"

4. ❌ **长期维护成本高**
   - 需求列表中明确包含：Trion兵、Trion炮台、传送站
   - 这些新增实体会导致Comp体系膨胀

## 第二部分：技术深度对比

### 2.1 Trion能量管理

**v2.0的能量消耗模型：**
```
每Tick消耗 =
  维持消耗(0.5)
  + 基础动作(1-5)
  + 伤口泄漏(伤口严重程度 × 0.5)
  + 激活消耗(一次性5-50)
  + 技能释放(50+)
```

**v3.0的能量消耗模型：**
```
每Tick消耗 = LeakRate
└─ LeakRate由策略层在OnDamageTaken时更新
```

**分析：**
- v2.0：清晰，可精确控制游戏感
- v3.0：简化，易于维护，但缺少细节
- **融合方案：** 在CompTrion中保留v2.0的四维消耗模型，Strategy层可覆写LeakRate计算

### 2.2 组件激活流程

**v2.0的激活机制：**
```
Request Activation
  → Check Energy Sufficient?
  → Activate Guiding (1-5 Tick)
    (Cannot switch during guiding)
  → Finish Activation
    (Consume energy immediately)
  → Component Active
```

**v3.0的激活机制：**
```
TryActivate(ComponentDef)
  → 简化的激活逻辑，细节由Strategy实现
```

**分析：**
- v2.0：状态完整，易于追踪和调试
- v3.0：灵活，但需要每个Strategy自己实现
- **融合方案：** 在TrionMountController中保留v2.0的状态机，Strategy层可配置参数

### 2.3 战斗体与肉身转换

**v2.0的转换流程：**
```
启动Trigger
  → SaveSnapshot(肉身数据)
  → GenerateCombatBody(虚拟实体)
  → CompTick() 维持战斗体
  → 战斗结束
    → DissolveCombatBody()
    → RestorePawn(恢复肉身)
```

**v3.0的转换流程：**
```
Strategy_HumanOverlay.OnInitialize()
  → _snapshot = new PawnSnapshot()
Strategy_HumanOverlay.OnDepleted()
  → _snapshot.Restore()
```

**分析：**
- v2.0：状态转换细节多，易于定制化
- v3.0：简洁，但快照机制缺乏详细说明
- **融合方案：** 快照逻辑写在Strategy_HumanOverlay中，核心数据结构由CompTrion管理

### 3.2 核心Comp定义（融合版）

```csharp
/// <summary>
/// 通用Trion实体核心组件（融合v2.0细节 + v3.0架构）
/// </summary>
public class CompTrion : ThingComp
{
    // ================= 核心数据 (v2.0水准) =================
    public float CurrentTrion { get; private set; }
    public float MaxCapacity { get; private set; }
    public float LeakRate { get; private set; }  // 伤口泄漏速率
    public float OutputPower { get; private set; } // 输出功率

    // ================= Trion能量消耗参数 (v2.0细节) =================
    public float MaintenanceCost { get; set; } = 0.5f;      // 维持消耗
    public float BaseActionCost { get; set; } = 1f;         // 基础动作
    public float ActivationCost { get; set; } = 20f;        // 激活消耗

    // ================= 策略与挂载 (v3.0架构) =================
    private ILifecycleStrategy _strategy;
    private List<TrionMountController> _mounts = new();

    public override void PostSpawnSetup(bool respawningAfterLoad)
    {
        base.PostSpawnSetup(respawningAfterLoad);

        // 工厂模式：根据宿主类型初始化策略（v3.0）
        if (parent is Pawn p && p.RaceProps.Humanlike)
            _strategy = new Strategy_HumanOverlay(this);        // 注入v2.0的血肉
        else if (parent is TrionSoldier)
            _strategy = new Strategy_Mechanoid(this);
        else if (parent is Building)
            _strategy = new Strategy_Building(this);

        _strategy.OnInitialize();
    }

    public override void CompTick()
    {
        // 计算消耗（v2.0的四维模型 + v3.0的通用接口）
        float drain = CalculateTotalDrain();

        // 执行消耗
        Consume(drain);

        // 策略层Tick
        _strategy.OnTick();
    }

    // 四维消耗计算（v2.0的精华）
    private float CalculateTotalDrain()
    {
        float drain = 0f;

        // 1. 维持消耗（战斗体存在）
        if (_strategy.IsActive)
            drain += MaintenanceCost;

        // 2. 基础动作消耗（如果有行动）
        // TODO: 根据Pawn.CurJobDef判断
        drain += BaseActionCost;

        // 3. 伤口泄漏（由策略维护）
        drain += LeakRate;

        // 4. 挂载点消耗（护盾、隐身等）
        foreach (var mount in _mounts)
            drain += mount.GetDrainThisTick();

        return drain;
    }

    public void Consume(float amount)
    {
        CurrentTrion = Mathf.Max(0, CurrentTrion - amount);

        if (CurrentTrion <= 0)
        {
            _strategy.OnDepleted();  // 回滚 或 爆炸 或 停机
        }
    }
}
```

### 3.3 人类覆写策略（融合版）

```csharp
/// <summary>
/// 人类战斗体策略（融合v2.0所有逻辑）
/// </summary>
public class Strategy_HumanOverlay : ILifecycleStrategy
{
    private CompTrion _comp;
    private PawnSnapshot _snapshot;           // v2.0的快照
    private List<WoundData> _wounds = new();  // v2.0的伤口追踪
    private ActivationState _currentState;    // v2.0的状态机

    public void OnInitialize()
    {
        // 1. 深度快照（v2.0）
        _snapshot = new PawnSnapshot(_comp.Pawn);

        // 2. 视觉覆写（v2.0）
        TrionUtility.ApplyOverlay(_comp.Pawn);

        // 3. 初始化挂载点（v3.0）
        InitializeMounts();  // LeftHand, RightHand
    }

    public void OnDamageTaken(float amount, BodyPartRecord hitPart)
    {
        // v2.0的伤口与泄漏系统
        var wound = new WoundData
        {
            bodyPart = hitPart,
            severity = amount * 0.1f
        };
        _wounds.Add(wound);

        // 更新泄漏速率（供CompTrion.CompTick()使用）
        _comp.LeakRate = _wounds.Sum(w => w.LeakageRate);

        // 部位摧毁判定（v2.0）
        if (amount > 50)
        {
            var mount = _comp.GetMountByPart(hitPart);
            mount?.ForceShutdown();
        }
    }

    public void OnDepleted()
    {
        // 1. 回滚肉身（v2.0）
        _snapshot.Restore();

        // 2. 惩罚效果（v2.0）
        _comp.Pawn.health.AddHediff(TrionDefOf.TrionExhaustion);

        // 3. Bail Out判定（v2.0）
        if (_comp.HasModule("BailOut"))
        {
            TeleportUtility.DoBailOut(_comp.Pawn);
        }

        // 4. 清理策略（v3.0）
        _comp.RemoveStrategy();
    }

    private void InitializeMounts()
    {
        _comp.AddMount(new TrionMountController
        {
            Tag = "LeftHand",
            MaxCapacity = 4,  // 可装配4个组件
            Pawn = _comp.Pawn
        });
        _comp.AddMount(new TrionMountController
        {
            Tag = "RightHand",
            MaxCapacity = 4
        });
    }
}
```

```csharp
enum ActivationState { Inactive, Activating, Active, Cooling, Destroyed }
```
→ 放入TrionMountController

✅ **肉身快照与回滚**
```csharp
class PawnSnapshot { Save(), Restore() }
```
→ 放入Strategy_HumanOverlay

✅ **伤口追踪系统**
```csharp
class WoundData { bodyPart, severity, leakageRate }
```
→ 放入Strategy_HumanOverlay或CompTrion

✅ **参数完全外挂化**
```xml
<maintenanceCost>0.5</maintenanceCost>

```
→ 配置CompTrion的Props

✅ **通用ECS框架**

### 5.3 简化/移除v3.0中的元素

❌ **事件总线（移除）**
- 原因：RimWorld是单线程游戏，事件系统带来不必要的复杂度和性能开销
- 替代：用C#原生event或直接方法调用
  ```csharp
  // 原来：_eventBus.Publish(new TrionDepletedEvent())
  // 改为：_strategy.OnDepleted()  或  OnDepleted?.Invoke()
  ```

❌ **过度接口化（简化）**
- v3.0定义了过多接口（IMountController, ITriggerComponent等）
- 融合方案：只保留最核心的ILifecycleStrategy，其他用具体类

**步骤1：用CompTrion替换4个Comp**
