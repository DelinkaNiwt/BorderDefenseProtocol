## 第一部分：设计概览

### 1. 核心设计理念

**Trion系统的本质：** 一套严密的生物能量资源管理、配置转换、输出控制的全过程框架

**框架的核心价值：**
- 实现高度可配置的能量分配与战术权衡系统
- 支持模块化的触发器组件生态
- 管理复杂的战斗体与肉身状态转换
- 追踪能量的多维消耗流程

### 2. 设计核心支柱

```
                    Trion能量资源层
                    ↓
    ┌──────────────────────────────────┐
    │  个体Trion数据管理               │
    │  - 总量容量管理                  │
    │  - 预留占用计算                  │
    │  - 可用余量追踪                  │
    └──────────────────────────────────┘
                    ↓
    ┌──────────────────────────────────┐
    │  模块化触发器系统                 │
    │  - 组件插槽管理                  │
    │  - 组件激活逻辑                  │
    │  - 能量占用计算                  │
    └──────────────────────────────────┘
                    ↓
    ┌──────────────────────────────────┐
    │  战斗体与肉身转换系统            │
    │  - 状态切换管理                  │
    │  - 伤口与泄漏追踪                │
    │  - 数据快照与回滚                │
    └──────────────────────────────────┘
                    ↓
    ┌──────────────────────────────────┐
    │  能量消耗与管理引擎              │
    │  - 多维消耗计算                  │
    │  - 损伤诱发泄漏                  │
    │  - Bail Out触发判定              │
    └──────────────────────────────────┘
```

### 3. 框架复杂度评估

| 维度 | 等级 | 说明 |
|------|------|------|
| **系统耦合度** | 🔴 高 | 4个核心模块深度耦合，能量计算影响全局 |
| **状态复杂度** | 🔴 高 | 战斗中涉及实时能量跟踪、伤口管理、多组件状态同步 |
| **数据模型复杂度** | 🟡 中 | 需要精密的Trion数据结构，但核心模型清晰 |
| **API依赖复杂度** | 🟢 低-中 | 主要依赖Comp系统和Pawn，RimWorld内建API充足 |

**结论：** 这是一个中-高复杂度的框架项目。核心在于如何清晰地组织能量管理逻辑，避免状态混乱。

## 第二部分：架构决策（权衡分析）

### 决策1：架构模式选择

**备选方案对比：**

| 方案 | 特点 | 优缺点 | 复杂度 |
|------|------|--------|--------|
| **A: 单体Comp模式** | 所有逻辑集中在一个TrionPawn Comp中 | ✓简单直观 ✗易爆炸，难扩展 | 高 |
| **B: 分层Comp模式** ⭐ | 能量层、触发器层、战斗体层分离 | ✓清晰职责 ✓易测试 ✗略显复杂 | 中 |
| **C: 事件驱动模式** | 所有状态变化通过事件通知 | ✓解耦性最强 ✗学习曲线陡 | 中-高 |

**选中方案：B - 分层Comp模式**

**理由：**
- ✓ 可行性：RimWorld Comp系统原生支持，无版本兼容问题
- ✓ 简洁性：3个Comp层职责分工明确，新手易上手
- ✓ 可维护性：未来可演进为事件驱动
- ℹ️ 代价：需要设计好Comp间通信接口

**RiMCP验证状态：** ⏳ 待验证（见API清单）

---

### 决策2：能量管理的计算模型

**备选方案：**

| 方案 | 计算方式 | 应用场景 | 复杂度 |
|------|---------|---------|--------|
| **A: 离散化消耗模型** | 每次行动后计算一次消耗 | 简化计算，适合回合制 | 低 |
| **B: 连续流消耗模型** ⭐ | 实时跟踪消耗速率，每Tick更新 | 逼真，适合动作游戏感 | 中 |
| **C: 混合模型** | 基础维持用流消耗，大动作离散计算 | 平衡精度和性能 | 中 |

**选中方案：B - 连续流消耗模型**

**理由：**
- ✓ 逼真性：能反映战斗中"伤口流血"的实时泄漏感
- ✓ 战术深度：玩家需要实时管理能量，而不是简单的离散计算
- ✓ 可行性：RimWorld的Tick系统原生支持
- ℹ️ 代价：需要精细的消耗参数调优

---

### 决策3：组件激活与切换机制

**核心问题：** 如何在战斗中实现"激活引导延迟+瞬间切换"的机制？

**备选方案：**

| 方案 | 实现方式 | 优缺点 |
|------|---------|--------|
| **A: Job系统** | 用Job来管理激活引导 | ✓RimWorld原生支持 ✗不够灵活 |
| **B: Hediff预留** | 用临时Hediff标记引导状态 | ✓灵活 ✗概念有些生硬 |
| **C: 自定义状态机** ⭐ | 在Comp中维护激活状态 | ✓最灵活 ✓便于追踪 ✗需自己管理 |

**选中方案：C - 自定义状态机**

**理由：**
- ✓ 灵活性：可精确控制激活流程、冷却、能量占用等
- ✓ 可观测性：容易调试和记录组件状态变化
- ✓ 可维护性：集中管理所有激活逻辑
- ℹ️ 代价：需要手工实现状态转移逻辑

---

### 决策4：伤口与泄漏的表现方式

**背景：** 战斗体受伤后需要追踪伤口位置、伤口大小、泄漏速率

**备选方案：**

| 方案 | 数据结构 | 优缺点 |
|------|---------|--------|
| **A: Hediff列表** | 每个伤口= 1个Hediff | ✓RimWorld原生 ✓易于序列化 ✗不够细粒度 |
| **B: 自定义伤口数据类** ⭐ | 在Comp中维护伤口列表 | ✓灵活度高 ✓性能好 ✗需自己序列化 |
| **C: 身体部位状态追踪** | 按Body Part维护状态 | ✓符合生物学 ✗RimWorld Body Part系统复杂 |

**选中方案：B - 自定义伤口数据类**

**理由：**
- ✓ 可控性：可以精确定义伤口属性（位置、大小、泄漏速率）
- ✓ 性能：避免Hediff系统的开销
- ✓ 序列化：配合Scribe系统，可靠地保存伤口状态
- ℹ️ 代价：需要手工管理伤口序列化

### 3.2 核心Comp设计

**1. TrionPawnComp** - 个体Trion数据管理
```csharp
public class TrionPawnComp : ThingComp
{
    // Trion总量管理
    float totalTrionCapacity;      // 个体总容量
    float currentTrionAmount;      // 当前总量

    // 预留管理
    float bailOutReserve;          // Bail Out预留占用
    float modulePreliminaryReserve; // 其他模块预留占用

    // 能量输出能力
    float peakOutputRate;          // 瞬时输出功率（电压）
    float sustainedOutputRate;     // 持续输出功率（电流）

    // 计算属性
    float GetAvailableTrion() { }  // 可用余量
    float GetReservedTrion() { }   // 预留占用总计
    void RequestBailOut() { }      // 触发紧急脱离
}
```

**2. TriggerSystemComp** - 模块化触发器系统
```csharp
public class TriggerSystemComp : ThingComp
{
    // 组件槽位
    List<ComponentSlot> mainSlots = new(4);
    List<ComponentSlot> subSlots = new(4);

    // 激活状态管理
    Dictionary<int, ActivationState> slotActivationStates;

    // 组件激活接口
    void RequestActivation(int slotIndex);
    void FinishActivation(int slotIndex);
    void DeactivateAll();

    // 能量占用计算
    float CalculatePreliminaryReserve();
}
```

**3. CombatBodyComp** - 战斗体与肉身状态管理
```csharp
public class CombatBodyComp : ThingComp
{
    // 状态追踪

```

### 3.3 Comp间通信设计

```
TrionPawnComp (能量数据中心)
    ↑ 查询获取
    ├─ TriggerSystemComp (需要查询可用Trion)
    ├─ CombatBodyComp (需要扣除消耗)
    └─ TrionConsumptionEngine (需要读写能量值)

CombatBodyComp (战斗体状态中心)
    ↑ 通知变化
    ├─ TrionPawnComp (设定肉身数据快照)
    └─ TrionConsumptionEngine (战斗体变化影响消耗)

TriggerSystemComp (触发器激活中心)
    ↑ 激活请求
    ├─ TrionPawnComp (检查能量是否足够)
    └─ TrionConsumptionEngine (添加激活消耗)
```

## 第五部分：关键设计考量

### 5.1 关键风险与缓解策略

| 风险 | 表现 | 缓解策略 |
|------|------|---------|
| **能量计算不精确** | 能量值不准确导致战斗逻辑混乱 | 严格的单元测试，参数外挂化 |
| **状态同步混乱** | Comp间状态不一致 | 明确的通信接口，状态验证 |
| **性能问题** | 大量Pawn时Tick卡顿 | 缓存优化，局部更新策略 |
| **存档兼容性** | 版本更新后存档崩溃 | 预留扩展字段，版本管理 |

### 5.2 可维护性与扩展性

**可维护性：**
- 清晰的职责划分使得维护和调试相对容易
- 参数配置外挂化，便于平衡调整

**扩展性：**
- 可在 TrionConsumptionEngine 基础上添加更多消耗方式
- 可在 TriggerSystemComp 基础上扩展更多组件类型
- 可在 CombatBodyComp 基础上完善伤口系统（如部位特化伤害）
