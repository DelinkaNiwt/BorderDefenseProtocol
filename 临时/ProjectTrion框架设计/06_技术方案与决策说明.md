# Trion系统技术方案与决策说明

## 第一部分：核心架构决策

### 决策1：分层Comp模式 vs 单体Comp模式

**问题背景：** 如何组织Trion系统的代码逻辑？

#### 方案对比

| 维度 | 单体Comp模式 | 分层Comp模式（选中） | 事件驱动模式 |
|------|------------|------|---------|
| **代码复杂度** | 🔴 高 (一个Comp ~2000行) | 🟡 中 (4个Comp各300-500行) | 🔴 高 |
| **职责清晰度** | 🔴 低 (混合多种逻辑) | 🟢 高 (各司其职) | 🟢 高 |
| **测试难度** | 🔴 难 (难以隔离测试) | 🟢 易 (可独立测试) | 🟡 中 |
| **扩展性** | 🔴 差 (修改涉及整个Comp) | 🟢 好 (局部扩展) | 🟢 优 |
| **学习曲线** | 🟢 快 (一个模式) | 🟡 中 (理解多层协作) | 🔴 陡 (事件驱动复杂) |
| **性能开销** | 🟢 最低 | 🟡 低 (多Comp查询) | 🔴 中等 (事件系统) |

#### 选中方案：分层Comp模式

**理由：**
1. ✅ **可行性**：完全利用RimWorld的Comp系统，无API兼容性问题
2. ✅ **简洁性**：4个独立Comp相比于单体Comp的数千行代码，更容易理解
3. ✅ **可维护性**：清晰的模块边界，新手易上手，维护成本低
4. ✅ **扩展空间**：可在此基础上演进为事件驱动

**代价：**
- ℹ️ Comp间通信需要通过GetComp()查询，比单体Comp多一层间接
- ℹ️ 需要定义清晰的Comp间接口，避免循环依赖

**版本兼容性：** ✅ 已验证RimWorld v1.6.4633 支持多Comp

---

### 决策2：连续流能量消耗模型 vs 离散化消耗

**问题背景：** 每Tick如何计算Trion消耗？

#### 方案对比

| 维度 | 离散化（回合制） | 连续流（选中） | 混合模型 |
|------|------------|------|---------|
| **计算方式** | 行动后一次性计算 | 每Tick实时流失 | 基础连续+技能离散 |
| **逼真度** | 低 (跳跃式下降) | 高 (平滑衰减) | 中 |
| **战术感** | 低 (计算简单) | 高 (需时刻警惕) | 中 |
| **性能** | 🟢 最优 | 🟡 良好 | 🟢 优 |
| **参数调优** | 简单 | 复杂 (需精调) | 中等 |

#### 选中方案：连续流消耗模型

**理由：**
1. ✅ **逼真性**：能模拟"伤口持续漏能"的真实感
2. ✅ **战术深度**：玩家需要实时管理能量，而不是离散的行动轮
3. ✅ **RimWorld原生适配**：Tick系统原生支持持续消耗（如饥饿）
4. ✅ **伤口泄漏自然表现**：伤口导致消耗加速，非常符合原作

**代价：**
- ℹ️ 参数调优复杂（需反复测试找到平衡点）
- ℹ️ 玩家需要适应"柔和但持续"的消耗模式，而非"突然降速"的感受

**版本兼容性：** ✅ RimWorld Tick系统自v1.0开始稳定

### 决策3：自定义激活状态机 vs Job系统 vs Hediff

**问题背景：** 如何实现"组件激活引导(1-5 Tick) → 瞬间切换"？

#### 方案对比

| 方案 | 实现方式 | 优点 | 缺点 |
|------|---------|------|------|
| **Job系统** | 创建一个ActivateComponentJob | ✅原生支持 | ❌不够灵活，难以追踪细节 |
| **Hediff** | 用临时Hediff标记激活状态 | ✅RimWorld原生 | ❌概念生硬，过度设计 |
| **自定义状态机** ⭐ | 在TriggerSystemComp中维护 | ✅灵活高效 | ❌需手工管理 |

#### 选中方案：自定义状态机

**实现思路：**
```csharp
public enum ActivationState
{
    Inactive,      // 槽位上没有激活的组件
    Activating,    // 正在引导激活（计时中）
    Active,        // 已激活（可以使用）
    Cooling,       // 冷却中（上一个组件还在冷却）
    Destroyed      // 组件被摧毁，无法激活
}

// 每Tick更新状态转移
public void OnTick()
{
    foreach (var (slotIndex, counter) in activationCounter)
    {
        counter--;
        if (counter <= 0)
        {
            // 转移为Active状态
            slots[slotIndex].activationState = ActivationState.Active;
        }
    }
}
```

**理由：**
1. ✅ **灵活性**：可精确控制激活流程、冷却、能量占用
2. ✅ **可观测性**：易于调试（直接读取状态机）
3. ✅ **性能**：相比Job/Hediff系统开销更小
4. ✅ **易于扩展**：可在此基础上添加冷却、连续激活等机制

**代价：**
- ℹ️ 需要手工实现状态转移逻辑
- ℹ️ 需要考虑边界情况（如快速切换）

**版本兼容性：** ✅ 完全自定义，无依赖

### 决策4：自定义伤口数据 vs Hediff伤口

**问题背景：** 如何追踪战斗体伤口和泄漏速率？

#### 方案对比

| 方案 | 数据结构 | 序列化 | 泄漏计算 | 集成 |
|------|---------|--------|---------|------|
| **Hediff列表** | RimWorld原生 | 自动 | 需要拦截 | 不自然 |
| **自定义伤口** ⭐ | WoundData类 | 手工Scribe | 直接 | 灵活 |

#### 选中方案：自定义伤口数据

**数据结构：**
```csharp
public class WoundData
{
    public BodyPartRecord bodyPart;      // 伤口位置
    public float severity = 1f;         // 伤口严重程度 (1-10)
    public float leakageRate            // 计算属性：severity * 0.5f
    {
        get => severity * 0.5f;
    }
    public bool isLethal = false;       // 是否致命伤口
    public long tickWhenOccurred;       // 用于恢复计算
}
```

**理由：**
1. ✅ **控制力**：完全掌控伤口数据和泄漏计算
2. ✅ **性能**：避免Hediff系统的开销（RimWorld Hediff搜索很慢）
3. ✅ **清晰性**：与框架的设计逻辑无缝配合
4. ✅ **序列化**：配合Scribe系统可靠地保存和恢复

**代价：**
- ℹ️ 需要手工序列化伤口列表
- ℹ️ 不能直接利用RimWorld的医疗系统（但可调用接口扩展）

**版本兼容性：** ✅ Scribe系统自v1.0开始稳定

**假设场景：** 100个Trion战士同时战斗

| 计算项 | 每Pawn | 100个Pawn | 占比 |
|--------|--------|----------|------|
| 维持消耗 | 0.01ms | 1ms | 低 |
| 伤口泄漏计算 | 0.02ms | 2ms | 低 |
| Comp查询 | 0.01ms | 1ms | 低 |
| 总计 | ~0.04ms | ~4ms | 可接受 |

**优化策略：**
- ✅ 缓存GetComp()结果（减少频繁查询）
- ✅ 批量计算多Pawn的消耗
- ✅ 伤口列表排序，优化查询

**结论：** 性能可满足需求，无明显瓶颈

## 第三部分：集成方案

### 3.1 与RimWorld系统的集成方式

**集成点1：Pawn创建**
```
Pawn创建 → SpawnSetup() → 注册TrionPawnComp
  ↓
自动计算初始Trion容量（基于属性、天赋）
自动装配Comp到Pawn
```

**集成点2：战斗触发**
```
玩家启动Trigger → GenerateCombatBody()
  ↓
保存肉身快照 → 进入战斗体状态
  ↓
CompTick() → 每Tick消耗Trion
  ↓
战斗结束 → DissolveCombatBody() → 恢复肉身
```

**集成点3：伤害系统**
```
造成伤害 → RegisterWound()
  ↓
加速Trion消耗 (泄漏)
  ↓
若致命 → 触发Bail Out
```

### 3.2 API依赖清单

| API | 位置 | 用途 | 验证状态 |

## 第四部分：风险评估与缓解

### 4.1 已识别风险

| 风险 | 概率 | 影响 | 缓解策略 |
|------|------|------|---------|
| **能量计算不准确** | 中 | 高 | 严格单元测试，参数外挂化 |
| **状态同步混乱** | 低 | 高 | 明确的Comp通信协议 |
| **性能下降** | 低 | 中 | 缓存优化，批量计算 |
| **存档兼容性** | 低 | 中 | 版本管理，字段扩展 |
| **跨mod冲突** | 低 | 中 | API隔离，通信协议 |

### 4.2 风险缓解策略

**风险1：能量计算不准确**
- 实现 TrionDataValidator 进行数据一致性检查
- 所有参数外挂到XML，不硬编码
- 建立详细的测试用例（维持、消耗、恢复、泄漏等）

**风险2：状态同步混乱**
- 定义清晰的Comp间通信接口
- 避免循环依赖（A->B->C，不允许C->A）
- 使用事件系统（可选）降低耦合

**风险3：性能下降**
- 缓存GetComp()结果到局部变量
- 伤口数量上限检查
- 定期性能测试
