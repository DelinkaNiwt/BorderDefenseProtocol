# Trion系统需求规格与系统拆解

## 第一部分：需求概览

### 1.1 模组目标

基于《境界触发者》原作设定，为 ProjectTrion 模组设计和实现一套底层的 Trion 能量资源管理框架，支持：

- **能量资源的严密管理**：每个殖民者/Trion兵拥有独立的Trion总量，框架追踪其分配、消耗、恢复流程
- **模块化触发器系统**：支持组件插槽化装备，组件激活与切换涉及能量占用和引导延迟
- **战斗体转换机制**：肉身与战斗体的状态转换，伤口与能量泄漏追踪
- **多维能量消耗**：维持、动作、伤口泄漏、组件激活等多种消耗方式的统一计算模型

### 1.2 版本与兼容性约束

| 条件 | 值 |
|------|-----|
| **RimWorld版本** | 1.6.4633 |
| **DLC支持** | 暂无强制依赖，框架设计保留扩展空间 |
| **编程语言** | C# (RimWorld MOD标准) |
| **核心依赖** | RimWorld原生 Comp、Pawn、Tick系统 |

## 第二部分：功能需求清单

### 2.1 核心功能 (优先级：P0 - 必须实现)

#### F1: 个体Trion容量管理

**描述：** 每个Pawn拥有独立的Trion总量，框架在其Comp中存储和管理该数据。

**需求细节：**
- 每个Pawn可配置基础Trion总量（范围：500~5000，推荐中位值1000）
- 支持个体天赋加成（±20% ~ ±100%）
- Trion总量初始化后相对固定，但可通过特定事件增长（青少年期加速）
- 数据须严格序列化保存

**用例：**
- 创建新Pawn → 根据属性计算初始Trion总量
- 加载存档 → 恢复Pawn的Trion数据
- 升级天赋 → 增加Trion容量上限

---

#### F2: Trion可用值与预留占用管理

**描述：** Trion总量分为"预留占用"和"战斗可用余量"，框架动态计算两者关系。

**需求细节：**
- **预留占用** = Bail Out占用 + 其他模块占用
- **战斗可用余量** = 总量 - 预留占用
- 当装配新组件时，预留占用动态更新
- 可用余量不能为负（违反可能导致游戏逻辑错误）

**用例：**
- 装配Bail Out系统 → 预留占用增加400（假设配置值）
- 装备其他组件 → 每个组件增加20-50占用
- 查询战斗时的可用能量 → 返回 总量 - 预留占用

---

#### F3: 能量实时消耗与Tick计算

**描述：** 战斗中，Trion能量以连续流的方式被消耗，框架每Tick计算各类消耗。

**消耗来源：**
1. **维持消耗** (0.5/Tick)：维持战斗体存在
2. **动作消耗** (1-5/Tick)：移动、攻击等行动
3. **激活消耗** (5-20/激活)：组件激活时的一次性消耗
4. **伤口泄漏** (2-10/Tick)：战斗体受伤后的能量流失
5. **高强度输出** (50/次)：大范围技能如旋空切击

**需求细节：**
- 每Tick扣除相应的能量值
- 消耗计算需考虑当前激活的组件
- 伤口泄漏速率随伤口严重程度增加
- 瞬时消耗（如技能释放）需检查峰值输出功率

**用例：**
- 战斗体维持1秒（10 Tick） → 消耗约5-15点能量
- 受伤导致左臂伤口 → 后续每Tick增加2点泄漏
- 释放旋空切击 → 检查输出功率是否≥50，消耗50点能量

---

#### F4: 组件激活与预留占用

**描述：** Trion触发器采用模块化设计，每个组件装配时预先占用Trion。

**需求细节：**
- 主触发器4个槽位，副触发器4个槽位，特殊槽位1个（Bail Out等）
- 每个组件在装配时刻就占用相应的Trion（不等到激活）
- 组件激活时的一次性消耗从"战斗可用余量"扣除
- 同一手臂同时只能有1个组件处于激活状态
- 激活需要引导时间（1-5 Tick），期间无法切换其他组件

**用例：**
- 装备弧月（10占用）+ 旋空（10占用） → 预留占用+20
- 激活护盾（一次性消耗20，引导2 Tick） → 期间不能激活其他组件
- 战斗中弧月被砍坏 → 该组件对应的占用和激活状态清除

---

#### F5: 战斗体生成与肉身保护

**描述：** 启动触发器时，肉身进入亚空间休眠，由Trion构建的战斗体取代其参战角色。

**需求细节：**
- 生成战斗体时保存肉身的完整快照（生理数据、装备、状态）
- 战斗体操作不影响肉身（包括伤口、断肢、死亡）
- 战斗体必须维持的核心部位：心脏/供能器官
- 战斗体状态下无法增删装备（物品被禁用）
- 战斗结束（主动或强制）→ 肉身完全恢复到快照状态

**用例：**
- 启动触发器时，殖民者有一个枪伤 → 战斗体不继承该伤口
- 战斗体被砍掉左臂 → 肉身解除后完整恢复
- 战斗中伤口蔓延导致供能器官受损 → 触发Bail Out
- 主动解除触发器 → 肉身恢复，战斗体消散

---

#### F6: 伤口与能量泄漏追踪

**描述：** 战斗体受伤后会产生伤口，伤口处持续漏能量。

**需求细节：**
- 每个伤口记录：部位、严重程度、泄漏速率
- 伤口严重程度决定泄漏速率（范围：1-10点/Tick）
- 多个伤口的泄漏速率叠加
- 伤口可通过医疗或自愈恢复（暂不实现，框架预留接口）
- 特殊伤口（如供能器官损伤）触发Bail Out

**用例：**
- 被斩击，产生一个中度伤口（严重程度3） → 泄漏速率+3/Tick
- 同时被击中两处 → 两个伤口泄漏叠加为5-6/Tick
- 供能器官被击中 → 立即触发Bail Out
- 伤口过多导致泄漏速率>维持+消耗速率 → 能量迅速耗尽

---

#### F7: Bail Out紧急脱离机制

**描述：** 当Trion能量耗尽或核心部位受损时，自动触发Bail Out脱离战斗。

**需求细节：**
- Bail Out 占用约 400 点 Trion（具体值可配置）
- 触发条件：
  - 战斗可用余量 ≤ 0
  - 供能器官被摧毁
  - 玩家主动解除触发器
- 触发时：战斗体崩解，肉身传送回指定位置（如基地锚点）
- 战斗结束后，未消耗的Trion自动回收（仅限主动解除时）

**用例：**
- 战斗中能量消耗至0 → Bail Out自动触发，战士传送回基地
- 战斗体被彻底摧毁 → Bail Out触发，肉身安全返回
- 主动解除触发器 → 未消耗的预留Trion回收，战斗体消散

---

### 2.2 扩展功能 (优先级：P1 - 后续迭代优化)

#### F8: 瞬时输出功率限制

**描述：** 大规模技能（如旋空切击）需要高瞬时输出功率，低Trion个体可能无法释放。

**需求细节：** （框架预留，暂不实现）
- 每个Pawn有基础输出功率（由Trion器官性能决定）
- 技能释放检查：输出功率 ≥ 技能所需功率
- 输出功率不足 → 技能释放失败，但仍消耗能量（提示不足）

---

#### F9: Trion再生与冷却

**描述：** 战斗结束后，Trion逐渐恢复到初始值。

**需求细节：** （框架预留，暂不实现）
- 恢复速率：10点/小时（可配置）
- 可通过特殊设施（休养舱）加速恢复
- 受伤状态下恢复速率降低

---

#### F10: 个体天赋与Trion特质

**描述：** 不同角色拥有不同的Trion相关天赋，影响容量、恢复速率、输出功率等。

**需求细节：** （框架预留，暂不实现）
- 天赋示例：「Trion天才」（+50% 容量）、「虚弱体质」（-30% 容量）
- 天赋影响参数配置，框架动态读取

---

## 第三部分：系统边界与架构拆解

### 3.1 系统边界定义

**Trion框架的职责范围：**

| 范围 | 包含 | 不包含 |
|------|------|--------|
| **能量管理** | ✓ 记录、计算、消耗Trion | ✗ 具体战斗伤害计算 |
| **触发器系统** | ✓ 组件插槽、激活状态、占用计算 | ✗ 具体技能效果实现 |
| **战斗体转换** | ✓ 肉身快照、状态切换、伤口追踪 | ✗ 战斗体的视觉表现 |
| **消耗引擎** | ✓ Tick周期消耗、泄漏计算 | ✗ 消耗显示UI |

**外部模块的职责：**
- **战斗系统**：调用框架接口进行战斗（扣能量、检查技能可行性）
- **UI系统**：读取框架数据展示能量条、组件状态
- **伤害系统**：在造成伤害时调用框架接口注册伤口

### 3.2 系统拆解：4个核心模块

```
Trion能量资源管理框架
├── 模块1：TrionPawnComp (个体Trion数据管理)
│   ├─ 职责：存储和维护Pawn的Trion数据
│   ├─ 数据：总量、预留、可用、输出功率
│   └─ 接口：GetAvailableTrion(), RequestBailOut()
│
├── 模块2：TriggerSystemComp (模块化触发器系统)
│   ├─ 职责：管理触发器组件的装备、激活、占用计算
│   ├─ 数据：槽位配置、激活状态、预留占用
│   └─ 接口：EquipComponent(), ActivateSlot(), CalcPreliminaryReserve()
│
├── 模块3：CombatBodyComp (战斗体与肉身转换)
│   ├─ 职责：管理战斗体生成/销毁、肉身快照、伤口追踪
│   ├─ 数据：战斗体状态、肉身快照、伤口列表
│   └─ 接口：GenerateCombatBody(), RestorePawn(), RegisterWound()
│
└── 模块4：TrionConsumptionEngine (能量消耗计算)
    ├─ 职责：在Tick周期内计算各类能量消耗
    ├─ 数据：消耗参数、消耗速率、Bail Out触发条件
    └─ 接口：OnCombatBodyTick(), CheckBailOutCondition()
```

### 3.3 模块职责详解

#### 模块1：TrionPawnComp

**核心数据：**
```
个体Trion总量 → 预留占用 + 可用余量
```

**职责：**
1. 初始化Pawn时计算基础Trion总量
2. 维护Trion总量、预留占用、可用余量三个数据
3. 提供查询接口：GetTotalTrion(), GetAvailableTrion(), GetReservedTrion()
4. 触发Bail Out：RequestBailOut()
5. 数据序列化与保存

**与其他模块的交互：**
- ← TriggerSystemComp 通知预留占用变化
- ← TrionConsumptionEngine 通知能量消耗
- → CombatBodyComp 查询当前可用能量

**设计要点：**
- 单一职责：只管数据，不涉及消耗计算逻辑
- 原子性：确保三个值始终满足 总量 = 预留 + 可用

---

#### 模块2：TriggerSystemComp

**核心数据：**
```
组件槽位
├── 主槽位 × 4
├── 副槽位 × 4
└── 特殊槽位 × 1 (Bail Out等)

每个槽位 {
  状态: 未装备 / 已装备 / 激活中 / 冷却中
  组件类型: weapon / shield / skill / utility
  占用: 10-20
  激活消耗: 5-50
  激活时间: 1-5 Tick
}
```

**职责：**
1. 管理8+1个槽位，支持组件装备与卸除
2. 追踪每个槽位的激活状态
3. 计算预留占用总和：Σ(已装备组件的占用)
4. 处理组件激活请求：检查能量、启动引导、完成激活
5. 处理组件切换：同一臂上新旧组件的状态转移

**与其他模块的交互：**
- → TrionPawnComp 提交预留占用变化
- → TrionConsumptionEngine 提交激活消耗请求
- ← TrionConsumptionEngine 查询当前激活组件

**设计要点：**
- 组件与组件之间的切换逻辑需清晰
- 同一手臂同时只能1个组件激活
- 预留占用与激活消耗的区分（预留 vs 消耗）

---

#### 模块3：CombatBodyComp

**核心数据：**
```
战斗体状态 (启用 / 禁用)
↓
肉身快照 {
  健康状态
  装备物品列表
  其他Hediff
}
↓
伤口列表 [
  { 部位, 严重程度, 泄漏速率 }
  ...
]
```

**职责：**
1. 在启动触发器时保存肉身快照，禁用物品
2. 生成并维持战斗体的虚拟状态
3. 注册伤口：RegisterWound(part, severity)
4. 追踪伤口泄漏：GetTotalLeakageRate()
5. 检查核心部位受损：IsCoreDamaged()
6. 解除战斗体时恢复肉身：RestorePawn()

**与其他模块的交互：**
- ← TrionConsumptionEngine 通知伤口泄漏更新
- → TrionConsumptionEngine 提供伤口泄漏速率
- → TrionPawnComp 触发Bail Out时通知肉身恢复

**设计要点：**
- 肉身与战斗体的完全隔离（任何伤害都不影响肉身）
- 伤口数据的精确性（用于计算泄漏）
- 快照机制的可靠性（确保恢复时数据完整）

---

#### 模块4：TrionConsumptionEngine

**核心职责：**
```
每Tick执行:
1. 计算维持消耗
2. 计算动作消耗
3. 计算伤口泄漏
4. 累加扣除能量
5. 检查Bail Out条件
```

**消耗来源：**

| 来源 | 计算方式 | 触发条件 |
|------|---------|--------|
| 维持 | 固定值 0.5/Tick | 战斗体激活中 |
| 动作 | 基于行动类型 1-5/Tick | 执行移动/攻击 |
| 激活 | 组件特定 5-50/激活 | 激活新组件 |
| 泄漏 | 伤口严重程度 2-10/Tick | 战斗体有伤口 |
| 高强度 | 技能特定 50/释放 | 释放旋空等技能 |

**与其他模块的交互：**
- → TrionPawnComp 扣除消耗能量
- ← TriggerSystemComp 查询激活组件状态
- ← CombatBodyComp 查询伤口泄漏速率
- → TrionPawnComp 触发Bail Out

**设计要点：**
- 消耗计算的准确性（参数调参数据驱动）
- Tick的高频执行需性能考虑（缓存、批量计算）
- 瞬时消耗（技能）与持续消耗的区分

## 第五部分：系统约束与假设

### 5.1 设计约束

| 约束 | 影响 | 处理方式 |
|------|------|---------|
| **Trion总量固定** | 无法通过一般方式改变总量 | 预留扩展天赋系统 |
| **战斗体不可以肉身互动** | 不能捡东西、进行社交等 | 框架层面强制禁用 |
| **Bail Out必须安全脱离** | 无法在特殊地点实现条件脱离 | 框架层面锁定传送点 |
| **能量只能线性恢复** | 无法通过特殊事件快速恢复 | 预留设施加速机制 |

### 5.2 设计假设

| 假设 | 原因 | 风险 |
|------|------|------|
| 战斗体性能 >> 肉身性能 | 符合原作设定 | 如果不满足，影响战斗平衡 |
| Trion泄漏速率线性 | 简化计算 | 现实可能非线性，可后续优化 |
| 同时最多N个战斗体 | 性能考虑 | 需通过测试确定合理值 |
| 伤口自动恢复 | 框架简化 | 可后续扩展医疗系统 |

## 第六部分：优先级与工作分解

### 6.1 功能优先级矩阵

```
高价值 + 低复杂度（优先做）
├── F1: 个体Trion容量管理
├── F2: 预留占用管理
├── F4: 组件激活与占用
├── F5: 战斗体生成与肉身保护
└── F7: Bail Out脱离

高价值 + 高复杂度（仔细计划）
├── F3: 能量实时消耗
├── F6: 伤口与泄漏追踪
└── F11: 参数配置外挂

低价值 + 低复杂度（可选）
├── F8: 瞬时输出功率限制
├── F9: Trion再生与冷却
└── F10: 个体天赋与特质
```

### 6.2 工作分解结构

| 工作项 | 依赖 | 估算工作量 | 风险 |
|--------|------|----------|------|
| 设计数据结构 | 无 | 1-2h | 低 |
| 实现TrionPawnComp | 上 | 2-3h | 低 |
| 实现TriggerSystemComp | 上 | 3-4h | 中（组件激活逻辑复杂） |
| 实现CombatBodyComp | 上 | 2-3h | 中（快照机制）
| 实现TrionConsumptionEngine | 1-4 | 3-4h | 高（参数调优需反复测试） |
| 集成与测试 | 全部 | 4-5h | 中 |

**总估算：** 15-25小时核心开发 + 后续调优
