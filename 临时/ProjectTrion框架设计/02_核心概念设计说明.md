# ProjectTrion Trion系统核心概念设计

## 第一部分：模组是什么

### 1.1 核心定位

**ProjectTrion** 是基于《境界触发者》(World Trigger) 原作 Trion 系统的 RimWorld 模组框架。

通过一套完整的能量资源管理系统，将原作中的"Trion能量驱动、模块化触发器、战斗体转换"等设定引入RimWorld，为模组开发者和玩家提供：

- **严密的资源优化框架**：Trion不仅是血槽，而是战略级别的资源配置博弈
- **模块化的装备系统**：触发器可灵活装备不同能力组件，每个选择都有代价
- **硬核的生存机制**：低能量个体需要精明的战术和团队协作才能生存

### 1.2 为什么这个设定值得做？

#### 原作的系统设计优势

1. **能量守恒的战斗哲学**
   - 传统游戏：能量多 = 自动赢
   - Trion系统：能量多只是基础，真正的胜负取决于如何分配、消耗和回收能量
   - **游戏深度提升**：加入了资源管理维度

2. **模块化的职业分化**
   - 高能量者可以单兵压制，低能量者必须依赖团队
   - **包容性强**：弱者有生存路径，不是简单的碾压

3. **信息战的重要性**
   - 隐身 vs 雷达的能量消耗博弈
   - **战术深度**：除了物理战斗，还有信息战维度

#### 对RimWorld的增强价值

RimWorld 的战斗系统相对简单，Trion 系统可以：
- ✅ 增加战斗的战术层数
- ✅ 为弱小殖民者提供"弯道超车"的机制
- ✅ 创建一个独特的能力系统生态

## 第二部分：核心创意方向

### 2.1 设定与原作的对应

| 原作概念 | RimWorld 对应 | 框架设计位置 |
|---------|--------------|------------|
| **Trion器官** | Pawn属性 | TrionPawnComp.totalCapacity |
| **Trion能量** | 资源值 (0-10000) | TrionPawnComp.currentAmount |
| **Trigger触发器** | 装备品类 | TriggerSystemComp.slots |
| **战斗体** | 虚拟实体状态 | CombatBodyComp.isCombatActive |
| **肉身休眠** | 快照保存 | CombatBodyComp.pawnSnapshot |
| **能量消耗** | Tick周期扣除 | TrionConsumptionEngine |
| **伤口泄漏** | 伤害诱发消耗加速 | CombatBodyComp.wounds |
| **Bail Out** | 自动传送脱离 | 触发器系统 |

### 2.2 游戏感目标

**玩家体验：**

```
战前：
  选择装备触发器组件 → 权衡火力、防御、生存
  低能量殖民者精心选配 → 形成特色战术

战中：
  观察能量条实时变化 → 感受"资源争夺"的紧张
  伤口增加导致能量加速消耗 → "消耗战"变成真实挑战
  必须考虑是否足以支撑下一次攻击 → 每一步都有风险

战后：
  存活殖民者能量缓慢恢复 → 给予休整的时间成本
  合理分配团队角色 → 低能量者也有用处
```

**核心游戏感：**
> 胜负不由天赋决定，而由如何在有限的资源下做出最优选择

## 第三部分：创意特色

### 3.1 与原作的适配程度

| 维度 | 适配度 | 说明 |
|------|--------|------|
| **能量资源概念** | 🟢 100% | 完全复现原作的能量资源博弈 |
| **模块化装备** | 🟢 90% | 支持组件插槽化，略简化原作 |
| **战斗体转换** | 🟡 70% | 框架层支持状态转换，具体表现留给上层 |
| **伤口泄漏** | 🟡 60% | 框架追踪伤口，泄漏计算简化为线性模型 |
| **Bail Out安全脱离** | 🟢 95% | 完全符合原作设定 |

**结论：** Trion系统框架能够捕捉原作的核心精髓，为后续的应用层实现奠定坚实基础。

### 3.2 扩展创意方向（非本框架范围，但框架预留空间）

1. **跨越型战斗**：支持多方参战者的Trion资源博弈
2. **特殊Trigger**：黑色Trigger（强大但危险）的实现
3. **副作用系统**：高能量个体产生的超能力效应
4. **生态应用**：Trion能源驱动的非战斗设施（传送、防护罩等）

## 第四部分：目标与价值

### 4.1 模组开发者的价值

✅ **快速原型**：无需从零开始实现能量系统，直接使用框架
✅ **高度可定制**：参数外挂化，无需修改代码即可调整平衡
✅ **清晰架构**：易于扩展新的消耗方式、组件类型、伤口效果
✅ **知识积累**：框架设计本身可作为其他系统的参考

### 4.2 玩家的价值

✅ **战略深度**：每个选择都有代价，无最优解只有权衡
✅ **包容性**：弱小个体可通过聪慧的配置和团队合作翻盘
✅ **紧张感**：实时追踪的能量条增加了战斗的节奏感
✅ **沉浸感**：对标原作的设定让粉丝感到"正统"

### 4.3 项目成功的定义

| 指标 | 目标 | 衡量方式 |
|------|------|---------|
| **功能完整性** | 4个核心模块正常工作 | 单元测试通过率 ≥ 95% |
| **性能** | 大量Pawn时无明显卡顿 | Tick消耗 < 1ms/frame |
| **可维护性** | 新开发者能轻松上手 | 代码文档清晰，接口直观 |
| **可扩展性** | 支持第三方模组使用 | 提供公共API接口 |

## 第五部分：限制与假设

### 5.1 框架的限制

| 限制 | 影响 | 后续解决 |
|------|------|---------|
| **不处理伤害计算** | 框架只追踪能量，不定义伤害如何造成 | 由应用层（战斗系统）负责 |
| **不处理视觉效果** | 框架数据驱动，不提供UI动画 | 由UI系统读取框架数据并渲染 |
| **不处理伤口医疗** | 框架只记录伤口，不定义如何恢复 | 由医疗系统调用框架接口治疗 |
| **不支持Trion共享** | 个体Trion独立，无法借用他人能量 | 如需支持，需在框架层扩展 |

### 5.2 设计假设

| 假设 | 原因 | 验证方法 |
|------|------|---------|
| 战斗体性能 >> 肉身 | 符合原作设定 | 通过游戏测试验证平衡性 |
| Trion总量在战斗中固定 | 简化设计 | 通过参数调整验证可行性 |
| 同时最多10个战斗体 | 性能与剧情考虑 | 根据实际测试调整上限 |
| 能量线性恢复 | 计算简化 | 可后续扩展为非线性模型 |

## 第六部分：与其他模组的协作

### 6.1 框架暴露的公共接口

**其他模组可以调用：**

```csharp
// 查询能量状态
float available = pawn.GetComp<TrionPawnComp>().GetAvailableTrion();

// 扣除能量
pawn.GetComp<TrionPawnComp>().ConsumeTrion(amount);

// 注册伤口
pawn.GetComp<CombatBodyComp>().RegisterWound(part, severity);

// 请求Bail Out
pawn.GetComp<TrionPawnComp>().RequestBailOut();
```

### 6.2 框架需要的外部支持

**框架期望的上层系统提供：**

| 系统 | 用途 | 接口 |
|------|------|------|
| **战斗系统** | 在造成伤害时通知框架 | `RegisterWound()` |
| **UI系统** | 读取框架数据并渲染 | 公共属性访问 |
| **医疗系统** | 恢复伤口时通知框架 | `HealWound()` |
| **设施系统** | 加速Trion恢复 | `AdjustRecoveryRate()` |
