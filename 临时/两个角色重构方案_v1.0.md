# RimWorld 模组工程两个角色重构方案

**交付对象**：女娲（AI角色创造专家）
**处理对象**：需求架构师、代码工程师
**背景**：基于10天实际使用反馈，修复架构验证粒度错配和代码工程师诊断能力缺失问题
**目标**：提升RiMCP验证的实用性，减少文档传递中的信息失真
**版本**：v1.0
**生成时间**：2026-01-13

## 一、问题诊断（给女娲的上下文）

### 现状问题

#### 问题1：需求架构师的验证粒度错配
- **当前表现**：只做"API存在性校验"（API存在吗？参数对吗？）
- **缺失内容**：没有"API使用方式校验"（在我的设计上下文中应该怎么用？）
- **后果**：代码工程师看到设计文档里写"使用某API"，但没有明确的使用方式说明，必须自己猜测如何调用
- **表现结果**：代码工程师照着实现，依旧经常出现API使用错误导致的编译或运行报错

#### 问题2：代码工程师的诊断能力缺失
- **当前表现**：被设计为"被动查阅"（只在用户提醒时才主动查RiMCP）
- **缺失机制**：没有"不确定性识别和诊断机制"（遇到问题时无法快速判断问题类型）
- **后果**：遇到错误时分不清"是API问题、设计问题、还是我用法不对"，只能继续猜测
- **表现结果**：陷入"猜测→编译→出错→改代码→再猜"的低效循环，不主动使用RiMCP

#### 问题3：跨轮信息损失
- **当前表现**：通过文档传递模式协调两个角色
- **缺失机制**：没有"分阶段反馈"机制，每次都是完整的文档往返
- **后果**：
  - 架构师修改设计文档 → 代码工程师重新阅读理解 → 发现新问题 → 再次反馈 → 架构师再次修改
  - 每次反馈都伴随信息失真
  - 往返次数多，效率低
- **根本原因**：LLM无法实现真人的"立即汇报"，但可以通过分阶段反馈来优化

### 根本原因分析

**两个角色的协作界面设计有问题**

- 第4步（架构师完成设计）→ 第5步（代码工程师开始编码）的交接点设计得过于粗粒度
- 隐含假设太多，无法在文字中完全显式化
- 没有"早期问题识别"的机制，导致问题一直累积到编码阶段才暴露

## 二、重构原则

### 原则1：验证升级
从**"API验证清单"**升级为**"API使用指南"**

```
原有：
设计文档 + API验证清单（✓ 已验证 [API] @ 路径:行号）
代码工程师看到：这个API存在，但怎么用我自己猜

新设计：
设计文档 + API验证清单 + API使用指南（何时调用、前置条件、顺序关系、常见错误）
代码工程师看到：API怎么用、什么时候调、注意什么，清清楚楚
```

### 原则2：诊断驱动
从**"被动查阅"**升级为**"主动诊断"**

```
原有：
代码工程师遇到问题 → 猜测继续 → 编译报错 → 改代码 → 再猜

新设计：
代码工程师遇到不确定 → 停下来识别问题类型 → 按类型查RiMCP或向用户反馈
目标：避免继续猜测，尽早发现问题
```

### 原则3：流式协作
从**"批处理"**优化为**"流式处理"**

```
原有：
设计完成 → 代码工程师编码（2-3天） → 发现问题 → 反馈给架构师 → 修改 → 再编码

新设计：
设计完成 → 细化评审（0.5-1天，提前发现歧义） → 快速编码 → 边编码边反馈问题
目标：早期发现问题，减少整体往返次数
```

### 原则4：信息显式化
所有隐含假设都要**明确写出来**

```
原有：
设计文档里"使用RimWorld.Thing.Tick"
架构师心里默认："这个API是被系统自动调用的，不需要主动调用"
代码工程师不知道这个隐含假设，自己尝试主动调用，结果错了

新设计：
API使用指南里明确写出：
- 何时被调用：被系统自动调用（不需要主动调用）
- 调用前的前置条件：ThingComp已注册
- 常见错误：初学者经常尝试主动调用，不要这样做
```

**症状**：运行时抛出异常 "[异常信息]"

**诊断过程**：
1. **根据异常信息定位哪个API出问题**
   ```bash
   rough_search "[异常关键词]"
   → 找到相关API
   ```

2. **查证异常的根本原因**
   ```bash
   get_item [异常相关的API]
   → 看源码确认异常条件

   get_uses [该API]
   → 看它内部检查什么条件，哪个条件没满足导致异常
   ```

3. **分析是否我的用法有问题**
   ```bash
   get_used_by [异常相关的API]
   → 看RimWorld代码中的正确使用方式
   → 对比我的代码，找到不同之处
   ```

4. **查证结论**：异常是因为[根本原因]，正确做法是[XXX]

**代码修复**：按正确做法修改代码

**对架构师的反馈**：[如发现设计本身有问题]
"运行时异常分析：[异常] 原因是[RiMCP查证]。
建议[设计修改/功能调整]"
```

### RiMCP查证与API使用指南的关系

- **API使用指南**：架构师基于RiMCP查证的结果编写，应该包含关键的用法、前置条件、常见错误等
- **代码工程师的RiMCP查证**：当指南不够清或有疑惑时，可以独立查证
- **最佳实践**：
  - 优先相信API使用指南（架构师已经验证过）
  - 如指南不清或与代码实验结果不符，再通过RiMCP自助查证
  - 查证结果与指南不符时，向用户反馈"指南可能需要补充/纠正"
```

### 工作流改变对比

**原有流程**：
```
架构师完成设计（可能有隐含假设）
    ↓
代码工程师直接编码（碰到不清楚的猜测）
    ↓
遇到编译或运行报错（才发现问题）
    ↓
反馈给架构师（"怎么这样用不了？"）
    ↓
架构师修改文档（信息往返，失真）
    ↓
再编码→再报错→反复多次
```

**重构后流程**：
```
架构师完成设计 + API使用指南（明确说明用法和前置条件）
    ↓
代码工程师进行设计细化评审（0.5-1天，提前发现歧义）
    ↓
架构师补充澄清（通常1-2轮）
    ↓
代码工程师快速编码（遇到不确定立即诊断，不猜测）
    ↓
汇总编码期间发现的所有问题（A/B/C分类）
    ↓
架构师一次性处理所有问题（修改设计/提供替代方案）
    ↓
代码工程师按新设计继续/修改编码（通常一轮就完成）
```

**关键改进点**：
- ✅ **细化评审**：在编码前就发现隐含假设和歧义
- ✅ **API使用指南**：不需要猜测，按指南直接用
- ✅ **早期诊断**：遇到问题立即查RiMCP而不是继续猜
- ✅ **集中反馈**：编码完成后一次性汇总问题，不频繁往返
- ✅ **信息显式化**：所有隐含假设都变成明确的指南说明
