## 功能 #1：Trion能量管理系统

### 基本信息

| 项 | 内容 |
|-----|--------|
| **功能ID** | F1.1 |
| **功能名** | Trion Energy Management |
| **优先级** | P0 |
| **依赖功能** | 无 |

---

### 功能概述

**一句话说明**：管理战斗体的Trion能量四要素（Capacity、Reserved、Consumed、Available），并计算实时可用量。

**目的**：为所有其他功能提供能量数据基础，跟踪消耗/恢复/可用的状态。

**触发条件**：战斗体存在期间持续工作。

---

### 工作流程

```
【初始化阶段】
1. CompTrion.PostSpawnSetup() 执行
   ├─ 如果是新建（不是读档）
   │  ├─ 调用 Strategy.GetInitialTalent(this)
   │  │  └─ Strategy负责：生成/读取天赋，返回TalentGrade?
   │  ├─ 如果返回null → 跳过容量计算，由Strategy自己管理
   │  └─ 如果返回TalentGrade → 框架调用 RecalculateCapacity(talentGrade)
   │     └─ 根据天赋等级设置Capacity（应用层可提供查表）
   │
   ├─ 初始 Reserved = 0
   ├─ 初始 Consumed = 0
   └─ 计算 Available = Capacity - Reserved - Consumed

   └─ 如果是读档
      └─ Capacity从存档恢复，无需重新计算

【生成战斗体时】
2. 应用层Strategy通知框架占用量
   ├─ 框架更新 Reserved（从Available扣除）
   └─ 计算新的 Available

【持续运行中】
3. 每个CompTick执行一次（每60 Tick执行）
   ├─ 累加所有消耗源（基础消耗 + 组件消耗 + 泄漏）
   ├─ 执行 Consume(totalConsumption)
   │  └─ Consumed += totalConsumption
   │  └─ Consumed = Math.Min(Consumed, Capacity)
   └─ 重算 Available = Capacity - Reserved - Consumed

【恢复阶段】
4. Strategy允许恢复时
   ├─ 执行 Recover(recoveryAmount)
   │  └─ Consumed -= recoveryAmount
   │  └─ Consumed = Math.Max(Consumed, 0)
   └─ 重算 Available

【耗尽检查】
5. 检查 Available <= 0
   ├─ true: 调用 Strategy.OnDepleted()（框架交给应用层处理）
   └─ false: 继续正常运行
```

### 涉及的数据

#### 输入数据

| 数据名 | 类型 | 来源 | 说明 |
|--------|------|------|------|
| Capacity | float | 应用层 | 战斗体的能量容量上限（固定值） |
| RecoveryRate | float | 应用层 | 每次恢复计算的基础恢复量 |
| totalConsumption | float | CompTick计算 | 本次Tick的总消耗（基础+组件+泄漏） |

#### 处理数据

| 数据名 | 计算方式 | 说明 |
|--------|---------|------|
| Available | Capacity - Reserved - Consumed | 实时可用能量 |
| consumedThisTick | Strategy.GetBaseMaintenance() + sum(mount.consumption) + LeakRate | 本Tick需要消耗的总量 |

#### 输出数据

| 数据名 | 类型 | 存储位置 | 说明 |
|--------|------|---------|------|
| Consumed | float | CompTrion属性 | 累计已消耗量（需保存到存档） |
| Reserved | float | CompTrion属性 | 当前占用量（需保存到存档） |
| Available | float | CompTrion计算属性 | 实时派生值，只读 |

---

### 公开接口

```csharp
// 公开方法
public void Consume(float amount)
{
    Consumed += amount;
    if (Consumed > Capacity) Consumed = Capacity;
}

public void Recover(float amount)
{
    Consumed -= amount;
    if (Consumed < 0) Consumed = 0;
}

public float GetAvailable() => Capacity - Reserved - Consumed;
```

---

### 特殊情况处理

#### 情况1：Capacity为0或负数

**触发条件**：应用层设置了无效的Capacity

**预期行为**：
- 框架不检查，应用层负责保证有效值
- 如果Capacity < 0，Available会异常，由应用层Strategy处理异常

#### 情况2：Consumed溢出

**触发条件**：单次消耗超过Capacity

**预期行为**：
- Consume()中进行Clamp：`Consumed = Math.Min(Consumed, Capacity)`
- Available最小为0

#### 情况3：恢复量超过已消耗量

**触发条件**：Recover(100)但Consumed仅为50

**预期行为**：
- Recover()中进行Clamp：`Consumed = Math.Max(Consumed, 0)`
- 不会导致Available超过Available

---

### 验收标准

- [ ] CompTrion正确持有Capacity、Reserved、Consumed三个属性
- [ ] Available正确计算为 Capacity - Reserved - Consumed
- [ ] Consume()正确更新Consumed并Clamp到Capacity
- [ ] Recover()正确减少Consumed并Clamp到0
- [ ] 四要素在存档/读档中正确序列化

## 功能 #2：生命周期策略接口 (ILifecycleStrategy)

### 基本信息

| 项 | 内容 |
|-----|--------|
| **功能ID** | F1.2 |
| **功能名** | Lifecycle Strategy Interface |
| **优先级** | P0 |
| **依赖功能** | F1.1 |

---

### 功能概述

**一句话说明**：定义ILifecycleStrategy接口，框架通过此接口从应用层Strategy获取数值和执行业务逻辑。

**目的**：实现"依赖注入"模式，让框架与应用层完全解耦。

**触发条件**：战斗体生成时注入Strategy；此后每个CompTick都会调用Strategy方法。

---

### 工作流程

```
【初始化】
1. 战斗体生成前
   ├─ 应用层通过SetStrategy()注入Strategy实现
   └─ 框架保存Strategy引用

【使用阶段】
2. 战斗体运行中，框架调用Strategy各方法
   ├─ CompTick()时调用：
   │  ├─ Strategy.GetBaseMaintenance() → 获取基础消耗
   │  ├─ Strategy.OnTick() → 让Strategy执行自己的逻辑
   │  └─ Strategy.ShouldRecover() / GetRecoveryModifier() → 决定恢复
   │
   ├─ 伤害时调用：
   │  ├─ Strategy.ShouldInterceptDamage() → 是否拦截
   │  └─ Strategy.OnDamageTaken() → 伤害处理逻辑
   │
   └─ 部位丧失时调用：
      └─ Strategy.OnBodyPartLost() → 部位丧失处理

【回调方式】
3. 应用层通过StrategyProvider回调提供Strategy实现
   └─ 框架在PostSpawnSetup中调用：
      var strategy = CompTrion.StrategyProvider(this, parent);
      SetStrategy(strategy);
```

### 接口定义

```csharp
public interface ILifecycleStrategy
{
    // 初始化钩子
    void OnInitialize();

    // 每Tick调用
    void OnTick();

    // 伤害处理
    bool ShouldInterceptDamage();
    void OnDamageTaken(float damageAmount, BodyPartRecord hitPart);

    // 部位丧失
    void OnBodyPartLost(BodyPartRecord part);

    // 耗尽处理
    void OnDepleted();

    // 恢复控制
    bool ShouldRecover();
    float GetRecoveryModifier();

    // 数据提供
    float GetBaseMaintenance();
    float GetBaseOutputPower();
    float ModifyOutputPower(float baseOutput);
    float GetLeakageRate(VirtualWound wound);
    float GetBodyPartLossLeakage(BodyPartRecord part);
}
```

### Strategy选择机制

```csharp
// 框架中的委托（未初始化，由应用层启动时设置）
public static Action<CompTrion, ThingWithComps> StrategyProvider { get; set; }

// 应用层初始化（在mod启动时执行）
public static void InitializeFramework()
{
    CompTrion.StrategyProvider = (comp, thing) =>
    {
        // 应用层的策略选择逻辑
        if (thing.def.HasModExtension<TrionSoldierMarker>())
            return new Strategy_TrionSoldier(comp);

        if (thing is Pawn)
            return new Strategy_HumanCombatBody(comp);

        if (thing is Building)
            return new Strategy_TrionBuilding(comp);

        return null;  // 如果无法确定，返回null（框架会报错）
    };
}

// 框架在PostSpawnSetup中使用
public override void PostSpawnSetup(bool respawningAfterLoad)
{
    var strategy = CompTrion.StrategyProvider(this, parent);
    if (strategy == null)
        throw new InvalidOperationException("StrategyProvider返回null");

    SetStrategy(strategy);
    strategy.OnInitialize();
}
```

---

### 特殊情况处理

#### 情况1：StrategyProvider未初始化

**触发条件**：应用层忘记在启动时设置StrategyProvider

**预期行为**：
- 框架在PostSpawnSetup中抛出异常
- 明确错误信息便于调试

#### 情况2：Strategy返回null

**触发条件**：StrategyProvider无法确定单位类型

**预期行为**：
- 框架抛出异常
- 应用层需要补充判断逻辑

---

### 验收标准

- [ ] ILifecycleStrategy接口定义清晰，包含所有必需方法
- [ ] StrategyProvider委托正确运作
- [ ] 框架正确调用Strategy各方法
- [ ] 应用层能通过StrategyProvider注入自定义Strategy

## 功能 #3：虚拟伤害拦截系统

### 基本信息

| 项 | 内容 |
|-----|--------|
| **功能ID** | F1.3 |
| **功能名** | Virtual Damage Interception |
| **优先级** | P0 |
| **依赖功能** | F1.1, F1.2 |

---

### 功能概述

**一句话说明**：框架提供伤害拦截入口，将指向战斗体的伤害转化为Trion消耗。

**目的**：实现虚拟战斗系统，让战斗体受伤而肉身不受伤。

**触发条件**：Pawn受到伤害，且拥有CompTrion且战斗体激活时。

---

### 工作流程

```
【伤害拦截流程】
1. 伤害发生时（Pawn.TakeDamage或PreApplyDamage）
   ├─ Harmony补丁拦截
   └─ 调用 comp.PreApplyDamage(damageAmount, hitPart)

2. 框架在PreApplyDamage中
   ├─ 询问 Strategy.ShouldInterceptDamage()
   │  ├─ true: 继续处理虚拟伤害
   │  └─ false: 让伤害走原系统
   │
   └─ 调用 Strategy.OnDamageTaken(damageAmount, hitPart)
      └─ 应用层决定：
         ├─ 消耗多少Trion
         ├─ 创建什么hediff
         ├─ 产生什么特殊效果
         └─ 框架不干预

3. 原系统伤害
   └─ 框架补丁返回true（允许继续）
      └─ 由应用层Strategy决定是否继续原流程
```

### 框架代码接口

```csharp
// 框架中的公开方法
public void PreApplyDamage(float damageAmount, BodyPartRecord hitPart)
{
    if (strategy.ShouldInterceptDamage())
    {
        strategy.OnDamageTaken(damageAmount, hitPart);
        // 框架不决定是否真的拦截，应用层Strategy决定
    }
}
```

### Harmony补丁（应用层负责提供）

```csharp
// 应用层需要提供这个补丁
[HarmonyPatch(typeof(Pawn_HealthTracker), "PreApplyDamage")]
public class Patch_InterceptDamage
{
    static bool Prefix(Pawn ___pawn, DamageInfo dinfo)
    {
        var comp = ___pawn.TryGetComp<CompTrion>();
        if (comp != null)
        {
            comp.PreApplyDamage(dinfo.Amount, dinfo.HitPart);
            // Prefix返回true = 让原系统继续
            // 应用层可根据需要返回false来拦截原系统
        }
        return true;
    }
}
```

---

### 特殊情况处理

#### 情况1：战斗体破裂时伤害继续到肉身

**触发条件**：Strategy.OnDamageTaken()中Trion不足以消耗全部伤害

**预期行为**：
- 应用层Strategy负责决定如何处理剩余伤害
- 框架不决定，可选择：
  - 消耗所有Trion，剩余伤害走原系统
  - 完全拦截伤害
  - 削弱伤害后再走原系统

#### 情况2：伤害为0

**触发条件**：damageAmount <= 0

**预期行为**：
- 框架仍然调用Strategy.OnDamageTaken()
- 应用层Strategy应该忽略或记录日志

---

### 验收标准

- [ ] PreApplyDamage()能正确被框架调用
- [ ] Strategy.ShouldInterceptDamage()返回值被正确使用
- [ ] Strategy.OnDamageTaken()接收正确的参数
- [ ] 应用层Harmony补丁能与框架配合工作
- [ ] 战斗体与肉身分离（伤害不影响肉身血量）

## 功能 #4：组件挂载系统 (TriggerMount)

### 基本信息

| 项 | 内容 |
|-----|--------|
| **功能ID** | F1.4 |
| **功能名** | Component Mount State Machine |
| **优先级** | P0 |
| **依赖功能** | F1.1 |

---

### 功能概述

**一句话说明**：TriggerMount管理挂载点的组件生命周期，实现状态机（未连接→休眠→激活）。

**目的**：组织组件的激活/休眠/切换，计算组件消耗。

**触发条件**：战斗体存在且有组件装备时。

---

### 状态机流程

```
【状态定义】
未连接 (Disconnected)
   ↓ 战斗体生成
休眠 (Dormant)
   ↓ 激活（支付费用+引导）
激活中 (Activating)
   ↓ 引导完成
激活 (Active)
   ↑↓ 切换到其他组件
休眠 (Dormant)
   ↓ 战斗体解除或部位毁
未连接 (Disconnected)

【激活流程】
1. 玩家请求激活组件（应用层调用TryActivate）
   ├─ 检查是否已有其他组件在激活中：
   │  ├─ 有: 拒绝激活（返回false）
   │  └─ 无: 继续
   │
   ├─ 检查是否有足够的Trion支付激活费用：
   │  ├─ 不足: 拒绝激活，恢复已扣Trion（返回false）
   │  └─ 足够: 继续
   │
   ├─ 检查是否满足OutputPower要求：
   │  ├─ 不满足: 拒绝激活（返回false）
   │  └─ 满足: 继续
   │
   ├─ 进入"激活中"状态
   ├─ 扣除激活费用
   └─ 设置引导剩余Tick数

2. 每Tick进行激活引导
   ├─ 引导计数器递减
   ├─ 引导完成时：
   │  ├─ 旧组件关闭（如有）
   │  ├─ 新组件激活
   │  └─ 状态转为"激活"

3. 持续消耗
   ├─ TickAndGetConsumption()计算此挂载点的消耗
   └─ 返回当前激活组件的持续消耗值
```

### 涉及的数据

#### 输入数据

| 数据名 | 类型 | 来源 | 说明 |
|--------|------|------|------|
| componentDef | TriggerComponentDef | 应用层 | 要激活的组件定义 |
| activationCost | float | 组件Def | 激活此组件的一次性消耗 |
| sustainCost | float | 组件Def | 激活后每Tick的持续消耗 |
| activationDelay | int | 组件Def | 引导需要的Tick数 |
| requiredOutputPower | float | 组件Def | 激活所需的最小输出功率 |

#### 处理数据

| 数据名 | 计算方式 | 说明 |
|--------|---------|------|
| activationTicksRemaining | 每Tick递减 | 记录引导还需多少Tick |

#### 输出数据

| 数据名 | 类型 | 说明 |
|--------|------|------|
| activeTrigger | TriggerComponent | 当前激活的组件 |
| activatingTrigger | TriggerComponent | 正在引导的组件 |
| state | TriggerState | 当前状态 |
| consumption | float | 本Tick此挂载点的消耗 |

**触发条件**：引导期间（Activating状态）部位被毁

**预期行为**：

## 功能 #5：框架调度核心 (CompTick)

### 基本信息

| 项 | 内容 |
|-----|--------|
| **功能ID** | F1.5 |
| **功能名** | Framework Scheduling - CompTick |
| **优先级** | P0 |
| **依赖功能** | F1.1, F1.2, F1.4 |

---

### 功能概述

**一句话说明**：CompTick是框架的"心跳"，每60 Tick执行一次，协调消耗计算、恢复、策略更新等全流程。

**目的**：保证框架各模块有序运作，性能可控。

**触发条件**：每60 Tick自动执行（RimWorld固有机制）。

---

### 工作流程

```
【完整CompTick流程】
if (TicksGame % 60 != 0) return;  // 仅每60 Tick执行一次

// ===== 第1步：累加所有消耗源 =====
float totalConsumption = 0;

1.1 基础消耗（由Strategy提供）
    totalConsumption += strategy.GetBaseMaintenance();

1.2 组件消耗（由每个TriggerMount计算）
    foreach (var mount in Mounts)
    {
        totalConsumption += mount.TickAndGetConsumption();
    }

1.3 泄漏消耗（由Strategy根据伤口计算）
    totalConsumption += LeakRate;  // Strategy维护此值

// ===== 第2步：执行消耗 =====
2.1 调用框架接口
    Consume(totalConsumption);
    // CompTrion.Consumed += totalConsumption
    // CompTrion.Consumed = Math.Min(Consumed, Capacity)

2.2 检查是否已耗尽
    if (Available <= 0)
    {
        // 触发耗尽逻辑（由Strategy处理）
        strategy.OnDepleted();
        return;  // 本次Tick不再继续
    }

// ===== 第3步：执行恢复 =====
3.1 询问是否允许恢复
    if (strategy.ShouldRecover())
    {
        3.1.1 获取恢复修正系数
              float modifier = strategy.GetRecoveryModifier();

        3.1.2 计算实际恢复量
              float recoveryAmount = RecoveryRate * modifier;

        3.1.3 执行恢复
              Recover(recoveryAmount);
              // CompTrion.Consumed -= recoveryAmount
              // CompTrion.Consumed = Math.Max(Consumed, 0)
    }

// ===== 第4步：策略Tick =====
4.1 让Strategy执行自己的定期逻辑
    strategy.OnTick();
    // 应用层可在此检查状态、更新AI等

// ===== 第5步：输出功率重算 =====
5.1 重算OutputPower（基于Strategy）
    RecalculateOutputPower();
    // OutputPower = Clamp(strategy.ModifyOutputPower(...), 0, 100)
```

### 性能考虑

```
【Tick间隔】
- 每60 Tick执行一次 = 间隔1秒（游戏时间）
- 目的：避免每Tick都做复杂计算
- 代价：消耗/恢复精度以秒级计算

【理想的消耗计算】
- 消耗速率 = X点/秒
- 实际消耗 = X点 * (60 Tick / 60) = X点/60 Tick
- 每Tick消耗 = X点 / 60

【应用层需要注意】
- 所有数值应该按"每60 Tick"的单位定义
- 例如：GetBaseMaintenance()返回的是60 Tick期间的消耗
```

---

### 涉及的数据

#### 输入数据

| 数据名 | 类型 | 来源 | 说明 |
|--------|------|------|------|
| GetBaseMaintenance() | float | Strategy | 此次Tick的基础消耗 |
| mount.TickAndGetConsumption() | float | TriggerMount | 每个挂载点的消耗 |
| LeakRate | float | Strategy维护 | 泄漏速率（由伤口决定） |
| ShouldRecover() | bool | Strategy | 是否允许恢复 |
| RecoveryRate | float | 应用层 | 基础恢复速率 |
| GetRecoveryModifier() | float | Strategy | 恢复修正系数 |

#### 处理数据

| 数据名 | 计算方式 | 说明 |
|--------|---------|------|
| totalConsumption | 基础 + 组件 + 泄漏 | 本次Tick的总消耗 |
| recoveryAmount | RecoveryRate * modifier | 实际恢复量 |

#### 输出数据

| 数据名 | 类型 | 说明 |
|--------|------|------|
| Consumed | float | 已消耗量（保存到存档） |
| Available | float | 可用量（派生值） |
| OutputPower | float | 输出功率（UI显示） |

---

### 特殊情况处理

#### 情况1：Recovery导致Available超过Capacity

**触发条件**：不会发生（Recover()中Clamp到0，不会负数）

**预期行为**：无特殊处理需要

#### 情况2：Strategy.OnDepleted()中修改Consumed

**触发条件**：Strategy在OnDepleted()中调用Recover()等

**预期行为**：
- 框架允许Strategy修改状态
- CompTick返回后不再检查
- 下一Tick会重新计算

#### 情况3：CompTick期间Strategy异常

**触发条件**：Strategy方法抛出异常

**预期行为**：
- 异常应该被应用层处理（框架不try-catch）
- 异常信息对调试有帮助

- [ ] 耗尽判定正确
