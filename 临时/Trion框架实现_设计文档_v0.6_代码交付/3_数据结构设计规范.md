## 1. CompTrion 数据结构

### 1.1 类定义

```csharp
public class CompTrion : ThingComp
{
    // 所有属性都必须在Scribe中序列化
}
```

### 1.2 属性列表

#### 能量相关属性

| 属性名 | 类型 | 范围 | 默认值 | 必须 | 说明 |
|--------|------|------|--------|------|------|
| **Capacity** | float | > 0 | 应用层初始化 | 是 | Trion能量容量上限，由天赋决定 |
| **Reserved** | float | 0 ~ Capacity | 0 | 是 | 当前占用量 |
| **Consumed** | float | 0 ~ Capacity | 0 | 是 | 当前已消耗量 |
| **LeakRate** | float | >= 0 | 应用层设置 | 是 | 泄漏速率（由Strategy维护） |
| **RecoveryRate** | float | >= 0 | 应用层设置 | 是 | 基础恢复速率 |
| **OutputPower** | float | 0 ~ 100 | 应用层计算 | 是 | 输出功率（仅供UI显示） |

#### 天赋相关说明

| 项 | 说明 |
|-----|-----|
| **天赋定义位置** | 隐藏属性，存储在Pawn.modData或应用层指定的位置 |
| **天赋等级** | S / A / B / C / D / E |
| **初始化时机** | CompTrion.PostSpawnSetup() 新建时执行 |
| **初始化流程** | 框架调用 Strategy.GetInitialTalent(this) → 应用层返回TalentGrade? → 框架调用RecalculateCapacity(talent)计算Capacity |
| **读档时** | 从存档直接恢复Capacity值，无需重新计算 |
| **框架责任** | 定义初始化接口，调用Strategy回调，计算Capacity |
| **应用层责任** | 实现天赋生成/读取逻辑，提供查表函数（TalentGrade→Capacity） |

#### 策略相关属性

| 属性名 | 类型 | 说明 |
|--------|------|------|
| **strategy** | ILifecycleStrategy | 当前使用的策略实现（私有，不序列化） |

#### 挂载点列表

| 属性名 | 类型 | 说明 |
|--------|------|------|
| **Mounts** | List<TriggerMount> | 所有组件挂载点 |

### 1.3 派生属性（只读）

| 属性名 | 计算公式 | 说明 |
|--------|---------|------|
| **Available** | Capacity - Reserved - Consumed | 实时可用量 |

### 1.4 存档序列化规范

```csharp
public override void ExposeData()
{
    // 必须序列化的字段
    Scribe_Values.Look(ref Capacity, "capacity", 0f);
    Scribe_Values.Look(ref Reserved, "reserved", 0f);
    Scribe_Values.Look(ref Consumed, "consumed", 0f);
    Scribe_Values.Look(ref LeakRate, "leakRate", 0f);
    Scribe_Values.Look(ref RecoveryRate, "recoveryRate", 0f);
    Scribe_Values.Look(ref OutputPower, "outputPower", 0f);

    // Mounts序列化（复杂类型，需要自定义处理）
    Scribe_Collections.Look(ref Mounts, "mounts", LookMode.Deep);

    // Strategy不序列化（在PostSpawnSetup重新注入）
}

// 约束检查
public override void PostLoadData()
{
    // 验证加载数据的有效性
    if (Capacity <= 0)
        Log.Warning("CompTrion: Capacity <= 0, 设置默认值");
    if (Reserved < 0 || Reserved > Capacity)
        Reserved = 0;
    if (Consumed < 0 || Consumed > Capacity)
        Consumed = 0;
}
```

## 2. TriggerMount 数据结构

### 2.1 类定义

```csharp
public class TriggerMount
{
    // 挂载点的完整状态
}
```

### 2.2 属性列表

| 属性名 | 类型 | 范围 | 默认值 | 必须 | 说明 |
|--------|------|------|--------|------|------|
| **slotTag** | string | 任意非空字符串 | - | 是 | 槽位标签（如"LeftHand", "RightHand"） |
| **boundPart** | BodyPartRecord | 有效部位或null | null | 否 | 绑定的身体部位（可选） |
| **equippedList** | List<TriggerComponent> | 0 ~ N | 空列表 | 是 | 装备在此挂载点的所有组件 |
| **activeTrigger** | TriggerComponent | equippedList中或null | null | 是 | 当前激活的组件 |
| **activatingTrigger** | TriggerComponent | equippedList中或null | null | 是 | 正在引导的组件 |
| **activationTicksRemaining** | int | 0 ~ 1000 | 0 | 是 | 引导剩余Tick数 |
| **isFunctional** | bool | true/false | true | 是 | 该挂载点是否工作（部位毁时为false） |

### 2.3 方法

| 方法名 | 返回值 | 参数 | 说明 |
|--------|--------|------|------|
| **TryActivate** | bool | TriggerComponentDef | 尝试激活组件 |
| **TickActivation** | void | - | 推进激活计数器 |
| **TickAndGetConsumption** | float | - | 返回本Tick消耗 |
| **OnPartDestroyed** | void | - | 部位被毁时调用 |
| **DisconnectAll** | void | - | 断开所有组件 |

### 2.4 存档序列化规范

```csharp
// TriggerMount本身是Deep模式序列化的
// 需要手动实现Scribe方法

public void ExposeData()
{
    Scribe_Values.Look(ref slotTag, "slotTag", "");
    Scribe_References.Look(ref boundPart, "boundPart");

    // equippedList 序列化
    Scribe_Collections.Look(ref equippedList, "equippedList", LookMode.Deep);

    // 激活状态序列化
    Scribe_References.Look(ref activeTrigger, "activeTrigger");
    Scribe_References.Look(ref activatingTrigger, "activatingTrigger");
    Scribe_Values.Look(ref activationTicksRemaining, "activationTicksRemaining", 0);
    Scribe_Values.Look(ref isFunctional, "isFunctional", true);
}
```

## 3. TriggerComponent 数据结构

### 3.1 类定义

```csharp
public class TriggerComponent
{
    // 组件实例（每个组件一个实例）
}
```

### 3.2 属性列表

| 属性名 | 类型 | 说明 |
|--------|------|------|
| **def** | TriggerComponentDef | 组件定义（Def配置） |
| **state** | TriggerState | 当前状态（Disconnected/Dormant/Activating/Active） |

### 3.3 虚拟方法（由应用层重写）

| 方法名 | 调用时刻 | 说明 |
|--------|---------|------|
| **OnEquipped** | 装备时 | 应用层可在此初始化 |
| **OnActivated** | 激活时 | 应用层可在此创建视觉效果 |
| **OnTick** | 每Tick | 应用层可在此更新逻辑 |
| **OnDeactivated** | 关闭时 | 应用层可在此清理效果 |
| **OnUnequipped** | 卸载时 | 应用层可在此销毁资源 |

### 3.4 状态定义

```csharp
public enum TriggerState
{
    Disconnected,  // 未连接（未生成战斗体或部位毁）
    Dormant,       // 休眠（战斗体存在但组件未激活）
    Activating,    // 激活中（正在引导）
    Active         // 激活（工作中）
}
```

### 3.5 存档序列化规范

```csharp
public override void ExposeData()
{
    // Def不序列化（通过defName查找）
    Scribe_Defs.Look(ref def, "def");

    // 状态序列化
    Scribe_Values.Look(ref state, "state", TriggerState.Disconnected);
}
```

## 4. TriggerComponentDef 配置结构

### 4.1 XML结构

```xml
<TriggerComponentDef>
    <defName>ComponentName</defName>
    <label>组件显示名</label>
    <description>组件描述</description>

    <!-- 能量相关 -->
    <occupyingCapacity>10</occupyingCapacity>      <!-- 占用值 -->
    <activationCost>5</activationCost>             <!-- 激活费用 -->
    <sustainCost>0</sustainCost>                   <!-- 持续消耗 -->
    <activationDelay>1</activationDelay>           <!-- 引导Tick数 -->
    <requiredOutputPower>0</requiredOutputPower>   <!-- 输出功率要求 -->
</TriggerComponentDef>
```

### 4.2 字段说明

| 字段名 | 类型 | 范围 | 必须 | 说明 |
|--------|------|------|------|------|
| **occupyingCapacity** | float | >= 0 | 是 | 装备此组件时占用的Trion容量 |
| **activationCost** | float | >= 0 | 是 | 激活此组件的一次性消耗 |
| **sustainCost** | float | >= 0 | 是 | 激活后每60 Tick的持续消耗 |
| **activationDelay** | int | >= 1 | 是 | 激活引导需要的Tick数 |
| **requiredOutputPower** | float | >= 0 | 是 | 激活所需的最小输出功率 |

### 4.3 约束规则

- occupyingCapacity + sustainCost >= 0（都非负）
- activationDelay >= 1（至少1 Tick）
- activationCost >= 0

## 7. 应用层需要补充的数据结构

### 7.1 Strategy实现所需

应用层的每个Strategy实现可能需要维护自己的数据：

```csharp
public class Strategy_HumanCombatBody : ILifecycleStrategy
{
    private CompTrion comp;

    // 应用层自己的数据（框架不关心）
    private int hitCount = 0;           // 击中计数
    private List<VirtualWound> wounds;  // 虚拟伤口列表
    private Hediff_SlowdownCounter slowdownHediff;
    private float leakageDueToWounds = 0f;

    // 框架会调用的方法
    public float GetLeakageRate(VirtualWound wound)
    {
        // 应用层根据VirtualWound计算泄漏速率
        return wound.severity * 2f;
    }

    public void OnDamageTaken(float damageAmount, BodyPartRecord hitPart)
    {
        // 应用层在此处理伤害
        // 可能创建VirtualWound、创建hediff等
    }
}
```

### 7.2 数据容器选择

应用层可以选择以下任何方式存储数据：

| 方式 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **Comp属性** | 框架能访问 | 复杂数据难以管理 | 简单状态 |
| **Hediff** | RimWorld原生 | 需要与原系统集成 | 效果叠加 |
| **自定义容器** | 完全自由 | 需要手动序列化 | 复杂逻辑 |

### 8.2 数据验证

```csharp
public override void PostLoadData()
{
    // 验证Capacity有效性
    if (Capacity <= 0)
    {
        Log.Error("CompTrion: Capacity无效");
        Capacity = 1000;  // 设置默认值
    }

    // 验证Consumed有效性
    if (Consumed > Capacity)
        Consumed = Capacity;
    if (Consumed < 0)
        Consumed = 0;

    // 验证Reserved有效性
    if (Reserved > Capacity)
        Reserved = Capacity;
    if (Reserved < 0)
        Reserved = 0;
}
```

## 9. 字段约束检查清单

创建/加载CompTrion时应检查：

- [ ] Capacity > 0
- [ ] Reserved >= 0 && Reserved <= Capacity
- [ ] Consumed >= 0 && Consumed <= Capacity
- [ ] LeakRate >= 0
- [ ] RecoveryRate >= 0
- [ ] OutputPower >= 0 && OutputPower <= 100
- [ ] Mounts != null（不能为null）
- [ ] 每个Mount的slotTag非空
- [ ] 每个Mount的equippedList != null

## 10. 性能考虑

### 10.1 内存占用

```
单个CompTrion占用：
├─ Capacity, Reserved, Consumed, LeakRate等浮点数：8个 * 4B = 32B
├─ Mounts列表：取决于Mount数量
│  └─ 单个Mount：~200B（包括状态和组件列表指针）
│  └─ 单个Component：~50B
└─ Strategy引用：8B

例：10个Mounts，每个5个组件：
= 32B + (10 * 200B) + (50 * 50B) + 8B
= 32 + 2000 + 2500 + 8
≈ 4.5KB 每个CompTrion
```

### 10.2 序列化性能

- Mounts使用LookMode.Deep可能影响加载速度
- 建议应用层为Mounts设置合理的初始容量
