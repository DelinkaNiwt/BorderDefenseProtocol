## 🔧 遇到的问题与解决方案

### 问题1：编译工具选择错误
**症状**：
- 最初尝试使用 MSBuild + PowerShell 编译失败
- 错误信息：`CommandNotFound`（编码问题和路径问题）
- MSBuild 文件存在但无法调用

**根本原因**：
- ProjectTrion.csproj 是 .NET Framework 4.7.2 项目
- 环境中 PowerShell 编码问题导致路径解析失败
- 过度依赖 MSBuild，忽视了 dotnet CLI 的能力

**解决方案**：
- 改用 `dotnet build ProjectTrion.csproj -c Release`
- dotnet CLI 能够自动处理 .NET Framework 项目
- 避免了路径和编码问题

**学到的经验**：
- dotnet build 是更通用的编译工具
- 当工具链出现问题时，应考虑使用更高层的抽象工具
- 之前的成功也是用 dotnet，应该保持一致

### 问题2：DLL 版本过旧
**症状**：
- 修改代码后，编译输出的 DLL 修改时间早于代码修改时间
- 编译命令执行但 DLL 未更新

**根本原因**：
- 第一次尝试编译失败，没有重新编译
- 旧的 DLL 仍在 1.6/Assemblies/ 目录中

**解决方案**：
- 确认代码修改时间：19:30:07
- 重新执行 dotnet build 编译
- 新的 DLL 生成时间：19:35+

**学到的经验**：
- 需要对比代码修改时间和编译输出时间
- 编译失败或修改后必须重新编译

## 💡 关键设计决策

### 1. 新建/读档分支分离
**为什么**：
- 新建时需要通过天赋计算Capacity
- 读档时Capacity已在存档中，无需重新计算
- 两个分支的初始化逻辑完全不同

**实现**：
```csharp
if (respawningAfterLoad)  // 读档分支
{
    // 直接恢复Strategy，无需重算
}
else  // 新建分支
{
    // 调用GetInitialTalent()获取天赋
    // 调用RecalculateCapacity()计算Capacity
}
```

### 2. 两层天赋初始化方式
**方式A（推荐）**：应用层通过委托提供天赋→容量映射
```csharp
CompTrion.TalentCapacityProvider = (talent) =>
{
    return talent switch { ... }
};
```

**方式B**：应用层在Strategy中直接设置Capacity
```csharp
public TalentGrade? GetInitialTalent(CompTrion comp)
{
    var talent = GenerateTalent();
    comp.Capacity = CalculateCapacity(talent);
    return talent;
}
```

**为什么两种方式**：
- 方式A：框架控制计算流程，应用层只需提供查表
- 方式B：应用层全权控制，更灵活
- 两种都支持，给应用层选择空间

### 3. null 天赋的处理
**设计**：Strategy可以返回null
**含义**：框架跳过自动计算，由应用层自己设置Capacity
**好处**：
- 支持应用层的特殊初始化逻辑
- 向后兼容旧的Strategy实现
- 增加系统灵活性
