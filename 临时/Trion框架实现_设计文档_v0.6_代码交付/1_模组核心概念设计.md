## 2. 核心理念

### 2.1 设计出发点

**痛点**：
- RimWorld原生系统基于血量和装备，难以支持"虚拟能量"驱动的战斗体系
- 现有模组多采用硬编码方案，难以扩展到新的单位类型
- 缺乏统一的、可配置的能量管理框架

**解决方案**：
- 创建一个"纯框架"系统，只定义接口和通用机制
- 所有具体的Strategy实现由应用层决定
- 通过ILifecycleStrategy接口支持任意数量的单位类型
- 提供虚拟伤害系统，将伤害转化为Trion消耗

### 2.2 核心价值主张

**独特性**：
- **框架纯净性**：框架仅定义通用机制，不包含任何应用层代码
- **完全可扩展性**：新增单位类型无需修改框架代码
- **灵活的Strategy模式**：应用层通过回调决定策略选择和数值
- **虚拟战斗系统**：战斗体与肉身完全分离，可实现快照/回滚机制

## 3. 目标玩法

### 3.1 核心玩法流程

```
【玩家操作】
步骤1：装备Trion触发器
  ↓ 系统识别：该单位有Trion天赋吗？
步骤2：生成战斗体（可选装备组件）
  ↓ 系统计算：占用多少Trion容量？
步骤3：激活组件（支付激活费用）
  ↓ 系统管理：消耗、恢复、泄漏的实时计算
步骤4：战斗/移动（组件工作）
  ↓ 系统拦截：伤害转化为Trion消耗
步骤5：Trion耗尽或主动解除
  ↓ 系统回复：恢复到快照状态
最终结果：玩家体验虚拟战斗能力与肉身保护的二重性
```

### 3.2 使用场景示例

**场景1**：人类殖民者装备战斗体
- 背景：玩家有一个S级Trion天赋的殖民者
- 用户操作：穿戴触发器 → 激活战斗体 → 装备组件 → 战斗
- 期望结果：虚拟战斗体替代肉身参战，肉身保持冻结；战斗结束后快照恢复
- 实际表现：通过框架的虚拟伤害系统实现，应用层定制数值和规则

**场景2**：Trion兵单位自动战斗
- 背景：游戏中有人造Trion兵
- 用户操作：Trion兵作为NPC自动激活战斗能力
- 期望结果：应用层Strategy实现自动化策略，框架只负责调度
- 实际表现：框架提供钩子，应用层Strategy决定行为

## 4. 设计目标与约束

### 4.1 设计目标（What to achieve）

| 目标 | 优先级 | 说明 |
|------|--------|------|
| 框架纯净性 | P0 | 框架不包含任何应用层代码或具体实现 |
| 完全可扩展 | P0 | 支持无限数量的单位类型，无需修改框架 |
| 接口清晰 | P0 | ILifecycleStrategy接口明确定义框架与应用层的交互 |
| 性能可控 | P1 | CompTick性能影响<1ms/Tick（60FPS条件下） |
| 虚拟系统完整 | P1 | 伤害拦截、部位丧失、快照回滚等机制健全 |
| 灵活的参数化 | P1 | 所有数值由应用层XML/Strategy定义，框架不硬编码 |

### 4.2 设计约束（What NOT to do）

| 约束 | 原因 |
|------|------|
| 不在框架中实现Strategy子类 | 违反"框架只定义接口"的原则 |
| 不硬编码任何数值参数 | 所有参数由应用层提供，确保灵活性 |
| 不改变RimWorld原生伤害系统 | 通过拦截点而非修改原系统实现虚拟伤害 |
| 不实现特定的战斗逻辑 | 战斗逻辑完全由应用层Strategy决定 |
| 不限制应用层的数据结构 | 应用层可自由选择数据容器（hediff、comp、自定义） |

## 5. 核心概念定义

### 5.1 概念词表

| 术语 | 定义 | 例子 |
|------|------|------|
| **Trion能量** | 一种虚拟能量资源，储存于生物或人造物内，驱动所有战斗功能 | 人类殖民者的Trion天赋等级决定容量 |
| **战斗体** | 由Trion构成的虚拟身体形态，完全覆盖并替代肉身进行战斗 | 激活触发器后，殖民者进入战斗体状态 |
| **组件(Component)** | 可装备到触发器的功能模块，可激活或休眠 | 远程武器、护盾、能力组件等 |
| **触发器(Trigger)** | 可穿戴的装备，用于生成和管理战斗体，包含多个组件挂载点 | 左手/右手/特殊槽位的组件挂载点 |
| **生命周期策略(ILifecycleStrategy)** | 框架调用的接口，由应用层实现，决定单位的战斗体如何工作 | 不同的Strategy实现处理人类、兵、建筑等 |
| **虚拟伤害系统** | 框架机制，将指向战斗体的伤害转化为Trion消耗，不影响肉身血量 | 战斗体受100点伤害 → 消耗100点Trion |
| **快照机制** | 生成战斗体时保存肉身状态，解除时恢复快照 | 战斗结束后肉身自动恢复到战斗前状态 |
| **泄漏速率(LeakRate)** | 伤口导致的Trion持续泄漏速度，由Strategy计算 | 重伤口可能导致每Tick泄漏0.5点Trion |

## 6. 功能范围

### 6.1 第一个版本要包含什么

**功能A：Trion能量管理**
- 复杂度：中等
- 影响范围：全局（所有战斗体都依赖）
- 依赖：无
- 说明：管理Trion四要素（Capacity、Reserved、Consumed、Available）的数据结构和计算接口

**功能B：生命周期策略接口**
- 复杂度：中等
- 影响范围：框架级（应用层都需要实现）
- 依赖：功能A
- 说明：定义ILifecycleStrategy接口，框架通过此接口获取数值和执行业务逻辑

**功能C：虚拟伤害拦截**
- 复杂度：中等
- 影响范围：战斗系统
- 依赖：功能A、B
- 说明：框架提供PreApplyDamage钩子，应用层Strategy决定拦截和处理

**功能D：组件挂载系统**
- 复杂度：中等
- 影响范围：组件管理
- 依赖：功能A
- 说明：TriggerMount状态机管理组件的生命周期（未连接→休眠→激活）

**功能E：框架调度核心**
- 复杂度：中等
- 影响范围：全局（CompTick）
- 依赖：功能A、B、D
- 说明：CompTick中的通用调度逻辑（消耗计算→消耗执行→恢复→状态检查）

### 6.2 有意不包含的功能

- **具体Strategy实现**：Strategy_HumanCombatBody等由应用层实现
- **硬编码数值**：所有参数由应用层提供
- **战斗逻辑**：护盾判定、debuff创建等由应用层Strategy实现
- **UI相关代码**：UI层由应用层独立实现
- **存档序列化**：ExposeData由应用层自行管理

## 7. 关键决策与权衡

### 7.1 设计决策表

| 决策 | 选项A | 选项B | **选择** | 理由 | RiMCP验证 |
|------|-------|-------|---------|------|----------|
| **框架包含Strategy实现** | 包含具体类 | 仅定义接口 | 仅定义接口 | 违反"框架纯净"原则，包含具体类会导致难以扩展 | ✓ 已验证 |
| **参数硬编码位置** | 框架中 | 应用层 | 应用层 | 框架硬编码违反灵活性，所有参数应由应用层提供 | ✓ 已验证 |
| **伤害拦截方式** | 修改原系统 | 使用钩子 | 使用钩子 | 钩子方式不破坏原游戏系统，通过Harmony补丁实现 | ✓ 已验证 |
| **Strategy选择机制** | 框架决定 | 应用层决定 | 应用层决定 | 应用层通过回调告诉框架选择，框架不决定具体实现 | ✓ 已验证 |
| **数据容器** | 框架定义 | 应用层自由 | 应用层自由 | 应用层可使用hediff、comp或其他容器维护自己的数据 | ✓ 已验证 |

## 8. 非功能需求

### 8.1 性能指标

| 指标 | 目标 | 验证方法 |
|------|------|---------|
| CompTick消耗 | <1ms/Tick | 60FPS下测试单个战斗体的Tick性能 |
| 内存占用 | <10MB/战斗体 | 监测活跃战斗体的内存使用 |
| 初始化延迟 | <10ms | 测试战斗体生成到第一个Tick的时间 |

### 8.2 兼容性

| 项 | 要求 |
|-----|--------|
| **RimWorld版本** | 1.6.4633 |
| **其他模组兼容** | 与大多数combat/balance模组兼容（通过应用层Strategy管理） |
| **游戏模式** | 所有模式（和平/死亡难度均支持） |
| **语言支持** | 支持多语言标签（通过XML标签系统） |

### 8.3 可维护性

- **易于扩展**：新增单位类型只需创建新的Strategy实现，无需修改框架
- **配置灵活性**：所有数值通过XML和应用层Strategy定制
- **国际化**：框架层不含硬编码文本，所有文本由应用层提供

## 9. 与其他系统的关系

### 9.1 依赖关系

```
Trion框架
├─ RimWorld 核心系统
│  ├─ Verse.ThingComp（组件系统）
│  ├─ Verse.Pawn_HealthTracker（伤害系统）
│  ├─ Verse.HediffMaker（Hediff系统）
│  └─ Verse.Thing（基础物体系统）
└─ 应用层Strategy实现（注入依赖）
```

### 9.2 影响范围

- **玩家操作流程**：提供虚拟战斗能力的二重体验（肉身+战斗体）
- **游戏平衡**：框架本身不改变原游戏平衡；应用层可通过数值配置调整
- **其他模组**：框架通过Harmony补丁拦截伤害，可能与其他伤害相关模组产生交互（应由应用层Strategy管理）

## 11. 风险和考虑

### 11.1 技术风险

| 风险 | 可能性 | 影响 | 缓解方案 |
|------|--------|------|---------|
| Harmony补丁冲突 | 中 | 伤害拦截失效 | 应用层管理补丁，使用标准补丁模式 |
| 性能瓶颈 | 低 | 战斗卡顿 | 在CompTick中优化算法，避免复杂计算 |
| 存档兼容性 | 低 | 旧存档加载失败 | 应用层在Strategy中处理版本迁移 |

### 11.2 设计风险

| 风险 | 可能性 | 影响 | 缓解方案 |
|------|--------|------|---------|
| 应用层理解偏差 | 中 | 实现不符预期 | 提供详细的设计文档和代码示例 |
| 接口不充分 | 低 | 应用层功能受限 | 在反馈中迭代接口 |
