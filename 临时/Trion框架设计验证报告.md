### 1.2 部分覆盖但需补充的设定 ⚠️

#### 1.2.1 输出功率系统（OutputPower）

**设定文档要求**：
- 定义：操控Trion的质量指标，数值/等级
- 决定因素：Trion天赋（主要）、小人特性（辅助）、装备加成（临时）
- 作用1：限制能力释放（如弧月旋空需要≥50）
- 作用2：影响组件表现（武器伤害、射程、护盾减伤比例、AOE范围、持续时间）

**草案现状**：
- ✅ 提供了OutputPower字段（CompTrion数据层）
- ❌ 未设计OutputPower的计算逻辑
- ❌ 未设计OutputPower对Trigger的影响接口
- 标记为"应用层实现"

**问题**：
- 如果应用层实现，框架层是否需要提供`IOutputPowerModifier`接口？
- 能力释放判定（如弧月旋空）是框架层还是应用层负责？

**建议**：
- 框架层提供OutputPower字段和计算接口
- 应用层通过扩展计算逻辑（天赋、特性、装备加成）
- TriggerComponent提供`RequiredOutputPower`配置项
- TriggerMount.TryActivate()时检查OutputPower是否满足

#### 1.2.2 护盾机制详细设计

**设定文档要求**：
- 判定流程：受攻击 → 概率判定 → 成功则按比例减伤 → 扣护盾费用 → 不注册伤口
- 示例：原始10伤害，50%减伤，实际消耗5 Trion，无伤口无泄漏

**草案现状**：
- ✅ 提到了护盾判定逻辑（草案7.2节）
- ❌ 未详细设计护盾的具体流程
- ❌ 未说明护盾判定在Harmony拦截中的位置

**建议**：
- 在Strategy.OnDamageTaken()之前添加护盾判定步骤
- 设计护盾配置参数：`blockChance`（概率）、`damageReduction`（减伤比例）、`blockCost`（抵挡费用）
- 成功抵挡后修改damageAmount，继续走虚拟伤害流程

---

#### 1.2.3 部位损毁影响

**设定文档要求**：
- 手臂被毁 → 该手挂载点所有组件断连 + 增加泄漏速率
- 单腿被毁 → 移速下降 + 增加泄漏速率
- 双腿被毁 → 失去移动能力 + 增加泄漏速率

**草案现状**：
- ✅ 提到了`TriggerMount.OnPartDestroyed()`方法
- ❌ 未设计部位损毁的检测机制
- ❌ 未设计部位损毁后泄漏速率的计算

**建议**：
- Strategy.OnDamageTaken()中检测hitPart是否核心部位
- 如果是绑定部位（如左手），调用对应TriggerMount.OnPartDestroyed()
- OnPartDestroyed()设置`isFunctional = false`，所有组件进入Disconnected
- 增加泄漏速率：`LeakRate += 部位严重度相关值`

---

#### 1.2.4 组件激活引导机制

**设定文档要求**：
- 激活需要引导时间（1-5 Tick）
- 同槽位切换：新组件引导 → 引导完成 → 旧组件瞬间关闭 → 新组件激活
- **关键**：引导期间旧组件仍产生消耗

**草案现状**：
- ✅ 提到了`activationDelay`字段（1-5 Tick）
- ❌ 未说明引导期间旧组件是否还在消耗
- ❌ 未设计引导状态的表示

**建议**：
- TriggerComponent增加状态：`Activating`（引导中）
- TriggerMount.TryActivate()时：
  1. 新组件进入Activating状态
  2. 旧组件保持Active状态（继续消耗）
  3. 引导完成后：旧组件Deactivate() → 新组件进入Active
- CompTick()计算消耗时，Activating状态的组件不计算持续消耗

### 1.3 完全遗漏的设定 ❌

#### 1.3.1 Trion恢复机制

**设定文档要求**：
- 人类殖民者：仅自然恢复，保证稀缺性
- 恢复速率影响因素：
  - 饱食度：饥饿时不恢复（硬性规则）
  - 小人特性：模组新增特性影响基础恢复速率
  - 休养舱：加速debuff消退，不提升恢复速率
  - 恢复增幅建筑：附近时提升恢复速率（待设计）

**草案现状**：
- ❌ 完全未提及Trion恢复机制
- ❌ 未设计自然恢复速率
- ❌ 未设计恢复影响因素

**影响**：
- 无法实现Trion的自然恢复
- 无法实现饥饿时停止恢复

**建议**：
- CompTrion增加字段：`recoveryRate`（基础恢复速率）
- CompTick()中每60 Tick执行恢复逻辑：
  ```
  if (Consumed < Capacity) {
      if (!IsHungry()) {  // 检查饱食度
          Consumed -= recoveryRate * modifiers;
      }
  }
  ```
- 恢复速率调整接口：由应用层实现（特性、建筑加成）
