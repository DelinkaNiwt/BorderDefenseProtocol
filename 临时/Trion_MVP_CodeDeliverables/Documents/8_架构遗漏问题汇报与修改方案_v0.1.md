### 问题2: 战斗体生成流程不清楚

#### 代码工程师反馈
```
流程图显示：先装备触发器 → 然后"生成战斗体" → 才能激活组件
现在的代码中没有"战斗体生成"的触发点
应该是通过什么触发？是检测仪？还是建筑？还是手动命令？
```

#### 设计文档中的现状

**文档1**（第1章）说：
- "避免全局Harmony补丁的架构"
- "植入腺体 → 初始化CompTrion → 首次检测 → 生成天赋 → 装备组件 → 生成战斗体"

**文档2**（功能4）说：
- "玩家点击'激活战斗体'按钮"
- "CompTrion.GenerateCombatBody() 被调用"

**问题**：触发方式不一致，不清楚是哪个UI/命令触发

#### 应该包含的内容

**触发路径应该是**：

1. **触发源**：玩家主动操作（不是自动）
2. **触发点**：需要定义明确的UI/命令
3. **前置条件**：
   - ✓ Pawn有CompTrion
   - ✓ Pawn有天赋（已检测）
   - ✓ 有装备触发器和组件
4. **触发后流程**：
   - 保存快照
   - 标记_isInCombat = true
   - 计算Reserved
   - 开始消耗循环

#### 修改方案

✅ **选项A**（推荐）：通过**Gizmo（Pawn检查栏按钮）**触发
- 优点：玩家容易找到、直观
- 代码位置：CompTrion.GetGizmos()
- 显示条件：有CompTrion且已有天赋

✅ **选项B**：通过**建筑（配置台）**触发
- 优点：游戏内交互感好
- 缺点：需要建筑在附近

✅ **选项C**：通过**命令行（dev工具）**触发
- 优点：简单快速
- 缺点：不符合正常玩法

**建议**：采用**选项A + 选项C**的组合
- 正常游戏：Gizmo按钮
- 测试调试：命令行

### 问题5: 伤害→泄漏的映射

#### 代码工程师反馈
```
流程图显示受伤会增加Trion消耗（+2/单位、+5/单位等）
这需要Harmony补丁来拦截伤害并计算泄漏率
```

#### 设计文档中的现状

**文档3**（第4.2章）定义了泄漏规则：

**一次性消耗**：
```
受到伤害 → 伤害值(1:1转化) + 伤口注册
```

**持续消耗**：
```
伤口泄漏 = 轻伤+2/单位时间，断肢+5/单位时间
```

**问题**：定义了规则，但没有说明**如何拦截伤害**

#### 应该包含的内容

**关键问题**：RimWorld中伤害是如何处理的？

根据RiMCP验证，伤害处理流程：
```
Pawn.TakeDamage()
  → Pawn_HealthTracker.Notify_GeneralInjury()
  → HediffSet.AddDirect()
  → 创建Hediff_Injury
```

**需要决定**：在哪里拦截？

**方案选项**：

**方案A**：使用Harmony补丁拦截Pawn.TakeDamage()
```csharp
[HarmonyPatch(typeof(Pawn), nameof(Pawn.TakeDamage))]
public static class Patch_PawnTakeDamage
{
    public static void Prefix(Pawn __instance, DamageInfo dinfo)
    {
        CompTrion comp = __instance.GetComp<CompTrion>();
        if (comp?.IsInCombat ?? false)
        {
            // 拦截伤害，转化为Trion消耗
            dinfo.SetAmount(0); // 肉身不受伤
            comp.Consume(dinfo.Amount); // Trion消耗
        }
    }
}
```

**方案B**：使用CompTrion.Notify_PawnInjury()回调
- 当Hediff_Injury被添加时调用
- 缺点：无法阻止原始伤害

**方案C**：完全禁用肉身伤害，转为虚拟伤害
- 在战斗体激活时标记Pawn
- 所有伤害都转化为Trion消耗
- 需要更复杂的Harmony补丁

#### 修改方案

✅ **选择方案**：采用**方案A**（最简洁）

✅ **在代码指南中添加**：Harmony补丁的具体实现

✅ **在文档中说明**：伤害拦截的设计理由

## 附录：代码工程师的问题回答

### Q: MVP是要验证这整个流程吗？还是只验证其中某些部分？

**A**: MVP要验证**全流程**。但根据优先级分为：

**必须实现**（Core MVP）：
- ✓ 植入→初始化
- ✓ 天赋检测
- ✓ 装备组件
- ✓ 生成战斗体
- ✓ 消耗计算
- ✓ 快照恢复

**应该实现**（Extended MVP）：
- ✓ 伤口泄漏
- ✓ Bail Out脱离
- ✓ 读档持久化

**所以答案是**：是的，全流程都需要实现。

### Q: 如果是全流程，上述这些系统都需要实现吗？

**A**: 是的。需要实现的系统包括：

| 系统 | 需要 | 原因 |
|------|------|------|
| 触发器系统 | ✓ 是 | 组件必须挂载在触发器上 |
| 战斗体生成 | ✓ 是 | 框架的核心机制 |
| 快照机制 | ✓ 是 | 战斗体破裂时需要恢复 |
| 传送锚 | ✓ 是 | Bail Out的目标 |
| 伤害拦截 | ✓ 是 | 虚拟伤害系统的基础 |

所有5个系统都是**架构必需的**，不是可选的。
