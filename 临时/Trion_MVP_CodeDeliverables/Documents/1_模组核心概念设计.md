# Trion_MVP 模组核心概念设计 v0.3

## 摘要

**Trion_MVP** 是一个面向 RimWorld 1.6 的验证性模组，目的是验证 ProjectTrion 框架的核心功能是否能正确运行、与游戏系统兼容、性能无问题。

MVP 不追求完整的游戏体验，而是通过 **10个设计精良的测试场景** 来穷尽框架的各个功能维度。

---

## 设计目标

### 主目标（优先级1）
✓ **验证框架可行性**：ProjectTrion_Framework 的7大核心功能块能否正常工作
✓ **验证API有效性**：所有设计中使用的RimWorld API是否真实可用
✓ **验证数据持久化**：存档/读档是否能正确保留天赋和Capacity数据

### 次目标（优先级2）
✓ 发现框架中的设计缺陷或性能问题
✓ 为应用层开发积累最佳实践
✓ 建立可复用的MVP测试模板

### 不是目标（明确排除）
✗ 制作完整可玩的模组
✗ 设计复杂的UI系统
✗ 实现20+种组件和技能树
✗ 性能极限优化

## 核心创新点

### 1. 避免全局Harmony补丁的架构

**问题**：大多数系统级模组需要用Harmony补丁拦截Pawn生成，这对性能和兼容性有风险。

**创新方案**：
- Pawn 生成时什么都不做
- 玩家给Pawn植入 **"Trion腺体"** 时，CompTrion 自动初始化
- 首次用 **"天赋检测仪"** 时，天赋随机生成并存储

**优势**：
- ✓ 零全局补丁侵入
- ✓ 符合游戏逻辑（天赋确实在生成时拥有，但延迟初始化）
- ✓ UI交互自然（扫描仪是天然的初始化触发点）
- ✓ 易于卸载测试

---

### 2. 清晰的天赋→容量映射

**问题**：框架层定义了 TalentGrade 枚举，但应用层怎么根据天赋计算Capacity？

**创新方案**：
- 框架提供 `CompTrion.TalentCapacityProvider` 静态委托
- Mod启动时设置查表函数：TalentGrade → float
- 首次生成天赋后，框架自动调用委托重算Capacity

**优势**：
- ✓ 框架保持纯净（不知道具体的Capacity值）
- ✓ 应用层可灵活定义天赋等级映射
- ✓ 天赋系统可分离迭代

---

### 3. 三层植入物系统

**第一层：植入获取**
```
殖民者 + Trion腺体（植入物）→ CompTrion初始化
```

**第二层：天赋生成**
```
首次扫描检测仪 → 随机生成天赋 → 存入Pawn.modData
```

**第三层：能量系统**
```
装备组件 → Reserved计算 → 激活战斗体 → 消耗计算 → Available递减 → Bail Out
```

---

### 4. 触发器系统与装备体系的融合

**问题**：如何以模块化方式管理多个能力组件，支持热切换和冲突检测，同时与RimWorld穿戴系统深度融合？

**创新方案**：
- 触发器（TrionTrigger）继承自 **Apparel**，通过原生穿戴系统进行装备/脱装
- Pawn 通过 `pawn.apparel.Wear(trigger)` 装备触发器
- 触发器上可装配多个 **TriggerComponent**（5个具体实现：弧月、护盾生成器、炸裂弹、变色龙隐身、紧急脱离）
- 通过 TriggerMount 管理触发器的组件列表、槽位限制、冲突检测
- 配置建筑（Building_TriggerConfigBench）提供UI界面进行组件装配

**优势**：
- ✓ 复用RimWorld穿戴系统，无需自定义引用逻辑
- ✓ 组件系统模块化，易于扩展
- ✓ 槽位限制保证平衡性（左/右/特殊槽位）
- ✓ 冲突检测避免不合理的组件组合
- ✓ 战斗体激活后组件锁定，保证公平性
- ✓ 快照和传送锚系统支持高级脱离机制
- ✓ 穿戴系统自动处理持久化和存档恢复

## 什么是 Trion（概念层）

### 定义
Trion 是一种假想的生物能能量系统，由 ProjectTrion_Framework 管理：

```
Trion = {
  Capacity:  总容量（由天赋决定）
  Reserved:  被组件占用的量
  Consumed:  已消耗的量（不可逆）
  Available: 可用量 = Capacity - Reserved - Consumed
}
```

### 天赋（Talent）
- **定义**：Pawn固有的Trion容量等级（S/A/B/C/D/E）
- **来源**：Pawn生成时就拥有（但我们延迟初始化）
- **可见性**：玩家看不见（除非用检测仪或开dev模式）
- **影响**：决定Capacity大小（S→2000, E→1000）

### 触发器（Trigger）
- **定义**：装备品，可装配多个 TriggerComponent
- **作用**：通过Gizmo按钮激活战斗体，消耗Trion能量
- **限制**：战斗体激活后组件锁定，不能热修改

### 触发器组件（TriggerComponent）
- **定义**：可装配到触发器的能力单元（武器、护盾、隐身等）
- **分类**：近战/远程/辅助/特殊
- **特征**：每个都有消耗值（Reserved Cost）

### 战斗体（Combat Body）
- **定义**：激活触发器后，Pawn进入的特殊强化状态
- **快照（Snapshot）**：生成前保存Pawn当前状态（生命值、部位、装备、伤口）
- **恢复**：战斗体摧毁后恢复到快照（战斗中的新伤口不继承）

### 战斗体解除的两种方式

#### 安全解除（Manual Deactivation）
- **触发条件**：玩家主动点击"解除战斗体"按钮 + Available > 0
- **流程**：
  - Reserved 完全返还（不转化为 Consumed）
  - 无任何 debuff
  - Pawn状态完整恢复到快照
- **名称**：称为"解除"而非"破裂"，表示受控的、安全的退出

#### 破裂（Crashed Combat Body）
- **触发条件**（任一满足）：
  - Available ≤ 0 自动触发
  - 玩家手动触发 Bail Out
  - 供给器官被摧毁
  - 任何其他被迫情况
- **流程**：
  1. 快照恢复（如启用）
  2. Reserved 累加到 Consumed（不返还）
  3. 施加 debuff（心理创伤）
  4. Bail Out 传送（如有效锚点）
- **特征**：Reserved→Consumed 的转化意味着能量"消耗掉"了，无法恢复

### 传送锚（Bail Out Anchor）
- **定义**：建筑物，标记为破裂时的传送目标位置
- **作用**：战斗体破裂且 Bail Out 触发时，自动查询最近的传送锚
- **传送逻辑**：
  - 优先级1：有效的传送锚（isActive && 未被摧毁）
  - 优先级2：无有效锚点时，传送到基地附近（RCellFinder.RandomSpotJustOutsideColony）
  - 无法传送：返回失败状态，Pawn留在原位置

## 设计约束

### 世界观约束
- 天赋在Pawn生成时就拥有，但隐藏（只有检测仪能看）
- 植入腺体是获得Trion系统的**必要条件**（不是自动的）
- 战斗体摧毁后肉身伤口不继承（但心理状态继承）

### 技术约束
- 不使用全局Harmony补丁进行Pawn生成拦截
- 天赋存储在Pawn.modData（RimWorld标准方式）
- 组件装备在战斗体激活后锁定（不能热修改）
- 性能目标：消耗计算每60Tick <5ms

### 功能约束
- MVP只实现5个示例组件（不是完整的20+）
- 不实现复杂的组件升级、融合、突变系统
- 不实现带UI的配置系统（仅命令行/dev工具）
- 不支持多Strategy同时存在（MVP阶段）

## 架构5层

```
第1层：植入获取
  Hediff_TrionGland.PostAdd() → CompTrion初始化
        ↓
第2层：天赋初始化
  Building_TrionDetector.ScanPawn() → 生成天赋 → Capacity重算
        ↓
第3层：组件装备
  TriggerMount.TryAddComponent() → Reserved更新
        ↓
第4层：战斗与消耗
  GenerateCombatBody() → TickConsumption() → TriggerBailOut()
        ↓
第5层：快照恢复
  DestroyCombatBody() → Snapshot.RestoreToPawn()
```
