### 【问题1】TriggerComponent 的数据模型 ✅ **新方案已采纳**

**代码工程师的疑问**
```
在9_代码实现指南中：
- TriggerComponent 继承自 ThingWithComps
- TriggerComponent_Def 继承自 ThingDef
- 但原MVP设计中5个组件定义为ThingDef
关系不清楚
```

**架构师的决策**

✅ **采纳简化方案：只有一个 TriggerComponent**

```
定义统一为 ThingDef（物品定义）
  ├─ TriggerComponent_ArcBlade (ThingDef)
  ├─ TriggerComponent_Shield (ThingDef)
  ├─ TriggerComponent_ExplosiveBullet (ThingDef)
  ├─ TriggerComponent_Chameleon (ThingDef)
  └─ TriggerComponent_BailOut (ThingDef)

同一个物品，两种状态：
  ├─ 状态A：在地图/背包中 → 普通物品
  └─ 状态B：装备到触发器 → 虚拟组件对象

核心认识：对玩家而言这就是同一个东西，无需区分
```

**具体改法**

在代码中，TriggerComponent 改为**轻量级数据类**：

```csharp
public class TriggerComponent : IExposable
{
    private ThingDef _def;        // 引用 TriggerComponent_* 的ThingDef
    private bool _isActive;

    // 从 defName 创建
    public static TriggerComponent CreateFromDef(string defName)
    {
        var comp = new TriggerComponent
        {
            _def = DefDatabase<ThingDef>.GetNamed(defName)
        };
        return comp;
    }

    // 获取定义数据
    public ThingDef Def => _def;
    public float ReservedCost => _def.GetModExtension<TriggerComponentExt>()?.reservedCost ?? 10f;
    public float TrionCost => _def.GetModExtension<TriggerComponentExt>()?.trionCost ?? 1f;
}
```

**XML 中的定义保持不变**：

```xml
<ThingDef ParentName="TriggerComponentBase">
    <defName>TriggerComponent_ArcBlade</defName>
    <label>弧月</label>
    <description>Trion能量驱动的近战刃具</description>
    <thingClass>ProjectTrion_MVP.TriggerComponent</thingClass>
    <modExtensions>
        <li Class="ProjectTrion_MVP.TriggerComponentExt">
            <reservedCost>10</reservedCost>
            <trionCost>1</trionCost>
        </li>
    </modExtensions>
</ThingDef>
```

**确认事项**
- [x] 不需要区分"物品版本"和"组件版本"
- [x] TriggerComponent 不继承 ThingWithComps
- [x] TriggerComponent 在TriggerMount中只作为数据对象，不是游戏对象
- [x] 参数通过 ModExtension 存储（参考 Core 模组的做法）

### 【问题2】Pawn 装备 TrionTrigger 的拥有关系 ✅ **新方案已采纳**

**代码工程师的疑问**
```
TrionTrigger 是 ThingWithComps，如何与 Pawn 建立"装备关系"？
应该用 equipment.Primary / apparel.WornApparel / inventory / 自定义引用？
```

**架构师的决策**

✅ **采纳方案：TrionTrigger 继承自 Apparel**

**理由**：
1. 复用 RimWorld 现有的穿戴系统（最简洁）
2. UI 自动显示"已装备触发器"
3. 存档/读档自动处理穿戴关系
4. 无需自定义引用逻辑

**具体改法**

```csharp
public class TrionTrigger : Apparel  // 改为 Apparel
{
    private List<TriggerMount> _mounts = new List<TriggerMount>();

    public List<TriggerMount> Mounts => _mounts;

    // 获取穿着者（由 Apparel 系统提供）
    public Pawn GetWearer() => wearer;

    public CompTrion GetWearerCompTrion() => wearer?.GetComp<CompTrion>();

    // 自定义初始化
    public override void Wear(Pawn pawn)
    {
        base.Wear(pawn);

        // 初始化3个挂载点
        if (_mounts.Count == 0)
        {
            _mounts.Add(new TriggerMount(this, "LeftHand", 4));
            _mounts.Add(new TriggerMount(this, "RightHand", 4));
            _mounts.Add(new TriggerMount(this, "Special", 1));
            Log.Message("[Trion] 触发器已穿戴，挂载点初始化完成");
        }
    }

    // 序列化（Apparel 系统自动处理穿戴状态）
    public override void ExposeData()
    {
        base.ExposeData();
        Scribe_Collections.Look(ref _mounts, "mounts", LookMode.Deep);
    }
}
```

**XML 定义**（关键）：

```xml
<ThingDef ParentName="ApparelBase">
    <defName>TrionTrigger_Standard</defName>
    <label>标准触发器</label>
    <thingClass>ProjectTrion_MVP.TrionTrigger</thingClass>

    <!-- Apparel系统参数 -->
    <apparel>
        <bodyPartGroups>
            <li>Torso</li>  <!-- 穿在躯干 -->
        </bodyPartGroups>
        <layers>
            <li>OnSkin</li>  <!-- 贴身穿戴 -->
        </layers>
    </apparel>

    <!-- Trion自定义参数 -->
    <statBases>
        <Mass>2</Mass>
        <Flammability>0.5</Flammability>
    </statBases>
</ThingDef>
```

**Pawn 装备方式**：

```csharp
// 不需要任何特殊逻辑！
TrionTrigger trigger = /* ... 获取 */;
pawn.apparel.Wear(trigger);  // 就这么简单

// 穿戴系统会自动调用 TrionTrigger.Wear(pawn)
// 自动管理 wearer 属性
// 自动在存档中序列化穿戴状态
```

**确认事项**
- [x] TrionTrigger 继承 Apparel（不是 ThingWithComps）
- [x] 穿戴关系由 Apparel 系统全自动管理
- [x] 无需在 CompTrion 中持有引用（通过 GetWearer() 获取）

### 【问题3】战斗体激活的完整触发流程 ✅ **已确认**

**代码工程师的疑问**
```
玩家点击"激活战斗体"后，系统应该按什么顺序执行？
- 前置检查
- 快照捕获时机
- Reserved计算时机
- 消耗何时开始
```

**架构师的决策**

✅ **确认步骤顺序**（已补充详细时序图到文档4）

```
玩家点击 Gizmo 按钮"激活战斗体"
  ↓
【前置检查】
├─ ✓ Pawn.GetComp<CompTrion>() != null
├─ ✓ 已检测天赋（CompTrion.HasTalent）
├─ ✓ 穿戴了触发器（pawn.apparel.WornApparel 包含 TrionTrigger）
├─ ✓ Available > 0（可用值充足）
└─ ✓ !_isInCombat（未在战斗）
  ↓
【第1步】捕获快照（【重要】必须在 _isInCombat=true 之前）
├─ _snapshot = new CombatBodySnapshot()
├─ _snapshot.CaptureFromPawn(pawn)
└─ 保存：生命值、部位完整性、伤口、装备、背包
  ↓
【第2步】计算 Reserved
├─ RecalculateReserved()
├─ 从穿戴的 TrionTrigger 的所有 TriggerMount 求和
├─ _reserved = ∑(mount.totalReserved)
└─ 检查：Available >= _reserved（空间充足）
  ↓
【第3步】标记激活
├─ _isInCombat = true
├─ Log: "战斗体已激活，Available: {Available}"
└─ 消耗循环即刻开始（下一个 CompTick 触发）
  ↓
【第4步】策略回调
├─ _strategy.OnCombatBodyGenerated(this)
└─ （可执行特效、音效等）
  ↓
完毕，战斗体激活，Available 在下一个 60Tick 时开始递减
```

**关键时机**：
- ✓ 快照必须在激活前捕获（防止激活后才捕获状态改变）
- ✓ Reserved 计算完成前若不足应激活失败（返回警告）
- ✓ 激活后可查看已装备组件，但不能装卸

✅ **状态**：已在文档2、4中补充详细说明

### 【问题4】Strategy.GetBailOutTarget() 的 Map 访问 ✅ **已确认**

**代码工程师的疑问**
```
Strategy 是无状态工具类，无法直接访问 Map。
应该：
- 接收 Map 参数？
- 通过 Pawn.Map？
- 由 CompTrion 负责查询？
```

**架构师的决策**

✅ **确认方案：通过 Pawn.Map 访问**

```csharp
public class DefaultTrionStrategy : ILifecycleStrategy
{
    public IntVec3 GetBailOutTarget(Pawn pawn)
    {
        // Pawn 在战斗状态下一定在地图上
        Map map = pawn.Map;

        if (map == null)
            return IntVec3.Invalid;

        // 1. 优先：查询传送锚
        var anchors = map.listerThings.ThingsOfDef(
            DefDatabase<ThingDef>.GetNamed("Building_BailOutAnchor")
        );

        if (anchors.Count > 0)
        {
            // 选择最近的有效锚点
            Building_BailOutAnchor nearest = null;
            float nearestDist = float.MaxValue;

            foreach (var thing in anchors)
            {
                if (thing is Building_BailOutAnchor anchor && anchor.IsActive)
                {
                    float dist = pawn.Position.DistanceTo(anchor.Position);
                    if (dist < nearestDist)
                    {
                        nearestDist = dist;
                        nearest = anchor;
                    }
                }
            }

            if (nearest != null)
                return nearest.Position;
        }

        // 2. 备选：传送到基地附近（参考空投逻辑）
        return RCellFinder.RandomSpotJustOutsideColony(map);
    }
}
```

**确认事项**
- [x] 通过 `pawn.Map` 获取地图（Pawn 在战斗时总有 Map）
- [x] 查询逻辑在 Strategy 中（符合策略决策职责）
- [x] 优先级1：有效的传送锚 → 优先级2：基地附近

✅ **状态**：已确认，实现指南可直接参考

### 【问题5】快照恢复的触发条件与破裂定义 ✅ **新方案已采纳（重大改动）**

**代码工程师的疑问**
```
什么时候调用 RestoreToPawn()？
什么是"战斗体破裂"？
Bail Out 成功/失败的处理如何分别？
```

**架构师的决策**

✅ **采纳架构师的新理解：只有两种解除方式**

**重新定义**：

```
【方式1】安全解除（主动）

触发条件：Available > 0 且玩家点击"解除战斗体"按钮

处理流程：
├─ DestroyCombatBody(reason: Manual)
├─ 【不调用快照恢复】因为没有发生"破裂"
├─ Reserved 完全返还（Reserved 变为 0）
├─ Consumed 不变
├─ _isInCombat = false
├─ 无 debuff
└─ Log: "{Pawn}主动解除战斗体"

意义：玩家在安全的时候选择结束战斗状态，不有任何代价


【方式2】破裂（被迫）

触发条件（三选一）：
├─ Available <= 0（能量耗尽）
├─ 玩家手动触发 Bail Out 按钮
└─ Trion 供给器官被摧毁

处理流程：
├─ TriggerBailOut() 被调用（自动传送）
├─ 【快照恢复】RestoreToPawn() 被调用
│  ├─ 恢复 Pawn 状态到激活前
│  └─ 战斗中产生的新伤口被丢弃
├─ 【Reserved 流失】Reserved → Consumed（永久消耗）
├─ 【传送】GetBailOutTarget(pawn)
│  ├─ 优先级1：最近的有效传送锚
│  └─ 优先级2：基地附近（RCellFinder.RandomSpotJustOutsideColony）
├─ 【施加 debuff】Trion 枯竭（24小时内恢复-50%）
├─ _isInCombat = false
└─ Log: "{Pawn}战斗体破裂，执行紧急脱离"

意义：战斗体被迫解除，Pawn 失去所有 Reserved Trion，但获得安全


【关键区别】

安全解除 ≠ 破裂
├─ Available 值：> 0 vs ≤ 0
├─ 快照恢复：否 vs 是
├─ Reserved 处理：返还 vs 消耗
├─ 传送：否 vs 是
├─ debuff：否 vs 是
└─ 主要区别：是否是玩家主动控制


Bail Out 不会失败
├─ 无论什么情况，一旦触发就一定能传送
├─ 有锚点 → 传送到锚点
├─ 无锚点 → 传送到基地附近
└─ 结果：Bail Out 总是成功的脱离（但属于破裂处理）
```

**代码实现**：

```csharp
// CompTrion.cs

public void GenerateCombatBody()
{
    // ... 检查和初始化

    _snapshot = new CombatBodySnapshot();
    _snapshot.CaptureFromPawn(parent as Pawn);
    _isInCombat = true;

    Log.Message("[Trion] 战斗体已激活");
}

public void TriggerBailOut()  // 破裂处理
{
    if (!_isInCombat)
        return;

    // 快照恢复
    if (_snapshot != null)
    {
        _snapshot.RestoreToPawn(parent as Pawn);
    }

    // Reserved → Consumed
    _consumed += _reserved;
    _reserved = 0;

    // 获取传送目标并传送
    Pawn pawn = parent as Pawn;
    IntVec3 target = _strategy.GetBailOutTarget(pawn);

    pawn.Position = target;
    pawn.Notify_Teleported();

    // 施加 debuff
    var trionExhausted = HediffMaker.MakeHediff(
        DefDatabase<HediffDef>.GetNamed("Hediff_TrionExhausted"),
        pawn
    );
    pawn.health.AddHediff(trionExhausted);

    _isInCombat = false;
    Log.Message($"[Trion] {pawn.Name}战斗体破裂，传送到{target}");
}

public void DestroyCombatBody(bool isSafely = false)  // 安全解除
{
    if (!_isInCombat)
        return;

    if (isSafely && Available > 0)
    {
        // 安全解除：Reserved 返还
        _reserved = 0;  // 或者直接返还给 Available
        _isInCombat = false;

        Log.Message($"[Trion] {(parent as Pawn).Name}主动解除战斗体");
    }
    // 否则应该调用 TriggerBailOut() 处理破裂
}
```

**确认事项**
- [x] 只有"Available > 0 + 玩家手动"是安全解除
- [x] 其他所有情况都属于"破裂"（包括 Bail Out）
- [x] Bail Out 总是成功的（总能找到传送位置）
- [x] 快照恢复只在破裂时调用
- [x] Reserved 在破裂时流失，在安全解除时返还

✅ **状态**：已在文档1、2、4中完整更新

### 【问题6】嵌套对象的序列化架构 ✅ **已确认**

**代码工程师的疑问**
```
CompTrion→TriggerMount→TriggerComponent 的嵌套序列化如何设计？
对象所有权、引用关系、重新加载后的初始化？
```

**架构师的决策**

✅ **确认方案：轻量化序列化 + 运行时重建**

```csharp
// CompTrion.cs

public override void ExposeData()
{
    base.ExposeData();

    // 核心数据
    Scribe_Values.Look(ref _capacity, "capacity");
    Scribe_Values.Look(ref _reserved, "reserved");
    Scribe_Values.Look(ref _consumed, "consumed");
    Scribe_Values.Look(ref _isInCombat, "isInCombat");

    // 快照（完整序列化）
    Scribe_Deep.Look(ref _snapshot, "snapshot");

    // 组件列表（只存 defName，加载时重建）
    List<string> componentDefNames = new List<string>();

    if (Scribe.mode == LoadSaveMode.Saving)
    {
        // 保存时：提取所有装备的组件 defName
        TrionTrigger trigger = parent.apparel.WornApparel
            .OfType<TrionTrigger>()
            .FirstOrDefault();

        if (trigger != null)
        {
            foreach (var mount in trigger.Mounts)
            {
                foreach (var comp in mount.EquippedComponents)
                {
                    componentDefNames.Add(comp.Def.defName);
                }
            }
        }
    }

    Scribe_Collections.Look(ref componentDefNames, "componentDefNames");

    if (Scribe.mode == LoadSaveMode.LoadingVars)
    {
        // 加载时：从 defName 重建对象
        if (componentDefNames != null && componentDefNames.Count > 0)
        {
            // 重建逻辑在 PostSpawnSetup 中处理
            _cachedComponentDefNames = componentDefNames;
        }
    }
}

private List<string> _cachedComponentDefNames = new List<string>();

public override void PostSpawnSetup(bool respawningAfterLoad)
{
    base.PostSpawnSetup(respawningAfterLoad);

    if (respawningAfterLoad && _cachedComponentDefNames.Count > 0)
    {
        // 从缓存的 defName 列表重建组件
        TrionTrigger trigger = parent.apparel.WornApparel
            .OfType<TrionTrigger>()
            .FirstOrDefault();

        if (trigger != null)
        {
            foreach (var defName in _cachedComponentDefNames)
            {
                var def = DefDatabase<ThingDef>.GetNamed(defName);
                var comp = TriggerComponent.CreateFromDef(defName);
                trigger.Mounts[0].TryAddComponent(comp);  // 假设装到第一个挂载点
            }

            _cachedComponentDefNames.Clear();
        }
    }
}
```

**优势**：
- ✓ 存档文件最小化（只存 defName）
- ✓ 后续修改组件定义不影响老存档
- ✓ 加载时灵活重建（可添加/移除组件）
- ✓ 快照单独完整序列化（精确恢复）

**确认事项**
- [x] 组件对象不需要完整序列化（只存 defName）
- [x] 加载时从 defName 重建对象树
- [x] 快照保持完整序列化
- [x] 穿戴关系由 Apparel 系统自动处理

✅ **状态**：已在文档3、9中说明
