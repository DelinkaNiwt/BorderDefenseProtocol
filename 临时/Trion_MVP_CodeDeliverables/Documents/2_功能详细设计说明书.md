# Trion_MVP 功能详细设计说明书 v0.3

## 功能1：植入获取

**功能说明**：玩家给Pawn植入Trion腺体后，CompTrion自动初始化到Pawn上。

**工作步骤**：
1. Pawn进行"植入Trion腺体"手术
2. Surgery成功执行，Hediff_TrionGland.PostAdd() 被调用
3. Hediff检查Pawn是否已有CompTrion（避免重复添加）
4. 创建CompProperties_Trion实例并配置
5. 创建CompTrion实例，设置props和parent
6. 调用 CompTrion.PostSpawnSetup(false) 手动初始化
7. 日志记录："[Trion] {Pawn.Name}植入Trion腺体，获得Trion能量系统"

**验收标准**：
- [ ] 植入后CompTrion存在于Pawn.GetComp<CompTrion>()
- [ ] Strategy已初始化（可正常调用方法）
- [ ] Capacity = 1400（临时值）
- [ ] Reserved = 0, Consumed = 0
- [ ] 日志可见成功消息
- [ ] 重复植入不会添加多个CompTrion

## 功能2：天赋初始化

**功能说明**：首次用检测仪扫描Pawn时，系统随机生成天赋等级、存储到modData、重算Capacity。

**工作步骤**：
1. 玩家与Building_TrionDetector交互，调用 ScanPawn(pawn)
2. 检查pawn是否有CompTrion（无则提示"无腺体"返回）
3. 调用 TrionTalentManager.GetTalent(pawn)
4. 如返回null（首次）：
   - 调用 GenerateRandomTalent(pawn) 随机选S-E
   - 调用 TrionTalentManager.SetTalent(pawn, talent)
   - 调用 comp.RecalculateCapacityFromTalent(talent)
   - 日志："[Trion] {Pawn}首次扫描：天赋等级{talent}"
5. 如已有天赋（非首次）：直接返回existing talent
6. 显示扫描结果：名字、天赋等级、新Capacity值

**验收标准**：
- [ ] 首次扫描生成天赋（可观察modData或日志）
- [ ] 天赋值在S-E范围内
- [ ] Capacity更新为正确的天赋对应值
- [ ] 再次扫描显示相同天赋（不再生成）
- [ ] UI显示结果清晰
- [ ] 日志分别记录"首次"和"非首次"扫描

## 功能3：穿戴触发器和装备组件

**功能说明**：玩家通过穿戴系统（Apparel）装备触发器到Pawn上，再通过配置台（Building_TriggerConfigBench）将 TriggerComponent 物品装备到触发器上。

**工作步骤**：

### 步骤A：穿戴触发器
1. 玩家在Pawn的检查面板找到触发器物品
2. 调用 `pawn.apparel.Wear(trigger)`（RimWorld穿戴系统）
3. TrionTrigger 继承自 Apparel，自动进入穿戴系统
4. Apparel系统自动处理：
   - 穿着判定（检查bodyPartGroups: [Torso]、layers: [OnSkin]）
   - 冲突检测（不能同时穿超过限定数量的衣物）
   - 持久化（读档时自动恢复）
5. 日志："[Trion] {Pawn}已穿戴{Trigger.label}"

### 步骤B：装备触发器内的组件
1. 玩家在配置台选择穿着的触发器和TriggerComponent物品
2. 调用 TriggerMount.TryAddComponent(component)
3. 检查1：槽位是否足够（used + component.SlotSize <= 4）
4. 检查2：是否与已有组件冲突（如两个近战武器）
5. 检查3：战斗体是否激活（激活则不允许修改）
6. 通过所有检查：添加到 _equippedComponents 列表
7. 计算新Reserved值（如战斗体已激活则需更新）
8. 从地图移除组件物品：component.Destroy(DestroyMode.Vanish)
9. 日志："[TriggerMount] 装备TriggerComponent{component.ComponentName}"

**验收标准**：
- [ ] Pawn穿戴触发器后在apparel.WornApparel中可见
- [ ] 穿戴系统正确处理冲突和持久化
- [ ] 成功装备组件后在Mounts列表中
- [ ] UI显示组件列表正确
- [ ] Reserved值正确更新
- [ ] 槽位限制有效（不能超4）
- [ ] 冲突检查有效（如两武器互斥则拒绝）
- [ ] 战斗体激活时提示"不能修改"并拒绝

## 功能4：生成战斗体

**功能说明**：玩家通过Gizmo按钮激活战斗体，系统保存快照、计算Reserved、标记激活状态。

**工作步骤**：
1. **通过Gizmo按钮触发**：Pawn持有触发器时，inspect面板右侧显示Gizmo按钮"激活战斗体"
2. 玩家点击Gizmo按钮
3. CompTrion.GenerateCombatBody() 被调用
4. 检查：_isInCombat是否已true（是则警告返回）
5. 创建快照：_snapshot = new CombatBodySnapshot()
6. 捕获Pawn当前状态：_snapshot.CaptureFromPawn(pawn)
7. 标记激活：_isInCombat = true
8. 回调Strategy：_strategy.OnCombatBodyGenerated(this)
9. 日志："[Trion] {Pawn}的战斗体已生成，可用:{Available}"

**验收标准**：
- [ ] Snapshot包含Pawn的生命值、部位、装备、伤口等完整信息
- [ ] _isInCombat = true
- [ ] Reserved = Σ(所有装备组件的消耗值)
- [ ] Available = Capacity - Reserved - Consumed
- [ ] Capacity不变（仅Reserved生效）
- [ ] 日志可见激活消息

## 功能5：消耗计算

**功能说明**：每60Tick计算一次消耗，包括基础维持、激活组件、伤口泄漏。

**工作步骤**：
1. CompTick()被调用，检查_isInCombat（不激活则返回）
2. 每60Tick检查一次（使用IsHashIntervalTick）
3. 步骤1：baseMaintenance = Strategy.GetBaseMaintenance() [默认5f]
4. 步骤2：mountConsumption = Σ(IsActive的组件消耗值)
5. 步骤3：leakRate = GetLeakRate() [从伤口计算]
6. 步骤4：totalConsumption = base + mount + leak
7. 步骤5：Consume(totalConsumption) // Consumed += amount
8. 步骤6：检查 Available <= 0 则 TriggerBailOut()
9. 步骤7：Strategy.OnTick(this) 回调

**验收标准**：
- [ ] Available逐步递减
- [ ] 无激活组件时消耗 ≈ 5
- [ ] 激活1个消耗10的组件后消耗 ≈ 15
- [ ] 有伤口时消耗进一步增加
- [ ] 每60Tick执行一次（不会每Tick都算）

## 功能6：伤口泄漏

**功能说明**：Pawn受伤后，伤口严重程度决定Trion泄漏速率。关键部位伤口泄漏加倍。

**工作步骤**：
1. **需要Harmony补丁拦截伤害**：在DamageWorker_AddInjury或相关处理中调用 CompTrion.InvalidateLeakCache()
2. CalculateLeakRate() 遍历pawn.health.hediffSet.hediffs
3. 对每个Hediff_Injury：leak += 0.5 + severity * 0.1
4. 如伤口位置是关键部位（手臂、腿、躯干）：leak *= 2
5. GetLeakRate() 返回缓存值，缓存60Tick后过期
6. InvalidateLeakCache() 在伤口变化时调用，清除缓存

**验收标准**：
- [ ] 无伤口时泄漏 = 0
- [ ] 手臂伤口(severity=0.5)时泄漏 ≈ 0.55
- [ ] 关键部位伤口时泄漏翻倍 ≈ 1.1
- [ ] 泄漏缓存有效（60Tick内不重新计算）
- [ ] 新增伤口时缓存失效，下次重算

## 功能7：战斗体破裂与Bail Out

**功能说明**：当Available <= 0 或玩家手动触发Bail Out或供给器官被摧毁时，战斗体进行"破裂"处理。Pawn传送到传送锚或基地附近，Reserved累加到Consumed，施加心理debuff。

**工作步骤**：
1. TriggerBailOut() 被调用（来自消耗检查或器官摧毁或玩家手动）
2. 检查_isInCombat（不激活则返回）
3. 检查enableBailOut是否启用（未启用则直接破裂）
4. 查询传送目标：
   - 优先级1：调用Strategy.GetBailOutTarget(this) 查询有效的传送锚
   - 优先级2：如无有效锚点，使用 RCellFinder.RandomSpotJustOutsideColony(pawn) 传送到基地附近
5. 执行传送：pawn.Position = target
6. 回调：pawn.Notify_Teleported()
7. 执行破裂处理：DestroyCombatBody(DestroyReason.Crashed)
   - 快照恢复（如启用）
   - Reserved 累加到 Consumed（表示能量已消耗）
   - 标记_isInCombat = false
   - 施加心理debuff

**验收标准**：
- [ ] Available <= 0时自动触发
- [ ] 玩家手动触发Bail Out总是成功（无失败分支）
- [ ] Pawn传送到传送锚或基地附近
- [ ] Reserved → Consumed 正确转化
- [ ] 战斗体标记为Crashed
- [ ] 日志记录传送位置和破裂原因

## 功能8：安全解除战斗体

**功能说明**：玩家点击"解除战斗体"按钮（Available > 0且战斗体已激活）时，战斗体安全解除。Reserved返还，无debuff。

**工作步骤**：
1. 玩家点击"解除战斗体"按钮（仅当_isInCombat=true时可见）
2. 检查Available > 0（如果 <= 0 则拒绝，提示"能量不足，无法安全解除，只能Bail Out"）
3. 调用DestroyCombatBody(DestroyReason.Manual)
   - 快照恢复（如启用）
   - Reserved 完全返还（不累加到Consumed）
   - 无任何心理debuff
   - 标记_isInCombat = false
4. 日志："[Trion] {Pawn}安全解除战斗体，Reserved返还"

**验收标准**：
- [ ] Available > 0时"解除"按钮可用
- [ ] Available <= 0时"解除"按钮禁用，提示Bail Out
- [ ] 解除后Reserved完全返还
- [ ] 无心理debuff施加
- [ ] Pawn状态恢复到快照
- [ ] 日志正确记录

## 功能9：快照恢复与debuff

**功能说明**：战斗体摧毁时（无论是破裂还是安全解除），Pawn状态恢复到快照。伤口和部位完全恢复，但心理debuff继承。

**工作步骤**：
1. DestroyCombatBody(reason) 被调用
2. 如enableSnapshot且_snapshot非空：
   - _snapshot.RestoreToPawn(pawn)
   - 恢复：生命值、部位、装备、战前伤口
   - 丢弃：战斗体期间产生的新伤口
3. 根据reason处理Reserved和Consumed：
   - Crashed/TrionDepleted/BailOut（被动）：Reserved累加到Consumed、施加debuff
   - Manual（主动）：Reserved直接返还（不累加Consumed）、无debuff
4. 标记_isInCombat = false
5. 清除泄漏缓存
6. 回调Strategy.OnCombatBodyDestroyed(this, reason)

**验收标准**：
- [ ] 回滚后Pawn生命值恢复
- [ ] 回滚后Pawn部位完整
- [ ] 战斗体产生的新伤口不继承
- [ ] 战前旧伤口保留
- [ ] 心理状态Debuff只在破裂时继承
- [ ] 主动解除时Reserved返还（Consumed不变）
- [ ] 被动解除时Reserved→Consumed、施加debuff

## 功能10：读档持久化

**功能说明**：关闭游戏后重新读档，Capacity和天赋值不变。CompTrion和Strategy正确恢复。

**工作步骤**：
1. 玩家保存游戏
2. PostExposeData() 序列化：
   - _capacity, _reserved, _consumed 保存
   - _isInCombat 保存
   - _snapshot 保存
   - _mounts 保存
3. 玩家读档
4. PostExposeData() 反序列化（自动恢复）
5. PostSpawnSetup(respawningAfterLoad=true)：
   - 跳过新建分支（不重新计算Capacity）
   - 执行读档分支：InitializeStrategy() 重新注入Strategy
6. Strategy返回null天赋（因为已有Capacity）

**验收标准**：
- [ ] 存档前后Capacity值一致
- [ ] 存档前后天赋值一致（modData持久化）
- [ ] Strategy重新初始化成功
- [ ] CompTrion继续可用
- [ ] 存档大小合理（无冗余数据）
