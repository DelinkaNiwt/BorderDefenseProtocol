## 架构问题清单

### 【架构问题1】TriggerComponent 的数据模型混淆

**问题描述**

在9_代码实现指南中：
- TriggerComponent 继承自 `ThingWithComps`（游戏对象）
- TriggerComponent_Def 继承自 `ThingDef`（定义）
- 但原 MVP 设计中有 5 个组件（ArcBlade、Shield 等）定义为 ThingDef

**关键疑问**

现存的 `TrionComponent_ArcBlade` (ThingDef) 与新设计的 `TriggerComponent_ArcBlade` (ThingWithComps) 是什么关系？

**可能的架构方案**

**方案A**：分离物品与组件
```
TrionComponent_ArcBlade (ThingDef)     // XML中的物品定义
  ↓ (在配置台进行某种转换)
TriggerComponent_ArcBlade (ThingWithComps) // 触发器内的组件对象
```
- 物品在地图上、inventory中存在为 ThingDef
- 装备到触发器后变为 TriggerComponent

**方案B**：统一为组件
```
TriggerComponent_ArcBlade (thingClass)  // 只有一种对象
```
- 直接在 XML 中定义为 TriggerComponent
- 地图/inventory 中也是同一对象

**方案C**：继承关系重构
```
TrionComponentBase (ThingDef, Abstract)
  ├─ TrionComponent_ArcBlade (ThingDef) // 物品形态
  └─ TriggerComponent (comp)            // 装备后的组件形态
```

**架构师需要确认**：
- [ ] TriggerComponent 是否应该是独立的数据模型还是物品的特化状态？
- [ ] 原 5 个组件的 ThingDef 是否需要修改？改成什么样？
- [ ] 组件从"物品"变为"装备组件"的转换机制是什么？

### 【架构问题2】Pawn 装备 TrionTrigger 的拥有关系

**问题描述**

TrionTrigger 是 `ThingWithComps`（物品），但它如何与 Pawn 建立"装备关系"？

RimWorld 中物品与 Pawn 的关系主要有：
1. **equipment.Primary** - 武器槽
2. **apparel.WornApparel** - 穿戴装备槽
3. **inventory.innerContainer** - 背包

**关键疑问**

TrionTrigger 应该使用哪种关系？

**可能的架构方案**

**方案A**：作为装备（apparel）
```csharp
pawn.apparel.Wear(triggerTrigger as Apparel);
```
- 优点：有现成的界面和UI
- 缺点：需要 TrionTrigger 继承自 Apparel

**方案B**：作为武器（equipment.Primary）
```csharp
pawn.equipment.MakeRoomFor(triggerTrigger);
pawn.equipment.AddEquipment(triggerTrigger);
```
- 优点：装备位置显著
- 缺点：与 CompTrion 的语义不符（不是"武器"）

**方案C**：创建新的 "TriggerSlot" 机制
```csharp
// CompTrion 内部持有对 TrionTrigger 的引用
public TrionTrigger _equippedTrigger;
```
- 优点：完全独立，不依赖现有系统
- 缺点：需要自定义 UI

**方案D**：保持在 inventory 中
```csharp
// TrionTrigger 物理上在 Pawn.inventory
// CompTrion 通过查询 inventory 获取
```
- 优点：最灵活
- 缺点：需要特殊的查询逻辑

**架构师需要确认**：
- [ ] TrionTrigger 与 Pawn 的拥有关系应该使用什么机制？
- [ ] 这个关系应该被 UI 显示出来吗？
- [ ] CompTrion 是否需要持有对 TrionTrigger 的引用？

### 【架构问题4】Strategy.GetBailOutTarget() 的 Map 访问

**问题描述**

DefaultTrionStrategy 中的方法：
```csharp
public Building GetBailOutTarget(Pawn pawn)
```

这个方法需要查询地图上的所有 `Building_BailOutAnchor`，但 Strategy 是无状态的工具类，无法直接访问 Map。

**关键疑问**

Strategy 应该如何获取 Map 对象？

**可能的架构方案**

**方案A**：Strategy 方法接收 Map 参数
```csharp
public Building GetBailOutTarget(Pawn pawn, Map map)
{
    return map.GetComponent<BailOutAnchorManager>()?.GetNearestAnchor(pawn);
}
```
- 优点：显式依赖
- 缺点：改变了接口签名

**方案B**：通过 Pawn 获取 Map
```csharp
public Building GetBailOutTarget(Pawn pawn)
{
    Map map = pawn.Map;
    // ...
}
```
- 优点：Pawn 已有 Map 属性
- 缺点：依赖 Pawn 的状态

**方案C**：CompTrion 负责查询，Strategy 只选择
```csharp
// CompTrion.TriggerBailOut()
List<Building_BailOutAnchor> anchors = Find.Maps...;
Building target = _strategy.SelectBailOutTarget(anchors, pawn);
```
- 优点：职责分离
- 缺点：Strategy 不再"自主决策"

**架构师需要确认**：
- [ ] Strategy 的 GetBailOutTarget() 应该接收什么参数？
- [ ] 查询 Map 资源的职责应该在 Strategy 还是 CompTrion？
- [ ] 是否需要创建一个 MapComponent 来管理锚点？

### 【架构问题5】快照恢复的触发条件

**问题描述**

CombatBodySnapshot.RestoreToPawn() 应该在什么时机调用？

当前理解：
- 战斗体"破裂"时调用
- Available <= 0 且无有效传送锚时调用（Bail Out 失败时）

**关键疑问**

什么是"战斗体破裂"？如何定义和检测？

**可能的架构方案**

**方案A**：Available <= 0 定义为"破裂"
```
Available <= 0 ← 自动触发 Bail Out 或 破裂
  ├─ 有有效锚 → TriggerBailOut() + 传送
  └─ 无有效锚 → 破裂 + RestoreToPawn()
```

**方案B**：Bail Out 失败才定义为"破裂"
```
Available <= 0 → TriggerBailOut()
  ├─ 成功 → 传送到锚
  └─ 失败（无锚/被摧毁） → 破裂 + RestoreToPawn()
```

**方案C**：Pawn 死亡时定义为"破裂"
```
Pawn.Dead == true → 检查 IsInCombat
  ├─ 是 → 破裂
  └─ 否 → 正常死亡
```

**方案D**：显式的"战斗体系统崩溃"机制
```
// 新增一个 _combatBodyState 字段
enum CombatBodyState { None, Generating, Active, Crashing, Crashed }
```

**架构师需要确认**：
- [ ] "战斗体破裂"的定义是什么？什么条件下发生？
- [ ] 破裂和 Bail Out 的区别是什么？
- [ ] 如果破裂发生在 Pawn 死亡时，快照恢复还有意义吗？
- [ ] 是否需要一个专门的"战斗体状态机"来管理这些转移？

### 【架构问题6】嵌套对象的序列化架构

**问题描述**

数据结构的序列化链：
```
Pawn.GetComp<CompTrion>()  ← Scribe
  └─ _mounts: List<TriggerMount>
      └─ _equippedComponents: List<TriggerComponent>
          └─ _componentDef: TriggerComponent_Def
```

**关键疑问**

这个嵌套链的序列化设计是否清晰？特别是：

1. **对象所有权**：
   - TriggerComponent 是 CompTrion 的一部分吗？还是独立的 Thing？
   - 存档时，TriggerComponent 应该序列化在 CompTrion 里还是单独保存？

2. **引用关系**：
   - _componentDef 应该用 `LookMode.Def` 还是 `LookMode.Deep`？
   - _equippedComponents 中的组件是否应该序列化完整对象还是只序列化 defName？

3. **重新加载后的关系恢复**：
   - 读档后，_equippedComponents 中的对象是否需要重新"初始化"？
   - 它们是否需要引用回 TriggerMount 和 TrionTrigger？

**可能的架构方案**

**方案A**：完整序列化嵌套对象
```csharp
// CompTrion.ExposeData()
Scribe_Collections.Look(ref _mounts, "mounts", LookMode.Deep);
// TriggerMount.ExposeData()
Scribe_Collections.Look(ref _equippedComponents, "components", LookMode.Deep);
```
- 优点：完整性高
- 缺点：序列化文件变大

**方案B**：只序列化关键数据
```csharp
// CompTrion 只序列化组件的 defName 列表
// 读档时重新构建对象
Scribe_Collections.Look(ref _componentDefNames, "componentDefNames");
// PostLoadInit() 中恢复对象树
```
- 优点：文件小，可灵活改装
- 缺点：实现复杂

**方案C**：分离持久化和运行时对象
```
存档中：CompTrion 存 CombatBodyData（序列化数据）
运行时：CompTrion 维护 _mounts 对象树（非序列化）
初始化：从 CombatBodyData 恢复 _mounts
```
- 优点：清晰的分离
- 缺点：增加中间层

**架构师需要确认**：
- [ ] TriggerComponent 对象是否应该作为 CompTrion 的一部分序列化？
- [ ] 还是应该采用"defName + 运行时重建"的模式？
- [ ] 从存档恢复后，对象树是否需要"重新初始化"或"关系修复"？
- [ ] 如果 TriggerComponent_Def 的定义在更新后改变，应该如何处理已保存的对象？
