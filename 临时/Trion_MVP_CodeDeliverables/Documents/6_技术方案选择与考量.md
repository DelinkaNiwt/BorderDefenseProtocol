# Trion_MVP 技术方案选择与考量 v0.3

## 决策1：避免全局Harmony补丁

### 问题
如何自动给所有Pawn附加CompTrion，而不破坏兼容性和性能？

### 备选方案

**方案A：全局Harmony补丁（拦截Pawn生成）**
```
Pawn生成时 → Harmony补丁拦截 → 自动添加CompTrion
```
优点：
- 全自动，玩家无需手动操作
- 所有Pawn都能用

缺点：
- ✗ 全局补丁，性能风险
- ✗ 与其他mod冲突概率高
- ✗ MVP测试时难以"关闭"框架
- ✗ 违反"最小化侵入"原则

**方案B：植入物系统（选中）**
```
玩家给Pawn植入腺体 → Hediff.PostAdd → 自动添加CompTrion
```
优点：
- ✓ 零全局补丁
- ✓ 符合世界观（天赋确实在生成时拥有）
- ✓ UI交互自然
- ✓ 易于启用/禁用
- ✓ MVP测试时可完全卸载

缺点：
- 玩家需要主动操作（手术）
- 初期setup步骤较多

**方案C：配置Def**
```
修改Pawn.humanlike的Def，在comps中添加CompTrion
```
优点：
- 自动生成时添加

缺点：
- ✗ 需要修改或创建新种族Def
- ✗ 所有该种族Pawn都有（无法选择）
- ✗ 污染原版定义

### 决策结果
✓ **选择方案B（植入物系统）**

理由：
- MVP目的是验证框架，不是改造游戏
- 植入物方案保证了零侵入
- 符合设计的"投机取巧"理念
- 为应用层开发奠定好的基础

## 决策2：TrionTrigger 与 RimWorld Apparel 系统的融合

### 问题
触发器（TrionTrigger）应该怎样被Pawn装备和管理？是自定义引用关系，还是复用穿戴系统？

### 备选方案

**方案A：自定义引用关系（原方案）**
```csharp
public class CompTrion
{
    private TrionTrigger _equippedTrigger; // 自定义字段
}
```
缺点：
- ✗ 需要自定义穿戴/脱下逻辑
- ✗ 需要自定义冲突检测
- ✗ 需要手动序列化持久化
- ✗ 无法复用RimWorld的装备UI

**方案B：继承 Apparel 系统（选中）**
```csharp
public class TrionTrigger : Apparel
{
    // 复用Apparel的所有系统
}
// 装备：pawn.apparel.Wear(trigger)
// 脱下：pawn.apparel.Unfold(trigger)
```
优点：
- ✓ 复用RimWorld穿戴系统，无需自定义逻辑
- ✓ Apparel系统自动处理：穿戴判定、冲突检测、持久化、UI
- ✓ bodyPartGroups 和 layers 参数原生支持
- ✓ 与其他mod的装备无冲突
- ✓ 存档/读档自动恢复

缺点：
- Apparel本身的属性（armor、insulation）用不上

**方案C：ThingWithComps（旧方案）**
```csharp
public class TrionTrigger : ThingWithComps { }
```
缺点：
- ✗ 无穿戴系统，需要全部自定义
- ✗ 与装备UI不融合

### 决策结果
✓ **选择方案B（Apparel继承）**

理由：
- RimWorld穿戴系统已经完美解决了装备管理问题
- 复用现有系统比重新发明车轮更高效
- Pawn.apparel.WornApparel 自动跟踪穿着的触发器
- 架构上利用了RimWorld的最佳实践

## 决策3：天赋初始化时机（原决策2）

### 问题
天赋应该在什么时候初始化？

### 备选方案

**方案A：Pawn生成时即初始化**
```
Pawn生成 → CompTrion.PostSpawnSetup → 立即生成天赋
```
缺点：
- ✗ 需要Harmony补丁或修改Def
- ✗ 所有Pawn都有天赋（即使没植入腺体）

**方案B：首次使用检测仪时初始化（选中）**
```
玩家扫描检测仪 → 首次扫描生成天赋 → 存储+Capacity重算
```
优点：
- ✓ 避免全局补丁
- ✓ UI交互点自然
- ✓ 符合世界观（天赋隐藏）
- ✓ 延迟初始化，性能更优
- ✓ 玩家可选择何时"激活"天赋系统

缺点：
- 初始化不是"自动的"，需要玩家操作

**方案C：第一次建造触发器时初始化**
缺点：
- ✗ 交互点不够自然
- ✗ 需要额外检测逻辑

### 决策结果
✓ **选择方案B（检测仪初始化）**

理由：
- "投机取巧"地既符合世界观又避免补丁
- 检测仪是天然的UI交互点
- 首次扫描→生成天赋→显示结果，流程清晰
- 后续扫描直接返回已有天赋，无重复计算

## 决策4：天赋→Capacity映射（原决策3）

### 问题
框架怎样知道每个天赋对应多少Capacity？

### 备选方案

**方案A：CompTrion存储天赋的Capacity值**
```
CompTrion { talent: TalentGrade, talentCapacity: float }
```
缺点：
- ✗ 框架层包含应用层数据
- ✗ 不够灵活，修改查表需改框架

**方案B：Pawn.modData存储天赋（选中）**
```
Pawn.modData["Trion_Talent"] = "B"
CompTrion 通过 TalentCapacityProvider(talent) 查表
```
优点：
- ✓ 框架保持纯净（不知道具体映射）
- ✓ 应用层灵活定义映射
- ✓ 天赋可独立迭代
- ✓ modData是RimWorld标准方式

缺点：
- 需要在Mod启动时设置委托

**方案C：Strategy回调返回Capacity**
```
Strategy.GetInitialTalent() 返回 { talent, capacity }
```
缺点：
- ✗ Strategy知道太多细节
- ✗ 不同Strategy需要重复实现映射

### 决策结果
✓ **选择方案B（modData + 委托查表）**

理由：
- RimWorld标准做法
- 框架和应用层职责分离清晰
- 灵活性最高
- 易于测试和修改

## 决策5：快照恢复粒度（原决策4）

### 问题
战斗体摧毁时，应该恢复哪些Pawn状态？

### 备选方案

**方案A：完全恢复（包括心理状态）**
缺点：
- ✗ 失去游戏风险感（玩家无任何代价）

**方案B：完全恢复（除心理状态）（选中）**
```
恢复：生命值、部位、装备、伤口
不恢复：心理状态、Debuff
```
优点：
- ✓ 战斗体可以"重置"身体状态
- ✓ 但战斗仍有心理成本
- ✓ 平衡游戏难度

缺点：
- 实现较复杂

**方案C：仅恢复生命值**
缺点：
- ✗ 战斗体失去保护价值
- ✗ 伤口仍存在，战斗体就没意义

### 决策结果
✓ **选择方案B（完全恢复除心理状态）**

理由：
- 符合设计理念（战斗体是"临时躯体"）
- 保证游戏的风险感（心理成本存在）
- 平衡性最优

## 决策6：组件装备时机（原决策5）

### 问题
组件什么时候可以装备，什么时候不能装备？

### 备选方案

**方案A：任何时候都可以装备**
缺点：
- ✗ 战斗体激活中还能改组件，不平衡

**方案B：战斗体未激活时可装备，激活后锁定（选中）**
```
战斗体未激活 → 可装备/移除组件
战斗体激活 → 组件锁定，只能激活/关闭
```
优点：
- ✓ 游戏平衡（战中不能临时更换组件）
- ✓ 性能稳定（Reserved固定）
- ✓ 强制玩家提前规划装配

缺点：
- 灵活性降低

**方案C：战斗体激活中也可以装备**
缺点：
- ✗ 不平衡（可无限优化配置）
- ✗ Reserved需要动态重算

### 决策结果
✓ **选择方案B（激活后锁定）**

理由：
- 游戏平衡
- 代码简洁（Reserved无需动态更新）
- 符合"战斗体不可修改"的设定

## 决策7：性能优化策略（原决策6）

### 泄漏计算缓存

**问题**：每Tick计算伤口泄漏，遍历所有hediff，性能开销大

**方案**：
- 缓存leakRate值，60Tick后过期
- 伤口变化时清除缓存
- 下次消耗计算时重新计算

**效果**：
- ✓ 将O(伤口数) 的计算从60次/s降到1次/60Tick
- ✓ 性能提升 60x

### Reserved预计算

**问题**：每次消耗计算都要Σ激活组件

**方案**：
- Reserved在装备/激活时预计算
- 消耗时直接使用，不重算

**效果**：
- ✓ 消耗计算时间从O(组件数)降到O(1)

## 决策8：MVP范围（原决策7）

### 问题
MVP应该包含哪些功能？

### 选择标准
- **包含**：框架的核心验证路径（天赋、组件、消耗、快照）
- **不包含**：复杂的应用层（UI系统、升级系统、多Strategy）

### 组件选择

**包含的5个组件**：
- 弧月（测试近战消耗）
- 护盾（测试组件共存）
- 炸裂弹（测试远程消耗）
- 变色龙（测试辅助功能）
- 脱离（测试Bail Out）

**不包含的**：
- 融合组件
- 升级组件
- 特殊技能组件

理由：MVP只验证基础框架，不做应用层创新

## 决策9：伤害拦截方案（原决策8）

### 问题
如何在Pawn受伤时自动清除泄漏缓存，保证伤口变化能正确反映在消耗中？

### 备选方案

**方案A：轮询检测（低效）**
```
每Tick遍历pawn.hediffs比较当前和缓存
```
缺点：
- ✗ O(伤口数) 的计算，浪费性能
- ✗ 无法及时响应伤口变化

**方案B：Harmony补丁拦截（选中）**
```
Harmony.Patch(DamageWorker_AddInjury)
→ CompTrion.InvalidateLeakCache()
```
优点：
- ✓ 精确拦截伤害事件
- ✓ 只在伤口变化时清除缓存
- ✓ 性能开销最小（O(1)）
- ✓ 兼容性好（只拦截关键方法）

缺点：
- 需要Harmony库集成

**方案C：Strategy回调**
```
Strategy.OnPawnDamaged(damage) → InvalidateLeakCache()
```
缺点：
- ✗ 需要修改Strategy接口
- ✗ 应用层需要实现回调

### 决策结果
✓ **选择方案B（Harmony补丁拦截）**

理由：
- 性能最优，拦截点精确
- RimWorld常见做法
- 与伤口系统深度集成，无额外逻辑

## 决策10：战斗体生成触发机制（原决策9）

### 问题
玩家怎样触发战斗体生成？是通过命令、按钮、还是自动？

### 备选方案

**方案A：Inspector窗口的默认按钮（低兼容）**
```
修改Pawn的检查窗口，添加"激活战斗体"按钮
```
缺点：
- ✗ 需要修改游戏UI，兼容性差
- ✗ 按钮位置固定，不够灵活

**方案B：Gizmo按钮（选中）**
```
CompTrion.CompGetGizmosExtra()
→ 返回Gizmo("激活战斗体")
```
优点：
- ✓ 标准RimWorld做法
- ✓ 出现在inspect面板右侧，自然位置
- ✓ 支持点击直接调用
- ✓ 可根据状态禁用按钮
- ✓ 兼容性最好

缺点：
- Gizmo位置依赖UI顺序

**方案C：命令行/Mod设置**
```
/trion activate {pawnID}
```
缺点：
- ✗ 不够直观
- ✗ 玩家体验差

### 决策结果
✓ **选择方案B（Gizmo按钮）**

理由：
- RimWorld标准交互方式
- 玩家体验最佳
- 兼容性和易用性最高

## 决策11：触发器槽位设计（原决策10）

### 问题
触发器应该有多少个槽位？怎么分配左手、右手、特殊槽位？

### 备选方案

**方案A：无槽位限制（不平衡）**
```
可装备无限个组件
```
缺点：
- ✗ 游戏不平衡，玩家可无限强化
- ✗ 性能风险（消耗计算复杂度增加）

**方案B：左右手分离，特殊槽位（选中）**
```
总9槽位：
├─ 左手：4槽
├─ 右手：4槽
└─ 特殊：1槽

同时激活最多2个组件（左手1+右手1）
```
优点：
- ✓ 平衡性好（限制总消耗）
- ✓ 符合人体结构（左右手）
- ✓ 特殊槽位用于辅助能力（隐身、护盾）
- ✓ 强制玩家做出取舍

缺点：
- 配置相对复杂

**方案C：统一槽位（简单但无差别）**
```
总9个槽位，无左右手区分
```
缺点：
- ✗ 失去武器位置的差异性
- ✗ 游戏深度不足

### 决策结果
✓ **选择方案B（左右手分离+特殊槽位）**

理由：
- 平衡性和游戏深度最优
- 符合现实逻辑
- 强制玩家策略选择

## 决策12：战斗体解除的分类

### 问题
战斗体摧毁时应该分为哪些情况？每种情况的处理方式有什么区别？

### 背景
原方案中Bail Out没有成功/失败的区分，且Reserved处理没有明确的语义。新方案为了增强游戏系统的一致性和平衡性，需要明确定义两种解除方式。

### 备选方案

**方案A：统一处理（旧方案）**
```
所有战斗体摧毁 → Reserved统一处理 → 无区分
```
缺点：
- ✗ 语义不清（什么时候Reserved返还？什么时候消耗掉？）
- ✗ 难以平衡游戏难度
- ✗ Bail Out是否成功的结果没有区分

**方案B：分为"安全解除"和"破裂"两类（选中）**
```
安全解除（Manual）：
  ├─ Available > 0 + 玩家主动点击
  ├─ Reserved 完全返还（→ Available）
  ├─ 无debuff
  └─ 语义：受控的、有序的退出

破裂（Crashed）：
  ├─ Available ≤ 0 或 玩家Bail Out 或 器官摧毁
  ├─ Reserved → Consumed（永久消耗）
  ├─ 施加debuff
  └─ 语义：被迫的、有代价的退出
```
优点：
- ✓ 语义清晰（何时返还、何时消耗）
- ✓ 游戏平衡性提升（破裂有代价）
- ✓ Bail Out成功100%（不区分失败）
- ✓ 强化"风险vs收益"的游戏设计
- ✓ Reserved→Consumed的含义清楚（能量"烧掉了"）

缺点：
- 实现逻辑相对复杂（需要两个DestroyCombatBody分支）

**方案C：三分类**
```
主动解除、自动脱离、器官摧毁
```
缺点：
- ✗ 分类过细，增加复杂度
- ✗ 器官摧毁和自动脱离的处理结果是一样的

### 决策结果
✓ **选择方案B（安全解除 vs 破裂）**

理由：
- 游戏设计上形成"安全港湾"（Available > 0时可以安全退出）
- 鼓励玩家合理计划能量管理
- Bail Out作为"破裂"的一部分，总是成功，符合"紧急脱离"的设定
- Reserved→Consumed的转化形成永久代价，增强游戏难度
- 两分类足以涵盖所有情况
