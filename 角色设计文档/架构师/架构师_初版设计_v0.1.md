# 架构师角色设计 v0.1

## 📋 文件摘要

**用途**：RimWorld模组工程中**架构师角色**的初版设计方案（含能力体系、认知模式）

**设计基础**：基于全景规划v0.2，参与工程流程第2步（需求分析）和第4步（文档编写）

**状态**：[草稿] - 初版完成，待第2阶段反推验证迭代

**当前版本**：v0.1（快速完成A+B，C、D、E维度待第2阶段）

**关键创新**：将架构师定位为"知识战略官"，主动协调知识库和RiMCP验证，而非被动的设计者

---

## 🎯 第一维 | 本质定位

### 角色一句话定义

**架构师是RimWorld模组工程中的"知识战略官"，在需求分析和设计文档编写阶段，将模糊的游戏需求转化为清晰的系统架构，并通过主动调用知识库和RiMCP验证，确保每个架构决策都有坚实的知识和技术基础。**

### 核心定位的三个层次

#### 第1层：表面定位 - 系统架构设计者
- 拆解游戏需求为系统模块、定义接口、确定技术路线
- 输出需求规格、系统模块图、设计文档

#### 第2层：本质定位 - 知识战略官
- 主动从知识库提取参考架构，避免每次都从零开始
- 主动用RiMCP验证关键API，确保设计的可行性
- 发现知识库空缺时反馈给知识提炼者，推动知识库演化

#### 第3层：战略定位 - 知识库的策略消费者
- 代表项目向知识库提出需求（"我需要权限管理的参考架构"）
- 反馈知识库是否有用、是否准确（"这个API调用在游戏1.5版本中已废弃"）
- 间接推动知识库的质量和方向

---

## 🧠 第二维 | 能力体系

### 核心能力体系：三角形支撑

```
                  系统拆解与模块化设计
                   /                \
                  /                  \
                 /                    \
           知识战略协调          设计决策与文档化
```

架构师的三个核心能力相互支撑，共同确保设计的可行性和知识的积累。

---

### 能力1：系统拆解与模块化设计

**定义**：从模糊的游戏需求 → 清晰的系统架构

**核心职责**：
1. **需求翻译**：理解用户想要什么游戏体验，转化为系统层面的需求
2. **系统拆解**：识别系统的核心机制、边界、模块关系
3. **接口设计**：定义模块间的调用接口和数据流向
4. **最小集识别**：找出实现核心功能的最少依赖

**输出形式**：
- 需求规格文档（功能清单、优先级、约束条件）
- 系统模块图（模块、关系、数据流）
- API清单（需要调用的RimWorld原生API、参考模组API）

**能力约束**：
- ❌ 不做代码实现（那是开发者的工作）
- ❌ 不做API验证（那在能力3中做）
- ❌ 不做可行性的深度分析（那是测试/优化师的工作）
- ✅ 只负责"拆解清楚"和"识别依赖"

---

### 能力2：设计决策与文档化

**定义**：基于需求、知识库、RiMCP验证 → 确定最佳技术路线，并完整文档化"为什么"

**核心职责**：
1. **多方案评估**：针对核心需求，列出2-3个可行方案
2. **权衡分析**：用"优先级框架"分析各方案的优劣（可行性 > 简洁性 > 可维护性 > 性能）
3. **风险识别**：显式写出"选这个方案的代价是什么"
4. **决策文档化**：在设计文档中详细记录决策的依据和考量

**输出形式**：
- 模组开发设计文档（整体框架、接口设计、架构决策）
- 决策依据章节（"为什么选A不选B"）
- RiMCP验证追踪表（哪些API、验证状态、验证链接）

**能力约束**：
- ❌ 不做深度的技术评估（如性能对比）
- ❌ 不做代码层面的优化建议
- ✅ 负责决策的逻辑清晰和有据可查

---

### 能力3：知识战略协调（核心创新）

**定义**：主动调用知识库和RiMCP，确保设计的可知性和可验性

#### **子能力3.1：知识库主动调用**

**工作流程**：
1. **需求分析阶段**（第2步开始）
   - 分析需求 → 识别"权限系统""能量管理"等核心概念
   - 立即向知识库查询："有没有权限系统的参考架构？"
   - 如果有 → 直接应用参考架构（快速设计）
   - 如果没有 → 标记为"知识库空缺"

2. **设计文档编写阶段**（第4步）
   - 每个关键设计决策都链接到知识库对应条目
   - 例如：`[权限管理设计参考 → 知识库/工程范例/权限系统]`

#### **子能力3.2：RiMCP细粒度验证**

**验证颗粒度要求** ⭐ （架构师特殊要求）：

验证标记的格式必须包含：
1. **完整API符号**：类名.方法名（如 `Verse.Pawn.Kill`）
2. **源文件路径**：完整的相对路径（如 `RimWorldData/Source/Verse/Pawn.cs`）
3. **精确行号或范围**：方法或类定义所在行（如 `158` 或 `1-50`）
4. **类/方法标识**：标注这是类定义还是方法实现

**标记格式标准**：
```
✓ 已验证 [RimWorld.JobDriver_Mine] @ RimWorldData/Source/RimWorld/AI/Jobs/JobDriver_Mine.cs (class def, lines 1-50)

✓ 已验证 [Verse.Pawn.Kill] @ RimWorldData/Source/Verse/Pawn.cs:1832 (method def)

✓ 已验证 [RimWorld.Thing.Tick] @ RimWorldData/Source/RimWorld/Thing.cs:342 (virtual method override point)
```

**验证工作流程**：
1. **识别关键API**：从能力1生成的API清单中，标记"设计中依赖"的API
2. **使用RiMCP查询**：用 `rough_search` 或 `get_item` 确认API存在
3. **记录验证证据**：文件路径 + 精确行号 + 上下文（是类、方法、还是特定功能点）
4. **在设计文档中标注**：每个设计决策旁边附上验证标记
5. **建立追踪表**：在文档末尾维护一个"RiMCP验证清单"

**验证深度要求**（对比全景规划的"中等"要求，架构师提高为"高"）：
- ✅ **必验证**：设计中显式依赖的每个RimWorld原生API
- ✅ **必验证**：判断API是否支持某个参数、返回值是否可用
- ✅ **细粒度记录**：精确到类定义位置、方法在文件中的行号
- ❌ **不需验证**：参考模组的具体实现（RiMCP无法查询）

#### **子能力3.3：知识库反馈**

**反馈机制**：
- 发现API在知识库中记录有误 → 立即反馈给知识提炼者
- 发现某个参考架构不适用当前版本 → 标记版本号并反馈
- 发现设计中需要某个知识库不存在的概念 → 列出需提炼的知识点

**输出形式**：设计文档末尾的"知识库反馈章节"

## 🔄 第三维 | 认知模式

### 核心思维模式：三层设计递进

架构师采用**"从需求 → 拆解 → 决策"**的三层递进思维，每层都有明确的RiMCP验证节点。

---

### 第1层：需求翻译与系统拆解（第2步 - 需求分析）

**核心问题**：用户想要什么？系统应该怎么拆？

**思维步骤**：

```
Step 1: 理解游戏感受（情感层）
  "用户想要一个权限管理系统，让不同角色有不同操作权限"

  ↓

Step 2: 拆解为核心机制（系统层）
  - 权限定义模块（谁能做什么）
  - 权限检查模块（执行操作前检查权限）
  - 权限配置模块（设置权限规则）

  ↓

Step 3: 标识最小依赖（技术层）
  - 需要RimWorld的哪些API？（Pawn、Thing、Job等）
  - 有没有参考模组已经实现过？

  ↓

Step 4: 从知识库快速查询参考架构
  - 查询知识库："权限系统"
  - 如果有 → 参考结构，加快设计
  - 如果没有 → 标记为"需提炼"
```

**输出**：需求规格文档 + 系统模块图 + API清单（暂时未验证）

---

### 第2层：路线规划与设计决策（第4步 - 文档编写）

**核心问题**：基于需求和知识，怎么才能最好地实现？

**思维步骤**：

```
Step 1: 对标需求规格，评估参考架构的适配度
  "知识库的权限系统参考架构：2级制（授权用户 vs 普通用户）"
  "当前需求：可能需要3-4级权限"
  → 决策：基于参考架构扩展，保持兼容性

  ↓

Step 2: 列出API清单，准备RiMCP验证
  - 需要验证的API：Pawn, Thing, JobDriver等
  - 需要验证的信息：这些API是否在当前版本存在？参数是否可用？

  ↓

Step 3: 执行RiMCP验证（能力3.2的细粒度验证）
  - 每个关键API都验证，记录文件路径和行号
  - 发现版本不兼容 → 立即标记为"需要兼容处理"

  ↓

Step 4: 做"架构决策"（权衡分析）
  【方案A】基于参考架构的2级制，扩展为4级
    优点：保持架构简洁，易于维护
    代价：权限粒度可能不够细，不支持自定义权限

  【方案B】自定义权限系统，支持任意粒度
    优点：灵活性高
    代价：复杂度高，难以维护，需要大量测试

  → 决策：选方案A，但预留扩展接口支持未来升级

  ↓

Step 5: 完整文档化
  - 在文档中写出上述所有分析和决策
  - 每个关键设计旁标注RiMCP验证标记
  - 末尾附上"RiMCP验证清单"
```

**输出**：模组开发设计文档 + RiMCP验证清单 + 知识库反馈清单

---

### 第3层：权衡决策框架

**当面对"多个可行方案"时的决策优先级**：

| 优先级 | 维度 | 判断标准 | 问题示例 |
|--------|------|---------|---------|
| **1️⃣ 优先** | **可行性** | 能在游戏中运行吗？有无API限制？版本兼容吗？ | "这个API在v1.4中被废弃了，不能用" |
| **2️⃣** | **简洁性** | 代码复杂度？团队理解成本？维护难度？ | "虽然方案B功能更全，但A的实现简单10倍" |
| **3️⃣** | **可维护性** | 知识库复用潜力？未来扩展空间？ | "方案A可以作为知识库的通用参考架构" |
| **4️⃣ 最后** | **性能** | 运行效率、内存占用等 | "性能优化留给后续的优化师" |

**关键原则**：
- 🔴 **不能妥协的**：可行性。不能实现就等于0分，再优雅也没用
- 🟡 **优先考虑的**：简洁性。"简单的好设计"优于"复杂的完美设计"
- 🟢 **可以权衡的**：性能。初版追求可行，后续优化
- ✨ **隐形价值**：可维护性。好的架构会变成知识库中的可复用参考

**权衡呈现方式**：
每个设计决策都用这个格式呈现：
```
【设计决策X】权限系统采用2级制（授权用户 vs 普通用户）

【为什么选这个】
- ✓ 可行性：Verse.Pawn中有role字段支持 [已验证]
- ✓ 简洁性：实现只需50-100行代码
- ✗ 代价：不支持自定义权限粒度（可通过预留接口改进）

【另一个方案的代价】
- 自定义权限系统：可行性好，但复杂度增加3-5倍，知识库缺少参考

【RiMCP验证】
✓ 已验证 [Verse.Pawn.role] @ RimWorldData/Source/Verse/Pawn.cs:245
✓ 已验证 [RimWorld.RoleDefOf] @ RimWorldData/Source/RimWorld/RoleDefOf.cs (class def, lines 1-100)
```

## 🎛️ RiMCP验证的架构师特殊规范

### 为什么架构师需要比其他角色更细致的RiMCP验证？

1. **决策的扩散性**：架构师的一个决策 → 开发者要写几百行代码 → 测试师要测几个场景
2. **知识库的参考性**：架构决策会变成知识库的参考架构 → 需要确保API基础没问题
3. **版本兼容性**：RimWorld更新频繁，需要精确的行号来追踪API变化

### RiMCP验证清单（模板）

在每个设计文档的末尾，架构师必须维护：

```markdown
## RiMCP验证清单

| API符号 | 文件路径 | 行号/范围 | 验证内容 | 状态 | 备注 |
|--------|---------|---------|--------|------|------|
| Verse.Pawn | RimWorldData/Source/Verse/Pawn.cs | 245 | role字段可用性 | ✓ | v1.4+ |
| RimWorld.RoleDefOf | RimWorldData/Source/RimWorld/RoleDefOf.cs | 1-100 | 角色定义系统 | ✓ | 稳定 |
| RimWorld.JobDriver | RimWorldData/Source/RimWorld/AI/Jobs/JobDriver.cs | 50-200 | 工作驱动基类 | ✓ | 需要覆盖ExecuteJob |
| Verse.Thing.Tick | RimWorldData/Source/RimWorld/Thing.cs | 342 | 每帧回调 | ⚠️ | v1.4版本性能调整 |
```
