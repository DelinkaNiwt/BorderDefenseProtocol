## 设计原则

本设计遵循以下原则（按优先级排序）：

| 原则 | 在本设计中的体现 |
|------|----------------|
| **KISS** | IChipEffect接口仅4个方法；CompTriggerBody只管槽位和激活，不关心效果实现 |
| **SRP** | CompTriggerBody管槽位状态机、IChipEffect管效果、TriggerChipComp管芯片数据 |
| **OCP** | 新芯片类型只需实现IChipEffect接口，不修改任何现有代码 |
| **DIP** | CompTriggerBody依赖IChipEffect抽象接口，不依赖具体效果实现 |
| **数据驱动** | 槽位数量、冷却时间、消耗值全部通过XML CompProperties配置 |
| **YAGNI** | 不预设事件系统、不预设芯片组合逻辑，需要时再加 |
| **轮询优于事件** | CompTick驱动切换冷却和状态检查，符合RimWorld tick驱动架构 |

### 1.3 三种触发器体系的统一方式

```
BORDER触发器：
  ThingDef(触发体) + CompTriggerBody(allowChipManagement=true)
  槽位配置：Left(4) + Right(4) + Special(N)
  → 玩家可见芯片槽位UI，可自由装载/卸载芯片
  → 左右手槽：战斗体生成后玩家选择激活/切换
  → 特殊槽：战斗体生成时全部自动激活

近界触发器：
  ThingDef(触发体) + CompTriggerBody(allowChipManagement=false)
  槽位配置：Left(0) + Right(0) + Special(N)
  → 仅特殊槽，预装且锁定，对玩家不可见
  → 玩家看到的是"一体化武器"
  → 底层复用完全相同的CompTriggerBody + IChipEffect

黑触发器：
  ThingDef(触发体) + CompTriggerBody(allowChipManagement=false)
                  + equippedStatOffsets[BDP_TrionCapacity: +N]
  槽位配置：Left(0) + Right(0) + Special(N)
  → 同近界，仅特殊槽，预装锁定，对玩家不可见
  → 额外Trion容量通过Stat系统自动聚合
  → 无需自定义Comp，复用微内核已有机制（6.1 §5.2）
```

### 1.4 OCP验证

```
新增一种芯片效果类型（如投射物类）：
  → 新建 ProjectileChipEffect : IChipEffect
  → 不修改 CompTriggerBody、CompChipSlot、任何现有IChipEffect实现 ✓

新增一种触发器变体（如特殊近界触发器）：
  → 新建 ThingDef XML + 预装芯片配置
  → 不修改任何C#代码 ✓

黑触发器新增其他机制（未来）：
  → 纯数值类 → equippedStatOffsets/statOffsets（零C#）
  → 特殊行为类 → 新增专用Comp，不影响触发器核心系统 ✓
```


```
  //   ① 显式实现IVerbOwner.VerbProperties/Tools（VerbTracker.InitVerbs()通过接口调用，可正确拦截）
  //   ② 直接重写CompGetEquippedGizmosExtra()（装备后Gizmo的正确入口）
  //   ③ 直接访问Holder（装备者Pawn）和VerbTracker
  //   ④ 触发体ThingDef用Inherit="False"覆盖comps，去掉BaseWeapon继承的原版CompEquippable

  ├── ChipSlot[] rightHandSlots      右手槽（数量由Props.rightHandSlotCount决定）
  ├── ChipSlot[] specialSlots    特殊槽（数量由Props.specialSlotCount决定）
  │                              // v2.1新增：内置系统芯片，全部同时激活/关闭
  ├── SwitchState switchState   当前切换状态（Idle / Switching）——仅左右手槽参与
  ├── SwitchContext? pending    待切换的目标（切换中时有值）

  │                              // null=未锁定，非null=被该槽位的双手芯片锁定
  │                              // 运行时引用，不序列化，读档后由激活恢复逻辑重建
  │
  │── 激活上下文（v3.2新增）：
  ├── ChipSlot ActivatingSlot   当前正在激活/关闭的槽位引用（临时上下文）
  │                              // DoActivate/DeactivateSlot中与ActivatingSide同步设置/清除
  │                              // 供Effect类直接读取当前操作槽位的芯片DefModExtension

        │   // 当前所有芯片均为0（瞬间关闭），预留扩展

        └── ...                         各效果类型的专属配置
```

### 2.3 激活状态机

```
                    ActivateChip(slot)
                    （目标侧无激活芯片）
                         │
    ┌────────────────────▼────────────────────┐
    │              Idle（稳定）                 │
    │  leftActive = N 或 -1                    │◄──────────────┐
    │  rightActive  = M 或 -1                    │               │
    └────────────────────┬────────────────────┘               │
                         │ ActivateChip(slot)                  │
                         │ （目标侧已有激活芯片）                │
                         ▼                                     │
    ┌────────────────────────────────────────┐                │
    │           Switching（切换空窗）          │                │
    │  旧芯片 IChipEffect.Deactivate() 已调用  │                │
    │  新芯片等待 cooldownTick                 │                │
    └────────────────────┬───────────────────┘                │
                         │ tick >= cooldownTick                │
                         │ 调用新芯片 IChipEffect.Activate()   │
                         └────────────────────────────────────┘

  注：从空侧激活芯片（无旧芯片）→ 直接调用Activate()，不经过Switching状态
```

### 2.4 不变量

```
任何操作后必须成立：

  ① 每侧激活芯片数 ≤ 1（左右手槽）；特殊槽无此限制（全部激活或全部关闭）
  ② 已装载芯片数 ≤ 该侧槽位数（由Props决定）
  ③ hasRightHand == false 时，rightHandSlots 为空，不参与任何逻辑
  ④ switchState == Switching 时，pending != null
  ⑤ switchState == Idle 时，pending == null
  ⑥ isActive == true 的槽位，loadedChip != null
  ⑦ allowChipManagement == false 时，loadedChip 不可被玩家修改
  ⑧ dualHandLockSlot != null 时，另一侧不可激活新芯片（v2.1新增，v3.4修正类型）
  ⑨ 特殊槽芯片不参与切换状态机（无Switching状态）（v2.1新增）
  ⑩ exclusionTags有交集的芯片不能同时处于激活状态（v2.1新增）
  ⑪ 特殊槽芯片的激活/关闭必须全部同时进行，不可单独操作（v2.1）
  ⑫ activationWarmup 对特殊槽芯片无效（战斗体生成时立即激活）（v2.1）
  ⑬ IsCombatBodyActive==false时，不应有任何芯片处于isActive状态（v3.2新增）
  ⑭ ActivatingSlot仅在DoActivate/DeactivateSlot执行期间非null，其余时间为null（v3.2新增）
```

### 3.2 LoadChip / UnloadChip

```
bool LoadChip(SlotSide side, int slotIndex, Thing chip)

前置条件：
  Props.allowChipManagement == true
  GetSlot(side, slotIndex).loadedChip == null
  chip.TryGetComp<TriggerChipComp>() != null

行为：
  slot.loadedChip = chip
  chip从持有者库存移除（被触发体"吸收"）
  return true

后置条件：
  slot.loadedChip == chip
  slot.isActive == false（装载不等于激活）

─────────────────────────────────────────────────────────────

bool UnloadChip(SlotSide side, int slotIndex)

前置条件：
  Props.allowChipManagement == true
  slot.loadedChip != null

行为：
  if slot.isActive:
    DeactivateChip(side)
  chip = slot.loadedChip
  slot.loadedChip = null
  chip生成到Pawn库存（或地面）
  return true
```

### 3.3 ActivateChip（核心方法）

```
bool ActivateChip(SlotSide side, int slotIndex)

前置条件检查（任一不满足则return false）：
  slot.loadedChip != null
  CanActivateChip(side, slotIndex) == true
  switchState == Idle（切换中不允许再次切换）
  side != Special（特殊槽不走此方法，走ActivateAllSpecial）

CanActivateChip 检查内容（v2.1扩展）：
  · Trion可用量 >= activationCost
  · dualHandLock == false（双手芯片锁定期间不可激活）
  · minOutputPower检查：pawn.GetStatValue(TrionOutputPower) >= chip.Props.minOutputPower
  · exclusionTags检查：当前所有激活芯片的exclusionTags与新芯片无交集
  · effect.CanActivate(pawn, triggerBody) == true

行为流程：

  ┌── 双手芯片？（chip.Props.isDualHand） ──────────────────────────┐
  │                                                                  │
  │  YES：                                                           │
  │    对侧有激活芯片 → 先 DeactivateChip(对侧)                      │
  │    dualHandLock = true                                           │
  │    （后续走直接激活路径，双手芯片不经过Switching状态）              │
  └──────────────────────────────────────────────────────────────────┘

  ┌── 目标侧有激活芯片？ ──────────────────────────────────────────┐
  │                                                                  │
  │  YES（切换路径）：                                                │
  │    oldEffect.Deactivate(pawn, triggerBody)                       │
  │    统一注销旧芯片drainPerDay（如有）                              │
  │    oldSlot.isActive = false                                      │
  │    switchState = Switching                                       │
  │    // v2.1：冷却时间取max(触发体.switchCooldown, 新芯片.activationWarmup)
  │    cooldown = Max(Props.switchCooldownTicks, chip.Props.activationWarmup)
  │    pending = { side, slotIndex, tick + cooldown }                │
  │    return true（激活将在TryResolvePending中完成）                  │
  │                                                                  │
  │  NO（直接激活路径）：                                             │
  │    // v2.1：有引导时间时也走Switching状态
  │    if chip.Props.activationWarmup > 0:                           │
  │      switchState = Switching                                     │
  │      pending = { side, slotIndex, tick + activationWarmup }      │
  │      return true                                                 │
  │    DoActivate(slot)                                              │
  │    return true                                                   │
  └──────────────────────────────────────────────────────────────────┘

DoActivate(slot)（内部方法，切换冷却到期后也调用此方法）：
  CompTrion.Consume(chip.Props.activationCost)
  effect.Activate(pawn, triggerBody)
  slot.isActive = true
  // v2.1：统一注册持续消耗
  if chip.Props.drainPerDay > 0:
    CompTrion.RegisterDrain($"chip_{slot.side}_{slot.index}", chip.Props.drainPerDay)

设计决策：
  切换路径不立即消耗Trion，等空窗期结束后激活时再消耗。
  原因：如果空窗期内Trion耗尽，新芯片不应激活。
```

### 3.3.1 ActivateAllSpecial / DeactivateAllSpecial（v2.1新增）

```
void ActivateAllSpecial()

调用者：战斗模块（战斗体生成时）
行为：
  foreach slot in specialSlots:
    if slot.loadedChip != null && !slot.isActive:
      DoActivate(slot)  // 复用DoActivate，含activationCost消耗和drainPerDay注册

void DeactivateAllSpecial()

调用者：战斗模块（战斗体解除时）
行为：
  foreach slot in specialSlots:
    if slot.isActive:
      DoDeactivate(slot)  // 复用DoDeactivate，含drainPerDay注销

特殊槽规则：
  · 不经过Switching状态机（无切换冷却）
  · 不检查dualHandLock（特殊槽独立于左右手槽）
  · 全部同时激活/关闭，不可单独切换
  · activationWarmup对特殊槽芯片无效（战斗体生成时立即激活）
```

### 3.4 切换冷却结算（懒求值）

```
⚠️ 关键约束：装备后的武器CompTick()不会被调用。
   Pawn_EquipmentTracker.EquipmentTrackerTick()源码：
     if (thingWithComps.def.tickerType != TickerType.Normal)
         thingWithComps.GetComp<CompEquippable>().verbTracker.VerbsTick();
   只调用VerbsTick()，不调用CompTick()。
   武器装备后DeSpawn，TickManager也不再tick它。

方案：懒求值——在IsSwitching/SwitchProgress属性访问时检查并结算。
UI每帧重绘时自然访问这些属性，帧级精度（~60fps）对0.5秒冷却完全足够。

TryResolvePendingSwitch()（私有方法，由IsSwitching/SwitchProgress getter调用）：

  if switchState != Switching || pending == null: return
  if TicksGame < pending.cooldownTick: return（未到期）

  // 冷却到期
  if CanActivateChip(pending.side, pending.slotIndex):
    DoActivate(newSlot)
  // CanActivate失败（Trion不足等）→ 静默取消
  switchState = Idle
  pending = null

IsSwitching属性：
  TryResolvePendingSwitch()
  return switchState == Switching

SwitchProgress属性：
  TryResolvePendingSwitch()
  if pending == null: return 1f
  remaining = pending.cooldownTick - TicksGame
  return 1 - Clamp01(remaining / switchCooldownTicks)

Trion耗尽检查：
  由CompTrion自身的CompTick负责（CompTrion挂在Pawn上，正常tick）。
  CompTriggerBody不重复检查。

IChipEffect.Tick()：
  当前四种效果均不需要per-tick逻辑（武器=被动、护盾=事件驱动、
  隐身=RegisterDrain、辅助=Ability）。
  未来如有效果需要真正的per-tick驱动，升级为MapComponent tick注册表方案。
```

### 3.5 数据流总览

```
芯片激活生命周期（最终设计，v2.1澄清）：

  装备触发体 → 装载芯片（纯物理存放，无效果）
       → 生成战斗体（战斗模块驱动）
           → IsCombatBodyActive = true（v3.2新增）
           → Allocate所有芯片的allocationCost（芯片进入"注册"状态）
           → 特殊槽：ActivateAllSpecial()，全部立即激活
           → 左右手槽：芯片进入"可激活"状态，玩家选择激活/切换
       → 解除战斗体（战斗模块驱动）
           → DeactivateAll()，全部关闭+注销
           → Release所有芯片的allocationCost
           → IsCombatBodyActive = false（v3.2新增）
  注：开发阶段可通过调试Gizmo直接调用API测试，不依赖战斗模块。

  v3.2新增——四态槽位模型（UI视觉区分）：
    空槽 → 灰色
    挂载未注册（有芯片 + IsCombatBodyActive=false）→ 暗黄，图标50%透明
    注册未激活（有芯片 + IsCombatBodyActive=true + isActive=false）→ 暗蓝
    激活（isActive=true）→ 绿色/蓝色高亮

玩家点击Gizmo（左右手槽）：
  → ActivateChip(side, slot)
      → CanActivateChip() 检查（含minOutputPower、exclusionTags、dualHandLock）
      → 切换路径 or 直接激活路径
      → DoActivate: IChipEffect.Activate() + RegisterDrain（如有）
      → CompTrion.Consume(activationCost)

UI每帧重绘（Gizmo/Window）：
  → 访问 IsSwitching / SwitchProgress
  → TryResolvePendingSwitch()
  → 冷却到期时 DoActivate 新芯片

触发体卸下（Notify_Unequipped）：
  → DeactivateAll()
  → 所有IChipEffect.Deactivate() + UnregisterDrain
```

## 4. IChipEffect实现策略

### 4.1 接口定义

```csharp
// 扩展点：所有芯片效果的统一协议
// 新增芯片类型只需实现此接口，不修改任何现有代码
interface IChipEffect
{
    // 激活：芯片开始输出效果
```

### 4.3 消耗模型（v2.1修正）

```
芯片的Trion消耗分为四类，前两类由统一Props配置，后两类由效果专属逻辑处理：

  ┌──────────────────┬──────────────────────────────────────────────┐
  │  消耗类型          │  机制                                          │
  ├──────────────────┼──────────────────────────────────────────────┤
  │  一次性消耗        │  Props.activationCost                         │

  ├──────────────────┼──────────────────────────────────────────────┤
  │  持续消耗          │  Props.drainPerDay                            │
  │  （激活期间）      │  CompTriggerBody.DoActivate()统一RegisterDrain │
  │                   │  CompTriggerBody.DoDeactivate()统一UnregisterDrain│
  │                   │  由微内核CompTrion.CompTick()聚合结算           │

  │  按次消耗          │  效果专属逻辑（DefModExtension配置）            │

  │                   │  远程双射：Verb_BDPDualRanged按侧读取trionCost  │

  │  受击消耗          │  效果专属逻辑（DefModExtension配置）            │

  └──────────────────┴──────────────────────────────────────────────┘
```

### 4.4 WeaponChipEffect——武器类

```
核心思路：
  CompTriggerBody 继承 CompEquippable 并显式实现 IVerbOwner 接口，
  拦截 VerbTracker.InitVerbs() 的接口调用，返回芯片的 Verb/Tool 配置。

技术方案（编码验证后确定）：

  CompEquippable.VerbProperties/Tools 在源码中非 virtual，不能用 override。
  正确方案：CompTriggerBody 声明实现 IVerbOwner，显式实现接口属性：

    // 类声明
    public class CompTriggerBody : CompEquippable, IVerbOwner

    // 显式接口实现（VerbTracker.InitVerbs()通过IVerbOwner接口调用，此处可正确拦截）
    // v2.0变更：改为调用DualVerbCompositor，支持双武器合成
    // v5.1变更（根因修复：芯片Verb彻底脱离VerbTracker）：
    //   VerbProperties只返回parent.def.Verbs（占位），不再合并芯片Verb
    //   原因：芯片Verb若在VerbTracker.AllVerbs中，会被引擎两条路径错误拾取：
    //     ① Pawn_MeleeVerbs.GetUpdatedAvailableVerbsList — 近战选择池（IsMeleeAttack即入池）
    //     ② VerbTracker.GetVerbsCommands Path B — FirstOrDefault(IsMeleeAttack)绑定Y按钮
    //   hasStandardCommand=false只控制Path A（标准Gizmo生成），对①②无效
    //   芯片Verb改为在RebuildVerbs中手动创建，不进入VerbTracker.AllVerbs
    //   Tools始终返回parent.def.tools（触发体"柄"的近战gizmo）

    List<VerbProperties> IVerbOwner.VerbProperties
        => parent.def.Verbs;  // 只返回占位Verb

    List<Tool> IVerbOwner.Tools
        => parent.def.tools;  // 始终返回占位Tools

  原版流程：
    VerbTracker.InitVerbs()
      → directOwner.VerbProperties   ← directOwner类型为IVerbOwner
      → CompEquippable.VerbProperties（隐式接口实现）
      → parent.def.Verbs          ← 共享ThingDef

  显式实现后流程：
    VerbTracker.InitVerbs()
      → directOwner.VerbProperties   ← directOwner类型为IVerbOwner
      → CompTriggerBody.IVerbOwner.VerbProperties（显式实现优先）
      → ActiveVerbProperties ?? parent.def.Verbs  ← 每实例独立

  优势：
    ① 无需 Harmony Patch，无模组兼容性风险
    ② 显式接口实现语义精确，不依赖 virtual/override
    ③ 触发体ThingDef用Inherit="False"去掉BaseWeapon继承的原版CompEquippable

  ⚠️ ThingDef必须用Inherit="False"覆盖comps列表：
    原因：BaseWeapon自带<li><compClass>CompEquippable</compClass></li>
          继承后有两个CompEquippable（原版+CompTriggerBody）
          GetComp<CompEquippable>()返回第一个（原版），CompGetEquippedGizmosExtra()不生效
    方案：<comps Inherit="False">，保留CompProperties_Forbiddable、CompProperties_Styleable、
          CompProperties_Trion和CompProperties_TriggerBody（去掉原版CompEquippable）
    ⚠️ XML中必须写CompProperties_Forbiddable，不能写CompForbiddable（Comp类本身）

Activate(pawn, triggerBody):
  // v3.2变更：从ActivatingSlot读取WeaponChipConfig（T36数据源切换）
  var triggerComp = triggerBody.TryGetComp<CompTriggerBody>();
  var side = triggerComp.ActivatingSide ?? SlotSide.LeftHand;
  var cfg = GetConfig(triggerComp);  // 优先ActivatingSlot，回退AllActiveSlots

  // v4.2变更（B2修复）：近战芯片只有tools无verbProperties时，自动合成melee VerbProperties标记
  // 数据约定：近战芯片只需配置WeaponChipConfig.tools，verbProperties由此处自动合成
  var verbs = cfg?.verbProperties;
  if (verbs == null && cfg?.tools != null && cfg.tools.Count > 0)
      verbs = SynthesizeMeleeVerbProps(cfg.tools);  // verbClass=Verb_BDPMelee

  triggerComp.SetSideVerbs(side, verbs, cfg?.tools);
  RebuildVerbs(pawn, triggerBody);

SynthesizeMeleeVerbProps(tools):  // v4.2新增
  // 从tools[0]合成最小化melee VerbProperties标记
  // 目的：让ComposeVerbs识别此侧为近战（IsMeleeAttack=true需要meleeDamageDef非null）
  // verbClass必须是Verb_BDPMelee（不能是标准Verb_MeleeAttackDamage）
  // DamageDef通过Tool.capacities[0] → ManeuverDef.requiredCapacity匹配 → ManeuverDef.verb.meleeDamageDef查找
  return [VerbProperties { verbClass=Verb_BDPMelee, meleeDamageDef, meleeDamageBaseAmount, defaultCooldownTime }]

Deactivate(pawn, triggerBody):
  var triggerComp = triggerBody.TryGetComp<CompTriggerBody>();
  var side = triggerComp.ActivatingSide ?? SlotSide.LeftHand;
  triggerComp.ClearSideVerbs(side);
  RebuildVerbs(pawn, triggerBody);

GetConfig(triggerComp):  // v3.2新增辅助方法
  优先从triggerComp.ActivatingSlot.loadedChip.def.GetModExtension<WeaponChipConfig>()读取
  回退到遍历AllActiveSlots（兼容读档恢复等边界情况）

RebuildVerbs(pawn):  // v5.1根因修复：芯片Verb彻底脱离VerbTracker
  var comp = triggerBody.TryGetComp<CompTriggerBody>();
  if comp?.VerbTracker == null: return

  // 步骤1：重建VerbTracker（只包含占位Verb + Tool产生的"柄"Verb）
  comp.VerbTracker.InitVerbsFromZero();

  // 步骤2：重新绑定caster（模拟Pawn_EquipmentTracker.Notify_EquipmentAdded的行为）
  // ⚠️ 关键：InitVerbsFromZero()创建全新Verb实例，caster为null
  //    原版中verb.caster只在装备时由Pawn_EquipmentTracker.Notify_EquipmentAdded()设置一次
  if pawn != null && comp.AllVerbs != null:
    foreach verb in comp.AllVerbs:
      verb.caster = pawn

  // 步骤3+4+5+6：合成芯片VerbProperties → 手动创建Verb实例 → 缓存
  CreateAndCacheChipVerbs(pawn)

  // 诊断日志（DevMode下输出VerbTracker.AllVerbs + 芯片Verb缓存状态）

CreateAndCacheChipVerbs(pawn):  // v5.1新增：手动创建芯片Verb实例
  leftHandAttackVerb = null
  rightHandAttackVerb = null
  dualAttackVerb = null

  // 合成芯片VerbProperties（DualVerbCompositor.ComposeVerbs）
  chipVerbProps = DualVerbCompositor.ComposeVerbs(
      leftHandActiveVerbProps, rightHandActiveVerbProps,
      GetActiveSlot(LeftHand), GetActiveSlot(RightHand))
  if chipVerbProps == null: return

  // 为每个芯片VerbProperties手动创建Verb实例
  foreach vp in chipVerbProps:
    verb = (Verb)Activator.CreateInstance(vp.verbClass)
    verb.loadID = $"BDP_Chip_{parent.ThingID}_{index}"
    verb.verbProps = vp
    verb.caster = pawn
    // 反射设置verbTracker（使verb.EquipmentSource正确指向触发体）
    // EquipmentSource = (DirectOwner as CompEquippable)?.parent
    fi_verbTracker.SetValue(verb, VerbTracker)  // Bug7修复：静态构造函数断言字段存在

    // 按type+label分配到缓存槽位
    if verb is Verb_BDPMelee or Verb_BDPShoot:
      side = ParseSideLabel(vp.label)
      if side == LeftHand: leftHandAttackVerb = verb
      elif side == RightHand: rightHandAttackVerb = verb
      else: dualAttackVerb = verb  // 无侧别标签 = 双手触发
    elif verb is Verb_BDPDualRanged:
      dualAttackVerb = verb

  // ⚠️ 手动创建的Verb不在VerbTracker.AllVerbs中，因此：
  //   · 不会被Pawn_MeleeVerbs近战选择池拾取
  //   · 不会被VerbTracker.GetVerbsCommands的Path B拾取
  //   · 只通过Command_BDPChipAttack gizmo使用
  //   · VerbTick/BurstingTick不被VerbTracker.VerbsTick()调用（→ §8.11约束）

  // ⚠️ 反射缓存：Verb.verbTracker是internal字段
  static fi_verbTracker = typeof(Verb).GetField("verbTracker", NonPublic|Public|Instance)
  // Bug7修复：静态构造函数中断言fi_verbTracker != null，避免静默失败

⚠️ T36关键约束：武器数据（verbProperties/tools）存放在WeaponChipConfig（DefModExtension）中，
   不能放在芯片ThingDef.Verbs/tools中。原因：ThingDef.IsWeapon检查category==Item && verbs/tools非空，
   芯片ThingDef有Verbs/tools会被误判为武器。

触发体ThingDef必须定义占位Tool：
  原因：ThingDef.IsWeapon检查verbs/tools是否非空（L958）
        无Verb/Tool的物品无法作为Equipment装备
  方案：定义一个低伤害近战Tool作为占位
        芯片激活后被重写属性替换
```

### 5.2 读档后恢复

```
PostExposeData (LoadSaveMode.PostLoadInit)：

  // 1. 校验槽位不变量
  foreach slot in allSlots:
    if slot.loadedChip == null:
      slot.isActive = false

  // 2. 恢复激活效果
  foreach slot in allSlots:
    if slot.isActive && slot.loadedChip != null:
      effect = slot.loadedChip.GetComp<TriggerChipComp>().GetEffect()
      effect.Activate(pawn, triggerBody)

  // 3. 切换状态保持（CompTick会继续计时）

设计决策：
  读档时重新调用Activate()而非恢复效果内部状态。

```

> - 任何芯片关闭时（Props.drainPerDay > 0）：CompTriggerBody.DoDeactivate()统一调用UnregisterDrain("chip_{side}_{index}")
> - 各IChipEffect实现不再自行调用RegisterDrain/UnregisterDrain

## 8. 双武器系统

### 8.1 设计背景

触发器可同时激活左右各一个槽位。当两侧都激活武器类芯片时，需要产生"双武器攻击"效果。
RimWorld原版不支持双持武器，需要自定义实现。

需求（按组合类型）：
- 近战+近战：左手槽连击N下 → 无停顿衔接右手槽连击M下 → 冷却max(左,右) → 下一轮
- 远程+远程：交替子弹类型，叠加连射数 → 冷却max(左,右)，射程取min
- 近战+远程：远距离射击，贴身自动近战（非双击，按距离切换）

### 8.4 DualVerbCompositor

```
static class DualVerbCompositor
{
    // ── 核心方法（v4.0重构：返回多Verb列表） ──

    // v4.0变更：新增ChipSlot参数，用于判断芯片是否相同和读取芯片label
    static List<VerbProperties> ComposeVerbs(
        List<VerbProperties> leftVerbs,
        List<VerbProperties> rightVerbs,
        ChipSlot leftSlot = null,    // v4.0新增
        ChipSlot rightSlot = null)   // v4.0新增
    {
        if (leftVerbs == null && rightVerbs == null) return null;
        if (leftVerbs == null) return EnsurePrimary(rightVerbs);  // 单侧：isPrimary=true
        if (rightVerbs == null) return EnsurePrimary(leftVerbs);

        bool leftMelee = IsMeleeOnly(leftVerbs);
        bool rightMelee = IsMeleeOnly(rightVerbs);
        bool sameChip = AreSameChip(leftSlot, rightSlot);

        if (leftMelee && rightMelee)
            return ComposeDualMelee(leftVerbs, rightVerbs, sameChip);
        if (!leftMelee && !rightMelee)
            return ComposeDualRanged(leftVerbs, rightVerbs, sameChip);

        return ComposeMixed(leftVerbs, rightVerbs, leftMelee);
    }

    // v4.0返回值变更（每种情况的Verb列表）：
    // v5.0变更（T44）：所有返回的VerbProperties统一设hasStandardCommand=false
    //   芯片Gizmo改由CompGetEquippedGizmosExtra用Command_BDPChipAttack生成
    //   标准路径只为占位Verb生成gizmo
    //
    // v5.1变更（Bug1/Bug2修复）：
    //   Bug1：EnsurePrimary不设label → ParseSideLabel返回null → Verb被错误分配到dualAttackVerb
    //         → 单芯片场景下无攻击Gizmo。修复：EnsurePrimary → TagSideAndEnsurePrimary，显式标记侧别。
    //   Bug2：ComposeMixed未设label → 两个Verb都被ParseSideLabel(null)→null
    //         → 互相覆盖dualAttackVerb，近战Verb丢失。修复：根据leftIsMelee确定各侧label。
    //
    //   单侧 → [该侧Verb(isPrimary=true, hasStandardCommand=false, label=侧别标签)]
    //   近战+近战 → [DualMeleeVerb(isPrimary=true, hasStandardCommand=false, meleeDamageDef, damage)]
    //              + [左Verb(label=BDP_LeftHand), 右Verb(如不同芯片, label=BDP_RightHand)]
    //   远程+远程 → [左Verb(label=BDP_LeftHand), 右Verb(如不同芯片, label=BDP_RightHand), DualRangedVerb(isPrimary=true)]
    //   近战+远程 → [近战Verb(label=近战侧标签), 远程Verb(label=远程侧标签, isPrimary=true)]
    //
    // 每个独立芯片Verb：hasStandardCommand=false（不走标准Gizmo路径），isPrimary=false
    // 双武器合成Verb：isPrimary=true（默认攻击），hasStandardCommand=false，无侧别label
    // 相同芯片：只保留1个独立Verb + 双武器Verb
    // v5.1新增：所有VerbProperties.label设置侧别标签（"BDP_LeftHand"/"BDP_RightHand"），
    //   供CreateAndCacheChipVerbs()按type+label匹配缓存Verb引用
    //
    // ⚠️ CopyVerbProps()浅拷贝避免修改原始DefModExtension数据
    // ⚠️ v4.2新增：CopyVerbProps补充meleeDamageDef/meleeDamageBaseAmount字段

    // ComposeTools不变（v2.0）
    static List<Tool> ComposeTools(left, right) { ... }
}

设计决策：
  Compositor是无状态纯函数——输入两侧数据，输出结果，不持有任何状态。
  v4.0变更：从返回单个合成Verb改为返回完整Verb列表，每个芯片独立Gizmo。
  近战+远程混合时不生成复合Verb，而是利用RimWorld原生的距离选择机制。
```

### 8.5 Verb_BDPMelee（近战攻击，v5.0重命名自Verb_BDPDualMelee）

```
继承：Verb_MeleeAttackDamage

v5.0重命名（T49）：
  · Verb_BDPDualMelee → Verb_BDPMelee
  · 原因：该类同时处理单侧近战和双侧近战，"Dual"名不副实
  · 近战共用1个类 vs 远程分2个类的原因：
    - 近战：单侧和双侧的核心差异只是"执行一侧 vs 两侧"，TryCastShot里一个分支搞定。
      两者都需要同一个ApplyMeleeDamageToTarget()重写（用芯片ThingDef替代触发体）。拆成两个类会大量重复。
    - 远程：Verb_BDPShoot（单发+Trion消耗）和Verb_BDPDualRanged（交替子弹+叠加连射）的burst逻辑完全不同，分开合理。

v5.2新增——重写OrderForceTarget（T50）：
  · 创建AttackMelee job并设置job.verbToUse = this
  · 原因：原版Verb.OrderForceTarget对近战verb创建AttackMelee时不设verbToUse，
    导致JobDriver_AttackMelee传null给TryMeleeAttack，走近战选择池选到拳头/柄
  · 设置verbToUse后，TryMeleeAttack优先使用它（Pawn_MeleeVerbs.cs:97）
  · 使用AttackMelee而非UseVerbOnThing，获得完整的跟随+持续攻击循环
  · 伪代码：
    OrderForceTarget(target):
      job = MakeJob(AttackMelee, target)
      job.verbToUse = this  // 关键：原版不设此字段
      job.playerForced = true
      if target is downed pawn: job.killIncappedTarget = true
      pawn.jobs.TryTakeOrderedJob(job)

v4.0变更（B3修复）：
  · 新增currentChipDef字段，跟踪当前攻击阶段使用的芯片ThingDef
  · 重写ApplyMeleeDamageToTarget()，使用currentChipDef作为DamageInfo的weapon参数
  · 修复伤害标签和战斗日志显示"测试触发体"而非芯片名的问题
  · 原因：原版DamageInfosToApply()使用EquipmentSource.def（触发体ThingDef）作为weapon

v4.2变更（B2修复）：
  · 现在是唯一的近战Verb（不再有标准Verb_MeleeAttackDamage与之竞争）
  · 单侧近战时也使用此类（只执行有tools的一侧，另一侧tools为null则跳过）
  · 由WeaponChipEffect.SynthesizeMeleeVerbProps确保verbClass=Verb_BDPMelee
  · v5.1变更：芯片Verb不在AllVerbs中，彻底隔离于近战选择池

数据获取路径：
  this.caster (Pawn)
    → pawn.equipment.Primary (触发体)
      → CompTriggerBody
        → GetActiveSlot(side).loadedChip.def.GetModExtension<WeaponChipConfig>()
        → WeaponChipConfig.tools（v3.2 T36：从DefModExtension读取，不从ThingDef.tools）

TryCastShot() 逻辑：
  triggerComp = caster.equipment.Primary.GetComp<CompTriggerBody>()

  // 解析侧别label，确定是单侧模式还是双侧模式
  singleSide = ParseSideLabel(verbProps.label)

  // 阶段1：左手槽连击（单侧模式为RightHand时跳过，tools为null则跳过）
  currentChipDef = 左手槽芯片.def  // v4.0新增：设置当前攻击芯片
  leftBurst = 从左手槽芯片读取meleeBurstCount
  for i in 0..leftBurst:
    // Bug6修复：目标已死亡时停止连击，避免对已销毁Thing继续攻击
    if currentTarget.ThingDestroyed: break
    base.TryCastShot() → ApplyMeleeDamageToTarget使用currentChipDef

  // 阶段2：右手槽连击（单侧模式为LeftHand时跳过，无停顿衔接，tools为null则跳过）
  currentChipDef = 右手槽芯片.def  // v4.0新增
  rightBurst = 从右手槽芯片读取meleeBurstCount
  for i in 0..rightBurst:
    // Bug6修复：目标已死亡时停止连击
    if currentTarget.ThingDestroyed: break
    base.TryCastShot()

  currentChipDef = null

ApplyMeleeDamageToTarget() 逻辑（v4.0新增重写）：
  if currentChipDef == null: return base.ApplyMeleeDamageToTarget()
  // 复制原版DamageInfosToApply逻辑，但weapon参数使用currentChipDef而非EquipmentSource.def
  构造DamageInfo(damageDef, damage, armorPen, ..., weaponDef=currentChipDef, ...)
  target.TakeDamage(dinfo)
  // 处理tool.extraMeleeDamages（同样使用currentChipDef）
```

### 8.7 近战+远程混合处理

```
当前实现（v2.0）：
  Compositor检测到一侧近战、一侧远程时：
  VerbProperties → 只返回远程侧的VerbProperties（isPrimary=true）
  Tools → 只返回近战侧的Tools（排除远程侧的枪托Tools）

  效果：
    触发体变成"有真正近战能力的枪"
    远距离 → PrimaryVerb是远程Verb → Pawn射击
    贴身 → RimWorld原版AI在被迫近战时使用Tools → 用近战芯片的刀刃攻击

  ⚠️ 行为细节：
    RimWorld原版AI持枪时倾向于保持距离射击
    只有敌人贴脸时才自动切换近战
```

### 8.10 攻击执行链（v5.2）

```
问题根因（Bug10）：
  原版攻击Job系统绕过job.verbToUse重新查找verb，导致芯片verb不被使用。

  远程攻击链路（原版）：
    Command_VerbTarget.ProcessInput() → verb.OrderForceTarget(target)
      → Verb.OrderForceTarget() → MakeJob(AttackStatic, target, verbToUse=this)
        → JobDriver_AttackStatic.tickIntervalAction
          → pawn.TryStartAttack(target)
            → TryGetAttackVerb() → 从VerbTracker.AllVerbs中查找最佳verb
            → 返回触发体"柄"近战verb（芯片verb不在AllVerbs中）
            → 不射击或用错verb

  近战攻击链路（原版）：
    Verb.OrderForceTarget() → MakeJob(AttackMelee, target)  // 不设verbToUse
      → JobDriver_AttackMelee → TryMeleeAttack(target, verbToUse=null)
        → Pawn_MeleeVerbs.GetUpdatedAvailableVerbsList() → 近战选择池
        → 选到拳头/柄（芯片verb不在AllVerbs中）

解决方案：

  远程（T51）：
    Verb_BDPRangedBase.OrderForceTarget() 重写：
      → MakeJob(BDP_ChipRangedAttack)  // 自定义JobDef
      → job.verbToUse = this
      → JobDriver_BDPChipRangedAttack.tickIntervalAction
        → 直接使用job.verbToUse.TryStartCastOn(target)
        → 不经过pawn.TryStartAttack()，不重新查找verb

  近战（T50）：
    Verb_BDPMelee.OrderForceTarget() 重写：
      → MakeJob(AttackMelee, target)
      → job.verbToUse = this  // 关键：原版不设此字段
      → JobDriver_AttackMelee → TryMeleeAttack(target, verbToUse=this)
        → verbToUse非null时优先使用它（Pawn_MeleeVerbs.cs:97）
        → 跳过近战选择池

JobDriver_BDPChipRangedAttack（v5.2新增）：
  · 复制AttackStatic的持续攻击循环（tickIntervalAction）
  · 核心区别：直接使用job.verbToUse而非pawn.TryStartAttack()
  · tickAction：手动调用verb.VerbTick()推进burst计时（→ §8.11）
  · 目标无效/已销毁/倒地 → EndJobWith(Succeeded)
  · 达到maxNumStaticAttacks → EndJobWith(Succeeded)
  · stance忙碌时跳过（等待冷却/瞄准完成）
  · 存档字段：startedIncapacitated, numAttacksMade

  XML定义（JobDefs_Trigger.xml）：
    <JobDef>
      <defName>BDP_ChipRangedAttack</defName>
      <driverClass>BDP.Trigger.JobDriver_BDPChipRangedAttack</driverClass>
      <reportString>用芯片射击 TargetA。</reportString>
      <allowOpportunisticPrefix>false</allowOpportunisticPrefix>
    </JobDef>

  BDP_DefOf引用：
    public static JobDef BDP_ChipRangedAttack;
```

### 8.11 VerbTick约束与芯片Verb生命周期（v5.2）

```
约束（v5.1设计的副作用）：
  芯片Verb不在VerbTracker.AllVerbs中（§8.3），因此：
    · VerbTracker.VerbsTick() 不会调用芯片Verb的VerbTick()
    · VerbTick()内部调用BurstingTick()推进burst计时
    · burstShotCount > 1的连射verb在第1发后：
      - state = Bursting
      - ticksToNextBurstShot需要每tick递减
      - 但BurstingTick不被调用 → ticksToNextBurstShot永远不递减
      - burst永久卡死，后续发射永远不触发

解决方案（T52）：
  JobDriver_BDPChipRangedAttack.tickAction中手动调用：
    job.verbToUse?.VerbTick()
  每tick推进burst计时，使连射verb正常工作。

  近战不受影响：
    Verb_BDPMelee在一次TryCastShot()中同步执行所有连击（for循环），
    不依赖burst机制（burstShotCount=1），无需VerbTick。

防御性重置（Bug5修复）：
  Verb_BDPRangedBase.OrderForceTarget()中：
    if (Bursting) Reset();
  原因：如果上一次burst被job中断（如玩家取消命令），
    芯片verb的state会永久卡在Bursting（因为BurstingTick不被调用来自然结束）。
    下次OrderForceTarget时先Reset()，确保verb处于干净状态。

芯片Verb生命周期总结：
  创建：RebuildVerbs() → CreateAndCacheChipVerbs() → Activator.CreateInstance
  使用：Command_BDPChipAttack gizmo → OrderForceTarget → Job → TryCastShot
  Tick：JobDriver_BDPChipRangedAttack.tickAction → verb.VerbTick()（仅远程）
  销毁：下次RebuildVerbs()时旧引用被覆盖（GC回收）
  存档：不序列化（RebuildVerbs在读档恢复时由WeaponChipEffect.Activate重建）
```
