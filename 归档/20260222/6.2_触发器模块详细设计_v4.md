## 设计原则

本设计遵循以下原则（按优先级排序）：

| 原则 | 在本设计中的体现 |
|------|----------------|
| **KISS** | IChipEffect接口仅4个方法；CompTriggerBody只管槽位和激活，不关心效果实现 |
| **SRP** | CompTriggerBody管槽位状态机、IChipEffect管效果、TriggerChipComp管芯片数据 |
| **OCP** | 新芯片类型只需实现IChipEffect接口，不修改任何现有代码 |
| **DIP** | CompTriggerBody依赖IChipEffect抽象接口，不依赖具体效果实现 |
| **数据驱动** | 槽位数量、冷却时间、消耗值全部通过XML CompProperties配置 |
| **YAGNI** | 不预设事件系统、不预设芯片组合逻辑，需要时再加 |
| **轮询优于事件** | CompTick驱动切换冷却和状态检查，符合RimWorld tick驱动架构 |

### 1.3 三种触发器体的统一方式

```
BORDER触发器：
  ThingDef(触发体) + CompTriggerBody(allowChipManagement=true)
  槽位配置：Left(4) + Right(4) + Special(N)
  → 玩家可见芯片槽位UI，可自由装载/卸载芯片
  → 左右手槽：战斗体生成后玩家选择激活/切换
  → 特殊槽：战斗体生成时全部自动激活

近界触发器：
  ThingDef(触发体) + CompTriggerBody(allowChipManagement=false)
  槽位配置：Left(0) + Right(0) + Special(N)
  → 仅特殊槽，预装且锁定，对玩家不可见
  → 玩家看到的是"一体化武器"
  → 底层复用完全相同的CompTriggerBody + IChipEffect

黑触发器：
  ThingDef(触发体) + CompTriggerBody(allowChipManagement=false)
                  + equippedStatOffsets[BDP_TrionCapacity: +N]
  槽位配置：Left(0) + Right(0) + Special(N)
  → 同近界，仅特殊槽，预装锁定，对玩家不可见
  → 额外Trion容量通过Stat系统自动聚合
  → 无需自定义Comp，复用微内核已有机制（6.1 §5.2）
```

### 1.4 OCP验证

```
新增一种芯片效果类型（如投射物类）：
  → 新建 ProjectileChipEffect : IChipEffect
  → 不修改 CompTriggerBody、ChipSlot、任何现有IChipEffect实现 ✓

新增一种触发器变体（如特殊近界触发器）：
  → 新建 ThingDef XML + 预装芯片配置
  → 不修改任何C#代码 ✓

黑触发器新增其他机制（未来）：
  → 纯数值类 → equippedStatOffsets/statOffsets（零C#）
  → 特殊行为类 → 新增专用Comp，不影响触发器核心系统 ✓
```

## 2. 核心数据模型

### 2.1 CompProperties_TriggerBody（XML可配置参数）

```xml
<li Class="BDP.Trigger.CompProperties_TriggerBody">

  <!-- ── 槽位配置 ── -->
  <leftHandSlotCount>4</leftHandSlotCount>
    <!-- 左手槽数量，默认4（原作设定） -->

  <rightHandSlotCount>4</rightHandSlotCount>
    <!-- 右手槽数量，默认4（原作设定） -->

  <specialSlotCount>2</specialSlotCount>
    <!-- 特殊槽数量，默认2 -->
    <!-- 内置系统芯片放在此处，战斗体生成时全部同时激活 -->
    <!-- 近界/黑触发器：仅有特殊槽（leftHandSlotCount=0, rightHandSlotCount=0） -->

  <!-- ── 切换配置 ── -->
  <switchCooldownTicks>30</switchCooldownTicks>
    <!-- 切换空窗期长度（ticks），默认30≈0.5秒 -->

  <!-- ── 权限配置 ── -->
  <allowChipManagement>true</allowChipManagement>
    <!-- 玩家是否可以装载/卸载芯片 -->
    <!-- BORDER触发器=true，近界/黑触发器=false -->

  <!-- ── 槽位侧配置 ── -->
  <hasRightHand>true</hasRightHand>
    <!-- 是否有右手槽。某些近界触发器可能只有单侧 -->

  <!-- ── 预装芯片（近界/黑触发器用） ── -->
  <preloadedChips>
    <!-- 首次生成时自动装载的芯片列表 -->
    <!-- BORDER触发器不配置此项 -->
  </preloadedChips>

  <!-- ── 自动激活 ── -->
  <autoActivateOnEquip>false</autoActivateOnEquip>
    <!-- 装备时是否自动激活预装芯片 -->
    <!-- 近界/黑触发器可设为true -->

</li>
```

### 2.2 数据结构层次

```
CompTriggerBody : CompEquippable, IVerbOwner（触发体上的核心Comp，同时也是武器的CompEquippable）
  // 继承CompEquippable的原因：
  //   ① 显式实现IVerbOwner.VerbProperties/Tools（VerbTracker.InitVerbs()通过接口调用，可正确拦截）
  //   ② 直接重写CompGetEquippedGizmosExtra()（装备后Gizmo的正确入口）
  //   ③ 直接访问Holder（装备者Pawn）和VerbTracker
  //   ④ 触发体ThingDef用Inherit="False"覆盖comps，去掉BaseWeapon继承的原版CompEquippable
  ├── ChipSlot[] leftHandSlots       左手槽（数量由Props.leftHandSlotCount决定）
  ├── ChipSlot[] rightHandSlots      右手槽（数量由Props.rightHandSlotCount决定）
  ├── ChipSlot[] specialSlots        特殊槽（数量由Props.specialSlotCount决定）
  │                                  // 内置系统芯片，全部同时激活/关闭
  ├── SwitchState switchState        当前切换状态（Idle / Switching）——仅左右手槽参与
  ├── SwitchContext? pending         待切换的目标（切换中时有值）
  ├── bool dualHandLock              双手芯片锁定标志
  │                                  // true时左右手槽均不可激活新芯片
  │
  │── 按侧Verb存储（双武器系统，§5）：
  ├── List<VerbProperties> leftHandActiveVerbProps   左手槽武器芯片注入的Verb数据
  ├── List<Tool> leftHandActiveTools                 左手槽武器芯片注入的Tool数据
  ├── List<VerbProperties> rightHandActiveVerbProps  右手槽武器芯片注入的Verb数据
  └── List<Tool> rightHandActiveTools                右手槽武器芯片注入的Tool数据
  // 按侧分开存储，由DualVerbCompositor（§5.2）合成最终结果
  // IVerbOwner.VerbProperties/Tools getter调用Compositor

ChipSlot（槽位数据类，非ThingComp，实现IExposable）
  ├── int index                 槽位编号
  ├── SlotSide side             Left / Right / Special
  ├── Thing? loadedChip         已装载的芯片物品（null=空槽）
  └── bool isActive             该槽位的芯片是否当前激活
  // 实现IExposable原因：ChipSlot列表通过Scribe_Collections.Look(LookMode.Deep)
  // 序列化，要求元素类型实现IExposable。loadedChip用Scribe_Deep存档。
  // ⚠️ 槽位容器必须用List<ChipSlot>，不能用ChipSlot[]数组：
  //    Scribe_Collections.Look<T>要求ref List<T>，数组无法推断类型参数（CS0411）

TriggerChipComp（芯片物品上的ThingComp）
  └── CompProperties_TriggerChip
        │
        │── 身份 ──
        ├── Type chipEffectClass        IChipEffect实现类（XML指定全限定类名）
        │   // 实例化：Activator.CreateInstance(chipEffectClass)
        │   // 要求：实现类必须有无参构造函数
        │   // IChipEffect无状态，每次Activate/Deactivate重建无副作用
        │
        │── Trion消耗（三维） ──
        ├── float activationCost        激活时一次性Trion消耗，默认0
        ├── float allocationCost        Trion占用/锁定量（战斗模块读取），默认0
        │   // 芯片声明自己需要锁定多少Trion
        │   // 战斗体激活时由HComp_TrionAllocate读取并调用CompTrion.Allocate()
        ├── float drainPerDay           持续消耗率（Trion/天），默认0
        │   // CompTriggerBody.DoActivate()统一调用RegisterDrain
        │   // CompTriggerBody.DoDeactivate()统一调用UnregisterDrain
        │
        │── 前置条件 ──
        ├── float minOutputPower        最低输出功率要求，默认0
        │   // CanActivate检查pawn.GetStatValue(TrionOutputPower) >= 此值
        │
        │── 时间参数 ──
        ├── int activationWarmup        激活引导时间（ticks），默认0
        │   // per-chip引导时间
        │   // 从空侧激活：等待activationWarmup后生效
        │   // 切换路径：等待max(触发体.switchCooldown, 新芯片.activationWarmup)
        ├── int deactivationDelay       关闭冷却（ticks），默认0
        │   // 关闭后到完全失效的延迟
        │   // 当前所有芯片均为0（瞬间关闭），预留扩展
        │
        │── 约束参数 ──
        ├── List<string> exclusionTags  互斥标签，默认null
        │   // 同标签芯片不能同时激活（跨侧也检查）
        │   // CanActivate检查：当前所有激活芯片的exclusionTags无交集
        ├── bool isDualHand             双手芯片，默认false
        │   // 激活时锁定双侧，对侧有激活芯片则先关闭
        │
        │── 效果专用（通过DefModExtension） ──
        └── ...                         各效果类型的专属配置
```

### 2.3 激活状态机

```
                    ActivateChip(slot)
                    （目标侧无激活芯片）
                         │
    ┌────────────────────▼────────────────────┐
    │              Idle（稳定）                 │
    │  leftActive = N 或 -1                    │◄──────────────┐
    │  rightActive  = M 或 -1                    │               │
    └────────────────────┬────────────────────┘               │
                         │ ActivateChip(slot)                  │
                         │ （目标侧已有激活芯片）                │
                         ▼                                     │
    ┌────────────────────────────────────────┐                │
    │           Switching（切换空窗）          │                │
    │  旧芯片 IChipEffect.Deactivate() 已调用  │                │
    │  新芯片等待 cooldownTick                 │                │
    └────────────────────┬───────────────────┘                │
                         │ tick >= cooldownTick                │
                         │ 调用新芯片 IChipEffect.Activate()   │
                         └────────────────────────────────────┘

  注：从空侧激活芯片（无旧芯片）→ 直接调用Activate()，不经过Switching状态
```

### 2.4 不变量

```
任何操作后必须成立：

  ① 每侧激活芯片数 ≤ 1（左右手槽）；特殊槽无此限制（全部激活或全部关闭）
  ② 已装载芯片数 ≤ 该侧槽位数（由Props决定）
  ③ hasRightHand == false 时，rightHandSlots 为空，不参与任何逻辑
  ④ switchState == Switching 时，pending != null
  ⑤ switchState == Idle 时，pending == null
  ⑥ isActive == true 的槽位，loadedChip != null
  ⑦ allowChipManagement == false 时，loadedChip 不可被玩家修改
  ⑧ dualHandLock == true 时，左右手槽均不可激活新芯片
  ⑨ 特殊槽芯片不参与切换状态机（无Switching状态）
  ⑩ exclusionTags有交集的芯片不能同时处于激活状态
  ⑪ 特殊槽芯片的激活/关闭必须全部同时进行，不可单独操作
  ⑫ activationWarmup 对特殊槽芯片无效（战斗体生成时立即激活）
```

### 3.2 LoadChip / UnloadChip

```
bool LoadChip(SlotSide side, int slotIndex, Thing chip)

前置条件：
  Props.allowChipManagement == true
  GetSlot(side, slotIndex).loadedChip == null
  chip.TryGetComp<TriggerChipComp>() != null

行为：
  slot.loadedChip = chip
  chip从持有者库存移除（被触发体"吸收"）
  return true

后置条件：
  slot.loadedChip == chip
  slot.isActive == false（装载不等于激活）

─────────────────────────────────────────────────────────────

bool UnloadChip(SlotSide side, int slotIndex)

前置条件：
  Props.allowChipManagement == true
  slot.loadedChip != null

行为：
  if slot.isActive:
    DeactivateChip(side)
  chip = slot.loadedChip
  slot.loadedChip = null
  chip生成到Pawn库存（或地面）
  return true
```

### 3.3 ActivateChip（核心方法）

```
bool ActivateChip(SlotSide side, int slotIndex)

前置条件检查（任一不满足则return false）：
  slot.loadedChip != null
  CanActivateChip(side, slotIndex) == true
  switchState == Idle（切换中不允许再次切换）
  side != Special（特殊槽不走此方法，走ActivateAllSpecial）

CanActivateChip 检查内容：
  · Trion可用量 >= activationCost
  · dualHandLock == false（双手芯片锁定期间不可激活）
  · minOutputPower检查：pawn.GetStatValue(TrionOutputPower) >= chip.Props.minOutputPower
  · exclusionTags检查：当前所有激活芯片的exclusionTags与新芯片无交集
  · effect.CanActivate(pawn, triggerBody) == true

行为流程：

  ┌── 双手芯片？（chip.Props.isDualHand） ──────────────────────────┐
  │                                                                  │
  │  YES：                                                           │
  │    对侧有激活芯片 → 先 DeactivateChip(对侧)                      │
  │    dualHandLock = true                                           │
  │    （后续走直接激活路径，双手芯片不经过Switching状态）              │
  └──────────────────────────────────────────────────────────────────┘

  ┌── 目标侧有激活芯片？ ──────────────────────────────────────────┐
  │                                                                  │
  │  YES（切换路径）：                                                │
  │    oldEffect.Deactivate(pawn, triggerBody)                       │
  │    统一注销旧芯片drainPerDay（如有）                              │
  │    oldSlot.isActive = false                                      │
  │    switchState = Switching                                       │
  │    cooldown = Max(Props.switchCooldownTicks, chip.Props.activationWarmup)
  │    pending = { side, slotIndex, tick + cooldown }                │
  │    return true（激活将在TryResolvePending中完成）                  │
  │                                                                  │
  │  NO（直接激活路径）：                                             │
  │    if chip.Props.activationWarmup > 0:                           │
  │      switchState = Switching                                     │
  │      pending = { side, slotIndex, tick + activationWarmup }      │
  │      return true                                                 │
  │    DoActivate(slot)                                              │
  │    return true                                                   │
  └──────────────────────────────────────────────────────────────────┘

DoActivate(slot)（内部方法，切换冷却到期后也调用此方法）：
  CompTrion.Consume(chip.Props.activationCost)
  effect.Activate(pawn, triggerBody)
  slot.isActive = true
  // 统一注册持续消耗
  if chip.Props.drainPerDay > 0:
    CompTrion.RegisterDrain($"chip_{slot.side}_{slot.index}", chip.Props.drainPerDay)

设计决策：
  切换路径不立即消耗Trion，等空窗期结束后激活时再消耗。
  原因：如果空窗期内Trion耗尽，新芯片不应激活。
```

### 3.4 ActivateAllSpecial / DeactivateAllSpecial

```
void ActivateAllSpecial()

调用者：战斗模块（战斗体生成时）
行为：
  foreach slot in specialSlots:
    if slot.loadedChip != null && !slot.isActive:
      DoActivate(slot)  // 复用DoActivate，含activationCost消耗和drainPerDay注册

void DeactivateAllSpecial()

调用者：战斗模块（战斗体解除时）
行为：
  foreach slot in specialSlots:
    if slot.isActive:
      DoDeactivate(slot)  // 复用DoDeactivate，含drainPerDay注销

特殊槽规则：
  · 不经过Switching状态机（无切换冷却）
  · 不检查dualHandLock（特殊槽独立于左右手槽）
  · 全部同时激活/关闭，不可单独切换
  · activationWarmup对特殊槽芯片无效（战斗体生成时立即激活）
```

### 3.5 切换冷却结算（懒求值）

```
⚠️ 关键约束：装备后的武器CompTick()不会被调用。
   Pawn_EquipmentTracker.EquipmentTrackerTick()源码：
     if (thingWithComps.def.tickerType != TickerType.Normal)
         thingWithComps.GetComp<CompEquippable>().verbTracker.VerbsTick();
   只调用VerbsTick()，不调用CompTick()。
   武器装备后DeSpawn，TickManager也不再tick它。

方案：懒求值——在IsSwitching/SwitchProgress属性访问时检查并结算。
UI每帧重绘时自然访问这些属性，帧级精度（~60fps）对0.5秒冷却完全足够。

TryResolvePendingSwitch()（私有方法，由IsSwitching/SwitchProgress getter调用）：

  if switchState != Switching || pending == null: return
  if TicksGame < pending.cooldownTick: return（未到期）

  // 冷却到期
  if CanActivateChip(pending.side, pending.slotIndex):
    DoActivate(newSlot)
  // CanActivate失败（Trion不足等）→ 静默取消
  switchState = Idle
  pending = null

IsSwitching属性：
  TryResolvePendingSwitch()
  return switchState == Switching

SwitchProgress属性：
  TryResolvePendingSwitch()
  if pending == null: return 1f
  remaining = pending.cooldownTick - TicksGame
  return 1 - Clamp01(remaining / switchCooldownTicks)

Trion耗尽检查：
  由CompTrion自身的CompTick负责（CompTrion挂在Pawn上，正常tick）。
  CompTriggerBody不重复检查。

IChipEffect.Tick()：
  当前四种效果均不需要per-tick逻辑（武器=被动、护盾=事件驱动、
  隐身=RegisterDrain、辅助=Ability）。
  未来如有效果需要真正的per-tick驱动，升级为MapComponent tick注册表方案。
```

### 3.6 数据流总览

```
芯片激活生命周期：

  装备触发体 → 装载芯片（纯物理存放，无效果）
       → 生成战斗体（战斗模块驱动）
           → 特殊槽：ActivateAllSpecial()，全部立即激活
           → 左右手槽：芯片进入"可激活"状态，玩家选择激活/切换
       → 解除战斗体（战斗模块驱动）
           → DeactivateAll()，全部关闭+注销
  注：开发阶段可通过调试Gizmo直接调用API测试，不依赖战斗模块。

玩家点击Gizmo（左右手槽）：
  → ActivateChip(side, slot)
      → CanActivateChip() 检查（含minOutputPower、exclusionTags、dualHandLock）
      → 切换路径 or 直接激活路径
      → DoActivate: IChipEffect.Activate() + RegisterDrain（如有）
      → CompTrion.Consume(activationCost)

UI每帧重绘（Gizmo/Window）：
  → 访问 IsSwitching / SwitchProgress
  → TryResolvePendingSwitch()
  → 冷却到期时 DoActivate 新芯片

触发体卸下（Notify_Unequipped）：
  → DeactivateAll()
  → 所有IChipEffect.Deactivate() + UnregisterDrain
```

### 4.3 消耗模型

```
芯片的Trion消耗分为四类，前两类由统一Props配置，后两类由效果专属逻辑处理：

  ┌──────────────────┬──────────────────────────────────────────────┐
  │  消耗类型          │  机制                                          │
  ├──────────────────┼──────────────────────────────────────────────┤
  │  一次性消耗        │  Props.activationCost                         │
  │  （激活时）        │  CompTriggerBody.DoActivate()统一调用Consume() │
  ├──────────────────┼──────────────────────────────────────────────┤
  │  持续消耗          │  Props.drainPerDay                            │
  │  （激活期间）      │  CompTriggerBody.DoActivate()统一RegisterDrain │
  │                   │  CompTriggerBody.DoDeactivate()统一UnregisterDrain│
  │                   │  由微内核CompTrion.CompTick()聚合结算           │
  ├──────────────────┼──────────────────────────────────────────────┤
  │  按次消耗          │  效果专属逻辑（DefModExtension配置）            │
  │  （使用时）        │  远程武器：Verb执行时调用CompTrion.Consume()    │
  │                   │  能力芯片：Ability释放时调用CompTrion.Consume() │
  ├──────────────────┼──────────────────────────────────────────────┤
  │  受击消耗          │  效果专属逻辑（DefModExtension配置）            │
  │  （事件驱动）      │  护盾：HediffComp拦截伤害时调用Consume()       │
  └──────────────────┴──────────────────────────────────────────────┘

  设计原则：
    · 一次性消耗和持续消耗是所有芯片共有的维度 → 统一Props
    · 按次消耗和受击消耗是效果类型专属的 → DefModExtension
    · CompTriggerBody不做按次/受击消耗结算，由产生消耗的具体逻辑自行处理
```

### 4.4 WeaponChipEffect——武器类

```
核心思路：
  CompTriggerBody 继承 CompEquippable 并显式实现 IVerbOwner 接口，
  拦截 VerbTracker.InitVerbs() 的接口调用，返回芯片的 Verb/Tool 配置。

技术方案：

  CompEquippable.VerbProperties/Tools 在源码中非 virtual，不能用 override。
  正确方案：CompTriggerBody 声明实现 IVerbOwner，显式实现接口属性：

    // 类声明
    public class CompTriggerBody : CompEquippable, IVerbOwner

    // 显式接口实现（VerbTracker.InitVerbs()通过IVerbOwner接口调用，此处可正确拦截）
    // 调用DualVerbCompositor，支持双武器合成
    List<VerbProperties> IVerbOwner.VerbProperties
        => DualVerbCompositor.ComposeVerbs(leftHandActiveVerbProps, rightHandActiveVerbProps)
           ?? parent.def.Verbs;

    List<Tool> IVerbOwner.Tools
        => DualVerbCompositor.ComposeTools(leftHandActiveTools, rightHandActiveTools)
           ?? parent.def.tools;

  原版流程：
    VerbTracker.InitVerbs()
      → directOwner.VerbProperties   ← directOwner类型为IVerbOwner
      → CompEquippable.VerbProperties（隐式接口实现）
      → parent.def.Verbs          ← 共享ThingDef

  显式实现后流程：
    VerbTracker.InitVerbs()
      → directOwner.VerbProperties   ← directOwner类型为IVerbOwner
      → CompTriggerBody.IVerbOwner.VerbProperties（显式实现优先）
      → ActiveVerbProperties ?? parent.def.Verbs  ← 每实例独立

  优势：
    ① 无需 Harmony Patch，无模组兼容性风险
    ② 显式接口实现语义精确，不依赖 virtual/override
    ③ 触发体ThingDef用Inherit="False"去掉BaseWeapon继承的原版CompEquippable

  ⚠️ ThingDef必须用Inherit="False"覆盖comps列表：
    原因：BaseWeapon自带<li><compClass>CompEquippable</compClass></li>
          继承后有两个CompEquippable（原版+CompTriggerBody）
          GetComp<CompEquippable>()返回第一个（原版），CompGetEquippedGizmosExtra()不生效
    方案：<comps Inherit="False">，保留CompProperties_Forbiddable、CompProperties_Styleable、
          CompProperties_Trion和CompProperties_TriggerBody（去掉原版CompEquippable）
    ⚠️ XML中必须写CompProperties_Forbiddable，不能写CompForbiddable（Comp类本身）

Activate(pawn, triggerBody):
  var triggerComp = triggerBody.TryGetComp<CompTriggerBody>();
  var side = triggerComp.GetChipSide(this);  // 通过遍历槽位确定自己在哪一侧
  triggerComp.SetSideVerbs(side, myVerbProperties, myTools);
  // SetSideVerbs内部：将数据存入对应侧的leftHandActiveVerbProps/rightHandActiveVerbProps
  // 然后调用 RebuildVerbs()（Compositor自动合成最终结果）
  RebuildVerbs(pawn, triggerBody);

Deactivate(pawn, triggerBody):
  var triggerComp = triggerBody.TryGetComp<CompTriggerBody>();
  var side = triggerComp.GetChipSide(this);
  triggerComp.ClearSideVerbs(side);
  // ClearSideVerbs内部：将对应侧的VerbProps/Tools置null
  RebuildVerbs(pawn, triggerBody);

RebuildVerbs说明：
  武器VerbTracker + Pawn verbTracker双重重建。
  Pawn.TryGetAttackVerb()走pawn.verbTracker，只重建武器VerbTracker不够；两步缺一不可。

触发体ThingDef必须定义占位Tool：
  原因：ThingDef.IsWeapon检查verbs/tools是否非空（L958）
        无Verb/Tool的物品无法作为Equipment装备
  方案：定义一个低伤害近战Tool作为占位
        芯片激活后被重写属性替换
```

## 5. 双武器系统

### 5.1 设计背景

触发器可同时激活左右各一个槽位。当两侧都激活武器类芯片时，需要产生"双武器攻击"效果。
RimWorld原版不支持双持武器，需要自定义实现。

需求（按组合类型）：
- 近战+近战：左手槽连击N下 → 无停顿衔接右手槽连击M下 → 冷却max(左,右) → 下一轮
- 远程+远程：交替子弹类型，叠加连射数 → 冷却max(左,右)，射程取min
- 近战+远程：远距离射击，贴身自动近战（非双击，按距离切换）

### 5.2 架构总览

```
┌─────────────────────────────────────────────────────────────┐
│  CompTriggerBody                                            │
│    leftHandActiveVerbProps / leftHandActiveTools   ← 左手槽武器芯片设置│
│    rightHandActiveVerbProps / rightHandActiveTools ← 右手槽武器芯片设置│
│                                                             │
│    IVerbOwner.VerbProperties ──→ DualVerbCompositor         │
│    IVerbOwner.Tools          ──→ DualVerbCompositor         │
│                                                             │
│  DualVerbCompositor（纯静态工具类）                           │
│    输入：左手槽数据 + 右手槽数据                                 │
│    输出：合成后的 VerbProperties / Tools                     │
│    规则：                                                    │
│      仅一侧有值 → 直接返回该侧                               │
│      近战+近战 → 生成 Verb_BDPDualMelee                     │
│      远程+远程 → 生成 Verb_BDPDualRanged                    │
│      近战+远程 → 合并（远程VerbProps + 近战Tools）            │
│                                                             │
│  Verb_BDPDualMelee（继承 Verb_MeleeAttackDamage）           │
│    TryCastShot() → 左手槽连击 + 右手槽连击                      │
│                                                             │
│  Verb_BDPDualRanged（继承 Verb_Shoot）                      │
│    burst逻辑 → 交替子弹类型，叠加连射数                      │
└─────────────────────────────────────────────────────────────┘
```

特殊槽与双武器系统的关系：
  · 特殊槽芯片不参与 DualVerbCompositor（Compositor 只处理左右手槽）
  · 近界/黑触发器（仅特殊槽）的武器芯片直接通过 IVerbOwner 接口返回，
    leftHandActiveVerbProps/rightHandActiveVerbProps 均为 null，Compositor 返回 null，
    触发体退回 parent.def.Verbs（占位 Tool）
  · 近界/黑触发器暂不设计武器类芯片（YAGNI），特殊槽武器芯片生效路径留待后续

### 5.4 DualVerbCompositor

```
static class DualVerbCompositor
{
    // ── 核心方法 ──

    static List<VerbProperties> ComposeVerbs(
        List<VerbProperties> leftVerbs,
        List<VerbProperties> rightVerbs)
    {
        if (leftVerbs == null && rightVerbs == null) return null;
        if (leftVerbs == null) return rightVerbs;
        if (rightVerbs == null) return leftVerbs;

        // 双侧都有武器 → 判断组合类型
        bool leftMelee = IsMeleeOnly(leftVerbs);
        bool rightMelee = IsMeleeOnly(rightVerbs);

        if (leftMelee && rightMelee)
            return ComposeDualMelee(leftVerbs, rightVerbs);
        if (!leftMelee && !rightMelee)
            return ComposeDualRanged(leftVerbs, rightVerbs);

        // 近战+远程混合 → 只返回远程侧VerbProperties
        // 近战能力由ComposeTools提供（通过Tools）
        return leftMelee ? rightVerbs : leftVerbs;
    }

    static List<Tool> ComposeTools(
        List<Tool> leftTools,
        List<Tool> rightTools)
    {
        if (leftTools == null && rightTools == null) return null;
        if (leftTools == null) return rightTools;
        if (rightTools == null) return leftTools;

        // 双侧都有 → 判断组合类型（需配合ComposeVerbs的判断）
        // 近战+近战 → 由Verb_BDPDualMelee内部处理，此处返回合并列表
        // 远程+远程 → 返回合并列表（远程武器的近战Tools通常是弱枪托）
        // 近战+远程 → 只返回近战侧Tools（排除远程武器的枪托）
        // ⚠️ 近战+远程时排除远程侧Tools的原因：
        //    确保贴身近战时使用的是真正的近战武器（弧月刀刃），
        //    而非远程武器的弱近战（枪托）
        return leftTools.Concat(rightTools).ToList();
        // 近战+远程的特殊处理在实现时细化
    }

    // ── 辅助方法 ──

    static bool IsMeleeOnly(List<VerbProperties> verbs)
        => verbs.All(v => v.IsMeleeAttack);

    // ── 近战+近战合成 ──

    static List<VerbProperties> ComposeDualMelee(left, right)
    {
        return new List<VerbProperties> {
            new VerbProperties {
                verbClass = typeof(Verb_BDPDualMelee),
                isPrimary = true,
                // 冷却时间 = max(左手槽冷却, 右手槽冷却)
                // 其他参数由Verb_BDPDualMelee内部从CompTriggerBody读取
            }
        };
    }

    // ── 远程+远程合成 ──

    static List<VerbProperties> ComposeDualRanged(left, right)
    {
        return new List<VerbProperties> {
            new VerbProperties {
                verbClass = typeof(Verb_BDPDualRanged),
                isPrimary = true,
                range = Mathf.Min(leftRange, rightRange),
                warmupTime = Mathf.Max(leftWarmup, rightWarmup),
                // burstShotCount = 左连射数 + 右连射数
                // 其他参数由Verb_BDPDualRanged内部处理
            }
        };
    }
}

设计决策：
  Compositor是无状态纯函数——输入两侧数据，输出结果，不持有任何状态。
  近战+远程混合时不生成复合Verb，而是利用RimWorld原生的距离选择机制。
```

### 5.5 Verb_BDPDualMelee（近战连击）

```
继承：Verb_MeleeAttackDamage

数据获取路径：
  this.caster (Pawn)
    → pawn.equipment.Primary (触发体)
      → CompTriggerBody
        → leftHandActiveTools / rightHandActiveTools（两侧的原始Tool数据）

近战连击数来源：
  芯片XML配置 <meleeBurstCount>N</meleeBurstCount>
  存储在 CompProperties_TriggerChip 的 DefModExtension 中
  默认值 = 1（单次攻击）

TryCastShot() 逻辑：
  triggerComp = caster.equipment.Primary.GetComp<CompTriggerBody>()

  // 阶段1：左手槽连击
  leftBurst = 从左手槽芯片读取meleeBurstCount
  for i in 0..leftBurst:
    用 leftTool 的 power/armorPen/damageDef 构造 DamageInfo
    对目标施加伤害（独立命中/闪避判定）

  // 阶段2：右手槽连击（无停顿衔接）
  rightBurst = 从右手槽芯片读取meleeBurstCount
  for i in 0..rightBurst:
    用 rightTool 的参数构造 DamageInfo
    对目标施加伤害

  return true

冷却时间：
  max(左手槽cooldownTime, 右手槽cooldownTime)
  通过Compositor生成的复合VerbProperties.defaultCooldownTime设置

战斗日志：
  每次伤害产生独立的战斗日志条目
  视觉上表现为快速连击
```

### 5.6 Verb_BDPDualRanged（远程交替连射）

```
继承：Verb_Shoot

数据获取路径：同Verb_BDPDualMelee

连射数合成：
  总burstShotCount = 左连射数 + 右连射数
  例：左3发 + 右5发 = 总8发

子弹交替规则：
  交替直到一方用完，剩余补齐。
  例（左3右5）：L, R, L, R, L, R, R, R

  分配算法：
    leftRemaining = leftBurstCount
    rightRemaining = rightBurstCount
    for each shot:
      if leftRemaining > 0 && rightRemaining > 0:
        偶数发用左手槽，奇数发用右手槽
        对应侧remaining--
      elif leftRemaining > 0: 用左手槽, leftRemaining--
      else: 用右手槽, rightRemaining--

每发子弹使用对应侧的：
  - projectileDef（子弹类型/外观/伤害）
  - forcedMissRadius（散布）
  - 伤害/穿甲（由projectileDef决定）

射程：min(左射程, 右射程)
瞄准时间（warmupTime）：max(左, 右)
冷却时间：max(左, 右)
连射间隔（ticksBetweenBurstShots）：取平均或较大值

实现方式：
  重写burst相关逻辑，维护burstShotIndex追踪当前第几发
  每次burst shot根据序号选择对应侧的projectileDef
```

### 5.7 近战+远程混合处理

```
Compositor检测到一侧近战、一侧远程时：
  VerbProperties → 只返回远程侧的VerbProperties（isPrimary=true）
  Tools → 只返回近战侧的Tools（排除远程侧的枪托Tools）

效果：
  触发体变成"有真正近战能力的枪"
  远距离 → PrimaryVerb是远程Verb → Pawn射击
  贴身 → RimWorld原版AI在被迫近战时使用Tools → 用近战芯片的刀刃攻击

⚠️ 行为细节：
  RimWorld原版AI持枪时倾向于保持距离射击
  只有敌人贴脸时才自动切换近战
  这与原版枪械+枪托的行为一致
  对"弧月+枪"组合来说是合理的基线行为

未来扩展空间（保留）：
  如果要实现"近身时一边砍一边射击其他目标"：
  → 新增 Verb_BDPHybridAttack
  → Compositor检测到近战+远程时生成复合Verb
  → 当前不实现，但架构不阻碍
```

## 6. 存档与生命周期

### 6.1 存档字段

```
CompTriggerBody.PostExposeData() 存档的字段：

  ChipSlot[] leftHandSlots / rightHandSlots / specialSlots
    每个槽位存档：
      Thing loadedChip    Scribe_Deep（芯片物品完整序列化）
      bool  isActive      Scribe_Values

  切换状态：
    SwitchState switchState    Scribe_Values
    SlotSide    pendingSide    Scribe_Values（切换中时）
    int         pendingSlot    Scribe_Values（切换中时）
    int         cooldownTick   Scribe_Values（切换中时）

  双手锁定：
    bool        dualHandLock   Scribe_Values

TriggerChipComp：无额外存档字段
IChipEffect：无状态（纯行为接口）
```

### 6.2 读档后恢复

```
PostExposeData (LoadSaveMode.PostLoadInit)：

  // 1. 校验槽位不变量
  foreach slot in allSlots:
    if slot.loadedChip == null:
      slot.isActive = false

  // 2. 恢复激活效果
  foreach slot in allSlots:
    if slot.isActive && slot.loadedChip != null:
      effect = slot.loadedChip.GetComp<TriggerChipComp>().GetEffect()
      effect.Activate(pawn, triggerBody)

  // 3. 切换状态保持（CompTick会继续计时）

设计决策：
  读档时重新调用Activate()而非恢复效果内部状态。
  原因：IChipEffect无状态，Activate()是幂等的重建操作。
```

### 6.3 生命周期事件

```
PostSpawnSetup(bool respawningAfterLoad)：
  if !respawningAfterLoad:
    leftHandSlots = new List<ChipSlot>(Props.leftHandSlotCount)   // ⚠️ 必须用List<T>，非数组
    if Props.hasRightHand:
      rightHandSlots = new List<ChipSlot>(Props.rightHandSlotCount)   // Scribe_Collections.Look要求List<T>
    if Props.specialSlotCount > 0:
      specialSlots = new List<ChipSlot>(Props.specialSlotCount)
    if Props.preloadedChips != null:
      foreach config in Props.preloadedChips:
        LoadChipInternal(config.side, config.slot, config.chipDef)

EnsureSlotsInitialized()（私有方法，懒初始化）：
  if leftHandSlots == null:
    leftHandSlots = InitSlots(Left, Props.leftHandSlotCount)
  if Props.hasRightHand && rightHandSlots == null:
    rightHandSlots = InitSlots(Right, Props.rightHandSlotCount)
  if Props.specialSlotCount > 0 && specialSlots == null:
    specialSlots = InitSlots(Special, Props.specialSlotCount)
  // 调用时机：LeftHandSlots/RightHandSlots getter、Notify_Equipped
  // 兼容CharacterEditor等绕过PostSpawnSetup直接装备武器的外部工具

// ⚠️ OwnerPawn必须用CompEquippable.Holder，不能用parent.ParentHolder as Pawn：
//    装备武器时parent.ParentHolder是Pawn_EquipmentTracker（非Pawn），直接as Pawn返回null
//    Holder = (parent.ParentHolder as Pawn_EquipmentTracker)?.pawn（正确）
Notify_Equipped(Pawn pawn)：          // ⚠️ 正确方法名（非PostEquip）
  EnsureSlotsInitialized()            // 懒初始化（兼容外部工具）
  if Props.autoActivateOnEquip:
    自动激活预装芯片

Notify_Unequipped(Pawn pawn)：        // ⚠️ 正确方法名（非PostUnequip）
  DeactivateAll()
  base.Notify_Unequipped(pawn)        // ⚠️ 必须调用：CompEquippable.Notify_Unequipped清理VerbTracker

PostDestroy()：
  DeactivateAll()
  已装载芯片随触发体一起销毁
```

### 6.4 版本升级兼容

```
新增槽位字段：Scribe_Values.Look的defaultValue保证旧存档兼容
槽位数量变化：PostLoadInit中检测并扩展/截断数组
新增IChipEffect类型：不影响已有存档
```
