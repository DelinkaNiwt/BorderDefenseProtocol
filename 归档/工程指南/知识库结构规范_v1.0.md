# 知识库结构规范

## 📋 文件摘要

**用途**：定义RimWorld模组工程项目的知识库的分类结构、存储规范和管理方式

**适用范围**：所有参与RimWorld模组开发工作的AI角色和用户

**版本**：v1.0（初稿）

**最后更新**：2026-01-01

**关键文件**：
- 参考文档：《RimWorld模组工程全景规划_v0.2.md - 三、协作关系与知识流动》
- 制定者：姜子牙（架构设计师）+ 知识提炼者角色设计输入

---

## 🎯 知识库的战略定位

**定义**：知识库是模组开发的"集体记忆"，由知识提炼者管理和维护

**作用**：
- 避免重复工作（已解决的问题无需再研究）
- 降低学习成本（新项目可直接查询相关知识）
- 沉淀最佳实践（参考模组的架构模式、代码框架等）
- 建立知识反馈循环（项目完成后，新知识反哺库）

**流动机制**：
- **调取**：任何角色需要时，直接从知识库查询
- **提炼**：知识提炼者从参考模组、项目过程中提炼新知识
- **反哺**：每个项目完成后，总结经验，新增知识库条目
- **演化**：知识库在循环中不断充实和优化

---

## 📁 知识库的五分类结构

知识库采用5个顶级分类，每个分类对应不同的知识类型：

### 1️⃣ 技术基础（RimWorld_API / Harmony系统）

**定位**：游戏引擎和框架层面的知识

**包含内容**：
- RimWorld API文档（核心类、方法、属性的说明）
- Harmony Patch系统的最佳实践
- 性能优化指南和常见陷阱
- RimWorld版本兼容性信息
- 官方代码范例分析

**知识提炼者的典型产出**：
- API用法总结（基于RiMCP验证）
- Patch常见模式（如何正确Hook游戏函数）
- 版本差异对比表

**例子**：
- `/KB/技术基础/RimWorld_API/Thing系统.md` - Thing类的主要方法和生命周期
- `/KB/技术基础/Harmony系统/Patch最佳实践.md` - 常见的Harmony Patch模式

---

### 2️⃣ 工程范例（权限系统 / 能量管理系统 / 虚像协议实现等）

**定位**：具体的游戏功能实现的架构和设计模式

**包含内容**：
- 完整的系统设计架构（权限系统、能量管理等）
- 代码框架和实现指南
- 参考实现（开源模组中的真实代码）
- 常见的设计模式和权衡

**知识提炼者的典型产出**：
- 系统设计架构报告
- 代码框架（可直接参考的代码结构）
- 参考实现分析（某个开源模组的逆向分析）
- 常见模式对比（多种实现方式的优缺点）

**例子**：
- `/KB/工程范例/权限系统/两级权限设计.md` - 简单权限系统的架构
- `/KB/工程范例/能量管理系统/容器设计模式.md` - 能量容器的实现框架
- `/KB/工程范例/虚像协议实现/状态机设计.md` - 虚像协议的核心逻辑

---

### 3️⃣ 项目记录（MVP试点项目 / ProjectAldon参考资料）

**定位**：具体项目的分析、设计决策和执行记录

**包含内容**：
- 项目分析报告（某个参考模组的完整分析）
- 设计对比（多个实现方案的对比分析）
- 技术决策记录（为什么选择这个方案而不是那个）
- 项目总结文档（MVP项目完成后的经验总结）

**知识提炼者的典型产出**：
- 参考模组的逆向分析报告
- 项目完成后的经验文档
- 技术决策的论证记录

**例子**：
- `/KB/项目记录/MVP试点项目/设计决策记录.md` - MVP项目中的关键决策和理由
- `/KB/项目记录/ProjectAldon参考资料/功能模块分析.md` - ProjectAldon各模块的架构分析
- `/KB/项目记录/参考模组分析/权限系统模组_v1.2_分析报告.md` - 某个权限系统模组的详细分析

---

### 4️⃣ 验证清单（API_RiMCP验证清单 / 可行性评估报告）

**定位**：项目可行性验证和技术风险评估

**包含内容**：
- RiMCP API验证清单（所有涉及的API是否都已验证）
- 兼容性报告（与其他模组、RimWorld版本的兼容性）
- 技术风险评估（项目中不确定的技术点）
- 性能和优化分析

**知识提炼者的典型产出**：
- API验证结果清单（哪些API已验证，哪些还需验证）
- 兼容性测试报告
- 技术可行性评估

**例子**：
- `/KB/验证清单/MVP试点项目/API_RiMCP验证清单.md` - MVP项目涉及的10个API的验证结果
- `/KB/验证清单/MVP试点项目/可行性评估.md` - MVP项目的可行性评估和风险分析
- `/KB/验证清单/兼容性报告/1.5.3_兼容性测试报告.md` - 与RimWorld 1.5.3的兼容性测试

---

### 5️⃣ 参考资料（外部资源链接与摘要）

**定位**：外部知识源的汇总和索引

**包含内容**：
- 官方文档引用（RimWorld官方文档、教程等）
- 社区讨论（论坛、Discord讨论的有价值内容）
- 其他开发者的经验分享（博客、GitHub项目等）
- 工具和库的使用指南（Harmony、Json等）

**知识提炼者的典型产出**：
- 资源索引（链接汇总）
- 资源摘要（关键内容摘录）
- 推荐阅读清单

**例子**：
- `/KB/参考资料/官方文档索引.md` - RimWorld官方文档的链接和目录
- `/KB/参考资料/Harmony使用指南.md` - Harmony库的使用教程和最佳实践（来自官方或社区）
- `/KB/参考资料/社区讨论精选.md` - 高价值的社区讨论汇总

## 📝 文件命名规范

所有知识库条目都应遵循以下命名规范：

### 命名格式
```
[主题]_[子主题]_[版本或日期].md
```

### 示例
- `Thing系统.md` - 简单主题，不需要版本号
- `权限系统_v1.0.md` - 有版本演进的主题
- `API验证清单_2026-01-01.md` - 日期敏感的文档

### 文件内容规范

每个知识库条目应包含以下部分：

```markdown
# [标题]

## 📋 文件摘要
- **作者**：[谁写的]
- **最后更新**：[日期]
- **验证状态**：[✓已验证/✗待验证/?不确定]
- **相关API**：[涉及的RimWorld API，应带验证标记]

## 核心内容
[详细内容]

## 应用示例
[具体的代码或实现例子]

## 相关链接
- [其他相关知识库条目]
- [外部参考资源]

## 版本历史
| 版本 | 日期 | 改动 |
|------|------|------|
| v1.0 | 2026-01-01 | 初稿 |
```

## ✅ 知识提炼者的输出标准

所有由知识提炼者生成的知识库条目都必须满足：

### 验证标记要求
- 所有RimWorld API引用都必须带有验证标记
- 格式：`✓ 已验证 [API类名.方法名] @ RimWorldData/Source/路径/文件名.cs:行号`

### 来源标注
- 如果来自参考模组：标注模组名称和版本
- 如果来自官方文档：标注文档链接
- 如果来自RiMCP验证：标注验证时间和结果

### 示例
```markdown
该系统使用 Thing 类的 Tick 方法实现每帧更新：
✓ 已验证 [RimWorld.Thing.Tick] @ RimWorldData/Source/RimWorld/Thing.cs:342

参考实现来自开源模组 PowerSystem_v1.2：
- GitHub链接：[xxx]
- 分析日期：2026-01-01
```

## 🔄 知识库的维护流程

### 1. 日常维护（任何角色）
- 发现知识库条目过时或错误 → 报告给知识提炼者
- 需要查询知识库 → 直接查询，不需要经过中介

### 2. 知识提炼（知识提炼者）
- 接收新项目需求 → 查询知识库是否有相关内容
- 如无相关内容 → 搜索参考模组，进行逆向解析
- 完成分析 → 生成新的知识库条目

### 3. 知识反哺（项目完成后）
- 项目完成 → 知识提炼者总结经验
- 生成新的知识库条目 → 补充或优化现有条目
- 更新相关的可行性评估和API验证清单

### 4. 版本升级
- 当知识库分类结构需要升级时 → 通知所有角色
- 根据"全景规划v0.2 - 5.x节"的规范演进流程执行

---

## 📌 关键约束

1. **权限管理**：只有知识提炼者有权在知识库中创建新条目
2. **版本控制**：所有知识库文件都在Git版本控制中，每次修改都有提交记录
3. **验证优先**：所有涉及RimWorld API的条目都必须通过RiMCP验证
4. **可追踪性**：每个条目都必须明确标注来源和作者
5. **不中断服务**：知识库的分类结构不会频繁变化，已有的条目不会被删除（只会归档或标记为废弃）

---
