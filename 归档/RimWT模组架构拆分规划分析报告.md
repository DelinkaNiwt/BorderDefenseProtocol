### 1.2 分析方法

本报告采用**自底向上提取 + 自顶向下验证**的双向方法：

1. **自底向上**：从第二阶段的30个条目（15游戏实体+4数据属性+1游戏系统+10概念）和36条关系中，识别出哪些需要独立的代码模块
2. **自顶向下**：从"微内核+模块化"的架构目标出发，验证模块划分是否满足高内聚低耦合原则
3. **交叉验证**：用第三阶段的RimWorld框架知识验证每个模块的技术可行性

### 1.3 关键原则

- **最简最小可行原则**：每个模块只包含实现其核心功能所必需的最少内容
- **单一数据源**：同一份数据只在一个地方持有，其他地方通过引用或查询获取
- **语义正确性**：每个RimWorld系统只承担符合其原始语义的职责
- **可独立验证**：每个模块应能独立编译和基本测试，不强依赖未完成的模块

## 二、微内核识别：为什么是Trion能量系统

### 2.1 微内核的定义标准

在"微内核+模块化"架构中，微内核必须满足以下条件：

| 条件 | 说明 |
|------|------|
| **被所有模块依赖** | 其他模块都需要它才能运作 |
| **不依赖其他模块** | 它自身可以独立存在和运行 |
| **提供统一接口** | 为所有模块提供一致的交互方式 |
| **足够小** | 只包含最核心的功能，不膨胀 |

### 2.2 候选分析

从第二阶段提炼的三大核心技术系统中筛选：

| 候选系统 | 被依赖程度 | 自身独立性 | 接口统一性 | 体量 | 结论 |
|---------|-----------|-----------|-----------|------|------|
| **Trion能量系统** | 极高——所有实体（Pawn/建筑/物品/载具）都需要Trion | 高——可独立存在为资源系统 | 高——`thing.GetComp<CompTrion>()`统一访问 | 小——核心仅CompTrion+Stat | **✓ 微内核** |
| 触发器系统 | 中——仅战斗员Pawn需要 | 低——依赖Trion能量 | 中——仅面向Pawn | 大——三层架构+芯片系统 | ✗ 模块 |
| 战斗体系统 | 中——仅战斗员Pawn需要 | 低——依赖Trion+触发器 | 低——仅面向Pawn | 中——状态机+视觉切换 | ✗ 模块 |

### 2.3 Trion能量系统作为微内核的核心理由

**理由一：跨实体通用性**（来源：2.6 v2.1跨实体认知修正）

Trion不仅是Pawn的属性，它是整个世界观的基石资源：

```
Trion载体类型：
├── Pawn（战斗员）：Trion腺体产生，用于战斗体/触发器
├── Pawn（Trion兵）：制造时注入，自身Trion池
├── Building（制造设施）：消耗Trion制造Trion兵
├── Building（母触发器）：巨型Trion储能
├── Item（触发体/黑触发器）：装备时影响使用者Trion池
└── Vehicle（远征船）：Trion作为燃料
```

只有基于ThingComp的方案才能覆盖所有载体类型——这正是技术选型报告得出CompTrion方案的根本原因。

**理由二：所有其他系统都消耗Trion**（来源：2.4全生命周期、2.6关系矩阵）

从2.6的14条需实现关系中提取Trion相关的：
- R1：Trion腺体→Trion（产生）
- R2：Trion接收器→Trion（增强）
- R5：Trion→副作用（触发）
- R6：黑触发器→Trion（叠加）
- R12：触发体→Trion（消耗）
- R19：制造设施→Trion兵（消耗Trion制造）
- R34：远征船→Trion（消耗燃料）

**7/14条关系直接涉及Trion**——超过一半。

**理由三：体量足够小**

微内核的核心仅包含：
- `CompTrion`：数据容器（cur/max/allocated三个字段 + Consume/Recover/Allocate/Release四个API）
- `StatDef`：三个聚合属性（TrionCapacity/TrionOutputPower/TrionRecoveryRate）

这就是全部。Gene/Need/Hediff等都属于Pawn适配层，是模块而非内核。

### 2.4 微内核边界定义

```
┌─────────────────────────────────────────────────┐
│              微内核：Trion能量核心                  │
│                                                   │
│  CompTrion : ThingComp                           │
│  ├── 数据：cur / max / allocated                 │
│  ├── API：Consume / Recover / Allocate / Release │
│  ├── 事件：OnTrionChanged / OnTrionDepleted      │
│  └── 挂载：任何ThingWithComps                    │
│                                                   │
│  StatDef（三个聚合属性）                           │
│  ├── TrionCapacity（容量上限）                    │
│  ├── TrionOutputPower（输出功率阈值）             │
│  └── TrionRecoveryRate（恢复速率因子）            │
│                                                   │
│  接口约定                                         │
│  └── thing.GetComp<CompTrion>() → 统一访问       │
└─────────────────────────────────────────────────┘
```

**微内核不包含**：Gene_TrionGland、Need_Trion、任何Hediff、任何UI——这些全部属于各功能模块。

## 三、模块全景拆分

### 3.1 拆分依据

模块拆分基于三个维度的交叉分析：

1. **设定维度**：第二阶段识别的15个游戏实体自然形成功能聚类
2. **技术维度**：RimWorld框架的系统边界（Gene/Need/Hediff/Comp/Ability/Job/Faction各自独立）
3. **耦合维度**：高内聚的功能放在同一模块，低耦合的功能拆分到不同模块

### 3.2 模块清单（8个功能模块）

以下按从核心到外围的顺序列出。每个模块标注了它对应的第二阶段实体编号和关系编号。

---

#### 模块A：Pawn Trion适配模块

**职责**：让Pawn（人类殖民者）具备Trion能力——产生、储存、恢复、显示Trion。

**对应实体**：#2 Trion腺体、#1a Trion天赋、#1c Trion恢复速率
**对应关系**：R1（腺体→Trion产生）

**包含内容**：

| 组件 | RimWorld系统 | 职责 | 来源依据 |
|------|-------------|------|---------|
| Gene_TrionGland | GeneDef | Trion腺体——配置天生属性（容量上限、输出功率、恢复速率），初始化时写入CompTrion | 2.4§1（Trion腺体是产生者+储存容器）、2.6（GeneDef） |
| Need_Trion | NeedDef | Trion需求条——驱动恢复循环+提供GUI显示，代理模式读写CompTrion | 技术选型报告（Need作为恢复驱动器+GUI） |
| Hediff_TrionDepleted | HediffDef | Trion耗尽状态——当Trion降至极低时施加debuff | 2.4§4（Trion耗尽后果）、3.1§2（Need极端值效果） |
| StatPart系列 | StatDef | 从Gene/Hediff多源聚合三个Trion属性 | 技术选型报告（Stat聚合层） |

**为什么独立成模块**：这些组件全部是Pawn专属的适配层，与微内核（CompTrion）的跨实体通用性形成清晰边界。没有这个模块，CompTrion仍然可以工作（只是Pawn没有Trion能力）。

**对应实体**：#5 触发体、#6 触发芯片、#8 黑触发器、#17 紧急脱离
**对应关系**：R7（触发体⊃触发芯片装载）、R12（触发体→Trion消耗）

**包含内容**：

| 组件 | RimWorld系统 | 职责 | 来源依据 |
|------|-------------|------|---------|
| ThingDef_TriggerHolder | ThingDef（装备） | 触发体——物理硬件平台，附带CompTriggerSlots | 2.3§1（触发体8槽系统）、2.6（ThingDef） |
| CompTriggerSlots | ThingComp | 芯片槽位管理——主4+副4，装载/卸载/锁定 | 2.3§1.2（芯片装载约束） |
| ThingDef_TriggerChip | ThingDef（物品） | 触发芯片——小型配置模块，定义能力类型和Trion占用量 | 2.3§2.2（芯片注册本质）、2.6（ThingDef物品） |

| Gizmo_TriggerPanel | Gizmo | 触发器控制面板UI——槽位显示、芯片切换按钮 | 机械师借鉴§2（控制组面板→芯片槽位面板） |

**为什么独立成模块**：触发器系统是一个完整的"硬件+软件+UI"体系，内部高度耦合（触发体-芯片-激活三者不可分），但与外部的接口清晰（消耗Trion、输出Trion表现）。

**注意**：黑触发器（#8）虽然在设定上与普通触发器有本质区别（生命意志 vs 技术手段），但在游戏实现层面它是触发体的一个特殊变体（ThingDef），因此归入本模块。其特殊获取方式（R35死亡事件→生成）由事件模块处理。
