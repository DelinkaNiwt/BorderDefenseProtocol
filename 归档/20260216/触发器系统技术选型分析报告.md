# 触发器系统技术选型分析报告

## 1. 问题定义

### 1.1 触发器系统的本质

触发器是一个**"元装备系统"**——不是武器本身，而是一个能动态生成武器/护盾/效果的可配置平台。它是CompTrion能量微内核之上最复杂的功能模块。

核心三层架构（详见"2.3_触发器三层架构过程分析"）：
- **第一层（触发体）**：硬件平台，提供芯片槽位和内置系统
- **第二层（触发芯片）**：可配置的能力模块，定义"做什么"
- **第三层（Trion表现）**：芯片激活后生成的实际武器/护盾/效果

### 1.2 核心设计原则

**通过世界观洞悉抽象本质，但不被世界观束缚。**

具体体现：
- "内置系统"和"触发芯片"本质上是同一种东西——**能力模块**，差异仅是配置参数
- "4+4"槽位布局是配置参数，不是硬编码常量
- "主/副"分侧是配置参数，不同触发体可有不同分组方案
- 三种触发器体系（BORDER/近界/黑触发器）共享同一套底层，差异全部通过配置表达

### 1.3 选型目标

1. **模块化**：触发器系统作为CompTrion微内核上的独立模块，通过已定义的API（Consume/Allocate/Release）与能量系统交互
2. **数据驱动**：芯片类型、槽位配置、组合规则等全部XML可配置，新增内容不改代码
3. **复用引擎**：优先使用RimWorld已有系统（Ability/Verb/Hediff/Comp），最小化自建基础设施
4. **统一底层**：三种触发器体系共享同一套代码，差异通过配置参数表达

### 1.4 与CompTrion能量系统的关系

触发器系统是CompTrion的**上层消费者**，紧密集成：

```
CompTrion（微内核）
  ↑ Allocate()  — 芯片注册时锁定占用量
  ↑ Release()   — 主动解除时释放占用量
  ↑ Consume()   — 激活/使用/维持/流失消耗
  ↑ ForceDeplete() — 被动破裂时强制耗尽
  │
触发器系统（模块）
  ├── CompTriggerSlots — 管理芯片配置和激活状态
  ├── Ability集成 — 芯片激活动作
  └── Trion表现 — 武器/护盾/效果的生成和维持
```

## 2. 系统拆解

触发器系统拆解为8个子问题，每个对应一个需要技术选型的环节：

| # | 子问题 | 本质抽象 | 核心挑战 |
|---|--------|---------|---------|
| P1 | 触发体表示 | 承载芯片的可配置平台 | 如何在RimWorld中表示一个"生成装备的装备" |
| P2 | 触发芯片表示 | 能力模块的数据定义+物理载体 | 静态定义与物理道具的双层设计 |
| P3 | 芯片管理系统 | 统一的模块插槽管理（含"内置"和"可配置"） | 参数化的分组、数量、锁定规则 |
| P5 | 芯片注册与Trion占用 | 模块激活时的资源预留 | 与CompTrion.Allocate/Release的集成 |
| P6 | 芯片激活与切换 | 模块的启动/持续/关闭状态机 | 持续型激活、组内互斥、多组并行 |
| P7 | Trion表现生成 | 模块输出的具体效果 | 三种输出形态各需不同RimWorld系统 |
| P8 | 合成弹 | 多模块输出的组合规则 | 数据驱动的组合规则表 |
| P9 | 三种体系统一 | 同一底层的配置差异化 | 配置参数覆盖所有体系差异 |

## 3. 各子问题技术方案

### 3.1 P1 触发体表示 → Equipment（ThingWithComps装备）

**候选方案对比：**

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| Equipment（装备槽） | 物理道具，可制造/交易/掉落；占据手部符合设定 | 占用武器槽，激活后Trion武器需另外处理 | ✓ 推荐 |
| Apparel（衣物槽） | 不占武器槽 | 语义不符，衣物层级冲突 | ✗ |
| Hediff_Implant | 永久存在 | 无法交易/转让，语义不符 | ✗ |
| 纯Comp | 最灵活 | 无物理存在感 | ✗ |

**推荐理由**：触发体是"手持的物理平台"，Equipment是最自然的映射。激活后触发体不消失（随真身被收容），战斗体手中的武器来自Trion表现——这个转换在P6中处理。

**原版先例**：所有武器装备（ThingWithComps + CompEquippable）。

### 3.2 P2 触发芯片表示 → 双层设计（TriggerChipDef + ThingDef）

**推荐方案：双层设计**

- **TriggerChipDef**（自定义Def）：芯片的静态能力定义——类型、Trion占用量、引导时间、输出表现类型、激活方式等
- **ThingDef**（物理道具）：芯片的物理存在——可制造、可交易、可存储。通过字段引用TriggerChipDef

芯片装载到触发体后，物理道具存入触发体的内部容器（ThingOwner），类似弹药装入枪中。

**为什么双层而非单层？**
- 单用ThingDef：静态能力数据混在物品定义中，不够干净
- 单用自定义Def：无物理存在，无法制造/交易
- 双层：TriggerChipDef专注能力定义，ThingDef专注物理存在，职责分离

**原版先例**：PawnKindDef + ThingDef的双层设计（种类定义与物种定义分离）。

### 3.3 P3 芯片管理系统 → CompTriggerSlots（统一芯片管理Comp）

**核心设计**：CompTriggerSlots是挂载在触发体Equipment上的ThingComp，统一管理所有芯片（包括"内置"和"可配置"）。

**配置层（CompProperties_TriggerSlots）：**

```
slotGroups: List<SlotGroupDef>     // 槽位分组（参数化，非硬编码）
builtInChips: List<TriggerChipDef> // 内置芯片（不占槽、不可拆）
maxActivePerGroup: int             // 每组同时激活上限
maxActiveTotal: int                // 总同时激活上限
isPlayerConfigurable: bool         // 玩家是否可配置槽位
```

**SlotGroupDef（槽位分组定义）：**

```
defName, label                     // 标识和显示名
slotCount: int                     // 该组槽位数量
```

**运行时层（CompTriggerSlots）：**

```
loadedChips: Dictionary<(group, index), TriggerChipDef>  // 已装载芯片
activeChips: Dictionary<SlotGroupDef, TriggerChipDef>    // 各组当前激活芯片
isLocked: bool                     // 战斗中锁定
```

**"内置"与"可配置"的统一**：内置芯片和可配置芯片走同一套激活/注册/消耗逻辑，区别仅是：
- 内置芯片：`occupiesSlot=false, isRemovable=false`（TriggerChipDef上的配置）
- 可配置芯片：`occupiesSlot=true, isRemovable=true`

**配置UI**：ITab_TriggerConfig（检查面板标签页），选中触发体时显示槽位配置界面。

**原版先例**：ITab_Storage（仓储区配置）、CompTransporter（运输舱内容物管理）。

### 3.4 P5 芯片注册与Trion占用 → CompTrion直接集成

不需要新系统。芯片注册/注销是CompTriggerSlots调用CompTrion已有API的直接集成：

```
战斗体激活时：
  CompTriggerSlots.RegisterAllChips()
    → 遍历所有已装载芯片（含内置）
    → 累加各芯片的trionOccupation
    → CompTrion.Allocate(totalOccupation)
    → isLocked = true

主动解除时：
  CompTriggerSlots.UnregisterAllChips()
    → CompTrion.Release(totalOccupation)  // 占用量返还
    → isLocked = false

被动破裂时：
  CompTrion.ForceDeplete()  // 占用量不返还，直接耗尽
  CompTriggerSlots.ForceReset()
    → isLocked = false
```

### 3.5 P6 芯片激活与切换 → Ability触发 + CompTriggerSlots状态管理

**架构决策**：Ability负责"激活这个动作"，CompTriggerSlots负责"激活后的持续状态管理"。

**为什么不让Ability管理持续状态？**
- 原版Ability是"施放→完成"的一次性模型
- 触发芯片需要"施放→持续→关闭"的持续模型
- 强行扩展Ability为持续型会侵入引擎设计
- 让CompTriggerSlots管理持续状态更干净——Ability只是触发入口

**激活流程：**

```
玩家点击芯片Gizmo（或AI决策）
  → Ability.StartCast()（warmupTicks = 芯片引导时间）
  → warmup期间：旧表现已消散，新表现未生成（空窗期）
  → Ability.Cast()完成
    → CompTriggerSlots.ActivateChip(group, chipDef)
      → 检查组内互斥，关闭同组旧芯片
      → 设置activeChips[group] = chipDef
      → CompTrion.Consume(激活消耗)
      → 生成Trion表现（P7）
```

**切换流程：**

```
玩家点击另一个芯片Gizmo
  → CompTriggerSlots.DeactivateChip(group)
    → 消散旧Trion表现
    → activeChips[group] = null
  → 新芯片的Ability.StartCast()（进入空窗期）
  → ... 同上
```

**双持**：两个SlotGroup各自独立维护activeChip，互不干扰。

**Ability的来源**：芯片装载到触发体时，CompTriggerSlots为每个芯片动态创建对应的Ability并注册到Pawn_AbilityTracker。卸载时移除。

**原版先例**：Gene授予Ability（Gene.PostAdd → pawn.abilities.GainAbility）。

### 3.6 P7 Trion表现生成 → 按输出类型分治，统一由芯片配置驱动

Trion表现的本质是"芯片输出的可配置效果"。不同输出类型使用不同的RimWorld系统实现，但统一由TriggerChipDef的配置字段驱动。

| 输出类型 | RimWorld系统 | 实现方式 | 原版先例 |
|---------|-------------|---------|---------|
| 近战武器实体 | Verb注入 + 自定义渲染 | 激活时向Pawn添加MeleeVerb，附带视觉渲染 | Pawn_MeleeVerbs |
| 远程投射物 | Verb注入 + 自定义Projectile | 激活时向Pawn添加RangedVerb，发射自定义投射物 | CompEquippable的Verb管理 |
| 护盾 | HediffComp + 伤害拦截 | 作为战斗体Hediff的动态子效果 | CompShield / CompProjectileInterceptor |
| 持续效果 | HediffComp + Effecter | 隐身/蜘蛛线等作为Hediff效果 | 各种HediffComp |

**关键设计**：TriggerChipDef通过配置字段指定输出类型和参数，具体生成逻辑由对应的**ManifestationWorker**子类处理（Worker委托模式）：

```
TriggerChipDef
├── manifestationWorkerClass: Type  // 指向具体的Worker子类
├── manifestationDef: Def           // 输出物的定义（VerbDef/ProjectileDef/HediffDef等）
└── manifestationParams: ...        // 输出参数

ManifestationWorker（基类）
├── OnActivate(pawn, chipDef)   // 芯片激活时生成表现
├── OnSustain(pawn, chipDef)    // 持续维持（每tick）
├── OnDeactivate(pawn, chipDef) // 芯片关闭时消散表现
│
├── ManifestationWorker_MeleeWeapon   // 近战武器
├── ManifestationWorker_RangedWeapon  // 远程武器
├── ManifestationWorker_Shield        // 护盾
├── ManifestationWorker_Effect        // 特殊效果
└── ...（可扩展）
```

新增芯片类型只需：1）定义新的TriggerChipDef（XML），2）如果需要新的输出类型则实现新的ManifestationWorker子类（C#）。

### 3.7 P8 合成弹 → 组合规则Def + 自定义Projectile

**本质**：两个激活芯片的输出按规则合并为新输出。

**技术方案**：数据驱动的组合规则表。

```
CompositeRuleDef（组合规则定义，XML）
├── chipA: TriggerChipDef    // 输入芯片A
├── chipB: TriggerChipDef    // 输入芯片B
├── resultProjectile: ThingDef  // 输出投射物
├── trionCostMultiplier: float  // 消耗倍率
└── requiresBothActive: bool    // 是否要求两个芯片同时激活
```

**触发逻辑**：CompTriggerSlots检测到多组各有子弹型芯片激活时，查询CompositeRuleDef数据库。匹配到规则则提供合成弹选项（Gizmo），玩家可选择发射合成弹或各自独立发射。

**原版先例**：RecipeDef（配方系统，输入→输出的数据驱动规则）。

### 3.8 P9 三种体系统一 → 同一底层，配置参数差异化

三种触发器体系共享CompTriggerSlots + TriggerChipDef + ManifestationWorker的完整底层，差异全部通过配置参数表达：

| 差异维度 | 配置参数 | BORDER | 近界 | 黑触发器 |
|---------|---------|--------|------|---------|
| 槽位可配置 | `isPlayerConfigurable` | true | false | false |
| 芯片可拆卸 | TriggerChipDef.`isRemovable` | true | false | false |
| 槽位分组 | `slotGroups` | 2组(4+4) | 1组(N) | 1组(1) |
| 性能数值 | TriggerChipDef各属性值 | 标准 | 较高 | 极高 |
| Trion来源 | CompTrion配置 | 仅自身 | 仅自身 | 自身+额外源 |
| 装备条件 | Equipment的CanEquip检查 | 训练 | 各异 | 人格兼容 |

**对玩家的呈现差异**：
- BORDER触发器：有ITab_TriggerConfig配置界面，玩家可拖拽芯片
- 近界触发器：无配置界面（`isPlayerConfigurable=false`），能力固定
- 黑触发器：无配置界面，但有独特的视觉效果和增强数值

**代码层面完全统一**——一个CompTriggerSlots类处理所有三种体系。

### 4.3 技术映射总览图

一张图看清"原作概念 → 抽象本质 → RimWorld技术方案"的三层对应关系：

```
  原作概念                    抽象本质                   RimWorld技术方案
  ─────────────────────────────────────────────────────────────────────

  触发体(无刃柄状装置)  ──→  可配置平台        ──→  Equipment (ThingWithComps)
       │                                              │
       ├─ 8槽(主4+副4)   ──→  参数化插槽管理   ──→  CompTriggerSlots
       │                                              + SlotGroupDef
       ├─ 内置系统        ──→  不可拆卸的       ──→  TriggerChipDef
       │  (紧急脱离等)         能力模块(同类)         (occupiesSlot=false)
       │                                         │
  触发芯片(小型模块)    ──→  能力模块          ──→  ├─ TriggerChipDef (能力定义)
       │                     (统一抽象)              └─ ThingDef (物理道具)
       │
       ├─ 芯片注册        ──→  资源预留          ──→  CompTrion.Allocate()
       │
       ├─ 芯片激活        ──→  模块启动          ──→  Ability (触发入口)
       │  (引导→生成)          状态机                  + CompTriggerSlots (状态)
       │

  合成弹(两芯片组合)    ──→  组合规则          ──→  CompositeRuleDef (数据驱动)
```

### 4.5 关键技术选型总结

| 子问题 | 选用的RimWorld系统 | 选型理由 |
|--------|-------------------|---------|
| 触发体 | Equipment（ThingWithComps） | 物理道具，可制造/交易/掉落 |
| 触发芯片 | TriggerChipDef（自定义Def）+ ThingDef（物理道具） | 能力定义与物理存在分离 |
| 芯片管理 | CompTriggerSlots（自定义ThingComp） | 统一管理内置+可配置芯片，参数化槽位 |
| Trion占用 | CompTrion.Allocate/Release（直接集成） | 无需新系统，复用已有API |
| 芯片激活 | Ability（触发入口）+ CompTriggerSlots（状态管理） | Ability提供warmup/Gizmo，Comp管理持续状态 |
| Trion表现 | ManifestationWorker（Worker委托模式） | 按输出类型分治，可扩展 |
| 合成弹 | CompositeRuleDef（数据驱动规则表） | XML可配置，新增规则不改代码 |
| 三种体系 | 配置参数差异化 | 同一底层，零代码分支 |

### 4.7 数据流图

#### 4.7.1 芯片激活数据流

```
玩家点击芯片Gizmo
  │
  ▼
Ability.StartCast()
  │  warmupTicks = TriggerChipDef.warmupTime
  │
  ├── warmup期间（空窗期）:
  │     CompTriggerSlots.DeactivateChip(group)  ← 如果同组有旧芯片
  │       → ManifestationWorker.OnDeactivate()  ← 旧表现消散
  │
  ▼ warmup完成
Ability.Cast()
  │
  ▼
CompTriggerSlots.ActivateChip(group, chipDef)
  │
  ├── CompTrion.Consume(chipDef.activationCost)  ← 激活消耗
  │
  ├── activeChips[group] = chipDef
  │
  └── ManifestationWorker.OnActivate(pawn, chipDef)
        │
        ├── _MeleeWeapon: 向Pawn注入近战Verb + 启动武器渲染
        ├── _RangedWeapon: 向Pawn注入远程Verb + 配置Projectile
        ├── _Shield: 添加护盾HediffComp到战斗体Hediff
        └── _Effect: 添加效果HediffComp + 启动Effecter
```

#### 4.7.2 战斗持续消耗数据流

```
每tick（战斗体存续期间）:

  CompTriggerSlots（通过战斗体Hediff的HediffComp驱动）
    │
    ├── 遍历所有激活中的芯片:
    │     if chipDef.sustainCostPerTick > 0:
    │       CompTrion.Consume(sustainCostPerTick)
    │
    ├── ManifestationWorker.OnSustain()
    │     └── 各Worker维持自身表现（Verb保持/护盾维持/效果持续）
    │
    └── 耗尽检测:
          if CompTrion.Available <= 0:
            → 触发被动破裂流程
            → CompTrion.ForceDeplete()
            → 所有ManifestationWorker.OnDeactivate()
            → 移除Hediff_CombatBody
```

#### 4.7.3 合成弹判定数据流

```
两组各有子弹型芯片激活时:

  CompTriggerSlots.CheckComposite()
    │
    ├── chipA = activeChips[groupA]
    ├── chipB = activeChips[groupB]
    │
    ├── 查询 DefDatabase<CompositeRuleDef>
    │     .Where(rule => rule.Matches(chipA, chipB))
    │
    ├── 匹配成功:
    │     → 显示合成弹Gizmo（玩家可选择）
    │     → 发射时使用 rule.resultProjectile
    │     → CompTrion.Consume(baseCost × rule.costMultiplier)
    │
    └── 无匹配:
          → 各自独立发射
```

## 6. 风险与待决事项

### 6.1 已识别风险

| # | 风险 | 影响 | 缓解措施 |
|---|------|------|---------|
| 1 | Ability系统是一次性模型，芯片需要持续激活 | 需要Ability与CompTriggerSlots的职责划分足够清晰 | Ability仅作触发入口，持续状态完全由Comp管理 |
| 2 | Verb动态注入可能与原版装备Verb冲突 | 战斗体状态下Pawn同时有触发体Equipment的Verb和Trion武器的Verb | 战斗体激活时禁用触发体自身的Verb，仅使用Trion武器Verb |
| 3 | 双持（两组同时激活）的Verb管理复杂度 | 两个Verb同时存在时的优先级和切换逻辑 | 参考原版近战Verb选择机制（Pawn_MeleeVerbs） |
| 4 | ManifestationWorker的渲染层实现复杂 | Trion武器需要自定义视觉表现（不是真实装备） | 可先用简化渲染，后续迭代完善 |

### 6.2 待决事项

| # | 事项 | 决策点 | 影响范围 |
|---|------|--------|---------|
| 1 | TriggerChipDef是否需要继承Def基类 | 继承Def可进入DefDatabase，但增加复杂度 | P2 芯片数据定义 |
| 2 | 芯片Ability的动态注册/注销时机 | 装载时注册 vs 战斗体激活时注册 | P6 激活流程 |
| 3 | 近战Trion武器的Verb注入具体方案 | 直接添加到Pawn.meleeVerbs vs 通过临时Equipment | P7 近战表现 |
| 4 | 合成弹是自动触发还是玩家手动选择 | 影响操作复杂度和战术深度 | P8 合成弹交互 |
| 5 | 触发体Equipment在战斗体状态下的处理 | 隐藏/卸下/保持装备但禁用 | P1/P6 状态切换 |
