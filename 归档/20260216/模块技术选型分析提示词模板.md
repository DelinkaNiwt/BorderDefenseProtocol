只看我指明的文件，不要看其它文件。

## 核心原则（必须严格遵守）

### 原则1：洞悉本质，不被世界观束缚
通过世界观设定去理解系统的抽象本质骨架，但绝不能把世界观设定当作技术设计的硬约束。
- 错误示范：因为世界观说"内置系统不占槽位"，就把内置系统设计成独立于芯片的另一套逻辑
- 正确做法：识别出"内置系统"和"芯片"本质上是同一种抽象（能力模块），差异仅是配置参数
- 所有数值、数量、分组方式都应该是可配置参数，不是硬编码常量

### 原则2：架构优先，预留扩展性
做的是架构规划，不是功能实现。关注：
- 系统的抽象本质是什么？
- 哪些是核心不变量，哪些是可配置变量？
- 不同子问题各自适合哪种RimWorld技术方案？
- 各方案的权衡是什么？

### 原则3：微内核+模块化
CompTrion能量系统是微内核，当前分析的模块是插在上面的功能模块。
- 模块通过CompTrion已定义的API（Consume/Allocate/Release等）与能量系统交互
- 模块内部也应遵循模块化——用Worker委托模式、数据驱动等方式保持可扩展性

### 原则4：复用引擎，最小自建
优先使用RimWorld已有系统（Comp/Ability/Hediff/Verb/Worker/Stat等），只在引擎无法满足时才自建。
每个技术选型都要给出：候选方案 → 对比 → 推荐 → 理由 → 原版先例。

## 工作流程

1. **完整阅读**所有指定的输入材料，理解该模块的世界观机制
2. **系统拆解**：将模块拆解为若干子问题，每个子问题对应一个需要技术选型的环节
3. **逐一提问**（每次只问一个问题）：
   - 分析范围？
   - 与已有系统（CompTrion等）的集成方式？
   - 报告深度？
   - 其它需要澄清的关键决策点？
4. **提出2-3种整体架构方案**，分析权衡，给出推荐
5. **分段呈现设计**，每段获得确认后继续
6. **生成报告文档**，保存到：c:\NiwtDatas\Projects\RimworldModStudio\临时\RimWT\技术选型\[模块名称]技术选型分析报告.md

## 报告输出格式

参考已完成的报告，包含以下章节：
1. 问题定义（本质、设计原则、选型目标、与CompTrion的关系）
2. 系统拆解（子问题列表表格）
3. 各子问题技术方案（候选对比表 + 推荐 + 理由 + 原版先例）
4. 整体架构概览（模块关系图 + 根据模块特点选择合适的补充图表 + 关键技术选型总结表）
5. 原版先例参考（借鉴点对照表）
6. 风险与待决事项

### 图表要求

所有图表使用ASCII art（Markdown代码块）。

**必选图（每份报告都要有）：**
- 模块关系图：展示模块内部的分层和组件关系

**可选图（根据模块特点自行判断是否需要，以及用什么形式表达）：**

| 图表类型 | 适用场景 | 不适用场景 |
|---------|---------|-----------|
| 状态流转图 | 系统有明确的多阶段生命周期或状态机 | 系统是无状态的或状态简单 |
| 技术映射图 | 原作概念与RimWorld系统的对应关系复杂，需要可视化辅助理解 | 映射关系简单，表格已足够 |
| 数据流图 | 数据在多个组件间流转，路径复杂 | 数据流向单一直接 |
| 配置对比图 | 同一底层有多种变体，差异通过配置表达 | 系统无变体 |
| 层级/继承图 | 系统有清晰的层级结构或继承关系 | 系统是扁平的 |

**原则：图是为了让文字难以表达的关系变得直观。如果表格或文字已经足够清晰，不需要强加图表。每张图都应该有明确的"这张图帮读者看清了什么"的回答。**

报告深度：架构概述级——分析到"用什么RimWorld系统实现什么功能"的层面，不深入到字段级设计。

## 注意事项

- 主动"解释原因"——每个选型决策都要说明为什么选这个、为什么不选其它
- 中文回答，回答完留下签名（当前AI模型名称）
- 文档遵循元信息模板格式
- 积极查询网络最新知识，不过多依赖预训练数据

## 从实践中提炼的经验教训

1. **最容易犯的错误**：按世界观设定写死逻辑。AI倾向于把世界观描述直接翻译为代码结构，需要明确提醒"洞悉本质，不被束缚"
2. **容易遗漏的事**：子问题之间的交叉依赖。拆解后要检查子问题之间是否有隐含的耦合关系
3. **深度控制**：架构概述级足够。过早深入字段级设计会消耗大量上下文且容易在后续被推翻
4. **已完成报告的价值**：每完成一份报告，都应加入后续分析的参考列表，让AI理解已有的架构决策和设计风格
