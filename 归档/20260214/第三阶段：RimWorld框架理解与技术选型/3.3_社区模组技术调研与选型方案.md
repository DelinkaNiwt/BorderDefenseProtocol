## 第1章 Trion能量系统选型 ★★★

> 2.6实体：Trion(数据)、Trion天赋/输出功率/恢复速率(数据) | 关系：R1/R2/R6/R12/R34

### 1.1 需求分析

Trion能量系统是RimWT的核心资源系统，具有以下特殊性：

**数值特征**：
- 连续浮点值（非离散整数），范围0~MaxTrion
- 多来源叠加：Trion腺体（基因天生）+ Trion接收器（植入体增幅）+ 黑触发器（装备叠加）
- 多消耗模式：占用（芯片注册锁定）、维持（战斗体持续消耗）、使用（芯片激活消耗）、流失（战斗体破裂全部流失）
- 战斗中不可恢复（战斗体状态下Trion只减不增）
- 非战斗状态下缓慢恢复

**行为特征**：
- 需要实时UI显示（Gizmo资源条）
- 需要阈值触发（Trion耗尽→临界状态→战斗体破裂）
- 需要Notify通知链（任何Trion变化→重算占用→可能触发紧急脱离）
- 三个派生属性：Trion天赋（上限决定因子）、输出功率（消耗速率因子）、恢复速率

### 1.2 方案对比：Need vs Stat vs Gene数值 vs 自定义资源池

| 方案 | 原版先例 | 优势 | 劣势 | 扩展性 |
|------|---------|------|------|--------|
| **A. 自定义Need** | Need_MechEnergy、Need_KillThirst | 原生UI支持（需求面板自动显示）；150 ticks自动调度NeedInterval；IsFrozen冻结机制可重写；MaxLevel可由Stat驱动 | 更新频率固定150 ticks（约2.5秒），战斗中可能不够精细；Need面板位置固定，自定义UI受限；Need设计上是"被动下降"模型，Trion的"多模式消耗"需要大量重写 | 中等——可继承Need重写，但偏离Need设计意图 |
| **B. 自定义Stat** | 带宽（StatDefOf.MechBandwidth） | Stat系统天然支持多来源叠加（statOffsets/statFactors）；与装备/Hediff/基因自动联动；GetStatValue一行获取总量 | Stat是"属性值"不是"资源池"——没有当前值/最大值的概念；无内置UI；无自动更新机制；无阈值触发 | 低——Stat不适合做动态资源 |
| **C. Gene_Resource** | Gene_Hemogen（血源素） | 原生资源Gizmo（GeneGizmo_Resource）；resourceLossPerDay自动消耗；与基因系统深度集成 | 强制绑定基因系统——非基因来源的Trion增幅难以接入；Gene_Resource的消耗模型过于简单（仅fallPerDay）；Gizmo样式固定 | 低——Gene_Resource设计上是单一基因资源 |
| **D. 自定义Tracker+Stat混合** | 机械师系统（Pawn_MechanitorTracker + StatDefOf.MechBandwidth） | Tracker管理资源池状态（当前值/占用/消耗模式）；Stat管理总量（多来源自动叠加）；完全自定义UI和更新频率；Notify链可自由设计 | 实现量最大；需要自行处理存档序列化；需要自行实现Gizmo | 最高——完全可控，不受原版框架限制 |

### 1.3 三个派生属性的实现方案

| 派生属性 | 语义 | 推荐挂载方式 | 理由 |
|---------|------|-------------|------|
| **Trion天赋** | Pawn的Trion上限决定因子（天生，不可改变） | 自定义StatDef `RimWT_TrionTalent` | 天赋是静态属性，Stat天然适合；可被基因的statOffsets影响 |
| **输出功率** | 战斗体消耗Trion的速率因子 | 自定义StatDef `RimWT_TrionOutputPower` | 输出功率受装备（触发体品质）和状态（芯片类型）影响，Stat自动叠加 |
| **恢复速率** | 非战斗状态下Trion恢复速率 | 自定义StatDef `RimWT_TrionRecoveryRate` | 恢复速率受Hediff（Trion接收器效率）和环境影响，Stat自动叠加 |

### 1.4 社区参考

| 模组/系统 | 实现方式 | 与RimWT的相关性 |
|----------|---------|----------------|
| **原版机械师带宽** | Stat叠加模型（`StatDefOf.MechBandwidth`），Tracker管理占用/空闲 | 高——Trion总量的多来源叠加直接参考此模式 |
| **原版Hemogen血源素** | Gene_Resource子类，resourceLossPerDay + GeneGizmo_Resource | 中——Gizmo设计可参考，但资源模型过于简单 |
| **VEF PipeSystem** | 自定义资源网络（PipeNetDef + CompResource），建筑间资源传输 | 低——PipeSystem是建筑级资源网络，非Pawn级资源池 |
| **原版Psyfocus焦点** | Pawn_PsychicEntropyTracker管理焦点值和熵值，双资源池 | 中——双资源池（焦点+熵）的Tracker设计可参考 |

### 1.5 推荐方案与理由

**推荐：方案D — 自定义Tracker + Stat混合**

理由：
1. **Stat管总量**：定义`StatDefOf.RimWT_MaxTrion`，让Trion腺体（基因statOffsets）、Trion接收器（Hediff statOffsets）、黑触发器（装备equippedStatOffsets）三条路径各自贡献，Tracker只需`pawn.GetStatValue()`一行获取总量——直接复用机械师带宽的Stat叠加模型
2. **Tracker管状态**：自定义`Pawn_TrionTracker`管理当前值、占用量、消耗模式、Notify链。战斗中每tick更新（不受150 ticks限制），非战斗中可降频
3. **Notify链**：`Notify_TrionChanged()`统一处理所有Trion变化事件，自动检查占用超额、触发临界状态、驱动紧急脱离——直接复用机械师的`Notify_BandwidthChanged()`模式
4. **自定义Gizmo**：方块网格可视化（借鉴`MechanitorBandwidthGizmo`），三色区分占用/已消耗/剩余

## 第2章 战斗体状态切换选型 ★★★

> 2.6系统：战斗体状态系统 | 关系：R9/R11/R13/R16/R17

### 2.1 需求分析

战斗体切换是RimWT最核心的机制——同一Pawn在"真身"和"战斗体"之间切换。

**核心需求**：
- **同一Pawn**：不是新建Pawn，是同一个Pawn的状态切换（保留身份、关系、记忆）
- **数据快照+恢复**：切换到战斗体时保存真身状态（装备、Hediff），恢复时还原
- **视觉切换**：战斗体有不同外观（Trion战斗服覆盖层）
- **伤害隔离**：战斗体受到的伤害不传递给真身（战斗体破裂后真身无伤）
- **能力变化**：战斗体拥有触发器赋予的特殊能力，真身没有
- **弱点系统**：头部/心脏部位缺失（被摧毁）触发战斗体失效

### 2.2 方案对比：MutantDef转化 vs 纯Hediff状态机 vs 混合方案

| 方案 | 原版先例 | 优势 | 劣势 | 扩展性 |
|------|---------|------|------|--------|
| **A. MutantDef转化** | Ghoul（食尸鬼）——可征召、能力白名单、专用外观 | Turn()/Revert()现成的20+副作用链；外观系统完整（皮肤/头型/体型/动画）；Need白名单机制；能力白名单机制 | Revert()是"部分恢复"——装备、基因、所有权不恢复；Turn()副作用链太重（清除基因、重置政策等），RimWT不需要这么多副作用；MutantDef设计上是"准永久转化"，不适合频繁切换 | 低——MutantDef框架过于刚性，难以定制切换行为 |
| **B. 纯Hediff状态机** | 非人化（Hediff_Inhumanized）——单Hediff+分散检查点 | 最轻量，完全可逆；Hediff的Stage系统可驱动属性变化；HediffComp可附加自定义逻辑；添加/移除Hediff即完成状态切换 | 外观切换需要额外处理（Hediff本身不改变外观）；伤害隔离需要Harmony Patch或自定义DamageWorker；无现成的装备处理逻辑 | 高——Hediff+HediffComp组合灵活 |
| **C. Hediff+自定义Comp混合** | 死眠三层架构（Gene+Need+Hediff）的简化版 | Hediff标记状态（战斗体激活/未激活）；自定义HediffComp_CombatBody管理数据快照、伤害隔离、视觉切换；与Trion Tracker联动（Hediff存在=消耗Trion） | 实现量中等；需要自行处理外观切换和装备快照 | 最高——Hediff标记+Comp逻辑+Tracker联动 |

### 2.3 伤害隔离的实现方案

| 方案 | 拦截位置 | 优势 | 劣势 |
|------|---------|------|------|
| **A. PreApplyDamage拦截** | `Pawn_HealthTracker.PreApplyDamage`阶段4 | 最早拦截，伤害不进入后续管线；可通过HediffComp的`PostPreApplyDamage`实现 | 需要区分"战斗体伤害"和"真身伤害"——战斗体状态下所有伤害都应被拦截并转化为Trion消耗 |
| **B. VEF HediffComp_Shield模式** | Hediff级别的`PostPreApplyDamage` | VEF已有成熟实现——支持近战/远程/全部吸收+能量系统+伤害反射；基于Hediff，与战斗体Hediff天然集成 | 依赖VEF框架（增加模组依赖）；Shield模式是"吸收伤害"而非"隔离伤害"，语义略有差异 |
| **C. 自定义HediffComp伤害转化** | HediffComp的`PostPreApplyDamage` | 完全自定义：伤害→Trion消耗的转化公式可自由设计；不依赖外部框架；可实现"部分穿透"（如弱点部位伤害穿透隔离） | 需要自行实现完整的伤害拦截逻辑 |

### 2.4 弱点系统的实现方案

弱点系统需求：头部或心脏部位被摧毁时，战斗体立即失效（强制解除）。

**推荐方案**：在`HediffComp_CombatBody.CompPostTick()`中周期性检查关键部位状态：
```
每N ticks检查：
  if (头部.Missing || 心脏.Missing)
    → 触发战斗体强制解除（等同紧急脱离路径B——被动破裂）
```

原因：原版`Pawn_HealthTracker.CheckForStateChange()`在部位摧毁时被调用，可以通过Harmony Postfix在此处插入弱点检查，比周期性轮询更高效。但考虑到实现简洁性，周期性检查（如每60 ticks）也完全可接受。

### 2.5 社区参考

| 模组/系统 | 实现方式 | 与RimWT的相关性 |
|----------|---------|----------------|
| **原版Ghoul** | MutantDef + Turn()/Revert()，可征召、能力白名单、专用外观 | 高——唯一可控的双向状态切换先例，外观和能力管理可参考 |
| **原版非人化** | Hediff_Inhumanized + 分散检查点，完全可逆 | 高——"Hediff作为状态标记"模式的典范 |
| **VEF HediffComp_Shield** | 基于Hediff的护盾——支持近战/远程/全部吸收+能量系统 | 高——伤害隔离的直接参考（选型方向小记已标注） |
| **AlteredCarbon（副本2重生）** | NeuralData.OverwritePawn()——意识转移到新身体 | 中——数据快照/恢复的思路可参考，但它是跨Pawn转移 |
| **原版死眠** | Gene+Need+Hediff三层架构，致命伤害触发ForceDeathrestOrComa | 中——紧急脱离的触发模式可参考 |

### 2.6 推荐方案与理由

**推荐：方案C — Hediff + 自定义HediffComp混合**

核心架构：
```
Hediff_CombatBody（状态标记）
  ├── HediffComp_CombatBodyCore    — 数据快照/恢复、状态机管理
  ├── HediffComp_DamageIsolation   — 伤害拦截→Trion消耗转化
  ├── HediffComp_CombatVisual      — 外观切换（Trion战斗服覆盖层）
  └── HediffComp_WeakpointMonitor  — 弱点部位监控
```

理由：
1. **不用MutantDef**：MutantDef的Turn()有20+副作用链，大部分RimWT不需要（不需要清除基因、重置政策、改变派系）。且Revert()不恢复装备和基因，而RimWT需要完整恢复
2. **Hediff标记+Comp逻辑**：Hediff的添加/移除天然对应战斗体的激活/解除；HediffComp的生命周期回调（PostAdd/PostRemoved/CompPostTick）提供完整的Hook点
3. **伤害隔离用自定义HediffComp**：不依赖VEF，自行实现`PostPreApplyDamage`中的伤害→Trion转化，可精确控制弱点穿透逻辑
4. **与Trion Tracker联动**：Hediff存在期间，Trion Tracker持续消耗维持Trion；Trion耗尽时Tracker通知HediffComp触发破裂

## 第3章 触发器装备系统选型 ★★★

> 2.6实体：触发体/触发芯片/黑触发器/紧急脱离 | 关系：R7/R11/R12/R18

### 3.1 需求分析

触发器是RimWT的核心装备系统，采用三层架构：

**触发体**（硬件平台）：
- 含8个槽位（主4+副4），同侧互斥（同时只能激活1个）
- 是战斗体的物理载体——装备触发体才能激活战斗体
- 有品质系统（影响输出功率、耐久度）
- 消耗Trion维持运行

**触发芯片**（可插拔功能模块）：
- 插入触发体槽位，定义战斗体的具体能力
- 注册时锁定占用Trion（预占用模型）
- 激活有引导时间和空窗期
- 不同芯片赋予不同的战斗能力（攻击型/防御型/辅助型）

**黑触发器**（特殊变体）：
- 事件生成（高Trion天赋者死亡时概率生成）
- 自主意识——不可量产，每个独一无二
- 叠加Trion总量（额外Trion来源）

### 3.2 触发体的载体选型

| 方案 | 原版先例 | 优势 | 劣势 | 扩展性 |
|------|---------|------|------|--------|
| **A. 武器类型** | 原版武器（ThingDef + equipmentType=Primary） | 原生装备/卸下逻辑；Comp系统完整；品质系统自动支持 | 占用武器栏——Pawn不能同时持有其他武器；触发体概念上不是"武器"而是"平台" | 中等 |
| **B. 附件层衣物** | 护盾腰带（Apparel + Belt层） | 不占武器栏；Apparel的CheckPreAbsorbDamage可用于伤害拦截；可与武器共存 | 衣物层冲突问题——需要自定义ApparelLayer避免与其他Belt层冲突；衣物没有原生品质影响战斗属性的机制 | 中等 |
| **C. 自定义装备类型** | 无直接先例（需自定义ThingClass） | 完全自定义行为；不占武器栏也不占衣物层；可定义专属装备逻辑 | 实现量最大；需要自行处理装备/卸下/存档；需要自行实现Gizmo | 最高 |
| **D. Hediff植入体** | 原版植入体（HediffDef + hediffClass=Hediff_Implant） | 不占任何装备栏；与身体部位绑定；Hediff系统成熟 | 植入体不是"可拆卸装备"——触发体需要频繁装卸；无品质系统；无物品实体（不能交易/掉落） | 低 |

### 3.3 触发芯片的槽位系统

| 方案 | 原版先例 | 优势 | 劣势 | 扩展性 |
|------|---------|------|------|--------|
| **A. 装备Comp管理** | Odyssey独特武器（CompUniqueWeapon + WeaponTraitDef） | WeaponTraitDef的exclusionTags实现互斥；Stat叠加（statOffsets/statFactors）自动生效；Worker生命周期回调成熟 | Odyssey特质是创建时随机固定的，不支持运行时插拔；特质数量1-3个，RimWT需要8个槽位 | 中等——需要扩展支持运行时增删 |
| **B. 独立物品+引用** | 无直接先例 | 芯片是独立ThingDef物品——可交易、可掉落、可制造；触发体通过引用列表管理已插入的芯片；物品实体保留品质/耐久度 | 需要自行实现"物品插入装备"的交互逻辑；存档序列化需要处理物品引用 | 高 |
| **C. 类Odyssey特质系统扩展** | Odyssey WeaponTraitDef + AncotLibrary运行时增删API | WeaponTraitDef已有完整的效果字段（statOffsets/equippedHediffs/abilityProps）；AncotLibrary提供运行时AddOrReplaceTrait/RemoveTrait API；exclusionTags实现同侧互斥 | 依赖AncotLibrary（或自行实现类似API）；WeaponTraitDef的字段可能不够用（需要DefModExtension扩展） | 中高 |

### 3.4 紧急脱离的实现方案

紧急脱离需求：战斗体破裂时，Pawn传送到安全位置（类似心灵跳跃）。

**推荐方案**：自定义Job + PawnFlyer

```
战斗体破裂触发
  → HediffComp_CombatBody.TriggerEmergencyExit()
    → 计算安全位置（后方N格，避开敌人）
    → 创建PawnFlyer（类似Skipshield的传送效果）
    → Pawn瞬移到安全位置
    → 移除Hediff_CombatBody（恢复真身）
    → 添加"紧急脱离后遗症"Hediff（短暂虚弱）
```

原版参考：`PawnFlyer`类用于心灵跳跃（Psycast Skip系列），实现Pawn的瞬间位移+视觉效果。

### 3.5 黑触发器的特殊处理

| 需求 | 实现方案 |
|------|---------|
| **事件生成** | Harmony Postfix `Pawn_HealthTracker.NotifyPlayerOfKilled()`或`Pawn.Kill()`，检查死亡Pawn的Trion天赋值，概率生成黑触发器物品 |
| **Trion叠加** | 黑触发器作为装备，通过`equippedStatOffsets`为`RimWT_MaxTrion`提供加值——与Stat叠加模型一致 |
| **不可量产** | ThingDef设置`recipeMaker=null`（无配方），仅通过事件生成；每个黑触发器有唯一名称（类似Odyssey独特武器的命名系统） |
| **自主意识** | 待深入调研——可能通过自定义ThingComp实现"黑触发器意志"（影响战斗体行为的随机事件） |

### 3.7 推荐方案与理由

**触发体载体：推荐方案B — 附件层衣物**

理由：
1. 不占武器栏——Pawn可以同时持有武器和触发体（真身状态下可能需要武器）
2. 自定义ApparelLayerDef `RimWT_Trigger`避免与其他Belt层冲突
3. Apparel的Comp系统完整，可附加自定义CompTriggerBody管理槽位
4. 品质系统通过Stat影响（品质→equippedStatOffsets→输出功率Stat）

**触发芯片槽位：推荐方案B — 独立物品+引用**

理由：
1. 芯片作为独立物品——可交易、可掉落、可制造、可存储，符合"可插拔模块"的设计意图
2. 触发体的CompTriggerBody维护`List<Thing> installedChips`（8个槽位）
3. 插入/拔出通过自定义Job实现（类似安装植入体的手术Job）
4. 芯片的效果通过ChipDef（自定义Def）定义，包含：Trion占用量、赋予的Ability列表、Stat修改、引导时间

## 第4章 副作用系统选型 ★★☆

> 2.6实体：副作用(Trait+Ability) | 关系：R5/R17

### 4.1 需求分析

副作用（Side Effect）是高Trion天赋者触发的永久能力扩展：

- **触发条件**：Trion天赋值超过阈值时，概率生成
- **等级分类**：S/A/B/C四级，等级越高越稀有、效果越强
- **永久性**：一旦获得不可移除（类似Trait的永久性）
- **跟随意识**：副作用自然存在于战斗体中，不需要触发器激活
- **表现形式**：Trait（被动效果）+ Ability（主动能力）的组合

### 4.2 方案对比

| 方案 | 原版先例 | 优势 | 劣势 | 扩展性 |
|------|---------|------|------|--------|
| **A. Trait + Ability** | 原版Trait的`abilities`字段（如Psycast特质赋予能力） | Trait天然永久；TraitDegreeData的`abilities`字段直接赋予Ability；`statOffsets/statFactors`提供被动效果；degree可对应S/A/B/C等级 | Trait不支持"仅在战斗体状态下生效"——需要额外逻辑控制；Trait的获得途径有限（需代码调用GainTrait） | 中等 |
| **B. Gene赋予** | Biotech基因的`forcedTraits` + `abilities` | Gene可同时赋予Trait和Ability；Gene有完整的添加/移除API；Gene的`statOffsets/statFactors`提供被动效果 | Gene绑定基因系统——副作用概念上不是"基因"；Gene的添加需要Xenogerm或代码调用；Gene有代谢成本（biostatMet），副作用不应有代谢成本 | 低——语义不匹配 |
| **C. Hediff赋予** | Hediff的`stages[].statOffsets` + HediffComp赋予Ability | Hediff可关联身体部位（副作用"存在于"Pawn身上）；HediffComp_GiveAbility可动态赋予能力；Severity可对应等级（S/A/B/C映射到不同Stage） | Hediff在健康面板显示——副作用概念上更接近"特质"而非"健康状况"；Hediff可被治疗/移除，需要设置`everCurableByItem=false` | 中等 |
| **D. Trait(被动) + Hediff(主动)混合** | 无直接先例 | Trait提供永久被动效果（statOffsets）；Hediff通过HediffComp_GiveAbility提供主动能力；两者分工明确 | 实现复杂度高——需要同时管理Trait和Hediff的生命周期；两者需要保持同步 | 中等 |

### 4.3 触发条件的实现

副作用的触发需要监控Trion天赋值并概率生成：

**推荐方案**：在`Pawn_TrionTracker`中实现副作用检查

```
Pawn_TrionTracker.CheckSideEffectGeneration()
  — 触发时机：Trion天赋值变化时（基因添加、成长事件等）
  — 条件：TrionTalent > 阈值 && 尚未拥有副作用
  — 概率：按TrionTalent值查曲线（越高概率越大）
  — 等级：按概率分布随机S/A/B/C
  — 执行：TraitSet.GainTrait(sideEffectTrait) + 赋予对应Ability
```

### 4.4 推荐方案与理由

**推荐：方案A — Trait + Ability**

理由：
1. **语义匹配**：副作用是"永久性角色特征"——这正是Trait的设计意图。Trait的degree可映射S/A/B/C四级（degree=4/3/2/1）
2. **TraitDegreeData功能完整**：`statOffsets`提供被动属性加成，`abilities`直接赋予主动能力，`description`可描述副作用效果
3. **永久性保证**：Trait设计上不随时间变化，不可被"治疗"移除
4. **战斗体限定**：通过在Ability的`CompAbilityEffect`中检查`pawn.health.hediffSet.HasHediff(Hediff_CombatBody)`来限制"仅战斗体状态下可使用主动能力"；被动效果（statOffsets）始终生效（符合"副作用跟随意识"的设定）

## 第5章 派系与敌对单位选型 ★★☆

> 2.6实体：BORDER/敌对惑星国家派系/近界民 | 关系：R21/R33

### 5.1 需求分析

RimWT需要定义以下派系结构：

- **BORDER**：中立防御组织，玩家所属。类似原版殖民地派系，但有Trion技术体系
- **敌对惑星国家派系**：多个敌对人类派系（阿夫托克拉托尔、加洛普拉等），各有不同科技水平和战斗风格
- **近界民**：来自近界的敌对人类Pawn，拥有Trion能力和触发器装备，是主要敌对力量

### 5.2 派系定义方案

**推荐方案**：标准FactionDef配置

| 派系 | FactionDef配置要点 |
|------|-------------------|
| **BORDER** | `isPlayer=false`（NPC版本，用于世界地图据点）；`techLevel=Spacer`；`goodwillMin/Max`设为中立偏友好；`pawnGroupMakers`定义BORDER队员生成规则 |
| **阿夫托克拉托尔** | `permanentEnemy=true`；`techLevel=Ultra`（超级科技）；`pawnGroupMakers`包含高级Trion兵和近界民精英 |
| **加洛普拉** | `permanentEnemy=true`；`techLevel=Spacer`；`pawnGroupMakers`侧重数量型近界民 |

**关键配置字段**：

```xml
<FactionDef>

  <label>阿夫托克拉托尔</label>
  <techLevel>Ultra</techLevel>
  <permanentEnemy>true</permanentEnemy>
  <humanlikeFaction>true</humanlikeFaction>
  <pawnGroupMakers>
    <li>
      <kindDef>Combat</kindDef>
      <options>
        <RimWT_Neighbor_Elite>10</RimWT_Neighbor_Elite>
        <RimWT_Neighbor_Standard>40</RimWT_Neighbor_Standard>
        <RimWT_TrionSoldier>30</RimWT_TrionSoldier>
      </options>
    </li>
  </pawnGroupMakers>
</FactionDef>
```

### 5.3 近界民Pawn的特殊配置

近界民是拥有Trion能力的敌对人类Pawn，与普通人类的差异：

| 维度 | 普通敌对人类 | 近界民 |
|------|------------|--------|
| **Trion能力** | 无 | 有Trion腺体基因→Trion天赋值→可激活战斗体 |
| **触发器装备** | 无 | PawnKindDef配置触发体+预装芯片 |
| **战斗行为** | 标准AI | 战斗开始时激活战斗体（自定义ThinkTree节点或DutyDef） |
| **倒地即死** | 按人口意图曲线 | `PawnKindDef.overrideDeathOnDownedChance`可自定义（如0.5=50%） |
| **掉落物** | 标准装备掉落 | 触发体+芯片可掉落（玩家可缴获） |

**PawnKindDef配置要点**：
- `race`：使用标准Human种族（近界民是人类）
- `xenotypeSet`：配置含Trion腺体基因的XenotypeDef
- `weaponTags`/`apparelTags`：配置触发体装备
- `combatPower`：根据Trion天赋等级设置不同战斗力

### 5.4 推荐方案与理由

**推荐：标准FactionDef + 自定义PawnKindDef + XenotypeDef**

理由：
1. FactionDef是派系定义的标准方式，无需自定义——纯XML配置
2. 近界民通过XenotypeDef获得Trion基因（Trion腺体→Trion天赋Stat），与基因系统自然集成
3. PawnKindDef控制装备生成规则，确保近界民自带触发体和芯片
4. 战斗体激活逻辑通过自定义DutyDef或ThinkTree节点实现（近界民AI在战斗开始时自动激活战斗体）

## 第6章 Trion兵系统选型 ★★☆

> 2.6实体：Trion兵 | 关系：R19/R21

Trion兵是无意识的人工战斗单位：

**推荐：方案A改造 — 基于机械体框架的Trion兵**

## 第7章 事件与袭击选型 ★☆☆

### 7.1 门（Gate）袭击的实现

门（Gate）是近界民入侵的通道——在地图上打开一个传送门，敌人从中涌出。

**推荐方案**：自定义RaidStrategyDef + PawnsArrivalModeDef

### 7.3 黑触发器生成事件

黑触发器在高Trion天赋者死亡时概率生成。

**推荐方案**：Harmony Postfix on `Pawn.Kill()`

```
Harmony Postfix: Pawn.Kill()
  → 检查死亡Pawn是否有Trion腺体基因
  → 检查TrionTalent Stat值是否超过阈值（如≥8.0）
  → 概率检查（TrionTalent越高概率越大）
  → 通过 → 在死亡位置生成黑触发器物品（ThingDef_BlackTrigger）
  → 发送信件通知玩家
```

替代方案：使用`HediffComp`在Trion腺体Hediff的`PostRemoved()`中触发（如果Trion腺体是Hediff而非Gene）。但由于Trion腺体推荐用Gene实现，Harmony Patch更合适。

### 7.4 推荐方案与理由

- **Gate袭击**：自定义RaidStrategy + ArrivalMode + Gate实体——原版袭击系统扩展性好，纯XML+少量C#即可实现
- **远征船**：基于穿梭机系统改造——待深入调研穿梭机API后细化
- **黑触发器生成**：Harmony Postfix on Pawn.Kill()——最直接的死亡事件Hook

## 第8章 UI与可视化选型 ★☆☆

> 横跨多个系统的UI需求

### 8.1 Trion资源池可视化

| 方案 | 原版先例 | 优势 | 劣势 |
|------|---------|------|------|
| **A. 方块网格** | MechanitorBandwidthGizmo（带宽方块网格） | 信息密度高——三色区分占用/已消耗/剩余；离散化显示直观；机械师系统已验证此模式 | Trion是连续浮点值，方块网格需要离散化（如每格=10 Trion）；格数过多时显示拥挤 |
| **B. 进度条** | Gizmo_EnergyShieldStatus（护盾能量条） | 连续值天然适合进度条；实现简单；原版护盾已验证 | 单一进度条难以同时显示占用/消耗/剩余三段信息；信息密度低 |
| **C. 分段进度条** | 无直接先例 | 一个进度条分三段着色（占用=蓝色、已消耗=红色、剩余=绿色）；连续值+多段信息兼得 | 需要自定义绘制逻辑；三段颜色区分需要清晰的视觉设计 |

**推荐：方案C — 分段进度条**

理由：Trion是连续浮点值，方块网格需要离散化会丢失精度。分段进度条既保留连续值的精确显示，又能通过颜色区分三段语义。实现上参考`Gizmo_EnergyShieldStatus`的绘制逻辑，扩展为三段着色。

### 8.2 触发芯片槽位面板

**推荐方案**：双列Gizmo面板（借鉴MechanitorControlGroupGizmo）

```
┌─────────────────────────────┐
│  主侧（4槽位）  │  副侧（4槽位）  │
│ [芯片A] ★激活  │ [芯片E]        │
│ [芯片B]        │ [芯片F] ★激活  │
│ [  空  ]       │ [芯片G]        │
│ [  空  ]       │ [  空  ]       │
└─────────────────────────────┘
```

- 每个槽位显示芯片图标+名称
- 当前激活芯片高亮（★标记）
- 引导中芯片显示进度条（类比孵化进度）
- 右键菜单切换芯片（类比工作模式切换）
- 空槽位显示"空"占位符

### 8.3 战斗体外观切换

**推荐方案**：衣物覆盖层 + 特效

| 组件 | 实现方式 |
|------|---------|
| **Trion战斗服** | 战斗体激活时自动穿戴的不可见衣物（Apparel），通过`renderNodeProperties`添加视觉覆盖层 |

| **颜色变化** | 战斗体Hediff的Stage中设置`skinColorOverride`或通过自定义渲染节点改变Pawn颜色 |

- Ghoul的外观系统（`MutantDef.skinColorOverride` + `forcedHeadTypes` + `renderNodeProperties`）
- 护盾气泡渲染（`CompShield.Draw()`的透明材质+尺寸插值）
