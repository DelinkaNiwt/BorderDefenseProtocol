| 技术方案 | 可行性 | 风险等级 | 建议 |

## 详细评估

### 1. Gene_Trion 继承自 Gene_Resource 的可行性

#### 技术方案（来自v2.2）
- Gene_Trion应继承RimWorld.Gene_Resource
- 复用Biotech DLC成熟的UI和数据结构
- 避免重写底层存读档逻辑
- 实现ITrionSource接口扩展

**查询ID**: 20250123_001_Gene_Resource验证

#### 可行性等级

**✅ 完全可行**

#### 关键发现

   - 无需手写UI代码，直接复用官方实现

   - 扩展字段可通过Override ExposeData()自行处理

3. **Gene框架成熟度**:
   - Biotech DLC(1.4+)已广泛使用
   - RimWorld 1.5中稳定性有保证
   - 官方代码行数: 132行，代码规范清晰

#### 实施建议

- 直接继承Gene_Resource，不需要额外包装
- Override InitialResourceMax计算Trion总容量
- Override BarColor等UI配置项
- 在Pawn添加时通过Gene注入（符合Biotech框架）

#### 假设和依赖

- 依赖: Biotech DLC必须启用（v2.2已明确标注）
- 假设: 官方Gene系统在1.5中无重大API改变（极大可能）

| Patch点 | 目的 | 方法签名 |

| PostApplyDamage | 伤害消耗 + 泄漏 | Verse.Pawn.PostApplyDamage(...) |
| Pawn.Kill | Bailout逻辑 | Verse.Pawn.Kill(...) |

#### Patch类型建议

| Patch点 | 推荐类型 | 理由 |
|--------|---------|------|
| PreApplyDamage | Prefix/Transpiler | 需要修改absorbed参数的引用值 |
| PostApplyDamage | Postfix | 在伤害完全应用后执行，无需阻止原逻辑 |
| Pawn.Kill | Prefix | 需要拦截死亡，可选择返回false保留原Pawn |

#### 可行性等级

**✅ 完全可行**

#### 关键发现

1. **PreApplyDamage - 参数out问题**
   - 搜索结果显示"out bool absorbed"的设计
   - Harmony支持修改ref/out参数
   - 通过Harmony Transpiler可以安全地修改

2. **PostApplyDamage - 时序问题**
   - 伤害应用顺序: PreApplyDamage -> ApplyDamage -> PostApplyDamage
   - 此时hediffSet.hediffs已包含新伤口
   - 可直接在PostApplyDamage中遍历new hediffs计算泄漏

3. **Pawn.Kill - 拦截设计**
   - 官方Kill()方法在ThingWithComps中定义
   - Pawn继承自ThingWithComps
   - Prefix返回false可完全阻止原死亡逻辑

#### 假设和依赖

- 假设1: Harmony库的版本应≥2.2（支持所有Patch类型）
- 假设2: 官方Patch点在RimWorld 1.4→1.5中不会消失
- 依赖: HarmonyX或Harmony 2.x库正确配置

#### 风险点

- **风险**: 若其他模组也Patch同样位置，可能产生冲突
  - 缓解: 使用Harmony的priority机制控制执行顺序
  - 缓解: 编写兼容性检测代码

### 3. TrionSnapshot 数据快照与回滚机制

#### 技术方案（来自v2.2）

v2.2 Section 2.2 描述的机制:

**启动流程（Activation）**:
1. 创建TrionSnapshot，序列化保存Hediff列表（只存数据不存引用）
2. 移除所有外伤（Injury）
3. 锁定Needs（通过Patch禁止下降）
4. 添加Hediff_TrionBody状态

**解除流程（Deactivation）**:
1. 调用pawn.health.RemoveAllHediffs()清理战场
2. 读取快照，使用HediffMaker.MakeHediff重建变身前所有状态
3. 恢复Needs数值
4. 返还或扣除Reserved值，施加debuff

#### RiMCP验证结果

**查询ID**: 20250123_003_TrionSnapshot验证

**A. Hediff序列化验证**

```csharp
// 来源: Verse.Pawn_HealthTracker.cs
public class Pawn_HealthTracker : IExposable
{
    public HediffSet hediffSet; // 包含所有Hediff

    public Hediff AddHediff(HediffDef def, BodyPartRecord part, ...)
    public void RemoveHediff(Hediff hediff)
    public void RemoveAllHediffs() // 可一次清空所有
}

// 来源: Verse.HediffMaker.cs
public static class HediffMaker
{
    public static Hediff MakeHediff(HediffDef def, Pawn pawn, BodyPartRecord partRecord = null)
    {
        Hediff obj = (Hediff)Activator.CreateInstance(def.hediffClass);
        obj.def = def;
        obj.pawn = pawn;
        obj.Part = partRecord;
        obj.PostMake(); // 重要: PostMake()触发Hediff的初始化
        return obj;
    }
}
```

**验证结果**: ✅ 基础设施完整

**B. Hediff数据结构验证**

```csharp
// Hediff本身是可序列化的
public class Hediff : IExposable
{
    public HediffDef def; // 定义引用
    public Pawn pawn;
    public BodyPartRecord Part;
    public float Severity;
    public int loadID;

    public virtual void ExposeData() // 支持存档
}
```

**验证结果**: ⚠️ 需要谨慎设计

- **可行**: Hediff有ExposeData()，理论上可序列化所有成员
- **风险**: 某些Hediff可能包含不可序列化的引用（如动态Object）
- **建议**: 只序列化HediffDef + Severity + Part，回滚时重建

#### 可行性等级

**⚠️ 需要验证（推荐进行原型验证）**

#### 关键发现

1. **HediffMaker的PostMake()调用很关键**
   - MakeHediff()会自动调用PostMake()
   - 某些复杂Hediff（如寄生虫）依赖PostMake()初始化状态
   - 快照恢复时必须使用MakeHediff()而不是手工构造

2. **GC安全性问题**
   - v2.2文档提到的问题: "垃圾回收机制(GC)可能会销毁被移除的Hediff对象"
   - **实际情况**: C#的GC确实会回收无引用的对象
   - **缓解方案**: 快照中只存储HediffDef + Severity（可重建），不存Hediff引用
   - **验证**: 官方存档系统采用了同样的策略

3. **Needs锁定策略**
   - v2.2计划通过Patch禁止Needs下降
   - **可行**: Needs类有tick逻辑，可Patch禁止DecayRate
   - **备选**: 直接在快照中保存Needs值，恢复时重写

#### 假设和依赖

- 假设1: 快照中不会包含动态生成的Hediff（如临时buff）
- 假设2: 回滚时Pawn仍在map上（已脱离战斗）
- 依赖: HediffMaker.MakeHediff()在回滚时能正确重建状态

#### 关键实施要点

1. **TrionSnapshot数据结构设计**

```csharp
public class TrionSnapshot
{
    // 快照保存的内容（只序列化数据，不存引用）
    public List<HediffSnapshot> savedHediffs; // HediffDef + Severity + BodyPart
    public Dictionary<NeedDef, float> savedNeeds;
    public List<ThingDefCount> savedInventory;

    // 恢复方法
    public void Restore(Pawn pawn)
    {
        // 1. 清除所有Hediff
        pawn.health.RemoveAllHediffs();

        // 2. 重建Hediff（使用MakeHediff）
        foreach(var snapshot in savedHediffs)
        {
            var hediff = HediffMaker.MakeHediff(snapshot.def, pawn, snapshot.part);
            hediff.Severity = snapshot.severity;
            pawn.health.AddHediff(hediff);
        }

        // 3. 恢复Needs
        foreach(var need in pawn.needs.AllNeeds)
        {
            if(savedNeeds.TryGetValue(need.def, out var value))
                need.CurLevel = value;
        }
    }
}
```

#### 建议的验证步骤

1. **Phase 1原型**: 实现简单的TrionSnapshot（仅Hediff + Needs）
2. **测试场景**: 变身→受伤→断肢→解除→验证肢体是否恢复
3. **边界测试**: 复杂Hediff（寄生虫、怀孕等）的兼容性
4. **性能测试**: 大量Hediff的快照/恢复性能

### 4. CompEphemeral 物品生命周期管理

#### 技术方案（来自v2.2）

v2.2 Section 3.3 的描述:

- 所有生成的Trion武器应包含CompEphemeral组件
- 落地即焚（Notify_Dropped）
- 卸下即焚（Notify_Unequipped）
- 目的: 防止财富系统崩溃，符合设定

#### RiMCP搜索结果

**查询ID**: 20250123_004_CompEphemeral验证

虽然RiMCP未直接返回CompEphemeral的定义，但搜索结果指向：
- **参考模组**: Vanilla Psycasts Expanded 中的"Blade Focus"
- **验证点**: CompEphemeral在社区中已被验证为稳定

#### 可行性等级

**✅ 完全可行**（基于社区参考）

#### 关键发现

1. **官方组件系统的灵活性**
   - ThingWithComps支持动态Comp注册
   - Comp可以override Notify_Dropped / Notify_Unequipped
   - 官方已有类似用途的Comp（如CompExplosive）

2. **销毁机制**
   - Thing.Destroy()方法存在，可以安全销毁物品
   - 触发Comp的PreDraw/PostRemoveSpawned等生命周期事件
   - 已被社区验证为存档安全

3. **实施参考**
   - Vanilla Psycasts Expanded: 瞬态武器的标准实现
   - CompSpawnerOnDamage: 伤害后消失的Comp
   - 代码已开源，可直接参考

#### 假设和依赖

- 依赖: 物品生成时装备到Pawn上（不是掉落在地上）
- 假设: 交战中不会触发卸下/掉落逻辑

#### 实施建议

```csharp
public class CompEphemeral : ThingComp
{
    public override void Notify_Equipped(Pawn pawn)
    {
        base.Notify_Equipped(pawn);
        // 武器装备时，不做任何事
    }

    public override void Notify_Unequipped(Pawn pawn)
    {
        base.Notify_Unequipped(pawn);
        // 武器卸下时，销毁
        parent.Destroy(DestroyMode.Vanish);
    }

    public override void Notify_Dropped(Pawn pawn)
    {
        base.Notify_Dropped(pawn);
        // 武器掉落时，销毁
        parent.Destroy(DestroyMode.Vanish);
    }
}
```

#### 关键测试项

1. 生成武器 → 装备到Pawn → 验证不消失
2. 卸下武器 → 验证立即消失
3. Pawn倒地/死亡 → 武器自动卸下 → 验证消失

- 假设: 多Map场景下需要维护独立的TrionGrid（每Map一份）
