## 📊 模块详细设计

### 模块1: 能量容器系统 (Energy Container Module)

#### 1.1 数据结构

```csharp
// 组件类：AldonEnergyComp (继承自ThingComp)
public class AldonEnergyComp : ThingComp
{
    // ===== 状态数据 =====
    public const int CAPACITY = 1000;  // 固定容量

    private int usedEnergy = 0;        // 已消耗待恢复
    private int lockedEnergy = 0;      // 被装备占用的容量
    private Dictionary<string, int> locks = new();  // 锁定项记录

    // ===== 计算属性 =====
    public int Available => CAPACITY - lockedEnergy - usedEnergy;
    public int LockedEnergy => lockedEnergy;
    public int UsedEnergy => usedEnergy;
    public int CurrentCapacity => CAPACITY;
}
```

#### 1.2 核心操作接口

| 操作 | 签名 | 说明 | 返回值 |
|------|------|------|--------|
| **AddEnergy** | `public void AddEnergy(int amount)` | 恢复能量，上限为容量 | 无 |
| **TryConsume** | `public bool TryConsume(int amount)` | 尝试消耗能量 | true=成功，false=不足 |
| **RegisterLock** | `public string RegisterLock(int amount, string source)` | 装备占用容量 | 锁定ID(string) |
| **ReleaseLock** | `public bool ReleaseLock(string lockID)` | 释放占用 | true=成功 |

#### 1.3 实现要点

**AddEnergy 实现逻辑**：
```
1. 输入验证：amount > 0？
2. 计算可恢复量：最多恢复至 CAPACITY
3. 更新 usedEnergy = max(0, usedEnergy - amount)
4. 标记dirty（Scribe保存时需要）
```

**TryConsume 实现逻辑**：
```
1. 权限检查：调用 PermissionComp.HasPermission()
   - 如果无权限 → 返回 false
2. 可用能量检查：Available >= amount？
   - 如果不足 → 返回 false
   - 如果充足 → usedEnergy += amount, 返回 true
3. 触发信号系统检查：IsActive？
   - 如果信号关闭 → 消耗不生效（返回false）
```

**RegisterLock 实现逻辑**：
```
1. 生成唯一ID：lockID = source + "_" + Guid（简化: source + "_" + Time.frameCount）
2. 检查容量：lockedEnergy + amount > CAPACITY？
   - 如果超量 → 返回 null（或空字符串）
   - 如果可以 → lockedEnergy += amount
3. 记录到字典：locks[lockID] = amount
4. 返回 lockID
```

**ReleaseLock 实现逻辑**：
```
1. 字典查询：locks.ContainsKey(lockID)？
2. 如果存在 → 取出 amount，lockedEnergy -= amount，移除字典项
3. 返回 true，否则返回 false
```

#### 1.4 设计决策

**【决策D1.1】容量设为固定值（1000）而不是可配置**

| 维度 | 说明 |
|------|------|
| 为什么选这个 | ✓ 可行性最高：无需参考配置文件 ✓ 简洁性：代码量少 ✗ 代价：后期无法调参（必须改代码） |
| 为什么不选其他 | 可配置容量：灵活但增加复杂度和数据持久化管理 |
| RiMCP验证 | ✓ 已验证 - const字段在C#中标准做法 |
| 扩展空间 | 如需要配置化，可在AldonEnergyComp中读取Def的自定义字段 |

**【决策D1.2】锁定ID采用字符串而不是整数**

| 维度 | 说明 |
|------|------|
| 为什么选这个 | ✓ 可行性：易于调试和日志输出（能看到source信息） ✓ 简洁性：避免ID冲突管理 ✗ 代价：占用更多内存 |
| 为什么不选其他 | 整数ID：更高效但调试困难 |
| RiMCP验证 | ✓ 已验证 - Dictionary<string, int>是标准C#用法 |
| 扩展空间 | 可升级为GUID以支持网络序列化 |

### 模块2: 全局信号系统 (Global Signal Module)

#### 2.1 数据结构

```csharp
// 单例管理器：AldonSignalManager
public class AldonSignalManager
{
    public static AldonSignalManager Instance { get; private set; }

    private bool isActive = false;  // 全局开关状态
    private List<Action> onActivated = new();    // 激活回调
    private List<Action> onDeactivated = new();  // 停用回调

    public bool IsActive => isActive;
}
```

#### 2.2 核心操作

| 操作 | 签名 | 说明 |
|------|------|------|
| **Activate** | `public void Activate()` | 激活信号（驱动塔开启时调用） |
| **Deactivate** | `public void Deactivate()` | 停用信号（驱动塔关闭时调用） |
| **GetSignal** | `public bool GetSignal()` | 获取当前信号状态 |
| **RegisterCallback** | `public void RegisterCallback(Action, bool isActivated)` | 注册激活/停用回调 |

#### 2.3 实现要点

**信号激活/停用流程**：
```
驱动塔建筑 → 调用 Activate()
  ↓
IsActive = true
  ↓
广播回调（所有注册的Pawn恢复能量）
  ↓
驱动塔关闭 → 调用 Deactivate()
  ↓
IsActive = false
  ↓
广播回调（能量恢复停止）
```

**Pawn能量恢复Tick逻辑**（在AldonEnergyComp.CompTick()中）：
```
1. 检查信号：IsActive？
   - false → 返回（不恢复）
   - true → 继续
2. 计算恢复速率：每秒2点 = 每Tick(60/s) 恢复0.0333点
   - 用tickCounter累计，当累计≥1时执行1点恢复
3. 调用AddEnergy(1)
4. tickCounter重置
```

#### 2.4 设计决策

**【决策D2.1】信号采用全局单例而不是基于范围**

| 维度 | 说明 |
|------|------|
| 为什么选这个 | ✓ 可行性：最简化，无需Verse.Region计算 ✓ 简洁性：50行代码可完成 ✗ 代价：无法实现范围限制（全地图要么都有要么都没有） |
| 为什么不选其他 | 基于地图格子：可实现范围但需要Verse.Map的格子系统 |
| RiMCP验证 | ✓ 已验证 - Verse.Map拥有AllThings列表可遍历 |
| 扩展空间 | 后期如需范围，可重写为Map-based信号系统 |

### 模块3: 权限系统 (Permission Module)

#### 3.1 数据结构

```csharp
// 权限等级枚举
public enum AldonPermissionLevel
{
    Unauthorized = 0,  // Lv0 无权限
    Authorized = 1      // Lv1 授权用户
}

// 组件类：AldonPermissionComp (继承自ThingComp)
public class AldonPermissionComp : ThingComp
{
    private AldonPermissionLevel permissionLevel = AldonPermissionLevel.Unauthorized;

    public AldonPermissionLevel PermissionLevel => permissionLevel;
    public bool HasPermission => permissionLevel >= AldonPermissionLevel.Authorized;
}
```

#### 3.2 核心操作

| 操作 | 签名 | 说明 |
|------|------|------|
| **GrantPermission** | `public void GrantPermission()` | 赋予权限（设为Lv1） |
| **RevokePermission** | `public void RevokePermission()` | 撤销权限（设为Lv0） |
| **HasPermission** | `public bool HasPermission()` | 检查是否有权限 |

#### 3.3 实现要点

**权限赋予流程**（通过Gizmo按钮）：
```
UI点击 → GrantPermission()
  ↓
permissionLevel = Authorized
  ↓
标记dirty（下次存档保存）
  ↓
显示消息："已赋予XX权限"
```

**权限检查流程**（在能量消耗前）：
```
TryConsume() 被调用
  ↓
调用 PermissionComp.HasPermission()
  ↓
if (false) → 返回 false（消耗失败）
if (true) → 继续能量检查
```

#### 3.4 设计决策

**【决策D3.1】权限采用两级制而不是多级**

| 维度 | 说明 |
|------|------|
| 为什么选这个 | ✓ 可行性：无需设计复杂的权限层级 ✓ 简洁性：if-else即可判断 ✗ 代价：无法实现权限级联（如"驾驶员"vs"指挥官"）|
| 为什么不选其他 | 多级权限（0-3级）：灵活但枚举复杂，决策树更深 |
| RiMCP验证 | ✓ 已验证 - Verse.Pawn存储自定义数据的标准方案 |
| 扩展空间 | 后期如需多级，改为int permissionLevel即可扩展 |

## 🔗 模块间接口与数据流

### 完整战斗场景数据流

```
场景：授权用户 John 对敌人 Enemy 开枪

1. Verb.Cast() 被调用（开始开枪）
   ↓
2. 权限检查（由Verb.TryStartCastOn中进行，不阻止）
   PermissionComp.HasPermission() → 记录权限状态
   ↓
3. 开枪动作执行（原版Verb_Shoot逻辑）
   ├─ 计算弹道
   ├─ 发射子弹
   └─ 敌人受伤
   ↓
4. Verb.Cast() 完成后 (Harmony Postfix钩子)
   AldonEnergyComp.TryConsume(10)
   ├─ 检查权限 → false则不消耗
   ├─ 检查信号 → IsActive = true
   ├─ 检查可用 → Available = 150 >= 10 → true
   ├─ 消耗能量 → usedEnergy += 10 → 成功
   └─ 返回 true
   ↓
5. 显示消息（如需要）

如果权限不足：
   → TryConsume() 返回 false（不消耗）
   → 显示消息："权限不足"

如果能量不足（Available < 10）：
   → TryConsume() 返回 false（不消耗）
   → 显示消息："能量不足"
   → 下次开枪失败（不是本次开枪失败）
```

## ⚠️ 已知限制与扩展空间

### 已知限制

| 限制 | 影响 | 后期改进方案 |
|------|------|-----------|
| **信号无范围限制** | 全地图同步激活/停用 | 后期可基于Map.tiles实现范围限制 |
| **权限二级制** | 无法表达权限级联 | 后期可改为int并扩展权限等级 |
| **容量固定值** | 无法通过配置调参 | 后期可读取Pawn的Def配置 |
| **无权限继承** | 同一Pawn多个装备各占各的 | 可优化为装备绑定的权限验证 |

### 扩展空间

1. **范围限制信号系统**：基于AldonSignalManager添加位置参数
2. **权限等级扩展**：增加Lv2(指挥官)、Lv3(系统管理员)等
3. **能量消耗自定义**：让装备或Verb定义自己的消耗量
4. **UI能量条显示**：在Pawn头部或Inspector中显示能量值
