## 📋 背景理解回顾

### World Trigger 系统概览

**Trion战斗系统的核心流程**（基于你提供的流程图）：

```
【阶段1：战斗体生成】
• 肉身生理状态快照保存 → 肉身停滞
• 组件占用量计算 → Reserved += 所有组件成本
• Available = Capacity - Reserved - Consumed
↓
【阶段2：战斗阶段】
• 每个CompTick计算: 基础维持消耗 + 组件消耗 + 泄漏 + 伤害转化
• 伤害拦截: 不影响肉身，全部转化为Trion流失
• 组件激活: 一次性费用 + 持续消耗
• 部位损毁: 如头部/躯干/Trion器官 → 触发Bail Out
• Trion耗尽 → 触发Bail Out
↓
【阶段3：战斗体摧毁与回滚】
• 主动解除 → 恢复到快照状态，保留已消耗值
• 被迫摧毁 → 恢复到快照状态，标记Trion枯竭debuff
```

此流程决定了框架的整体架构和疑问的重点。

#### Q2.2：CompTick的消耗计算流程

**问题核心**：CompTick的5步调度如何具体实现消耗计算？

**代码工程师的分析**（基于World Trigger流程图）：

CompTick应该每60 Tick执行一次，流程应该是：

```csharp
public override void CompTick()
{
    if (!isEnabled) return;

    // 步骤1：计算基础维持消耗
    float baseMaintenance = strategy.GetBaseMaintenance();

    // 步骤2：计算所有激活组件的消耗
    float mountConsumption = 0;
    foreach (var mount in Mounts.Where(m => m.IsActive))
        mountConsumption += mount.GetConsumptionRate();

    // 步骤3：计算泄漏
    float leak = LeakRate;

    // 步骤4：累加并执行消耗
    float totalConsumption = baseMaintenance + mountConsumption + leak;
    Consume(totalConsumption);

    // 步骤5：检查耗尽，触发回调
    if (Available <= 0)
        strategy.OnDepleted();
}
```

**代码工程师的推荐**：✅ **这个流程框架是合理的**

#### Q2.3：部位损毁与Bail Out的触发

**问题核心**：当战斗体的关键部位被摧毁时，如何触发Bail Out？

**代码工程师的分析**（基于World Trigger设定）：

World Trigger中关键部位是：头部、躯干核心、Trion器官。

根据系统设定文档v3.0第5.3节，Bail Out的触发条件有两个：

```
触发方式一：自动，Trion可用量 ≤ 0
触发方式二：自动，Trion供给器官被摧毁  ← 关键！
触发方式三：手动，玩家主动触发
```

**用户纠正**：特殊槽位（包含Bail Out系统）不会因为四肢/躯干损毁而失效。
Bail Out系统只有在Trion供给器官被摧毁时才会无法触发。

**代码工程师改正**：✅ **改为Tier 1（两个触发条件，AND关系）**

```csharp
public void TriggerBailOut()
{
    // 条件1：Trion供给器官被摧毁
    // 条件2：Trion可用量 ≤ 0

    if (!strategy.CanBailOut(this))
    {
        // 无法Bail Out，战斗体直接破裂
        DestroyBody(DestroyReason.ForcedDeactivation);
        return;
    }

    // 执行Bail Out传送
    var target = strategy.GetBailOutTarget(this);
    GenTeleport.TeleportTo(parent, target);

    // 被动解除战斗体
    DestroyBody(DestroyReason.PassiveDeactivation);
}
```

**用户决策**：
- [x] 改为Tier 1（仅两个触发条件）

#### Q2.4：战斗体摧毁时的状态回滚

```csharp
public class CombatBodySnapshot

    public List<Apparel> apparels;           // 穿戴服装

    public ThingOwner inventory;             // 携带物品

    // 不快照的内容（战斗中完整保留）
    // ✗ 技能等级和经验值
    // ✗ 心理状态和心情
    // ✗ 社交关系
```

**用户纠正**：回滚范围已在系统设定v3.0中明确定义。
**强调**：不仅限于"伤害回滚"，而是整个"物理状态回滚"
- ✅ 回滚：伤口、疾病、装备、物品

```csharp
    this.health.hediffSet.RestoreState(snapshot.hediffBefore);

    // 回滚装备和物品
    this.apparel.RestoreApparels(snapshot.apparels);
    this.inventory.container.TryLoadInContainers(snapshot.equipment);

    // 不回滚：心理状态、技能、经验（战斗中的成长和创伤保留）
}
```

**用户决策**：

#### Q4.2：性能基准与优化

**问题核心**：CompTick的性能目标应该是多少？

**代码工程师的分析**（基于World Trigger的大战场场景）：

- CompTick每60 Tick执行一次
- 如果有100个战斗体在场，每帧要处理 100/60 ≈ 1.67个CompTick
- 目标：每个CompTick < 0.5ms（允许总共1ms/帧用于Trion系统）

**代码工程师的推荐**：✅ **这个目标是合理的**

实现方式：
- 不在CompTick中做复杂计算（如AI决策）
- 复杂逻辑交给Strategy.OnTick()
- 使用缓存避免重复计算

```csharp
public override void CompTick()
{
    // 快速路径：只累加，不计算
    totalConsumptionThisTick = baseMaintenance + mountConsumption + leak;
    Consume(totalConsumptionThisTick);

    // 复杂逻辑延迟给Strategy处理
    strategy.OnTick();  // 在Strategy中做AI决策等
}
```

**需要你确认**：
- [ ] 目标 < 0.5ms/CompTick（严格）
- [x] 目标 < 1ms/CompTick（宽松）
- [ ] 不设性能目标，尽力而为
