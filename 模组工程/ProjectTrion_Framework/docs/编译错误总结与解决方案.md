## 错误分类与分析

### 错误类型1：Harmony库缺失（39个错误）

#### 错误表现
```
HarmonyLib related errors (5个):
- error CS0246: 未能找到类型或命名空间名称 "HarmonyLib"
- error CS0234: 类型 "HarmonyLib" 中不存在名称 "Harmony"
```

#### 直接原因
- `.csproj`文件中配置了错误的Harmony DLL路径：
  ```xml
  <Reference Include="0Harmony">
    <HintPath>C:\NiwtGames\Steam\steamapps\common\RimWorld\Mods\Harmony\1.2.2.0\Assemblies\0Harmony.dll</HintPath>
  </Reference>
  ```
- 此路径在用户的RimWorld 1.6.4633安装中不存在

#### 根本原因
- **阶段**：框架项目初始化阶段
- **责任角色**：框架设计师/项目初始化角色
- **错误根源**：对RimWorld 1.6 Harmony集成方式的假设错误
  - RimWorld 1.6已将Harmony内置到核心，不需要外部DLL引用
  - 模板可能基于旧版本（1.4/1.5）或其他项目配置
  - 未验证目标RimWorld版本的实际集成方式

#### 解决方案
1. **移除直接引用**：从`.csproj`删除`<Reference Include="0Harmony">`节点
2. **实现反射加载**：修改`HarmonyPatches/HarmonyInit.cs`使用运行时反射
   ```csharp
   var harmonyType = Type.GetType("HarmonyLib.Harmony, 0Harmony");
   var harmonyInstance = Activator.CreateInstance(harmonyType, HARMONY_ID);
   var patchAllMethod = harmonyType.GetMethod("PatchAll", new Type[] { typeof(Assembly) });
   patchAllMethod.Invoke(harmonyInstance, new object[] { typeof(HarmonyInit).Assembly });
   ```
3. **架构调整**：将Harmony补丁加载延迟到应用层（mod实现），框架层不再强制加载

#### 文件修改
- `ProjectTrion.csproj` - 删除Harmony引用
- `HarmonyPatches/HarmonyInit.cs` - 改用反射
- `GameParts/TrionDefOf.cs` - 移除[DefOf]属性

### 错误类型3：RimWorld 1.6 API兼容性问题（34个错误）

这是最复杂的错误类型，涉及多个API变化。

#### 错误3.1：BodyPartGroupDefOf缺失属性

**错误表现**
```
CS0117: "BodyPartGroupDefOf"未包含"Arms"的定义 (3个位置)
CS0117: "BodyPartGroupDefOf"未包含"Head"的定义 (3个位置)
```

**直接原因**
RimWorld 1.6的BodyPartGroupDefOf只包含这些属性：
- Torso, Legs, LeftHand, RightHand
- FullHead, UpperHead, Eyes
- LeftBlade, FrontLeftPaw

不包含通用的`Arms`或`Head`。

**根本原因**
- **阶段**：VitalPartUtil.cs和CompTrion.cs实现阶段
- **责任角色**：框架编码者
- **错误根源**：使用泛化的API名称，未针对RimWorld 1.6的具体API设计
  - 可能基于RimWorld 1.7+的文档或代码示例
  - 未使用RiMCP工具验证目标版本的实际API

**解决方案**
使用RiMCP查询实际可用的属性，替换为具体值：
```csharp
// 原代码 (错误)
return part.IsInGroup(BodyPartGroupDefOf.Arms) ||
       part.IsInGroup(BodyPartGroupDefOf.Head);

// 修复后
return part.IsInGroup(BodyPartGroupDefOf.LeftHand) ||
       part.IsInGroup(BodyPartGroupDefOf.RightHand) ||
       part.IsInGroup(BodyPartGroupDefOf.FullHead);
```

**文件修改**
- `Utilities/VitalPartUtil.cs` - 更新IsImportantPart()和GetLeakMultiplier()
- `Components/CompTrion.cs` - 更新IsImportantBodyPart()

**错误表现**
```

```

**直接原因**
RimWorld 1.6中没有`GenTeleport`实用工具类。

**根本原因**
- **阶段**：CompTrion.cs BailOut功能实现阶段
- **责任角色**：框架编码者
- **错误根源**：假设RimWorld标准库中存在GenTeleport
  - 可能是被删除或重命名的功能
  - 未验证实际API可用性

**解决方案**

```csharp
GenTeleport.TeleportTo(pawn, target, pawn.Map);

// 修复后
pawn.Position = target;
pawn.Notify_Teleported();
```

#### 错误3.4：Apparel.Wear()方法签名变化

**错误表现**
```
CS1503: 参数 1: 无法从"RimWorld.Apparel"转换为"Verse.ThingDef"
```

**直接原因**
- `CanWearWithoutDroppingAnything()`期望`ThingDef`参数，但传入了`Apparel`实例
- `Wear()`方法有两个可选参数在RimWorld 1.6中是必需的

**根本原因**
- **阶段**：CombatBodySnapshot.cs RestoreApparel()实现阶段
- **责任角色**：框架编码者
- **错误根源**：API使用错误导致的链式问题
  - 误解了CanWearWithoutDroppingAnything()的参数类型
  - Wear()方法签名变化未正确理解

**解决方案**
使用RiMCP查询正确签名：
```csharp
// Pawn_ApparelTracker.Wear()的正确签名
public void Wear(Apparel newApparel, bool dropReplacedApparel = true, bool locked = false)

// 修复前
pawn.apparel.Wear((Apparel)apparel);
pawn.apparel.CanWearWithoutDroppingAnything(apparel);  // 错误：传入Apparel

// 修复后
pawn.apparel.Wear(apparel, dropReplacedApparel: true, locked: false);
pawn.apparel.CanWearWithoutDroppingAnything(apparel.def);  // 正确：传入ThingDef
```

### 错误类型4：自动属性与Scribe序列化不兼容

#### 错误表现
```
CS0206: 非引用返回属性或索引器不能用作 out 或 ref 值
```

#### 直接原因
`TriggerMount.cs`中使用自动属性，但Scribe.Look()需要ref参数：
```csharp
// 问题代码
public bool IsActive { get; set; }  // 自动属性返回值，不是引用
Scribe_Values.Look(ref IsActive, ...);  // 需要ref，但属性返回值无法ref
```

#### 根本原因
- **阶段**：TriggerMount.cs组件设计阶段
- **责任角色**：框架设计师
- **错误根源**：使用现代C#特性（自动属性）而未考虑RimWorld序列化系统的限制
  - RimWorld使用Scribe系统进行保存/加载
  - Scribe.Look()只接受ref参数，需要实际的字段引用
  - 自动属性在编译时生成私有支持字段，无法直接ref

#### 解决方案
转换为显式backing field模式：
```csharp
// 修复前
public bool IsActive { get; set; }

// 修复后
private bool _isActive = false;
public bool IsActive
{
    get { return _isActive; }
    set { _isActive = value; }
}

// 序列化改为
Scribe_Values.Look(ref _isActive, "isActive");
```

**文件修改**
- `Components/TriggerMount.cs` - IsActive属性

### 错误类型5：字段遮蔽警告

#### 错误表现
```
CS0108: 'TriggerMountDef.description' 隐藏了继承的成员
```

#### 直接原因
`TriggerMountDef`继承自`Def`，而`Def`已包含`description`字段。子类重新定义了同名字段。

#### 根本原因
- **阶段**：TriggerMountDef.cs设计阶段
- **责任角色**：框架设计师
- **错误根源**：设计需求不清楚
  - 是否真的需要覆盖基类的description？
  - 还是应该使用基类的description？

#### 解决方案
添加`new`关键字抑制警告，表示意图遮蔽基类成员：
```csharp
// 修复前
public string description = "";

// 修复后
public new string description = "";
```

**文件修改**
- `Components/TriggerMountDef.cs` - 第58行

## 关键教训与建议

### 1. API兼容性问题根源
**教训**：框架代码对RimWorld版本的假设导致大量编译错误

**建议**：
- ✅ 使用RiMCP工具验证目标RimWorld版本的实际API
- ✅ 在框架设计文档中明确标注支持的RimWorld版本及API变化
- ✅ 建立版本兼容性测试清单

### 2. 框架与应用层责任划分
**教训**：Harmony补丁加载不应在框架层强制执行

**建议**：
- ✅ 框架提供基础设施（如HarmonyInit类）
- ✅ 应用层（mod实现）负责实际加载和应用补丁
- ✅ 减少框架的耦合度和强制要求

### 3. 序列化系统设计
**教训**：自动属性无法与RimWorld的Scribe系统兼容

**建议**：
- ✅ 所有可序列化字段使用显式backing field
- ✅ 在IExposable实现时文档化序列化字段要求
- ✅ 考虑提供基础组件类来简化常见序列化场景

### 4. 工具使用
**教训**：盲目修复导致多轮编译失败

**建议**：
- ✅ 优先使用RiMCP工具查询RimWorld API
- ✅ 不要假设API存在，先验证
- ✅ 建立"API查询 → 修复 → 验证"的标准流程
