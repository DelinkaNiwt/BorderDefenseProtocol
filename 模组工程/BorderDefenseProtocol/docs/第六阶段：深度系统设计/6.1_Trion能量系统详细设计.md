## 设计原则

本设计遵循以下原则（按优先级排序）：

| 原则 | 在本设计中的体现 |
|------|----------------|
| **KISS** | CompTrion是纯数据容器+操作，无事件回调，消费者自行轮询 |
| **SRP** | 每个组件只做一件事：CompTrion管数据+恢复驱动、Gene配置属性、Gizmo_TrionBar负责GUI显示 |
| **YAGNI** | 不预设事件系统、不预设多池模型、不预设Trion网络，需要时再加 |
| **数据驱动** | 数值通过XML Def配置，C#只实现机制框架 |
| **轮询优于事件** | 符合RimWorld tick驱动架构惯例，简单可靠 |

```

                                          ▼

  CompTrion.CompTick() ──RegisterDrain聚合结算──→ CompTrion.cur ↓
```

## 2. 核心数据模型：CompTrion

### 2.1 内部状态

```
CompTrion 内部字段：
┌─────────────────────────────────────────────────────┐
│                                                       │
│  核心数据（3个字段，Single Source of Truth）：           │
│  ───────────────────────────────────────────────────  │
│  float cur          当前Trion总值                     │
│  float max          当前最大容量                       │
│  float allocated    被占用量（芯片注册锁定）             │
│                                                       │
│  状态标志（1个字段）：                                   │
│  ───────────────────────────────────────────────────  │
│  bool frozen        恢复是否被冻结                     │
│                     由外部系统设置，CompTrion不关心原因  │
│                                                       │
│  内部控制（1个字段）：                                   │
│  ───────────────────────────────────────────────────  │
│  int refreshTick    下次从Stat刷新max的tick            │
│                                                       │
│  聚合消耗注册表（1个字段）：                              │
│  ───────────────────────────────────────────────────  │
│  Dictionary<string, float> drainRegistry               │
│                     外部模块注册的持续消耗源              │
│                     key=唯一标识, value=每天消耗量        │
│                                                       │
│  GUI缓存（1个字段，v1.5新增）：                           │
│  ───────────────────────────────────────────────────  │
│  Gizmo_TrionBar gizmo                                  │
│                     Gizmo缓存，避免每帧创建              │
│                                                       │
└─────────────────────────────────────────────────────┘
```

### 2.2 派生属性

```
Available = cur - allocated     可用量（战斗中可消耗的部分）
Percent   = max > 0 ? cur / max : 0    当前百分比
```

- `Available`：关键属性，战斗中所有消耗从此扣减
- `Percent`：便利计算属性，供Gizmo_TrionBar显示使用

### 2.3 不变量

```
不变量（任何操作后必须成立）：

  ① 0 ≤ cur ≤ max
  ② 0 ≤ allocated ≤ cur
  ③ max ≥ 0

推论（由①②自动保证）：
  · Available = cur - allocated ≥ 0
  · Percent ∈ [0, 1]
```

### 2.4 状态空间图

```
CompTrion的两种典型状态：

日常状态（frozen=false, allocated=0）：
┌─────────────────────────────────────────┐
│  cur                        │  空闲      │
│  ████████████████████████  │           │
│  Available = cur            │  max-cur  │
│  全部可用，CompTick()驱动恢复│           │
└─────────────────────────────────────────┘
0                            cur         max

战斗体状态（frozen=true, allocated>0）：
┌─────────────────────────────────────────┐
│  allocated  │  Available    │  已消耗    │
│  ██████████│  ████████████│  ░░░░░░░  │
│  芯片锁定   │  战斗可消耗   │  不可恢复  │
│  不可使用   │              │           │
└─────────────────────────────────────────┘
0          alloc          cur          max
```

## 3. 核心API规范：CompTrion

### 3.1 消耗操作

```csharp
bool Consume(float amount)
```

```
描述：从可用量中消耗Trion
调用者：触发器模块（芯片使用）、战斗模块（维持/流失消耗）、设施模块（建筑运作）

行为流程：
  amount ≤ 0 ──→ return false（无效参数）
  Available < amount ──→ return false（不足）
  cur -= amount
  return true

后置条件：
  · cur' = cur - amount
  · allocated 不变
  · Available' = Available - amount
  · 不变量①②③成立

消耗前：
┌──────────────────────────────────────┐
│  allocated  │  ████ Available ████  │
└──────────────────────────────────────┘
0           alloc                   cur

消耗后（Consume(N)成功）：
┌──────────────────────────────────────┐
│  allocated  │  ██ Available-N ██ ░░ │
└──────────────────────────────────────┘
0           alloc              cur'  旧cur
```

### 3.2 恢复操作

```csharp
void Recover(float amount)
```

```
描述：恢复Trion（日常状态下由CompTrion.CompTick()自驱动）
调用者：CompTrion.CompTick()（Pawn日常恢复）、Building适配器（补给）

行为流程：
  amount ≤ 0 ──→ return（无效参数）
  frozen == true ──→ return（静默忽略）
  cur = Mathf.Min(cur + amount, max)

设计决策——frozen时静默忽略而非抛异常：
  CompTrion.CompTick()每recoveryInterval ticks调用Recover()，
  不应每次都先检查frozen。让Recover()自己处理更简洁。

恢复前：
┌──────────────────────────────────────┐
│  ████████████ cur ████  │  空闲      │
└──────────────────────────────────────┘
0                        cur          max

恢复后（Recover(N)）：
┌──────────────────────────────────────┐
│  ████████████████ cur+N ██  │ 空闲  │
└──────────────────────────────────────┘
0                           cur'      max
```

### 3.3 占用操作

```csharp
bool Allocate(float amount)
```

```
描述：锁定占用量（芯片注册时调用）
调用者：战斗模块（HComp_TrionAllocate）

行为流程：
  amount ≤ 0 ──→ return false
  Available < amount ──→ return false（可用量不足以锁定）
  allocated += amount
  return true

后置条件：
  · allocated' = allocated + amount
  · cur 不变
  · Available' = Available - amount

锁定前（日常状态，allocated=0）：
┌──────────────────────────────────────┐
│  ████████████ Available ████████████ │
└──────────────────────────────────────┘
0                                    cur

锁定后（Allocate(N)）：
┌──────────────────────────────────────┐
│  ██ alloc ██ │  ████ Available' ████ │
└──────────────────────────────────────┘
0            alloc'                  cur
```

```csharp
void Release(float amount)
```

```
描述：释放占用量（主动解除时调用）
调用者：战斗模块（HComp_TrionAllocate，主动解除路径A）

行为流程：
  amount = Mathf.Min(amount, allocated)  // 防御性钳位
  allocated -= amount

后置条件：
  · allocated' = allocated - amount
  · cur 不变
  · Available' = Available + amount（可用量增加）
```

### 3.4 强制操作

```csharp
void ForceDeplete()
```

```
描述：强制耗尽所有Trion（被动破裂时调用）
调用者：战斗模块（被动破裂路径B）

行为流程：
  cur = 0
  allocated = 0

语义对应：

破裂前：
┌──────────────────────────────────────┐
│  allocated  │  ██ Avail ██  │  已消耗 │

ForceDeplete()后：
┌──────────────────────────────────────┐
│  ░░░░░░░░░░░░░░ 全部归零 ░░░░░░░░░░ │
└──────────────────────────────────────┘
cur=0, allocated=0, Available=0
```

### 3.5 状态控制

```csharp
void SetFrozen(bool value)
```

```
描述：设置恢复冻结状态
调用者：战斗模块（进入/退出战斗体状态时）
行为：frozen = value
```

```csharp
void RefreshMax()
```

```
描述：从Stat系统重新读取max值
调用时机：
  · 定期刷新（每statRefreshInterval ticks，默认250）
  · Gene添加/移除时（Gene_TrionGland.PostAdd/PostRemove）
  · Hediff变化时（触发角植入/移除）
  · 装备变化时（黑触发器装备/卸下）

行为流程：
  if parent is Pawn pawn:
    newMax = pawn.GetStatValue(BDP_StatDefOf.TrionCapacity)
  else:
    newMax = Props.baseMax    // Building/Item用XML配置值
  max = Mathf.Max(0, newMax)
  cur = Mathf.Clamp(cur, 0, max)   // max缩小时cur跟着缩
  allocated = Mathf.Min(allocated, cur)  // 保持不变量②

max缩小时的钳位行为：
  旧状态：max=100, cur=80, allocated=30
  触发角被移除 → newMax=60
  钳位后：max=60, cur=60, allocated=30
  Available: 80-30=50 → 60-30=30（可用量减少）
```

### 3.6 聚合消耗注册（v1.4新增）

```csharp
void RegisterDrain(string key, float drainPerDay)
```

```
描述：注册一个持续消耗源
调用者：触发器模块（隐身芯片激活）、战斗模块（战斗体维持）、未来其他持续消耗源

行为：
  drainRegistry[key] = drainPerDay

参数说明：
  key: 唯一标识（如 "combatBody", "stealth_main"）
  drainPerDay: 每天消耗的Trion绝对值

设计动机：
  多个模块可能同时产生持续Trion消耗。
  各模块独立调用Consume()存在浮点精度问题且无法统一展示总消耗速率。
  聚合后由CompTick统一间隔结算，一次Consume()。
```

```csharp
void UnregisterDrain(string key)
```

```
描述：移除一个持续消耗源
调用者：触发器模块（隐身芯片关闭）、战斗模块（战斗体解除）

行为：
  drainRegistry.Remove(key)
```

```csharp
float TotalDrainPerDay { get; }
```

### 3.7 生命周期方法

```csharp
override void PostSpawnSetup(bool respawningAfterLoad)
```

```
行为：
  drainRegistry = new Dictionary<string, float>()  // 始终初始化
  if !respawningAfterLoad:   // 首次生成（非读档）
    max = Props.baseMax
    cur = max * Props.startPercent
    allocated = 0
    frozen = false
  RefreshMax()               // 立即从Stat同步一次
  refreshTick = Find.TickManager.TicksGame + Props.statRefreshInterval
  // 注册建筑被动消耗（如有）
  if Props.passiveDrainPerDay > 0:
    RegisterDrain("passive", Props.passiveDrainPerDay)
```

```csharp
override void CompTick()
```

```
行为：
  // 定期刷新max
  if Find.TickManager.TicksGame >= refreshTick:
    RefreshMax()
    refreshTick = Find.TickManager.TicksGame + Props.statRefreshInterval

  // 聚合消耗结算（v1.4统一机制）
  // passiveDrainPerDay在PostSpawnSetup时已注册到drainRegistry
  // 所有持续消耗源统一在此间隔结算
  if drainRegistry.Count > 0 && Find.TickManager.TicksGame % Props.drainSettleInterval == 0:
    totalDrain = Sum(drainRegistry.Values)
    Consume(totalDrain * Props.drainSettleInterval / TICKS_PER_DAY)

  // Pawn恢复驱动（v1.5新增，原Need_Trion职责）
  if parent is Pawn pawn && max > 0 && !frozen
     && Find.TickManager.TicksGame % Props.recoveryInterval == 0:
    recoveryRate = pawn.GetStatValue(BDP_StatDefOf.TrionRecoveryRate)
    amount = recoveryRate * Props.recoveryInterval / TICKS_PER_DAY
    Recover(amount)

  // 非Pawn载体自动恢复（Pawn由上方逻辑驱动，此处不生效）
  if Props.recoveryPerDay > 0 && parent is not Pawn
     && Find.TickManager.TicksGame % Props.drainSettleInterval == 0:
    Recover(Props.recoveryPerDay * Props.drainSettleInterval / TICKS_PER_DAY)
```

## 4. Pawn适配层

### 4.1 Gene_TrionGland — 天赋配置器

```
┌─────────────────────────────────────────────────────┐
│  Gene_TrionGland : Gene                               │
│                                                       │
│  职责：表达"此Pawn拥有Trion腺体"这一先天事实           │
│        向Stat系统贡献基础属性值                         │
│        不持有任何Trion数据，不参与运行时逻辑            │
│                                                       │
│  继承Gene而非Gene_Resource的原因：                      │
│    Gene_Resource假设自身持有cur/max                     │
│    数据在CompTrion中 → 继承它会产生两份数据源           │
│    违反Single Source of Truth                          │
│                                                       │
│  运行时开销：零（无Tick逻辑）                           │
└─────────────────────────────────────────────────────┘

GeneDef XML配置：
  statOffsets:
    TrionCapacity:     +N    // 天赋对容量的基础贡献
    TrionOutputPower:  +N    // 天赋对输出功率的基础贡献
    TrionRecoveryRate: +N    // 天赋对恢复速率的基础贡献
  // v1.5：enablesNeeds已移除。
  // 原设计通过enablesNeeds激活Need_Trion，现已不需要。
  // 惰性激活改为纯max>0判断：Gene的statOffsets使TrionCapacity>0
  // → CompTrion.max>0 → 自动恢复+Gizmo显示。

C#方法：
  PostAdd():
    comp = pawn.GetComp<CompTrion>()
    comp?.RefreshMax()       // Stat重新聚合，更新max

  PostRemove():
    comp = pawn.GetComp<CompTrion>()
    comp?.RefreshMax()       // max可能缩小

为什么用statOffsets而不是在PostAdd()里直接设置max？

  直接设置：Gene.PostAdd() → comp.max = 100
    问题：触发角(Hediff)也贡献容量，两个来源冲突

  statOffsets：Gene贡献+80, Hediff贡献+20 → Stat聚合=100
    优点：多源自动叠加，无冲突

  ┌──────────────────────────────────────────┐
  │  Stat:TrionCapacity 聚合过程              │
  │                                          │
  │  Gene_TrionGland.statOffsets  ──→ +80    │
  │  Hediff_TriggerHorn.statOffsets ──→ +20  │
  │  ────────────────────────────────────     │
  │  聚合结果                      = 100     │
  │                                          │
```

### 5.2 Stat聚合流程

```
以TrionCapacity为例，引擎自动聚合过程：

  StatWorker.GetValueUnfinalized(pawn, TrionCapacity)
       │
       ├── defaultBaseValue = 0
       │
       ├── + Hediff_TriggerHorn.CurStage.statOffsets[TrionCapacity]  // 植入（先）

       ├── + Gene_TrionGland.def.statOffsets[TrionCapacity]    // 天赋（后）

    // 引擎内部有缓存，频繁调用开销可控
```

## 7. 游戏机制公式

### 7.1 公式总览

```
本节定义Trion系统的核心公式。
采用"公式优先"策略：用变量名和关系式表达机制，
具体数值留给平衡性测试阶段（6.9 平衡性参考表）。
```

### 7.2 容量公式

```
MaxCapacity = BASE_CAPACITY + TRIGGER_HORN_BONUS + TRAINING_BONUS

其中：
  BASE_CAPACITY      = GeneDef.statOffsets[TrionCapacity]
                       天赋决定的基础容量
  TRIGGER_HORN_BONUS = Hediff_TriggerHorn.stage.statOffsets[TrionCapacity]
                       Trion植入物增强
  TRAINING_BONUS     = 待定（可能通过经验系统实现）

所有项通过Stat系统自动聚合，CompTrion.RefreshMax()读取结果。

容量的原作参考范围（BORDER参数评分制）：
  操作员（非战斗）: ~1
  低级战斗员:       4
  普通战斗员:       7
  高级战斗员:       9
  异常值（千佳）:   38

  注：参数评分制是相对评分，非绝对Trion值。
  游戏中的绝对值需要平衡性测试确定。
```

### 7.3 恢复公式

```
RecoveryPerInterval = RECOVERY_RATE × (INTERVAL_TICKS / TICKS_PER_DAY)

其中：
  RECOVERY_RATE   = pawn.GetStatValue(TrionRecoveryRate)
                  = BASE_RECOVERY + HEDIFF_MODIFIERS
  INTERVAL_TICKS  = 150（NeedInterval固定间隔）
  TICKS_PER_DAY   = 60000

单位：RECOVERY_RATE的单位是"每天恢复的Trion绝对值"

示例（假设数值）：
  RECOVERY_RATE = 50（每天恢复50点Trion）
  每150 ticks恢复 = 50 × (150 / 60000) = 0.125点

恢复时间估算：
  从枯竭(cur=0)恢复到满(cur=max)所需天数：
  Days = max / RECOVERY_RATE

  高容量者恢复更慢（原作设定）：
  ┌──────────────────────────────────────────┐
  │  角色        max    rate    恢复天数      │
  │  低Trion者    40     50     0.8天         │
  │  普通战斗员   70     50     1.4天         │
  │  高Trion者   100     50     2.0天         │
  │                                          │
  │  注：以上为示意数值，非最终平衡数据        │
  └──────────────────────────────────────────┘
```

### 7.4 建筑被动消耗公式

```
DrainPerTick = PASSIVE_DRAIN_PER_DAY / TICKS_PER_DAY

其中：
  PASSIVE_DRAIN_PER_DAY = CompProperties_Trion.passiveDrainPerDay
  TICKS_PER_DAY         = 60000

在CompTrion.CompTick()中每tick调用Consume(DrainPerTick)。
```

## 9. 存档兼容性设计

### 9.1 存档字段

```
CompTrion.PostExposeData() 存档的字段：

  字段名            类型     说明
  ─────────────────────────────────────
  trionCur          float    当前Trion值
  trionMax          float    当前最大容量
  trionAllocated    float    被占用量
  trionFrozen       bool     恢复冻结状态
  trionDrains       Dict     聚合消耗注册表（v1.4新增）
```

### 9.2 读档后校验

```
读档后（PostLoadInit）执行不变量校验：

  max = Mathf.Max(0, max)
  cur = Mathf.Clamp(cur, 0, max)
  allocated = Mathf.Clamp(allocated, 0, cur)

目的：防止存档损坏或版本升级导致的数据不一致。
即使存档中的值违反不变量，读档后也能恢复到合法状态。
```

### 9.3 版本升级兼容

```
场景：玩家从旧版本升级到新版本

  新增字段：Scribe_Values.Look的defaultValue参数保证
            旧存档中不存在的字段使用合理默认值

  字段改名：避免。如果必须改名，提供迁移逻辑
            在PostExposeData中先尝试读新名，失败则读旧名

  字段删除：旧存档中多余的字段被Scribe自动忽略
```

## 11. 关键设计决策记录

| # | 决策 | 选择 | 理由 | 否决方案 |
|---|------|------|------|---------|
| D1 | 通知机制 | 纯轮询，无事件回调 | KISS+YAGNI+RimWorld惯例；消费者在自己的Tick中检查CompTrion状态 | 事件回调（增加复杂度，当前无必要） |
| D2 | Max同步策略 | 定期刷新(250 ticks)+事件触发双保险 | 简单可靠，有兜底机制；GetStatValue有引擎缓存，开销可控 | 懒加载+脏标记（需精确维护dirty，容易遗漏） |
| D3 | Building/Item适配器 | 独立Comp组合模式 | 组合优于继承；CompTrion保持通用性；符合RimWorld惯例 | 子类继承（违反通用容器定位） |
| D5 | Gene继承 | 继承Gene，不继承Gene_Resource | 避免两份数据源；Gene只做配置器，不持有cur/max | 继承Gene_Resource（两份数据需同步） |
| D6 | Recover()对frozen的处理 | 静默忽略 | CompTrion.CompTick()不需要每次检查frozen再决定是否调用 | 抛异常（过于严格，增加调用方负担） |
| D7 | Percent属性 | 保留为便利计算属性 | 多个消费者需要cur/max计算，减少重复；符合RimWorld惯例 | 去掉（消费者自己算，略增代码重复） |
| D8 | ~~基因-Need关联~~ | ~~onlyIfCausedByGene + enablesNeeds~~ | ~~v1.5废弃：Need_Trion已移除。惰性激活改为纯max>0判断~~ | — |
| D9 | Stat条件显示 | showIfUndefined="false" | defaultBaseValue=0时自动隐藏；有基因时statOffsets使值非零，自动显示 | 自定义StatWorker（过度工程） |
| D10 | Def扩展机制 | 通用TrionDefExtension | 一个类服务所有Def类型；符合RimWorld modExtensions惯例 | 每个Def类型单独Extension（类爆炸） |
| D13 | 持续消耗结算 | RegisterDrain聚合机制 | 多源统一结算避免浮点误差；可统一展示总消耗速率；间隔结算（60 ticks）平衡精度与性能 | 各模块独立Consume()（浮点累积误差、无法统一展示） |
| D14 | Need_Trion去留（v1.5） | 完全移除 | Trion是战斗资源不是生理需求；Need面板语义不匹配；GUI应在Gizmo栏显示 | 保留但隐藏（概念不干净）；保留做内部驱动（多余的间接层） |
| D15 | 职责重分配（v1.5） | 全部归入CompTrion | 单一职责集中，不再需要Need代理层；CompTrion已有CompTick()可复用 | 分散到多个组件（增加复杂度） |
| D16 | GUI风格（v1.5） | Gizmo_Slider+扩展（三段色） | 复用Gizmo_Slider框架减少开发量；三段色展示占用/可用/空 | 自定义Gizmo（开发量大）；纯Gizmo_Slider（无法展示占用段） |
| D17 | 三段色顺序（v1.5） | 占用→可用→空（左到右） | 占用是"已锁定的底层"在左，可用是"活跃的中间层" | 可用→占用→空（占用在中间不直观） |
| D18 | Gizmo来源（v1.5） | CompTrion.CompGetGizmosExtra() | CompTrion已预挂载到所有Human；max>0判断统一；Building/Item可复用 | Gene提供（Building无法复用） |
| D19 | 惰性激活简化（v1.5） | 纯max>0判断 | 移除Need后自然简化；不再依赖enablesNeeds | 保留enablesNeeds（多余） |
