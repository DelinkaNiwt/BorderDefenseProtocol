## 一、模块结构与分层

### 目录结构

```
BDP/
├── Core/                    L0（不变）
│   ├── Comps/              CompTrion
│   ├── Defs/               BDP_DefOf
│   ├── Genes/              Gene_TrionGland ← 扩展：外层FSM + 快照持有 + Gizmo
│   └── Gizmos/             Gizmo_TrionBar
│
├── Trigger/                 L1（不变）
│
└── Combat/                  L2（新建）
    ├── Hediffs/            HediffComp_CombatBodyActive（内层FSM + 破裂检测）
    ├── Snapshot/           CombatBodySnapshot（IExposable Memento容器）
    ├── Pipeline/           CombatBodyDamagePipeline（伤害管道逻辑）
    ├── Escape/             EscapeRouter（责任链，紧急脱离目标选择）
    └── Patches/            Harmony patch（伤害/死亡/倒地，共3个）
```

### 依赖规则

```
L0: Core/       ← 不知道Trigger/和Combat/的存在
L1: Trigger/    → Core/（已有，CompTriggerBody查询CompTrion）
L2: Combat/     → Core/ + Trigger/（单向，允许）
```

- Core/不依赖任何其他模块
- Combat/与Trigger/之间：Combat/可读Trigger/，Trigger/不知道Combat/
- 跨模块添加Hediff使用DefName字符串，无C#类型引用

## 二、双层FSM设计

### 设计决策

选择双层FSM（方案二）而非单层Gene全权驱动（方案一）或Pipeline化破裂检测（方案三）。

**理由**：Gene职责保持轻量（SRP），HediffComp在Active期完全自治，两层边界清晰。当前6个破裂条件稳定，不需要Policy接口体系的抽象开销。

### 外层（Gene_TrionGland，Core/，3态）

```
Inactive ──[InitiateChanneling()]──→ Active（添加Hediff）
   ▲                                    │
   │                                    │
Cooldown ←── Active
   │    [OnCombatBodyEnded(Voluntary)]  Release + 短CD
   │    [OnCombatBodyEnded(Passive)]    ForceDeplete + 长CD
   └──[CD到期]──→ Inactive
```

Gene只做三件事：持有外层状态枚举、持有快照引用、响应HediffComp回调。

### 内层（HediffComp_CombatBodyActive，Combat/，3态）

```
Channeling ──[计时完成]──→ Established ──[玩家Gizmo]──→ DismissChanneling
    │                          │                              │
    │[中断/破裂条件]           │[破裂条件任一]                │[计时完成]
    ↓                          ↓                              ↓
  Gene.OnCombatBodyEnded    Gene.OnCombatBodyEnded         Gene.OnCombatBodyEnded
  (Passive, 跳过快照还原)   (Passive)                      (Voluntary)
```

### 两层协议接口

Gene暴露给HediffComp的入口：

- `InitiateChanneling()` — Gizmo调用
- `DismissChanneling()` — "主动解除"Gizmo调用
- `OnCombatBodyEnded(CombatBodyEndReason)` — HediffComp唯一回调入口

HediffComp访问Gene：`hediff.pawn.genes.GetFirstGeneOfType<Gene_TrionGland>()`

## 三、Gizmo解耦——静态可否决事件

### 设计决策

Gizmo挂在Gene_TrionGland（Core/ L0），但激活检查需要CompTriggerBody（Trigger/ L1）。L0不能依赖L1。

**选择**：静态可否决事件（Vetoable Event），而非动态ThingComp（序列化陷阱）、多基因（语义不符）、接口注入（需Harmony）或推送bool（侵入性高）。

### 机制

- Core/：Gene_TrionGland定义静态事件 `QueryCanActivate`
- Combat/：模块启动时（`[StaticConstructorOnStartup]`）注册检查逻辑
- 事件携带EventArgs，订阅者可设置`Vetoed=true`+`BlockReason`否决激活
- Gene调用事件时不知道订阅者是谁，零耦合
- 未来新增激活路径：新增订阅者即可，Gene无需改动

## 四、快照系统（Memento）

### 设计决策

Gene_TrionGland持有CombatBodySnapshot引用。快照在**Channeling计时完成、激活操作执行前**拍摄，确保记录引导完成时的真实状态。

### 快照内容（NR-026）

| 数据类型 | 快照方式 | 还原方式 |
|---|---|---|
| Hediff列表 | 记录字段值快照（List<HediffRecord>，hediff留在pawn上不动） | 删除激活期间新增的hediff，按快照字段值重新AddHediff |
| 需求数值 | Dictionary<NeedDef, float>记录值 | need.CurLevel = savedValue |
| 衣物 | 物理转移至快照容器（ThingOwner） | 先从快照容器转移出，再pawn.apparel.Wear(dropReplacedApparel:false) |
| 物品 | 物理转移至快照容器（ThingOwner） | 使用ThingOwner.TryTransferToContainer恢复到pawn.inventory.innerContainer |

### M1阶段实际实现方案（2026-03-03）

**衣物方案：换装 + ThingOwner**
```
激活：原衣物 → originalApparelContainer（ThingOwner<Apparel>）
      战斗体衣物 → ThingMaker.MakeThing()创建 → Pawn.apparel.Wear()穿戴
解除：战斗体衣物 → combatApparelContainer（ThingOwner<Apparel>）
      原衣物 → Pawn.apparel（从originalApparelContainer转移 → Wear()穿戴）
```

**物品方案：转移 + ThingOwner + 阻止腐烂**
```
激活：物品 → originalInventoryContainer（ThingOwner<Thing>）
解除：物品 → Pawn.inventory（从originalInventoryContainer转移）
腐烂：Harmony patch CompRottable.Active属性，检测holdingOwner是否为CombatBodySnapshot
```

**关键实现细节**：
- 三个ThingOwner容器：`originalApparelContainer`（原衣物）、`combatApparelContainer`（战斗体衣物）、`originalInventoryContainer`（原物品）
- 衣物状态记录：`Dictionary<int, ApparelState>`记录locked/forced状态，key为thingIDNumber
- 物品状态记录：`Dictionary<int, ItemState>`记录NotForSale/UnpackedCaravan标记
- 阻止腐烂：Harmony Postfix patch `CompRottable.Active`属性getter，当容器owner是CombatBodySnapshot时返回false

### 快照排除机制

部分Hediff不应参与快照→回滚（如启灵神经、机控中枢），通过 `BDPSnapshotConfigDef` 配置：

- **按具体Def排除**：`excludedHediffs` 列表，精确匹配
- **按C#类排除**：`excludedHediffClasses` 列表，`IsAssignableFrom` 匹配含所有子类
- **零侵入**：不修改任何原版Def；其他模组可添加自己的 `BDPSnapshotConfigDef` 扩展排除名单
- **对称性约束**：记录步骤和移除步骤必须使用相同的排除过滤（`BDPSnapshotConfigDef.IsExcluded`），两边不对称会导致排除项在移除阶段丢失
- **默认配置**：`1.6/Defs/Combat/BDPSnapshotConfig.xml`（排除 `PsychicAmplifier`、`MechlinkImplant` 及其子类）

### 关键技术要点

- Hediff快照采用**字段值记录**而非实例持有：激活时遍历hediffSet记录每条hediff的字段值（defName/severity/bodyPartDefName/bodyPartIndex/isPermanent），hediff实例留在pawn上不动；激活期间新增的hediff打上BDP标记；还原时删除有标记的hediff，按快照字段值重新AddHediff
- 对”回滚时不希望触发的副作用”采用**受控还原上下文**（RestoreScope）+定向拦截，而非绕过整个健康管线
- 存档时CombatBodySnapshot通过IExposable序列化：Hediff列表使用Scribe_Collections+LookMode.Deep序列化List<HediffRecord>（自定义IExposable结构体，只含基础字段，无游离实例风险）；衣物/物品ThingOwner用Scribe_Deep.Look序列化，CombatBodySnapshot需实现IThingHolder
- 衣物/物品恢复必须遵循ThingOwner所有权规则：先转移容器，再穿戴/入包，避免holdingOwner冲突
- CombatBodySnapshot需实现IThingHolder接口（GetChildHolders + GetDirectlyHeldThings），ThingOwner构造时传入this作为owner，确保存读档时容器所有权正确重建

### M1阶段关键问题修复（2026-03-03）

**问题1：Gene.GetGizmos() NullReferenceException**
- **根因**：`base.GetGizmos()` 在某些游戏初始化状态下可能返回null，直接在foreach中使用导致异常
- **修复**：添加null检查，先将baseGizmos赋值给变量，检查非null后再迭代
- **教训**：RimWorld的基类方法在特殊状态下可能返回null而非空集合，必须防御性检查

**问题2：战斗体装甲纹理路径错误**
- **根因**：ThingDef中texPath和wornGraphicPath引用了不存在的ShieldBelt纹理
- **修复**：改用原版Duster纹理作为临时占位
- **教训**：装备ThingDef必须使用存在的纹理路径，否则渲染时报错

**问题3：CompRottable腐烂阻止**
- **根因**：最初尝试patch CompTick方法，但CompRottable没有CompTick，实际是CompTickRare→TickInterval→检查Active属性
- **修复**：Harmony Postfix patch `CompRottable.Active`属性getter，利用游戏原有控制流
- **教训**：优先patch属性而非方法，利用游戏原有逻辑，而非阻断执行流

## 五、伤害管道

### Harmony patch清单（3个，最小化）

| Patch | 拦截点 | 作用 |
|---|---|---|
| Patch_DamageInterception | Pawn_HealthTracker.PreApplyDamage | 完全拦截伤害，转入BDP管道 |
| Patch_DeathInterception | 死亡相关方法（具体方法留详设） | 头/心脏致死→被动破裂 |
| Patch_DownedInterception | Pawn_HealthTracker.MakeDowned | 倒地→被动破裂 |

### 管道流程（NR-042）

```
伤害x，部位P → 完全拦截（原版HP不变）
    ├─ Trion.Consume(f(x))         f(x)>x，直接消耗
    ├─ shadowHP[P] -= x            影子HP扣减
    └─ 分支：
        ├─ shadowHP>0 → 伤口Hediff创建/合并 + Trion持续流失注册
        └─ shadowHP≤0 → 部位被毁（NR-035逻辑）
            ├─ 头/心/双腿 → 被动破裂
            └─ 其他器官 → 注册高额Trion流失（间接破裂）
```

### 影子HP

- 住在HediffComp_CombatBodyActive内，Dictionary<BodyPartRecord, float>
- 激活时每部位从最大HP初始化，植入物覆盖的缺失部位视为完整（NR-027）
- 需序列化存档（NR-041），通过CompExposeData

### 伤口Hediff

- BDP专用Hediff类型，不继承原版伤口类（天然屏蔽出血/疼痛）
- 同部位多次受伤合并为一个（severity增大），不堆叠
- 每部位维持唯一一条Trion持续流失注册

## 六、关键时序

### 激活流

```
玩家点Gizmo → Gene.InitiateChanneling()
    ├─ 前置检查（Trion足够、不在CD、事件未否决）
    ├─ Trion.Allocate(cost)        预锁定能量
    ├─ AddHediff("BDP_CombatBodyActive")  DefName字符串
    └─ OuterState = Active

HediffComp.InnerState = Channeling（引导期，可被打断）
    └─ 计时完成 → 拍快照（原子操作第1步）→ 执行11步激活操作 → InnerState = Established
```

引导被打断时：丢弃（快照尚未拍摄）、Trion.Release、RemoveHediff、OuterState=Inactive、CD=0（接口保留）。

### 被动破裂流

```
破裂条件触发 → Gene.OnCombatBodyEnded(Passive)
    ├─ 检查紧急脱离（装备脱离芯片？有效目标？）
    │     信标建筑 → 殖民者区域 → 随机安全位置（责任链三级回退）
    │     先传送（战斗体形态），再还原（落点处恢复真身）
    ├─ 执行还原操作（Hediff/需求/衣物/物品回滚）
    ├─ Trion.ForceDeplete()        能量清零
    ├─ 添加枯竭debuff（NR-040）
    └─ OuterState = Cooldown（长CD）
```

### 主动解除流

```
玩家点Gizmo → HediffComp.InnerState = DismissChanneling
    ├─ 解除引导期（期间被动破裂优先级更高）
    └─ 计时完成 → Gene.OnCombatBodyEnded(Voluntary)
        ├─ 执行还原操作
        ├─ Trion.Release(cost)     能量归还
        └─ OuterState = Cooldown（短CD=0，接口保留）
```

## 八、完整架构决策表

| 决策项 | 结论 | 理由 |
|---|---|---|
| 模块位置 | 新建Combat/（L2） | 独立系统，不污染Core/和Trigger/ |
| 持久层 | Gene_TrionGland扩展 | 生命周期与基因一致，IExposable天然支持 |
| 活跃行为 | HediffComp_CombatBodyActive | Hediff驱动符合RimWorld原生风格 |
| 外层FSM | Gene管3态 | Gene轻量，仅状态+快照+事件 |
| 内层FSM | HediffComp管3态 | Active期完全自治，解耦清晰 |
| Gizmo解耦 | 静态可否决事件 | 零Harmony、零per-pawn注入、可扩展 |
| Hediff跨模块 | DefName字符串 | 无C#依赖，保持分层 |
| 快照策略 | API优先回滚 + Scribe_Deep快照 | 保持RimWorld健康系统不变量，避免直接改内部列表 |
| 影子HP | HediffComp内Dict，序列化 | 按部位追踪，非HP系统，是Trion缓冲层 |
| 伤害拦截 | 3个Harmony patch | 最小化：伤害/死亡/倒地各1个 |
| 紧急脱离 | 责任链三级回退 | 信标→基地→随机，逻辑清晰 |
| 破裂条件 | Tick轮询+Harmony即时 | 混合：状态类轮询，事件类即时 |
| CD机制 | 接口保留，默认值=0 | 架构先行，数值后调 |
