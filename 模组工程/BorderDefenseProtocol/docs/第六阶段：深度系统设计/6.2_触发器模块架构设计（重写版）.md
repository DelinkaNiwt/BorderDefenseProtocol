# 触发器模块架构设计

## 前置文档

- 5.1 BDP模组总体架构设计——微内核+多模块方案
- 6.1 Trion能量系统详细设计

## 设计原则（按优先级）

1. **KISS** — 能简单就不复杂
2. **SRP** — 每个类只干一件事
3. **OCP** — 新增芯片类型不改现有代码
4. **DIP** — 依赖抽象（IChipEffect），不依赖具体效果类
5. **数据驱动** — 变体通过XML Def配置，不硬编码
6. **YAGNI** — 不为假想需求增加复杂度
7. **零Harmony Patch** — 完全通过继承和接口实现，最大化模组兼容性

## 1. 架构总览

触发器模块是BDP总体架构（微内核+多模块）中的**第一层消费者模块**，仅依赖微内核（CompTrion能量系统），与角色模块、设施模块之间零依赖。

模块内部以**CompTriggerBody为中心编排器**，通过**策略模式**（IChipEffect接口）实现芯片行为的可扩展性，通过**状态机**管理槽位切换，通过**DualVerbCompositor组合器**合成双武器Verb配置。

三种触发器（BORDER / 近界 / 黑触发器）共享同一套CompTriggerBody代码，仅通过XML Props配置产生完全不同的玩家体验。

```
  ┌─ BDP.Trigger 模块 ──────────────────────────────────────────────┐
  │                                                                  │
  │  ┌──────── CompTriggerBody (编排器) ──────────────────────────┐ │
  │  │                                                             │ │
  │  │  ┌──────────┐  ┌──────────┐  ┌──────────┐                 │ │
  │  │  │ 左手槽位  │  │ 右手槽位  │  │ 特殊槽位  │ ← List<ChipSlot> │
  │  │  └─────┬────┘  └─────┬────┘  └─────┬────┘                 │ │
  │  │        └──────┬──────┘──────────────┘                      │ │
  │  │               ▼                                             │ │
  │  │     ┌── 状态机 (按侧独立：Idle/WindingDown/WarmingUp) ──┐     │ │
  │  │     │  后摇 / 前摇 / 激活 / 关闭                       │     │ │
  │  │     └──────────┬────────────────────┘                      │ │
  │  └────────────────┼───────────────────────────────────────────┘ │
  │                   │ 委托                                        │
  │                   ▼                                              │
  │  ┌─── IChipEffect (策略接口) ──────────────────────────────┐   │
  │  │  Activate() / Deactivate() / Tick() / CanActivate()     │   │
  │  └──┬──────────┬──────────┬────────────────────────────────┘   │
  │     ▼          ▼          ▼                                     │
  │  Weapon     Shield     Utility       ← 可扩展                  │
  │  (Verb注入)  (Hediff)   (Ability)                               │
  │     │                                                           │
  │     ▼                                                           │
  │  DualVerbCompositor (组合器) ──→ 自定义Verb族                   │
  │                                  ──→ Command_BDPChipAttack     │
  │                                  ──→ JobDriver (攻击执行)      │
  │                                                                  │
  │  ┌── UI层 ─────────────────────────────────────────────────┐   │
  │  │  Gizmo_TriggerBodyStatus  │  Window_TriggerBodySlots    │   │
  │  └─────────────────────────────────────────────────────────┘   │
  └──────────────────────────┬─────────────────────────────────────┘
                             │ 依赖
                             ▼
                  BDP.Core — CompTrion (微内核/能量系统)
```

### 组件职责一览

| 组件 | 职责 |
|------|------|
| CompTriggerBody | 编排器。管理槽位、状态机、激活逻辑、IVerbOwner实现、Gizmo生成、组合能力 |
| ChipSlot | 槽位数据容器（IExposable），持有芯片引用和激活状态 |
| SwitchContext | 按侧独立的切换上下文（IExposable），记录阶段(WindingDown/WarmingUp)、目标槽位、阶段结束tick |

| ShieldChipEffect | 护盾策略：添加护盾Hediff |

| WeaponChipConfig | DefModExtension，存储武器芯片的VerbProperties/Tools/连击数/射击消耗 |

| UtilityChipConfig | DefModExtension，存储辅助芯片的AbilityDef |
| ComboAbilityDef | 组合能力Def（继承Def），定义两芯片组合后授予的Ability |
| PreloadedChipConfig | 预装芯片配置POCO，用于CompProperties_TriggerBody.preloadedChips |
| DualVerbCompositor | 纯静态组合器，根据两侧武器数据合成Verb配置 |
| Verb_BDPRangedBase | BDP远程Verb基类，提供TryCastShotCore辅助方法替换equipmentSource，重写OrderForceTarget使用自定义JobDriver |
| Verb_BDPShoot | 单发远程Verb，射击前检查并消耗Trion |
| Verb_BDPDualRanged | 交替连射Verb，左右子弹交替发射 |
| Verb_BDPMelee | 近战Verb，hitIndex状态机驱动burst连击，每击独立前冲动画，支持按芯片配置不同burst间隔 |

| Command_BDPChipAttack | 自定义Command，按attackId控制Gizmo合并 |
| JobDriver_BDPChipRangedAttack | 自定义JobDriver，直接使用job.verbToUse并手动推进VerbTick |
| CompAbilityEffect_TrionCost | Ability的Trion消耗组件 |
| Gizmo_TriggerBodyStatus | 内联状态Gizmo，四态显示左右手芯片图标 |
| Window_TriggerBodySlots | 浮动窗口，显示所有槽位并支持点击激活/关闭 |

## 2. 槽位系统

### 数据模型

CompTriggerBody持有三组槽位列表，每组是`List<ChipSlot>`：

```
CompTriggerBody
├── leftHandSlots: List<ChipSlot>    // 左手槽，可装任意类型芯片，每次只能激活1个
├── rightHandSlots: List<ChipSlot>   // 右手槽，可装任意类型芯片，每次只能激活1个
├── specialSlots: List<ChipSlot>     // 特殊槽，全量同时激活/关闭（典型芯片待设计）
├── leftSwitchCtx: SwitchContext    // 左手侧切换上下文（null=Idle）
├── rightSwitchCtx: SwitchContext   // 右手侧切换上下文（null=Idle）
├── dualHandLockSlot: ChipSlot      // 双手武器锁定（非null时另一侧不可激活）
├── isCombatBodyActive: bool         // 战斗体是否激活（影响UI四态和allocationCost时机）
├── grantedCombos: List<ComboAbilityDef> // 已授予的组合能力列表
├── leftHandActiveVerbProps: List<VerbProperties>  // 左侧Verb配置（由WeaponChipEffect写入）
├── rightHandActiveVerbProps: List<VerbProperties> // 右侧Verb配置
├── leftHandActiveTools: List<Tool>                // 左侧近战Tool配置
├── rightHandActiveTools: List<Tool>               // 右侧近战Tool配置
├── leftHandAttackVerb: Verb         // 缓存：左手攻击Verb实例
├── rightHandAttackVerb: Verb        // 缓存：右手攻击Verb实例
└── dualAttackVerb: Verb             // 缓存：双手触发攻击Verb实例
```

**ChipSlot**（IExposable）：

```csharp
class ChipSlot : IExposable
{
    public int index;              // 该侧内的序号
    public SlotSide side;          // Left / Right / Special
    public Thing loadedChip;       // 已装载的芯片Thing（null=空槽）
    public bool isActive;          // 是否已激活
}
```

### CompProperties_TriggerBody（XML配置）

```xml
<compClass>BDP.Trigger.CompTriggerBody</compClass>
<leftHandSlotCount>2</leftHandSlotCount>      <!-- 左手槽位数 -->
<rightHandSlotCount>2</rightHandSlotCount>    <!-- 右手槽位数 -->
<specialSlotCount>1</specialSlotCount>        <!-- 特殊槽位数 -->
<hasRightHand>true</hasRightHand>             <!-- 是否有右手 -->
<switchCooldownTicks>30</switchCooldownTicks>  <!-- 切换冷却(tick) -->
<allowChipManagement>true</allowChipManagement> <!-- 允许玩家管理芯片 -->
<autoActivateOnEquip>false</autoActivateOnEquip> <!-- 装备时自动激活已装载芯片 -->
<preloadedChips>                               <!-- 出厂预装芯片 -->
  <li>                                         <!-- （近界/黑触发器用） -->
    <side>Special</side>
    <slotIndex>0</slotIndex>
    <chipDef>BDP_ChipShield</chipDef>
  </li>
</preloadedChips>
```

### CompProperties_TriggerChip（芯片配置）

```xml
<compClass>BDP.Trigger.TriggerChipComp</compClass>
<chipEffectClass>BDP.Trigger.WeaponChipEffect</chipEffectClass>
<!-- Trion消耗三维度 -->
<activationCost>20</activationCost>     <!-- 激活时一次性消耗 -->
<allocationCost>5</allocationCost>      <!-- 激活期间占用（降低可用上限） -->
<drainPerDay>10</drainPerDay>           <!-- 激活期间持续消耗/天 -->
<!-- 门槛与约束 -->
<minOutputPower>0.3</minOutputPower>    <!-- 最低输出功率门槛 -->
<exclusionTags>                         <!-- 互斥标签（同标签不可同时激活） -->
  <li>MeleeWeapon</li>
</exclusionTags>
<isDualHand>false</isDualHand>          <!-- 是否双手武器（锁定另一侧） -->
<!-- 时序控制 -->
<activationWarmup>0</activationWarmup>  <!-- 激活预热时间(tick)，特殊槽无效 -->
<deactivationDelay>0</deactivationDelay> <!-- 关闭后摇(tick)，切换时旧芯片保持激活的时长 -->
```

### 状态机

每侧（左/右手）各持有独立的SwitchContext（null=Idle），三态切换，通过懒求值驱动转换（原因见§7决策D2）：

```
  每侧独立状态机（leftSwitchCtx / rightSwitchCtx）：

  ┌───────┐  ActivateChip(有旧芯片,deactivationDelay>0)  ┌──────────────┐
  │ Idle  │ ─────────────────────────────────────────────→│ WindingDown  │
  │(null) │                                                │ (旧芯片后摇) │
  └───────┘                                                └──────┬───────┘
      ↑                                                           │ 后摇到期
      │                                                           ▼
      │  前摇到期                                          ┌──────────────┐
      │  TryResolveSideSwitch()                            │  WarmingUp   │
      └────────────────────────────────────────────────────│ (新芯片前摇) │
                                                           └──────────────┘

  ActivateChip(无旧芯片 或 deactivationDelay=0) → 直接进入 WarmingUp

  WindingDown阶段：旧芯片仍isActive=true（不变量⑭），后摇到期后DeactivateSlot
  WarmingUp阶段：cooldown = max(switchCooldownTicks, 新芯片activationWarmup)
  两侧互不阻塞：左手切换时右手可同时操作
```

### 激活流程

```
ActivateChip(side, index)
    │
    ▼
CanActivateChip() ── 前置检查 ──┐
    │                            │ 失败 → 返回false
    │  ✓ IsCombatBodyActive     │
    │  ✓ Trion ≥ activationCost │
    │  ✓ outputPower ≥ minOutputPower
    │  ✓ 无互斥标签冲突          │
    │  ✓ 无双手锁定冲突          │
    │                            │
    ▼                            │
本侧正在切换中？ ── 是 → 拒绝 ──┘
    │ 否
    ▼
该侧已有激活芯片？
    │
    ├── 否 → 进入WarmingUp（cooldown = max(switchCooldown, warmup)）
    │         cooldown=0时立即DoActivate
    │
    └── 是 → deactivationDelay > 0？
              │
              ├── 是 → 进入WindingDown（旧芯片仍isActive=true）
              │         后摇到期 → DeactivateSlot(旧) → 进入WarmingUp
              │
              └── 否 → DeactivateSlot(旧) → 进入WarmingUp

TryResolveSideSwitch()（懒求值，UI每帧触发）：
  WindingDown到期 → DeactivateSlot(旧) → WarmingUp
  WarmingUp到期 → DoActivate(新) → ctx=null（Idle）

DoActivate(slot):
    1. CompTrion.Consume(activationCost)     // 一次性消耗
    2. CompTrion.RegisterDrain(drainPerDay)   // 注册持续消耗
    3. 设置 ActivatingSide / ActivatingSlot 上下文
    4. effect.Activate(pawn, triggerBody)      // 委托给策略
    5. slot.isActive = true
    6. 清除上下文
```

### 特殊槽

特殊槽（SlotSide.Special）与左右手槽的行为差异：

- 全量操作：ActivateAllSpecial() / DeactivateAllSpecial() 同时激活/关闭所有特殊槽
- 不参与切换状态机（无Switching状态）
- activationWarmup对特殊槽无效
- 不受switchCooldownTicks影响
- 典型芯片类型：待设计

### 核心不变量

1. 每侧（左/右手）同时激活的芯片数 ≤ 1
2. isActive == true 的槽位必有 loadedChip != null
3. leftSwitchCtx/rightSwitchCtx 非null时 phase 为 WindingDown 或 WarmingUp；null 时该侧 Idle
4. dualHandLockSlot != null 时，另一侧不可激活新芯片
5. hasRightHand == false 时 rightHandSlots 为空数组
6. specialSlotCount == 0 时 specialSlots 为 null
7. allowChipManagement == false 时，玩家不可装载/卸载芯片
8. IsCombatBodyActive == false 时不可激活任何芯片
9. WindingDown 阶段旧芯片仍 isActive == true，后摇到期才 Deactivate

## 3. 芯片效果系统

### IChipEffect 接口

```csharp
public interface IChipEffect
{
    // 激活芯片效果。pawn=持有者，triggerBody=触发体Thing
    void Activate(Pawn pawn, Thing triggerBody);

    // 关闭芯片效果
    void Deactivate(Pawn pawn, Thing triggerBody);

    // 每tick回调（可选，大多数效果不需要）
    void Tick(Pawn pawn, Thing triggerBody);

    // 前置条件检查（可选，默认返回true）
    bool CanActivate(Pawn pawn, Thing triggerBody);
}
```

IChipEffect实现类要求：
- **无参构造函数**（通过`Activator.CreateInstance(chipEffectClass)`实例化）
- **无状态**（不持有实例字段，所有上下文通过参数传入）
- 需要知道当前操作的槽位时，从triggerBody获取CompTriggerBody，读取`ActivatingSide`和`ActivatingSlot`

### 三种策略与RimWorld机制映射

| 策略类 | RimWorld机制 | Activate做什么 | Deactivate做什么 |
|--------|-------------|---------------|-----------------|
| WeaponChipEffect | IVerbOwner + VerbTracker | 写入侧别Verb/Tool配置 → 触发RebuildVerbs | 清除侧别配置 → 触发RebuildVerbs |
| ShieldChipEffect | HediffDef | pawn.health.AddHediff(shieldHediffDef) | pawn.health.RemoveHediff |
| UtilityChipEffect | AbilityDef | pawn.abilities.GainAbility(abilityDef) | pawn.abilities.RemoveAbility |

> 注：同一策略分类下可有多种具体实现机制。例如辅助类芯片的Ability可间接添加Hediff（如隐身），不局限于单一机制。当前每类仅有一种实现，但这不是架构约束。

> 注：IChipEffect接口的第二个参数是`Thing triggerBody`而非`CompTriggerBody`。效果类需要访问CompTriggerBody时，通过`triggerBody.TryGetComp<CompTriggerBody>()`获取。

### Trion消耗模型

消耗分为两层，各司其职：

```
┌─ 共有维度（CompProperties_TriggerChip，CompTriggerBody统一处理）──┐
│  activationCost  — 激活时一次性消耗                               │
│  allocationCost  — 激活期间占用量（降低Trion可用上限）             │
│  drainPerDay     — 激活期间持续消耗/天（通过RegisterDrain注册）    │
└──────────────────────────────────────────────────────────────────┘

┌─ 效果专属维度（DefModExtension，各效果类自行处理）───────────────┐
│  trionCostPerShot        — WeaponChipConfig：每次射击消耗        │
│  trionCostPerDamageFactor — ShieldChipConfig：受击Trion消耗倍率  │
│  trionCostPerUse         — CompProperties_AbilityTrionCost：每次使用 │
└──────────────────────────────────────────────────────────────────┘
```

原因：共有维度是所有芯片都有的，提升到统一Props由编排器处理，避免每个效果类重复实现。效果专属维度只有特定类型才有，留在DefModExtension中由效果类自行处理。

### 组合能力系统

当左右手同时激活特定芯片组合时，CompTriggerBody自动授予额外的组合Ability。

`ComboAbilityDef`（继承Def，由DefDatabase管理）定义组合规则：

```csharp
class ComboAbilityDef : Def
{
    public ThingDef chipA;        // 组合所需芯片A
    public ThingDef chipB;        // 组合所需芯片B
    public AbilityDef abilityDef; // 组合成功后授予的Ability
    public bool Matches(ThingDef a, ThingDef b); // 对称匹配（AB=BA）
}
```

流程：芯片激活后，CompTriggerBody遍历`DefDatabase<ComboAbilityDef>`匹配当前左右手芯片组合，满足则`pawn.abilities.GainAbility()`；芯片关闭后检查并移除不再满足条件的组合能力。已授予的组合记录在`grantedCombos`列表中。

### 扩展：新增芯片类型

1. 实现`IChipEffect`接口（无参构造、无状态）
2. 如有效果专属消耗，创建对应的`DefModExtension`配置类
3. 编写XML Def：ThingDef（芯片物品）+ CompProperties_TriggerChip（指定chipEffectClass）
4. 无需修改CompTriggerBody或其他现有代码

## 4. 武器与攻击系统

这是触发器模块中最复杂的子系统。武器类芯片需要在RimWorld的Verb/Tool体系中注入自定义攻击行为，同时避免被引擎的多条自动路径错误拾取。

### 4.1 芯片Verb隔离方案

芯片Verb**不在VerbTracker.AllVerbs中**，而是由CompTriggerBody手动创建和缓存。

原因链（详见§7决策D1）：
1. VerbTracker.AllVerbs中的近战Verb会被`Pawn_MeleeVerbs.GetUpdatedAvailableVerbsList()`自动拾取进近战选择池
2. VerbTracker.AllVerbs中的Verb会被`VerbTracker.GetVerbsCommands()`的Path B绑定到Y按钮
3. `hasStandardCommand=false`只控制Path A（标准Gizmo生成），对上述两条路径无效
4. 因此芯片Verb必须彻底脱离VerbTracker

实现方式：
```csharp
// RebuildVerbs() 中手动创建芯片Verb
var verb = (Verb)Activator.CreateInstance(verbClass);
// 反射设置verb.verbTracker（Verb内部需要此引用）
typeof(Verb).GetField("verbTracker", ...).SetValue(verb, parent.VerbTracker);
// verb.tool 和 verb.maneuver 不在此处设置（不经过VerbTracker.InitVerb）
// 近战Verb由Verb_BDPMelee.EnsureToolAndManeuver()在每击前从芯片配置动态设置
// 缓存到CompTriggerBody字段，不进入AllVerbs
leftHandAttackVerb = verb;
```

### 4.2 武器数据存储

武器类芯片的Verb/Tool配置存放在**DefModExtension**（WeaponChipConfig）中，而非ThingDef.Verbs/tools。

原因：`ThingDef.IsWeapon`检查`category==Item && (verbs非空 || tools非空)`，芯片物品不应被识别为武器。

```csharp
public class WeaponChipConfig : DefModExtension
{
    public List<VerbProperties> verbProperties;  // 远程Verb配置
    public List<Tool> tools;                      // 近战Tool配置
    public int meleeBurstCount = 1;               // 近战连击次数
    public int meleeBurstInterval = 12;           // 近战连击间隔（ticks，默认12≈0.2秒）
    public float trionCostPerShot = 0f;           // 每次射击Trion消耗
}
```

WeaponChipEffect在Activate时，若芯片只有tools无verbProperties，会调用`SynthesizeMeleeVerbProps()`自动合成一个`verbClass=Verb_BDPMelee`的VerbProperties。原因：标准`Verb_MeleeAttackDamage`用触发体作为weapon来源，而`Verb_BDPMelee`用芯片def，需要专用Verb类。

### 4.3 DualVerbCompositor 组合规则

纯静态工具类，根据两侧的Verb数据合成最终结果：

```
输入：leftVerbs + rightVerbs（由WeaponChipEffect写入CompTriggerBody）

                    ┌─ 仅一侧有Verb ──→ TagSideAndEnsurePrimary(侧别label)
                    │                    该侧Verb (isPrimary=true)
                    │
ComposeVerbs() ─────┼─ 近战 + 近战 ───→ [左Verb, 右Verb, DualMeleeVerb*]
                    │                    相同芯片时不重复添加右侧独立Verb
                    │
                    ├─ 远程 + 远程 ───→ [左Verb, 右Verb, DualRangedVerb*]
                    │                    相同芯片时不重复添加右侧独立Verb
                    │
                    └─ 近战 + 远程 ───→ [近战Verb, 远程Verb*]

                    * = isPrimary=true（默认攻击）

所有返回的VerbProperties统一设置 hasStandardCommand = false
（XML源数据中可为true，组合器运行时覆盖为false）
（脱离标准Gizmo生成路径，由CompGetEquippedGizmosExtra自定义生成）
```

### 4.4 自定义Verb继承体系

```
Verb (RimWorld)
├── Verb_LaunchProjectile
│   └── Verb_Shoot
│       └── Verb_BDPRangedBase          // BDP远程基类（abstract）
│           │  提供TryCastShotCore(chipEquipment)辅助方法：
│           │  复制原版射击逻辑但替换equipmentSource为芯片Thing
│           │  → 战斗日志显示芯片名而非触发体名
│           │  重写OrderForceTarget：使用BDP_ChipRangedAttack替代AttackStatic
│           │
│           ├── Verb_BDPShoot            // 单发远程
│           │   射击前检查并消耗Trion (trionCostPerShot)
│           │
│           └── Verb_BDPDualRanged       // 交替连射
│               维护leftProjectile/rightProjectile
│               每发交替切换子弹类型
│
└── Verb_MeleeAttackDamage
    └── Verb_BDPMelee                   // 近战（单侧+双侧共用）
        覆盖ShotsPerBurst => verbProps.burstShotCount（基类硬编码1）

```

### 4.5 Gizmo生成：Command_BDPChipAttack

RimWorld原版的`Command_VerbTarget.GroupsWith()`按`ownerThing.def`合并Gizmo，导致同一触发体的所有芯片Verb只显示1个按钮。

解决方案：自定义`Command_BDPChipAttack`继承`Command_VerbTarget`，重写`GroupsWith()`基于`attackId`控制合并。

```
attackId 分类规则：
- 独立芯片攻击：attackId = chipDef.defName
  例："BDP_ChipArcMoon"
- 双手触发攻击：attackId = "dual:" + Sort(左chipDef, 右chipDef).Join("+")
  例："dual:BDP_ChipArcMoon+BDP_ChipScorpion"
- 组合技能攻击：attackId = "combo:" + comboAbilityDef.defName
  例："combo:BDP_ComboArcScorpion"

CompGetEquippedGizmosExtra() 生成流程：
1. 读取 leftHandAttackVerb → 生成 Command_BDPChipAttack(attackId=左芯片defName)
2. 读取 rightHandAttackVerb → 生成 Command_BDPChipAttack(attackId=右芯片defName)
3. 读取 dualAttackVerb → 生成 Command_BDPChipAttack(attackId="dual:...")
4. 多选pawn时，相同attackId的Gizmo自动合并
```

### 4.6 攻击执行链：自定义JobDriver

芯片Verb不在VerbTracker中，引擎不会调用VerbTick()推进burst计时。原版JobDriver也会通过TryGetAttackVerb()重新查找verb，返回触发体"柄"而非芯片verb。因此远程和近战各需自定义JobDriver。

**JobDriver_BDPChipRangedAttack**（远程）：

```
├── initAction: verb = job.verbToUse（直接使用，不重新查找）
├── tickAction (每tick):
│   └── verb.VerbTick()              // 手动推进burst计时
├── tickIntervalAction (间隔tick):
│   └── verb.TryStartCastOn(target)  // 尝试开始下一轮攻击
└── endCondition: 目标死亡/不可达/verb失效
```

**JobDriver_BDPChipMeleeAttack**（近战）：

```
├── tickAction (每tick):
│   ├── verb.VerbTick()                          // 手动推进burst计时
│   ├── (verb as Verb_BDPMelee).ApplyPendingInterval()  // 覆盖为芯片自定义间隔
│   └── 异常时调用SafeAbortBurst()（不用verb.Reset()，防止hitIndex残留导致burst无限重启）
├── tickIntervalAction (间隔tick):
│   ├── burst进行中 → 跳过（由VerbTick驱动后续打击）
│   ├── 跟随目标（||条件，匹配原版FollowAndMeleeAttack逻辑）
│   └── 到达近战范围 → pawn.meleeVerbs.TryMeleeAttack(target, job.verbToUse)
└── endCondition: 目标死亡/不可达/异常时EndJobWith(Errored)
```

**OrderForceTarget重写**：

```
Verb_BDPRangedBase.OrderForceTarget():
- 创建 Job(BDP_ChipRangedAttack, target, verbToUse=this)

Verb_BDPMelee.OrderForceTarget():
- 创建 Job(BDP_ChipMeleeAttack, target, verbToUse=this)
- 原因：原版Verb.OrderForceTarget对近战verb创建AttackMelee时不设verbToUse，
  导致走近战选择池选到拳头/柄而非芯片Verb
```

### 序列化策略

CompTriggerBody通过`PostExposeData()`序列化以下字段：

| 字段 | 方式 | 说明 |
|------|------|------|
| leftHandSlots / rightHandSlots / specialSlots | Scribe_Collections (LookMode.Deep) | ChipSlot实现IExposable |
| leftSwitchCtx / rightSwitchCtx | Scribe_Deep | SwitchContext (IExposable)，null=Idle |

| dualHandLockSlot | 不序列化 | 读档后从激活状态重建 |
| grantedCombos | 不序列化 | 读档后从激活状态重建 |

| *AttackVerb | 不序列化 | 读档后从激活状态重建 |

ChipSlot序列化：`loadedChip`用`Scribe_References`（Thing引用），`isActive`用`Scribe_Values`。

### 读档恢复

IChipEffect是无状态的，因此读档恢复策略是**重新调用Activate()**幂等重建：

```
PostExposeData() 读档后：
1. 遍历所有 isActive == true 的槽位
2. 对每个槽位调用 effect.Activate(pawn, this)
3. 武器类效果会重新写入Verb配置 → 触发RebuildVerbs
4. 护盾类效果会重新添加Hediff
5. 辅助类效果会重新授予Ability
6. dualHandLockSlot 从激活状态中重建
7. grantedCombos 从左右手芯片组合中重建
```

### 生命周期事件

| 事件 | 触发时机 | 行为 |
|------|---------|------|
| PostSpawnSetup | 触发体Thing生成时 | 初始化槽位数组；处理preloadedChips预装 |
| EnsureSlotsInitialized | 任何公开API调用前 | 懒初始化守卫，兼容外部工具直接操作 |
| Notify_Equipped | Pawn装备触发体时 | 恢复已激活芯片的效果；注册CompTrion.OnAvailableDepleted事件 |
| Notify_Unequipped | Pawn卸下触发体时 | 注销OnAvailableDepleted事件；关闭所有激活芯片的效果 |
| PostDestroy | 触发体Thing销毁时 | 清理所有激活效果 |

### 战斗体管理API

| 方法 | 说明 |
|------|------|
| DismissCombatBody() | 解除战斗体：DeactivateAll → Release(trion.Allocated) → IsCombatBodyActive=false |
| CanGenerateCombatBody() | 检查Trion是否足够生成战斗体（所有芯片allocationCost总和） |

### Trion耗尽自动解除

CompTrion新增`Action OnAvailableDepleted`事件，在Available从>0降至≤0时触发。触发点：`Consume()`、CompTick聚合消耗结算、`RefreshMax()`。

CompTriggerBody在Notify_Equipped时注册`OnTrionDepleted`回调，Notify_Unequipped时注销。回调逻辑：若IsCombatBodyActive则调用DismissCombatBody()。

原因：装备的CompTick()不被引擎调用，无法主动轮询。事件机制只在Available实际变化时触发，比每tick回调高效。事件字段不需要序列化（Notify_Equipped/PostLoadInit时重新注册）。

## 7. 关键设计决策

以下是当前设计中非显而易见的决策，每条解释"为什么是这样"。

### D1. 芯片Verb脱离VerbTracker

**决策**：芯片Verb通过`Activator.CreateInstance`手动创建，不进入`VerbTracker.AllVerbs`。

**原因**：VerbTracker.AllVerbs中的Verb会被引擎两条路径自动拾取——`Pawn_MeleeVerbs.GetUpdatedAvailableVerbsList()`（近战选择池，IsMeleeAttack即入池）和`VerbTracker.GetVerbsCommands()`的Path B（`FirstOrDefault(IsMeleeAttack)`绑定Y按钮）。`hasStandardCommand=false`只控制Path A（标准Gizmo生成），对这两条路径无效。唯一彻底的解决方案是让芯片Verb不存在于AllVerbs中。

### D2. 懒求值替代CompTick

**决策**：切换冷却结算采用懒求值（在IsSwitching等属性访问时检查并结算），而非CompTick主动轮询。

**原因**：RimWorld引擎中，装备后的武器`CompTick()`不会被调用。`Pawn_EquipmentTracker.EquipmentTrackerTick()`只调用`VerbsTick()`，不调用各Comp的`CompTick()`。UI每帧重绘时自然访问IsSwitching属性，触发懒求值结算。

### D3. 武器数据存DefModExtension

**决策**：武器类芯片的VerbProperties和Tools存放在`WeaponChipConfig`（DefModExtension）中，而非`ThingDef.Verbs`/`ThingDef.tools`。

**原因**：`ThingDef.IsWeapon`检查`category==Item && (verbs非空 || tools非空)`。如果芯片ThingDef直接配置verbs/tools，会被引擎识别为武器，产生一系列副作用（如出现在武器列表中、被Pawn自动拾取等）。

### D4. 自定义JobDriver

**决策**：远程芯片攻击使用`JobDriver_BDPChipRangedAttack`，近战芯片攻击使用`JobDriver_BDPChipMeleeAttack`，均为自定义JobDriver。

**原因**：原版AttackStatic的tickIntervalAction调用`pawn.TryStartAttack()`→`TryGetAttackVerb()`重新查找verb，返回触发体"柄"的近战verb而非芯片verb（因为芯片verb不在VerbTracker中）。原版AttackMelee的JobDriver也不调用VerbTick()，导致burst计时器不推进。自定义JobDriver直接使用`job.verbToUse`绕过重新查找，并在tickAction中手动调用VerbTick()推进burst。近战JobDriver额外提供：burst期间跳过tickIntervalAction的攻击尝试（由VerbTick驱动后续打击）、ApplyPendingInterval覆盖芯片自定义间隔、SafeAbortBurst异常保护。

### D5. 手动VerbTick推进burst

**决策**：JobDriver的tickAction中手动调用`verb.VerbTick()`。

**原因**：芯片verb不在VerbTracker.AllVerbs中，`VerbTracker.VerbsTick()`不会遍历到它们，导致`BurstingTick()`永远不被调用。burstShotCount>1的连射verb在第1发后state卡在Bursting，无法继续。手动调用VerbTick()推进burst计时器。

### D6. Command_BDPChipAttack重写GroupsWith

**决策**：自定义Command类基于`attackId`字符串控制Gizmo合并，而非原版的`ownerThing.def`。

**原因**：原版`Command_VerbTarget.GroupsWith()`按`ownerThing.def`合并，同一触发体的所有芯片Verb会被合并为1个Gizmo。需要每个芯片攻击模式（左手/右手/双手触发）各有独立Gizmo。

### D7. 零Harmony Patch

**决策**：整个触发器模块不使用任何Harmony Patch。

**原因**：Harmony Patch（尤其是Transpiler）是模组冲突的主要来源。通过继承CompEquippable + 显式实现IVerbOwner + 自定义Verb/JobDriver，完全在RimWorld框架内实现功能，最大化与其他模组的兼容性。

### D8. Props驱动三种触发器统一

**决策**：BORDER、近界、黑触发器共享同一个CompTriggerBody类，仅通过CompProperties_TriggerBody的配置差异化。

**原因**：三种触发器的核心逻辑（槽位管理、激活/关闭、Trion消耗）完全相同，差异仅在槽位数量和权限。用Props配置差异化避免了代码重复，也使得未来新增触发器类型只需写XML。

### D9. ActivatingSide/ActivatingSlot上下文

**决策**：CompTriggerBody在DoActivate/DeactivateSlot中设置临时上下文字段，供IChipEffect实现类读取。

**原因**：IChipEffect是无状态的（无参构造、不持有实例字段），但Activate()需要知道当前操作的是哪个槽位（侧别、序号）。通过编排器设置临时上下文，效果类通过`body.ActivatingSide`/`body.ActivatingSlot`读取，调用结束后清除。

### D10. 特殊槽全量激活/关闭

**决策**：特殊槽只支持ActivateAllSpecial/DeactivateAllSpecial批量操作，不支持单个激活/关闭。

**原因**：特殊槽的设计意图是提供一组被动能力，玩家不需要精细控制单个特殊槽。简化为全量操作降低了UI复杂度和状态管理复杂度。典型芯片类型待设计。

### D11. Inherit="False"防止Comp重复

**决策**：触发体ThingDef的comps列表使用`Inherit="False"`。

**原因**：CompTriggerBody继承自CompEquippable。如果不设Inherit="False"，XML继承链会导致CompEquippable出现两次（一次来自父ThingDef，一次来自CompTriggerBody自身），引发运行时错误。

### D12. IVerbOwner始终返回占位Verb + 芯片Verb

**决策**：CompTriggerBody的IVerbOwner.VerbProperties始终返回`parent.def.Verbs`（占位"柄"近战Verb），芯片Verb通过手动创建独立存在。

**原因**：占位Verb确保触发体在无芯片激活时仍可作为近战武器使用（用"柄"打人）。芯片Verb设`hasStandardCommand=false`脱离标准Gizmo路径，由CompGetEquippedGizmosExtra自定义生成独立Gizmo。

### D13. 近战Verb使用自定义JobDriver

**决策**：`Verb_BDPMelee.OrderForceTarget()`创建`BDP_ChipMeleeAttack` job并设置`job.verbToUse = this`。

**原因**：原版`Verb.OrderForceTarget()`对近战verb创建AttackMelee时不设verbToUse字段，导致后续走近战选择池（`Pawn_MeleeVerbs`）选到拳头或触发体"柄"而非芯片Verb。此外，原版AttackMelee的JobDriver不调用VerbTick()，BDP Verb的burst计时器永远不推进。使用自定义`BDP_ChipMeleeAttack` JobDef + `JobDriver_BDPChipMeleeAttack`同时解决这两个问题。

### D14. 组合能力系统

**决策**：左右手特定芯片组合自动授予额外Ability，通过`ComboAbilityDef`（继承Def）在XML中配置组合规则。

**原因**：组合能力是数据驱动的——哪两个芯片组合产生什么Ability完全由XML定义，无需修改C#代码。`ComboAbilityDef`继承Def使其自动被`DefDatabase`管理，匹配规则对称（AB=BA）。

### D15. 覆盖ShotsPerBurst

**决策**：`Verb_BDPMelee`覆盖`ShotsPerBurst => verbProps.burstShotCount`。

**原因**：`Verb`基类硬编码`ShotsPerBurst => 1`，只有`Verb_Shoot`覆盖为`BurstShotCount`（读取verbProps）。近战Verb继承链（`Verb_MeleeAttack` → `Verb_MeleeAttackDamage`）从未覆盖此属性，导致`WarmupComplete()`中`burstShotsLeft = ShotsPerBurst`永远=1，无论verbProps.burstShotCount设多少。

### D16. EnsureToolAndManeuver

**决策**：`Verb_BDPMelee`在每次`base.TryCastShot()`调用前，从当前芯片配置设置`tool`和`maneuver`字段。

**原因**：BDP的Verb通过`Activator.CreateInstance`手动创建，不经过`VerbTracker.InitVerb()`（那里会设置`verb.tool`和`verb.maneuver`）。`Verb_MeleeAttack.TryCastShot()`中`CreateCombatLog`访问`maneuver`字段生成战斗日志，`maneuver`为null时抛NullReferenceException。每击调用（非仅首击），因为左右芯片可能有不同的tool/maneuver。

### D17. Burst期间TryCastShot始终返回true

**决策**：`Verb_BDPMelee.TryCastShot()`在burst期间始终返回true，忽略`base.TryCastShot()`的返回值。

**原因**：`Verb_MeleeAttack.TryCastShot()`在miss和dodge时返回false。`TryCastNextBurstShot()`收到false后设`burstShotsLeft=0`取消整个burst。远程burst不受此影响（弹丸已发射，返回true），但近战burst中任何一击miss/dodge都会中断整个连击。miss/dodge的伤害和日志已由base处理完毕，返回true只是告诉burst系统"继续下一击"。
