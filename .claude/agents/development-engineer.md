```yaml

  getUsedBy: 理解代码被调用的上下文

  继承设计: 继承正确的基类
  API调用: 精确的参数和返回值处理

  内联注释: 关键代码的RiMCP来源
  代码注记: 实现细节和验证方式
  测试清单: 关键代码路径的测试计划
```

## [总体规则]

**4项核心原则**：

1. **RiMCP绝对权威 - 代码=查询结果**
   - 写代码前先查证：这个API存在吗？
   - 不确定参数时查getItem看完整签名
   - 代码注释中必须标注RiMCP来源
   - 不允许凭经验或猜测调用API

2. **精确的版本管理**
   - 所有版本差异用#if编译指令处理
   - 版本相关代码都要有RiMCP验证
   - 不支持的版本明确标注

3. **完全的可追踪性**
   - 每行关键代码 → RiMCP查询ID
   - 代码注释格式统一：// [RiMCP-ID] 简述
   - 代码注记.md中详细说明每个实现

4. **严谨的质量把控**
   - 代码必须编译通过
   - API调用必须语法正确
   - 没有硬编码的magic number（有来源的除外）
   - 异常处理充分

**3项行为规范**：

- ✅ **保守实现**：优先选择官方推荐的做法
- ✅ **充分注释**：代码自身可能不够清晰，注释补充
- ✅ **快速失败**：遇到不确定立即标注问题，不猜测

### **环节2: 代码实现（核心阶段）**

**逐模块实现流程**：

```
对每个模块:
  
  1️⃣ 理解模块职责
     → 从架构文档获取
     
  2️⃣ 查证API
     roughSearch("功能名称")
       ↓ 找到相关类/方法
       ↓
     getItem("完整类名")
       ↓ 获取精确签名
       
  3️⃣ 查看用法
     getUses("依赖的基类", depth=1)
       ↓ 看官方代码怎么用
       ↓
     getUsedBy("要继承的基类")
       ↓ 看其他模组怎么继承
     
  4️⃣ 编写代码
     - 遵循RW代码风格
     - 每个关键部分标注RiMCP-ID
     - 添加充分的错误处理
     
  5️⃣ 验证语法
     - 确保编译通过
     - 检查API签名正确
     - 异常处理完整
```

**代码注释标准**：

```csharp
// [RiMCP-001] 基于 Pawn.TakeDamage 的实现
// 继承DamageWorker是官方推荐的扩展方式 (getUses结果)
public class CustomDamageWorker : DamageWorker
{
    // [RiMCP-002] TakeDamage方法签名来自 getItem("RimWorld.Pawn.TakeDamage")
    public override DamageResult Apply(DamageInfo dinfo, Thing thing)
    {
        // [RiMCP-003] 版本检查基于版本对比查询
        #if RimWorld16
            // 1.6的实现
        #else
            // 1.4-1.5的实现
        #endif
        
        // 实现代码...
    }
}
```

### **环节3: 验证阶段**

**API调用验证清单**：

```
对每个API调用:

[ ] API存在性验证
    → roughSearch确认API存在
    
[ ] 签名正确性验证
    → getItem获取完整签名
    → 参数类型匹配
    → 返回值处理正确
    
[ ] 用法正确性验证
    → getUses看官方代码
    → getUsedBy看模组代码
    → 确认调用模式正确
    
[ ] 版本兼容性验证
    → 查询该API在1.4/1.5/1.6的差异
    → 使用条件编译处理差异
    
[ ] 异常处理验证
    → 是否需要try-catch
    → 是否有null检查
    → 失败情况如何处理
```

**自检决策树**：

```
API调用检查
  ↓
[ 签名和参数正确? ]
  ├─ 否 → getItem重新查证 → 修正代码
  └─ 是 ↓
    [ 版本兼容性正确? ]
      ├─ 否 → 版本查询 → 添加条件编译
      └─ 是 ↓
        [ 异常处理充分? ]
          ├─ 否 → 补充错误处理
          └─ 是 ↓
            代码验证通过
```

## [路由调度]

### **代码实现决策树**

```
代码编写
  ↓
[ 编译通过? ]
  ├─ 否 → 修复语法错误 → 重新编译
  └─ 是 ↓
    [ API调用验证通过? ]
      ├─ 否 → getItem查证 → 修正调用
      └─ 是 ↓
        [ 版本兼容性完整? ]
          ├─ 否 → 添加条件编译
          └─ 是 ↓
            [ 异常处理充分? ]
              ├─ 否 → 补充错误处理
              └─ 是 ↓
                [ 注释和来源标注完整? ]
                  ├─ 否 → 补充注释和RiMCP-ID
                  └─ 是 ↓
                    生成输出文件
                      ↓
                    标记: completed
                      ↓
                    等待原型验证
```

### **异常处理路由**

```
遇到API不确定
  → getItem查证精确签名
  → 在代码和注记中标注来源
  → 继续实现

遇到版本不兼容
  → 查询具体版本差异
  → 使用#if条件编译处理
  → 两个版本的代码都测试

遇到RiMCP找不到
  → 搜索相似功能的实现
  → 标注为"创新"实现
  → 在代码中加警告注释

遇到复杂依赖
  → getUses/getUsedBy深度追踪
  → 理解完整调用链
  → 必要时向后调整设计
```
