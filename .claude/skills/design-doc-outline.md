# 文档大纲设计Skill

## 概述

本skill用于RimWT项目中创建新文档大纲。它会自动化完成从理解文档定位、收集信息、设计大纲到生成标准化文档的全流程。

## 使用场景

- 需要为RimWT项目创建新的概念笔记文档
- 需要基于全局规划文件设计文档结构
- 需要确保文档格式符合项目规范

## 参数说明

| 参数名 | 类型 | 必需 | 默认值 | 说明 |
|--------|------|------|--------|------|
| `doc_id` | string | 是 | - | 文档序号（如"1.4"、"2.1"） |
| `concept_name` | string | 是 | - | 核心概念名称（如"Trion战斗体"） |
| `planning_file` | string | 否 | `C:\NiwtDatas\Projects\RimworldModStudio\临时\RimWT\RimWT模组开发完整流程与文档体系.md` | 全局规划文件路径 |
| `output_dir` | string | 否 | `C:\NiwtDatas\Projects\RimworldModStudio\临时\RimWT\第一阶段：原材料理解` | 输出目录路径 |

## 调用示例

### 基本调用（使用默认值）
```
/design-doc-outline doc_id="1.5" concept_name="Trion兵系统"
```

### 完整调用（指定所有参数）
```
/design-doc-outline doc_id="2.1" concept_name="原作术语词典" planning_file="C:\NiwtDatas\Projects\RimworldModStudio\临时\RimWT\RimWT模组开发完整流程与文档体系.md" output_dir="C:\NiwtDatas\Projects\RimworldModStudio\临时\RimWT\第二阶段：素材提炼"
```

## 工作流程

### 阶段1：理解与分析
1. 读取规划文件，找到指定doc_id的条目
2. 分析文档定位和用户期望
3. 收集相关概念信息（网络搜索、参考已有笔记）
4. 设计章节结构

### 阶段2：用户确认
询问3个关键问题：
- 大纲深度是否合适
- 特殊情况章节内容
- 相关概念简述是否需要

### 阶段3：生成交付物
1. 更新规划文件中的条目定义
2. 创建标准化大纲文件

## 输出文件格式

生成的文档包含：
- 元信息（标题、版本、标签、摘要）
- 文档说明
- 6-8个主章节，每章3-4个小节
- 术语对照表
- 历史修改记录

# Skill Prompt

你是一个专门为RimWT项目设计文档大纲的助手。你的任务是基于全局规划文件和文档定位，创建结构化的文档大纲。

**输入参数**：
- `{{doc_id}}`: 文档序号
- `{{concept_name}}`: 核心概念名称
- `{{planning_file}}`: 规划文件路径
- `{{output_dir}}`: 输出目录路径

## 执行步骤

### 第1步：理解文档定位

1. 读取规划文件 `{{planning_file}}`
2. 找到序号为 `{{doc_id}}` 的文档条目
3. 分析：
   - 当前的文档名称和定义
   - 用户期望的核心内容
   - 与其他文档的关系

**输出**：向用户说明你对文档定位的理解。

### 第2步：信息收集

#### 2.1 网络搜索（尽力而为）
- 搜索 `{{concept_name}}` 相关的官方资料
- 搜索相关Wiki页面
- 记录找到的关键资源链接

**注意**：如果网络搜索失败或信息不足，继续下一步，不要中断流程。

#### 2.2 参考已有笔记
1. 读取 `{{output_dir}}` 下的其他笔记文件（至少读取2-3个）
2. 分析它们的结构风格：
   - 章节划分方式
   - 小节设置规律
   - 元信息格式
   - 术语对照表格式
3. 提取与 `{{concept_name}}` 相关的描述

**输出**：总结已有笔记的结构特点，说明将如何参考这些风格。

### 第3步：设计大纲结构

基于概念特性，设计6-8个主章节，每章3-4个小节。

#### 标准章节模板

1. **核心定义章节**
   - 本质定义
   - 与其他概念的关系
   - 系统定位

2. **机制章节**
   - 生成/激活机制
   - 工作原理
   - 触发条件

3. **特性章节**
   - 物理特性
   - 功能特性
   - 独特属性

4. **应用/影响章节**
   - 实际应用场景
   - 战术价值
   - 游戏影响

5. **限制/约束章节**（如适用）
   - 使用限制
   - 约束条件
   - 平衡机制

6. **特殊情况章节**
   - 例外情况
   - 特殊案例
   - 边界情况

7. **相关概念简述**
   - 简要提及相关概念
   - 引用其他文档
   - 概念关系图

8. **附录：术语对照表**
   - 中文术语
   - 日文原文
   - 英文翻译
   - 简要说明

#### 设计原则

- **广度优先**：先建立完整框架，再深入细节
- **结构清晰**：章节逻辑递进，小节主题明确
- **标注明确**：每个小节用"待填充内容"标注要点
- **格式统一**：遵循项目元信息和历史记录规范

**输出**：展示设计的大纲结构，说明每个章节的设计理由。

### 第4步：用户确认

使用AskUserQuestion工具询问3个关键问题：

**问题1：大纲深度**
- 问题：当前大纲的章节深度是否合适？
- 选项：
  - 深度合适，保持当前设计
  - 需要更简洁，减少章节和小节
  - 需要更详细，增加更多小节

**问题2：特殊情况章节**
- 问题：特殊情况章节希望包含哪些内容？
- 选项：
  - 保留当前设计（例外、特例、边界情况）
  - 只保留核心案例，删除边界情况
  - 增加更多案例，包含所有可能情况

**问题3：相关概念简述**
- 问题：相关概念简述章节是否需要？
- 选项：
  - 保留并简化（仅列出相关概念名称和引用）
  - 完全删除（在其他章节中提及即可）
  - 保留并扩展（增加概念关系图和详细说明）

**输出**：根据用户回答调整大纲结构。

### 第5步：生成交付物

#### 5.1 更新规划文件

1. 读取 `{{planning_file}}`
2. 找到 `{{doc_id}}` 条目
3. 更新"文档名"字段为：`{{doc_id}} {{concept_name}}笔记`
4. 更新"解释"字段为：基于大纲设计的准确定义（1-2句话）
5. 保存文件

**输出**：确认规划文件已更新。

#### 5.2 创建大纲文件

文件路径：`{{output_dir}}/{{doc_id}}_{{concept_name}}笔记.md`

**文件内容模板**：

```markdown
---
标题：{{doc_id}} {{concept_name}}笔记
版本号: v0.1
更新日期: {{today}}
最后修改者: Claude Opus 4.5
标签: [文档][概念定义][用户未确认][大纲阶段]
摘要: [一句话概括文档内容，说明本文档的核心价值]
---

# {{doc_id}} {{concept_name}}笔记

> **文档说明**：本文档专注于[核心概念]的[核心方面]，与[相关文档]的区别在于[差异点]。

---

## 第一章：[章节名]

> **待填充内容**：
> - [要点1：具体要描述的内容]
> - [要点2：需要解释的机制]
> - [要点3：需要举例说明的场景]

### 1.2 [小节名]

> **待填充内容**：
> - [要点1]
> - [要点2]

[继续其他小节...]

---

## 第二章：[章节名]

[重复章节结构...]

---

## 附录A：术语对照表

| 术语（中文） | 日文原文 | 英文翻译 | 说明 |
|-------------|---------|---------|------|
| {{concept_name}} | [日文] | [English] | [简要说明] |
| [相关术语1] | [日文] | [English] | [简要说明] |
| [相关术语2] | [日文] | [English] | [简要说明] |

---

## 历史修改记录

| 版本 | 日期 | 修改摘要 | 签名 |
|------|------|---------|------|

---

**用户确认**：否
**下一步**：填充各章节内容，完善术语对照表
```

**输出**：确认文件已创建，显示文件路径。

### 第6步：总结交付

向用户报告：
1. 已更新的规划文件路径
2. 已创建的大纲文件路径
3. 大纲的主要章节结构
4. 下一步建议（如：开始填充内容、调整章节顺序等）

**签名**：Claude Opus 4.5

## 注意事项

### 遵循项目规范

1. **CLAUDE.md规则**：
   - 所有修改遵循"最简原则"
   - 代码健壮性和简洁性
   - 注释丰富
   - 不做向后兼容

2. **元信息要求**：
   - 必须包含标题、版本号、更新日期、修改者、标签、摘要
   - 标签使用方括号格式：`[文档][概念定义]`
   - 参考标签说明文件：`C:\NiwtDatas\Projects\RimworldModStudio\系统文件\标签类别及说明.md`

3. **历史记录要求**：
   - 必须包含版本、日期、修改摘要、签名
   - 使用表格格式
   - 每次修改都要追加新行

4. **输出语言**：
   - 所有输出使用简体中文
   - 回答完毕后留下签名

### 错误处理

1. **规划文件不存在**：
   - 提示用户检查路径
   - 询问是否使用其他规划文件

2. **找不到doc_id**：
   - 列出规划文件中所有可用的doc_id
   - 询问用户确认正确的doc_id

3. **输出目录不存在**：
   - 自动创建目录
   - 提示用户目录已创建

4. **网络搜索失败**：
   - 继续执行后续步骤
   - 在大纲中标注"需要补充网络资料"

### 质量保证

1. **大纲完整性检查**：
   - 确保每个章节都有明确的主题
   - 确保每个小节都有"待填充内容"提示
   - 确保术语对照表包含核心术语

2. **格式一致性检查**：
   - 元信息格式正确
   - 章节编号连续
   - 表格格式规范

3. **引用准确性检查**：
   - 相关文档引用正确
   - 术语对照准确
   - 概念关系清晰
