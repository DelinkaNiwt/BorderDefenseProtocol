  RimWorld 原生系统技术参考分析

  以下按"游戏中是什么" → "技术本质" → "对战斗体的参考价值"三层逐一分析。

  ---
  一、食尸鬼转化系统（Ghoul / MutantDef）

  游戏中是什么：
  Anomaly DLC 外科手术将殖民者转化为食尸鬼。可通过 Pawn_MutantTracker.Revert()
  逆向恢复。

  技术本质：
  - Turn() = 添加主Hediff + givesHediffs列表，选择性移除原有Hediff（removePerman
  entInjuries/removeChronicIllnesses等标志）
  - Revert() =
  逐个移除转化时添加的Hediff，恢复保存的originalFaction/originalIdeo
  - 关键缺陷：Ghoul转化不做Hediff快照——转化前的伤/病被直接删除而非存储，故Revert
  只能恢复派系等元数据，不能恢复原始伤情
  - 序列化：Pawn_MutantTracker.ExposeData() 通过 Scribe_* 保存运行时状态

  对战斗体的参考价值：
  NR-027（Hediff替换策略）的反面参考。Ghoul证明"移除Hediff再添加"这个操作在游戏
  框架里是可行的，但它缺少快照导致不可逆。战斗体必须在 Turn 等价操作前先完整序列
  化所有Hediff，解除时再反序列化还原。这是原版未做、但框架能支撑的扩展。

  ---
  二、冷冻舱挂起系统（Cryptosleep / Suspended）

  游戏中是什么：
  殖民者进入冷冻舱后所有需求停止消耗，出来后继续（不是恢复到快照值，而是冻结期间
  完全不动）。

  技术本质：
  Pawn.Suspended → 递归检查 ParentHolder 链
  → 找到 Building_CryptosleepCasket 即返回 true
  → Need.IsFrozen 检查 pawn.Suspended
  → NeedInterval() 开头 if (!IsFrozen) return → 直接跳过
  所有需求的IsFrozen都自动响应Suspended，不需要逐个禁用。

  对战斗体的参考价值：
  NR-029（禁用生理需求）有两条路径：
  路径A：仿照Cryptosleep利用Suspended机制——让战斗体Pawn进入某种"挂起"状态，所有
  需求自动冻结。但代价是心理需求（NR-030要求保持）也会被一起冻结，不符合需求。
  路径B（更适合）：直接使用 HediffStage.disablesNeeds 字段——只列出食物/休息/舒适
  度，精确禁用。代码证明这是现成的缓存字典机制，性能可接受。
  结论：战斗体应选路径B，禁用特定需求而非全体挂起。

  ---
  三、Deathrest系统（吸血鬼死亡休眠）

  游戏中是什么：
  吸血鬼进入特殊"死亡休眠"，有专属Need用于追踪恢复进度，状态完整序列化存档。

  技术本质：
  - Need_Deathrest.IsFrozen 覆写了基类的冻结判断
  - Gene_Deathrest.ExposeData() 序列化所有进度字段（ticks、容量等）
  - Hediff通过 def.DefName 引用存储，确保跨存档一致
  - lastDeathrestTick 字段用于判断当前是否处于Deathrest状态（tick比较法）

  对战斗体的参考价值：
  NR-041（影子HP序列化）的直接参考。Gene_Deathrest.ExposeData() 展示了如何用 
  Scribe_Values.Look 序列化自定义追踪数据。战斗体的每个部位影子HP字段可以完全用
  同样的模式序列化，无需复杂方案。
  另外"tick比较法"（lastDeathrestTick + 
  updateRate）是判断当前状态的轻量模式，可用于战斗体状态的持久化标记。

  ---
  四、伤害管道拦截系统（DamageWorker Pipeline）

  游戏中是什么：
  护盾装备（原版盾牌、心灵护盾）吸收伤害，不让伤害到达本体。

  技术本质（完整拦截链）：
  Thing.TakeDamage(dinfo)
    → ThingComp.PreApplyDamage(dinfo, absorbed)     ← 修改/阻止伤害
    → ThingComp.PostPreApplyDamage(dinfo, absorbed)  ← 护盾在此吸收
    → 实际伤害应用
    → CheckForDownedStatus()
    → ThingComp.PostApplyDamage(...)
    → [如死亡] PreDeath() → 返回false可拦截

  关键发现：
  - DamageInfo.SetAmount(0) 可使伤害变零，原版HP不变
  - PreDeath() 返回 false 可完全阻止死亡逻辑
  - 这些均通过 ThingComp 扩展实现，无需Harmony patch就能接入

  对战斗体的参考价值：
  NR-001（完全伤害隔离）、NR-012（死亡拦截）的核心技术基础。  
  - 伤害拦截：在 PostPreApplyDamage 中对战斗体Pawn将 
  dinfo.SetAmount(0)，然后转入自己的影子HP管道（NR-042）
  - 死亡拦截：PreDeath() 
  中判断如果Pawn在战斗体状态，拦截死亡，转为触发被动破裂（NR-008/009）
  - 重要：两个点均可通过 
  HediffComp（在战斗体Hediff上挂载）来实现，不越权到ThingComp层级

  ---
  五、倒地检测系统

  游戏中是什么：
  部位受伤累积到阈值后Pawn倒地，触发AI行为改变（被救援、被俘虏等）。

  技术本质：
  伤害应用 → CheckForDownedStatus() → 设置pawn.health.Down = true
  → 广播 OnDownedStateChanged 事件
  - Pawn.Downed 是只读属性，实际由 hediffSet 的痛苦/意识值决定
  - 倒地事件通过 HediffComp.CompPostTick() 或事件监听响应
  - HediffComp_KillOnDowned_Instant 是原版已有"倒地即触发"模式的直接实现

  对战斗体的参考价值：
  NR-036（倒地即破裂）可直接参照 HediffComp_KillOnDowned_Instant 
  模式——在战斗体Hediff的HediffComp中监听倒地状态，一旦 pawn.Downed == true 且处
  于战斗体激活状态，立即触发被动破裂流程（NR-009）。这是原版已验证可行的钩子点。

  六、传送系统（Psycast Skip / 跳跃能力）

  游戏中是什么：
  灵能跳跃（Skip）、传送门（DoorTeleporter），可跨地图传送Pawn。

  技术本质：
  pawn.teleporting = true;                          // 设置无敌保护标志
  pawn.ExitMap(allowedToJoinOrCreateCaravan: true);
  pawn.teleporting = false;
  GenSpawn.Spawn(pawn, targetCell, targetMap);       // 在目标位置生成
  pawn.Notify_Teleported();                          // 通知传送完成
  - pawn.teleporting 标志期间，离开地图不触发各种"玩家离开"副作用
  - GenSpawn.Spawn 是标准的Pawn生成API

  对战斗体的参考价值：
  NR-014/NR-015（紧急脱离传送）的直接技术路径。
  紧急脱离时：GenSpawn.Spawn(pawn, safeCell, map) 将真身生成到安全位置即可。
  轨道空投多级回退逻辑（轨道贸易代码）更接近NR-014描述的"殖民地建筑附近→随机安全
  位置"回退规则，值得进一步单独调研。

  ---
  七、衣物动态穿戴 / 物品容器系统

  游戏中是什么：
  强制给Pawn穿戴衣物（CharacterEditor等工具使用），物品在ThingOwner容器间转移。

  技术本质：
  - pawn.apparel.wornApparel（ThingOwner容器）直接 TryAdd() 即可穿戴，跳过UI
  - 物品：pawn.inventory.innerContainer.TryAdd() / Remove()
  - 穿戴前需 Apparel.DeSpawnOrDeselect() 清理地图存在状态
  - 冲突衣物需先 pawn.apparel.Remove() 移除

  对战斗体的参考价值：
  NR-028（衣物/物品转移到快照容器、激活时Spawn外观Apparel）的完整技术路径已明确
  。
  转移路径：pawn.apparel.wornApparel → 快照自定义ThingOwner → 解除时反向
  战斗体外观生成：GenSpawn.Spawn(apparelThing, ...) + 
  apparel.wornApparel.TryAdd()

  ---
  八、强制征召系统

  游戏中是什么：
  通过代码强制设置Pawn的征召状态（CompDraftable用于动物征召）。

  技术本质：
  if (pawn.drafter == null)
      pawn.drafter = new Pawn_DraftController(pawn);
  pawn.drafter.Drafted = true;

  对战斗体的参考价值：
  NR-004步骤10（激活时强制征召）、NR-039（解除保持状态）：
  激活时 pawn.drafter.Drafted = true。
  解除时不操作 Drafted 属性，直接保持当时值即可。技术上无任何障碍。

  补漏 10：精神崩溃阻止（NR-031）—— 确认原生支持

  HediffStage.blocksMentalBreaks 字段确实存在，上一轮标注"待定"是多虑了。

  // MentalStateHandler.cs
  public bool MentalBreaksBlocked()
  {
      foreach (Hediff hediff in pawn.health.hediffSet.hediffs)
          if (hediff.CurStage != null && hediff.CurStage.blocksMentalBreaks)
              return true;
      return false;
  }

  原版只有 Anomaly DLC 的 MindNumbSerum 使用了它。战斗体 Hediff 的 stage
  中加一行 <blocksMentalBreaks>true</blocksMentalBreaks> 即可，零代码。

  注意：forced = true 的 TryStartMentalState 调用可以绕过此检查。如果有 mod
  或事件强制触发精神崩溃，这个字段挡不住。但原版正常心情触发的崩溃都走
  CanHaveMentalBreak() → Blocked 检查，可以被阻止。

  补漏 11：环境免疫（NR-033）的精确实现路径

  体温免疫：原版没有布尔开关。温度伤害通过
  HediffGiver_Heat/Hypothermia.OnIntervalPassed() 施加，判断条件是
  ambientTemperature > SafeTemperatureRange().max。免疫方式是拉宽舒适区：

  <!-- 战斗体Hediff的statOffsets -->
  <statOffsets>
      <ComfyTemperatureMin>-200</ComfyTemperatureMin>
      <ComfyTemperatureMax>200</ComfyTemperatureMax>
  </statOffsets>

  参考：机械体用 -100 到 +250。设到 -200/+200 足以覆盖游戏中所有极端温度。

  中毒免疫：通过 Stat 实现，两个 stat 是乘法关系：

  num *= Mathf.Max(1f - pawn.GetStatValue(StatDefOf.ToxicResistance), 0f);
  // ToxicResistance = 1 → 乘数 = 0 → 完全免疫

  战斗体 Hediff 的 statOffsets 加 <ToxicResistance>1</ToxicResistance> 即可。

  疾病免疫：HediffStage.makeImmuneTo
  列表可以精确指定免疫哪些疾病。但这只阻止新感染，不冻结已有疾病进度。

  疾病进度冻结：原版没有外部注入的冻结机制。SeverityChangePerDay() 由疾病自身的
  HediffComp_Immunizable 驱动。要冻结需要 Harmony
  patch，或者利用快照机制——战斗体激活时已有疾病随 Hediff
  一起存入快照并移除，解除时从快照恢复，等效于"冻结"。

  补漏 12：自定义 HediffWithComps 天然屏蔽出血和自愈

  这是本轮最重要的发现，直接验证了 NR-042 的设计决策：

  原版出血和自愈系统硬编码了 is Hediff_Injury 类型检查：

  // 出血：Hediff基类
  public virtual float BleedRate => 0f;  // 只有Hediff_Injury override返回非零

  // 自愈：HediffSet
  public bool HasNaturallyHealingInjury()
  {
      for (int i = 0; i < hediffs.Count; i++)
          if (hediffs[i] is Hediff_Injury hd && hd.CanHealNaturally())  // 
  硬编码类型检查
              return true;
      return false;
  }

  Hediff_Injury），以下原版机制全部自动跳过：
