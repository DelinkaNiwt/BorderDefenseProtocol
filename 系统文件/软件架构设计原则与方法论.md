# 核心架构原则

## SOLID 原则
- **单一职责 (SRP)**：一个类只负责一件事
- **开闭原则 (OCP)**：对扩展开放，对修改封闭
- **里氏替换 (LSP)**：子类可替换父类
- **接口隔离 (ISP)**：接口小而专注
- **依赖倒置 (DIP)**：依赖抽象而非具体实现

## 分层架构
```
表现层 → 业务逻辑层 → 数据访问层 → 基础设施层
```
- 清晰层次边界，单向依赖
- 避免跨层调用、循环依赖

## 模块化原则
- **高内聚，低耦合**：相关功能聚合，模块间松散连接
- **依赖注入**：通过构造函数/属性注入依赖
- 避免在类内部 new 依赖对象

## 游戏模组特有原则

### 兼容性优先
- 使用 Harmony Postfix/Prefix 优于 Transpiler
- 检查其他模组存在，提供兼容补丁
- 使用 Def 系统而非硬编码
- 避免破坏原版保存/加载机制

### 性能敏感
- 缓存频繁访问的数据
- 使用对象池减少 GC
- Tick 方法避免重复计算
- 避免每帧遍历所有 Pawn

### 可配置性
- 通过 XML Def 配置数值
- 提供 Mod Settings
- 支持运行时调整

# 架构设计方法论

## 领域驱动设计 (DDD)
- 识别核心域（模组的核心价值）
- 建立统一语言（一致的术语）
- 限界上下文（划分子系统边界）

## 事件驱动架构
- 使用事件解耦模块
- 发布-订阅模式处理跨模块通信
- 避免事件链过长

## 数据驱动设计
- 游戏逻辑和数据分离
- 通过 Def 配置而非代码实现变体
- 使用表格/配置文件管理复杂数据

## 渐进式架构
- 从最小可行产品 (MVP) 开始
- 根据反馈迭代优化
- 避免过度设计 (YAGNI)

# 常见反模式（避免）

- **大泥球**：代码混乱，无清晰结构
- **过度工程**：为简单问题设计复杂方案
- **上帝对象**：一个类做太多事情
- **意大利面代码**：控制流混乱，难以追踪
- **循环依赖**：模块 A 依赖 B，B 又依赖 A

# 推荐设计模式

- **策略模式**：不同行为策略
- **工厂模式**：创建复杂对象
- **观察者模式**：事件通知
- **组合模式**：树形结构
- **命令模式**：封装操作

# RimWorld 模组特定建议

## 该做
- 研究原版代码，理解设计模式
- 使用 Def 系统，充分利用 XML 配置
- 遵循命名约定，与原版保持一致
- 提供调试工具（Dev Mode Gizmo）
- 编写兼容性补丁，主动适配热门模组
- 使用 Profiler 进行性能分析

## 避免
- 破坏保存兼容性
- 硬编码路径和文本
- 过度使用 Transpiler
- 忽略本地化

# 架构评估标准

好的架构应该：
- ✅ 可理解：新人能快速上手
- ✅ 可测试：容易编写单元测试
- ✅ 可维护：修改不会引发连锁反应
- ✅ 可扩展：添加新功能不需要大改
- ✅ 高性能：满足游戏性能要求
- ✅ 兼容性好：与其他模组和平共处

# 核心指导思想

**在简单性和灵活性之间找平衡**
- 从简单开始，根据实际需求逐步演进
- 避免过度设计，遵循 KISS 原则
- 对于游戏模组，额外考虑性能和兼容性
