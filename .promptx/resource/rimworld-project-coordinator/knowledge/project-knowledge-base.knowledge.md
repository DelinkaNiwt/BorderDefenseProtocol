## 第二部分：RiMCP工具完整使用指南

### 工具1：roughSearch（粗糙搜索）

```yaml
用途: 用自然语言快速找到相关符号
特点: 宽泛搜索，快速入门

参数：
  query (必填):
    类型: 字符串
    含义: 自然语言描述
    例子: "怎样计算伤害值"
  
  kind (可选):
    默认: 省略（返回全部）
    值: csharp|cs (仅C#) 或 def|xml (仅XML)
    用途: 过滤返回类型
  
  max_results (可选):
    默认: 20
    推荐: 10-30
    含义: 返回结果数量

  module (可选):
    值: "Verse", "RimWorld", 或社区模组名
    用途: 限定搜索范围
    例子: module: "RimWorld" 仅搜索官方

常见用法：

1️⃣ 初步探索（Base库）
   ```
   roughSearch("weapon damage system", kind: "csharp", max_results: 10)
   返回: [Pawn, StatWorker_MeleeAverageDPS, CompExplosive, ...]
   ```

2️⃣ 快速查看（Mods库）
   ```
   roughSearch("武器定义", kind: "xml", module: "示例")
   返回: [各个武器的ThingDef定义]
   ```

3️⃣ 宽泛搜索（全局）
   ```
   roughSearch("植物系统")
   返回: [C#类, XML定义, 模组参考, ...]
   ```

输出示例：
  • RimWorld.Pawn.TakeDamage (C#方法)
  • xml:Steel (XML定义)
  • ProjectTrion.DamageModifier (模组类)

何时使用：
  ✅ 不知道具体的类名或文件名时
  ✅ 想快速了解某个系统有哪些相关符号时
  ✅ 探索新领域，需要理解概要时
  ✅ Agent1/2阶段最常用
```

### 工具2：getItem（精确获取）

```yaml
用途: 获取某个符号的完整代码和元数据
特点: 精确、详细、完整

参数：
  symbol (必填):
    含义: 完整符号标识
    C#格式: NameSpace.ClassName 或 NameSpace.ClassName.MethodName
    XML格式: xml:DefinitionName
    例子: RimWorld.Pawn.TakeDamage 或 xml:Steel

  max_lines (可选):
    默认: 0 (显示全部)
    用法: max_lines: 100 仅显示前100行
    场景: 代码过长时用于预览

  module (可选):
    用途: 当同名符号在多个库中存在时，指定来源
    例子: module: "RimWorld" (官方) 或 module: "ProjectTrion" (模组)

常见用法：

1️⃣ 获取完整源代码（Base库）
   ```
   getItem("RimWorld.StatWorker_MeleeAverageDPS")
   返回: [完整的class代码，包括所有方法]
   ```

2️⃣ 获取单个方法（Base库）
   ```
   getItem("RimWorld.Pawn.TakeDamage")
   返回: [该方法的完整实现代码]
   ```

3️⃣ 获取XML定义（Mods库）
   ```
   getItem("xml:QBZ71_QS", module: "MilitaryWeapons")
   返回: [该武器的完整XML定义和数值]
   ```

4️⃣ 大文件的预览
   ```
   getItem("RimWorld.Pawn", max_lines: 50)
   返回: [前50行，了解类的结构]
   然后用getItem获取完整版本
   ```

输出示例：
  [完整C#代码]
  class StatWorker_MeleeAverageDPS : StatWorker
  {
      public override float GetBaseValueFor(...)
      {
          // 完整实现
      }
  }
  
  元数据：
  • 定义位置: RimWorld/Statistics/StatWorker_MeleeAverageDPS.cs
  • 文件行号: 1-45
  • 继承自: StatWorker
  • 相关类: [列表]

何时使用：
  ✅ 需要理解具体代码实现时
  ✅ 想看某个方法的完整逻辑时
  ✅ Agent3/5阶段最常用
  ✅ 作为getUses/getUsedBy的补充
```

### 工具3：getUses（正向依赖）

```yaml
用途: 查看某个符号使用了什么（它的依赖）
特点: 追踪调用链，理解依赖关系

参数：
  symbol (必填):
    含义: 要分析的符号
    例子: RimWorld.Pawn.TakeDamage

  depth (可选):
    默认: 1 (直接依赖)
    可选: 2 (间接依赖)
    含义: 追踪深度
    使用建议:
      depth=1: 大多数场景（更清晰）
      depth=2: 想理解完整的调用树时

  kind (可选):
    默认: all (全部)
    可选: csharp (仅C#) 或 xml (仅XML)
    用途: 过滤返回类型

  max_results (可选):
    默认: 50
    推荐: 20-50

常见用法：

1️⃣ 了解某个方法使用了什么
   ```
   getUses("RimWorld.StatWorker_MeleeAverageDPS.GetBaseValueFor", depth=1)
   返回:
   • Mathf.Max()
   • Pawn.GetStatValue()
   • VerbProperties
   • Tool
   说明: 这个方法依赖了这些东西
   ```

2️⃣ 追踪两层依赖
   ```
   getUses("RimWorld.Pawn.TakeDamage", depth=2)
   返回: 
   • 直接调用: [一层]
   • 间接调用: [两层]
   说明: 完整的依赖树
   ```

3️⃣ 仅查看C#依赖
   ```
   getUses("RimWorld.Thing", kind: "csharp", depth=1)
   返回: 该类使用的所有C#库或类
   ```

输出示例：
  RimWorld.Pawn.TakeDamage() uses:
  ├─ DamageWorker (自定义伤害处理)
  │  └─ HealthUtility.AdjustSeverity()
  ├─ HediffSet (健康状态管理)
  ├─ BodyPartRecord (身体部位)
  └─ Message (日志系统)

何时使用：
  ✅ 想知道"这个方法需要什么"时
  ✅ 理解某个模块的构成时
  ✅ Agent3/4阶段用来理解技术栈
  ✅ 检查某个功能是否可行时
```

### 工具4：getUsedBy（逆向依赖）

```yaml
用途: 查看什么使用了这个符号（它的使用者）
特点: 反向追踪，理解影响范围

参数：
  symbol (必填):
    含义: 要分析的符号
    例子: RimWorld.Pawn.TakeDamage

  depth (可选):
    默认: 1 (直接引用)
    可选: 2 (间接影响)
    使用建议:
      depth=1: 直接会受影响的代码（修改时重要）
      depth=2: 整个影响链（风险评估时重要）

  kind (可选):
    默认: all
    可选: csharp 或 xml

  max_results (可选):
    默认: 50

常见用法：

1️⃣ 了解谁调用了某个方法
   ```
   getUsedBy("RimWorld.Pawn.TakeDamage", depth=1)
   返回:
   • CompExplosive.Explode()
   • Verb_Shoot.Impact()
   • ProjectileHitUtility.HitPawn()
   说明: 这些代码都调用了TakeDamage
   ```

2️⃣ 评估修改的影响范围
   ```
   getUsedBy("RimWorld.DamageWorker", depth=2)
   返回: 直接和间接使用DamageWorker的所有代码
   说明: 修改DamageWorker会影响所有这些地方
   ```

3️⃣ 查看某个Def被谁使用
   ```
   getUsedBy("xml:Steel")
   返回: 所有引用Steel的ThingDef和RecipeDef
   说明: Steel这个资源在模组中用的很广
   ```

输出示例：
  RimWorld.Pawn.TakeDamage() is used by:
  ├─ ProjectileHitUtility (子弹命中处理)
  │  └─ 会间接调用UI显示
  ├─ CompExplosive (爆炸组件)
  ├─ Verb_Shoot (射击动词)
  ├─ FallDamage (坠落伤害)
  └─ [其他伤害源]

何时使用：
  ✅ 想知道"修改这里会影响什么"时
  ✅ 调试问题，追踪问题根源时
  ✅ Agent6调试修复阶段最常用
  ✅ 评估风险和修改影响时
  ✅ Agent3做兼容性检查时
```

### RiMCP工具的协作策略

```yaml
完整分析流程：

步骤1: 粗糙搜索定位
  ├─ 工具: roughSearch
  ├─ 目的: 找到相关的入口点
  └─ 结果: 获得符号列表

步骤2: 精确获取理解
  ├─ 工具: getItem
  ├─ 目的: 看清楚代码实现
  └─ 结果: 获得完整代码

步骤3: 正向追踪依赖
  ├─ 工具: getUses
  ├─ 目的: 了解这个符号需要什么
  └─ 结果: 获得依赖链

步骤4: 逆向追踪引用
  ├─ 工具: getUsedBy
  ├─ 目的: 了解有什么依赖这个符号
  └─ 结果: 获得影响范围
```

## 第三部分：RimWorld系统参考

### RimWorld的核心系统

```yaml
系统1：需求系统 (Needs)
  概念: 每个殖民者都有多个基础需求
  关键类: 
    • Pawn.Needs (需求集合)
    • Need (基类)
    • Need_Food (饥饿需求)
  常见需求: 饥饿、睡眠、快乐、社交、美感等
  扩展方式: 自定义Need类继承Need
  注意: 版本间可能有变化，必须查询

系统2：心情系统 (Mood)
  概念: 多个因素影响心情值，低于临界值会心理崩溃
  关键类:
    • Pawn.MentalState (心理状态)
    • MentalStateDef (心理状态定义)
  影响因素: 需求、近期事件、人际关系等
  注意: 心理崩溃后果严重，架构要考虑

系统3：技能系统 (Skills)
  概念: 8大技能树，通过工作积累经验
  关键类:
    • Pawn.skills (技能集)
    • Skill (单个技能)
    • SkillRecord (技能数据)
  8大技能: 射击、肉搏、建造、种植、烹饪、医疗、社交、知识
  扩展方式: 通过WorkGiver关联新技能

系统4：工作系统 (Jobs & Work)
  概念: 工作分配，影响殖民者的行为
  关键类:
    • JobDriver (工作驱动)
    • JobDef (工作定义)
    • WorkGiver (工作分配)
  流程: WorkGiver分配 → JobDriver执行 → 完成/失败
  常见工作: 采矿、建造、种植、烹饪等

系统5：战斗系统 (Combat)
  概念: 射击、近战、掩体、子弹飞行
  关键类:
    • Verb (动词，代表一个战斗动作)
    • Pawn.TakeDamage (受伤处理)
    • DamageWorker (伤害计算)
  关键概念: 
    • Accuracy (准确度)
    • Cover (掩体)
    • DamageDef (伤害类型)
  复杂性: 高，架构时需要深入查询

系统6：医疗系统 (Medical)
  概念: 伤口、手术、器官移植、药物
  关键类:
    • Hediff (健康状况，如伤口)
    • HediffSet (某个人的所有Hediff)
    • Operation (手术定义)
  关键操作: 处理伤口、手术、药物治疗
  复杂性: 高，许多模组依赖这个系统

系统7：关系社交系统
  概念: 好感度、婚姻、派系关系
  关键类:
    • Pawn_RelationTracker (关系追踪)
    • PawnRelationDef (关系定义)
    • Opinion (好感度)
  影响: 心情、工作效率、事件触发

系统8：贸易经济系统
  概念: 商人访问、物品定价、市场波动
  关键类:
    • TradeSession (交易)
    • PriceImpactCount (价格计数)
    • Caravan (商队)
  机制: 复杂的定价、库存、波动算法
  注意: 适配社区模组时容易出现冲突
```

### 常见的模组开发陷阱

```yaml
陷阱1：版本兼容性
  问题: RW 1.4/1.5/1.6 间有API差异
  表现: 代码在某个版本工作，在另一个版本崩溃
  预防:
    • 使用RiMCP查询不同版本的API
    • 使用条件编译处理差异
    • 提前充分测试

陷阱2：组件加载顺序
  问题: Comp加载顺序不对，导致初始化失败
  预防:
    • 在Def中指定CompProperties的顺序
    • 使用RiMCP查询已有的Comp加载顺序

陷阱3：Harmony补丁冲突
  问题: 多个模组Patch同一个方法，互相干扰
  预防:
    • 使用RiMCP的getUsedBy检查是否有冲突
    • 尽量使用事件而不是Patch

陷阱4：性能问题
  问题: 频繁创建对象、GC压力、主线程阻塞
  预防:
    • 研究RW官方的优化做法（getItem查看）
    • 使用对象池而不是频繁new
    • 耗时操作异步化

陷阱5：不正确的API使用
  问题: API用法不对，导致意外的副作用
  预防:
    • 用RiMCP的getItem看源代码的完整实现
    • 查看是否有特殊的初始化或清理步骤

陷阱6：模组之间的API冲突
  问题: 某个常用API的实现被改变了
  预防:
    • 使用RiMCP的rimworld-code-rag-mods查看社区做法
    • 设计时考虑多个模组共存的情况

陷阱7：Save/Load不兼容
  问题: 保存文件格式改变，导致旧存档崩溃
  预防:
    • 设计时考虑向后兼容性
    • 使用ScribeHelper进行版本兼容的数据处理
```

### 不同版本的关键差异

```yaml
RimWorld 1.4 → 1.5
  • DamageWorker系统有性能改进
  • JobDriver的生命周期有变化
  • 某些Def类型被重新组织
  
  检查方法:
  $ getItem("RimWorld.DamageWorker", version: "1.4")
  $ getItem("RimWorld.DamageWorker", version: "1.5")
  $ 对比两个版本的差异

RimWorld 1.5 → 1.6
  • 添加了新的系统（Anomaly DLC相关）
  • 修改了平衡参数
  • 优化了某些核心系统
  
  检查方法:
  $ 类似上面的方法对比1.5和1.6

跨版本兼容性最佳实践:
  1. 确定你要支持的最低版本
  2. 对该版本和目标版本都进行RiMCP查询
  3. 识别API差异
  4. 使用条件编译或运行时检查处理差异
  5. 充分测试所有版本组合
```

## 第五部分：用户沟通知识

### 识别用户的偏好和风格

```yaml
用户偏好识别：

Q1：用户的决策风格
  • 保守型：倾向于低风险、稳妥方案
  • 创新型：倾向于尝试新想法、接受风险
  • 平衡型：在两者之间权衡
  
  做法：观察用户在遇到选择时的反应

Q2：用户的技术水平
  • 非技术：不理解代码、系统等概念
  • 入门：有基本的OOP知识
  • 中级：能理解系统设计，但不深入代码
  • 高级：能深入理解代码实现细节
  
  做法：观察用户对技术细节的反应和提问

Q3：用户关心的优先级
  • 功能优先："最重要的是能用"
  • 质量优先："宁可慢一点也要稳定"
  • 时间优先："越快越好"
  • 兼容性优先："不能和其他模组冲突"
  • 美观优先："界面和体验很重要"
  
  做法：从用户的反馈和决策中推断

Q4：用户的沟通偏好
  • 详细型：喜欢看完整的分析
  • 简洁型：只想看结论
  • 可视型：喜欢图表和示意
  • 文字型：喜欢详细的文字说明
  
  做法：观察用户对不同形式信息的反应

根据这些识别，调整你的沟通方式，会大大提高效率和满意度。
```

### 常见的沟通陷阱

```yaml
陷阱1：过度技术化
  问题：用太多技术术语，用户听不懂
  表现：用户沉默，反复追问"什么意思"
  解决：用更简洁的语言重新表述，必要时用类比
  
陷阱2：隐瞒问题
  问题：发现了问题但不敢说，希望自己能修复
  表现：项目进度延缓，最后才告诉用户
  解决：发现问题立即报告，给出3个解决方案，让用户选择
  
陷阱3：强行推销某个方案
  问题：你个人倾向于某个方案，过度宣传其优点
  表现：用户感到被胁迫，可能反感你的建议
  解决：客观列出所有方案的优缺点，让用户自主选择
  
陷阱4：承诺不可能实现的事
  问题：为了讨好用户，承诺做不到的事
  表现：最后无法交付，失去信任
  解决：现实评估，宁可承诺不足而超额交付
  
陷阱5：对用户的反馈反应迟缓
  问题：用户问了问题，长时间不回应
  表现：用户开始怀疑你是否还在工作
  解决：即使没有完整答案，也要及时反馈进度
```
