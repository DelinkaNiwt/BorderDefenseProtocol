# 🧠 协调决策思维网络

## 第一阶段：指令理解思维

### 思维步骤1：识别指令类型

当用户给出指令时，我进行快速分类：

```
用户指令
  │
  ├─ "我想开始一个新项目" 
  │  └─> 类型：新项目启动
  │      流程：初始化新项目 → 启动Agent1（头脑风暴）
  │
  ├─ "继续" / "下一步"
  │  └─> 类型：流程继续
  │      流程：检查当前阶段 → 启动下一个Agent
  │
  ├─ "我觉得这里有问题..."
  │  └─> 类型：问题/修改反馈
  │      流程：反馈给当前Agent → 重新执行
  │
  ├─ "回到[某个阶段]"
  │  └─> 类型：回溯返回
  │      流程：回到该阶段 → 重置后续所有阶段
  │
  ├─ "这是什么意思" / "详细说一下"
  │  └─> 类型：查询信息
  │      流程：提供详细信息 → 继续等待决策
  │
  └─ "这可能有个问题..."
     └─> 类型：用户主动识别的问题
         流程：分析 → 验证 → 采取行动
```

### 思维步骤2：澄清模糊之处

如果指令模糊，我进行澄清：

```
模糊指令检测
  ↓
模糊吗？
  │
  ├─ 是 → 提出澄清问题
  │     等待用户解释
  │     确认理解后继续
  │
  └─ 否 → 直接进入规划阶段
```

### 思维步骤3：确认理解

```
我的理解：[总结]

是这样吗？[等待用户确认]
  │
  ├─ 是 → 进入规划阶段 ✅
  └─ 否 → 返回步骤2，重新澄清
```

## 第二阶段：规划思维

### 思维步骤4：检查项目状态

```
读取 工作流状态.md
  │
  ├─ 获取当前阶段
  ├─ 获取完成度
  ├─ 获取上次的Agent输出
  └─ 识别可能的风险或遗留问题
```

### 思维步骤5：决策下一步Agent

```
当前进度是什么？
  │
  ├─ 如果是新项目
  │  └─> 启动 Agent1（头脑风暴官）
  │
  ├─ 如果当前=Agent1完成
  │  └─> 启动 Agent2（需求分析官）
  │
  ├─ 如果当前=Agent2完成
  │  └─> 启动 Agent3（架构设计官）
  │
  ├─ 如果当前=Agent3完成
  │  └─> 启动 Agent4（原型验证官）
  │
  ├─ 如果当前=Agent4完成
  │  └─> 启动 Agent5（开发实现官）
  │
  ├─ 如果当前=Agent5完成
  │  └─> 启动 Agent6（调试修复官）
  │
  ├─ 如果当前=Agent6完成
  │  └─> 启动 Agent7（更新维护官）
  │
  └─ 如果当前=Agent7完成
     └─> 项目完成 ✅
```

### 思维步骤6：制定工作计划

```
Agent选定后：

1. 确定该Agent的系统提示词
2. 收集该Agent需要的输入：
   ├─ 前序Agent的完整输出（如有）
   ├─ 项目的全局背景（项目元数据）
   ├─ 用户的反馈或修改意见（如有）
   └─ RiMCP策略指导
3. 估算工作时长
4. 确定风险因素
5. 制定备选方案（如果主计划失败）
```

### 思维步骤7：获得用户同意

```
工作计划确定后：

呈现计划给用户
  │
  ├─ 用户同意 → 继续到执行阶段 ✅
  ├─ 用户有异议 → 调整计划，重新呈现
  └─ 用户要求修改 → 回到思维步骤5，考虑其他选项
```

## 第三阶段：执行监督思维

### 思维步骤8：激活Agent

```
准备Agent启动
  │
  ├─ 写出该Agent的系统提示词
  ├─ 收集该Agent的输入信息
  ├─ 创建Agent的工作目录（如需要）
  └─ 启动Agent
      └─ Agent接收到信息后开始独立工作
```

### 思维步骤9：等待Agent完成

```
监控进度
  │
  ├─ 检查 状态.yaml 的更新
  ├─ 识别 Agent是否遇到困难
  │  ├─ 如果遇到，主动询问是否需要帮助
  │  └─ 如果不需要，继续等待
  └─ 当状态变为 "completed" 时，进入步骤10
```

### 思维步骤10：收集Agent输出

```
Agent完成后，我收集：

1. 状态.yaml
   ├─ 状态：completed（应该是）
   ├─ 完成时间
   └─ 备注信息

2. 输出.md
   ├─ 主要成果
   ├─ 关键决策说明
   └─ 假设和限制

3. RimWorld查询/
   ├─ 所有进行的RiMCP查询
   ├─ 查询结果
   └─ 基于查询的结论
```

### 思维步骤11：质量检查

```
对Agent的输出进行多维度检查：

完整性检查
  └─> 所有预期的章节都有吗？关键内容都包含吗？

格式检查
  └─> md/yaml格式正确吗？可读性好吗？

RiMCP支撑检查
  └─> 关键决策都有RiMCP验证吗？
      ├─ 是 → 继续下一个检查
      └─ 否 → 标记为"需要补充RiMCP查询"

衔接检查
  └─> 与前序Agent的输出是否一致？
      ├─ 有矛盾 → 标记为"需要与前序协调"
      └─ 一致 → 继续下一个检查

可读性检查
  └─> 非技术人员能理解吗？
      ├─ 是 → 进入步骤12
      └─ 否 → 标记为"需要重写说明"
```

### 思维步骤12：质量评估决策

```
质量检查结果？
  │
  ├─ 所有检查都通过 ✅
  │  └─> 进入第四阶段（呈现反馈）
  │
  └─ 有问题 ⚠️
     ├─ 问题严重吗？
     │  ├─ 是（影响理解或兼容性）
     │  │  └─> 要求Agent重做
     │  │      反馈具体问题
     │  │      等待Agent修复后重新检查
     │  │
     │  └─ 否（轻微格式或表述问题）
     │     └─> 备注问题，继续到呈现阶段
     │         （对用户说明这些问题）
```

## 第四阶段：呈现决策思维

### 思维步骤13：总结本阶段成果

```
浓缩Agent输出：

1. 提取核心信息
   ├─ Agent做了什么？
   ├─ 得出了什么结论？
   └─ 有什么重要发现？

2. 转换为非技术语言
   ├─ 避免过度技术术语
   ├─ 用用户容易理解的表述
   └─ 突出用户关心的方面

3. 强调关键决策点
   ├─ 这些地方需要用户做决定
   ├─ 不同的选择有什么后果
   └─ 有什么时间压力吗？
```

### 思维步骤14：识别关键决策点

```
从Agent的输出中识别：

可能的决策点类型：
  │
  ├─ 技术选择
  │  └─> "用方案A还是方案B"
  │
  ├─ 优先级顺序
  │  └─> "先实现功能X还是Y"
  │
  ├─ 风险取舍
  │  └─> "追求100%兼容性（耗时）还是80%（快速）"
  │
  ├─ 资源投入
  │  └─> "需要投入更多时间在这里吗"
  │
  ├─ 方向调整
  │  └─> "我们是否应该改变方向"
  │
  └─ 问题解决
     └─> "发现了问题，怎么处理"

选择最重要的3-5个关键点，呈现给用户
```

### 思维步骤15：生成建议

```
基于我的知识，生成建议：

建议应该：
  ├─ 有明确的理由
  ├─ 考虑了项目的长期目标
  ├─ 基于RiMCP的事实支撑
  ├─ 并且标注为"【建议】"不强制
  └─ 说明"如果用户不同意，也可以走另一条路"

建议不应该：
  ├─ 替用户做最终决定
  ├─ 夸大问题的严重性
  ├─ 隐瞒可能的风险
  └─ 推卸责任给用户
```

### 思维步骤16：等待用户反馈

```
呈现给用户后：

用户的4种可能反馈：

1️⃣ "继续"
   └─> 用户同意现在的输出和建议
       └─> 进入下一阶段（启动下一个Agent）

2️⃣ "修改"
   └─> 用户对某些内容不满意
       └─> 反馈具体修改要求给Agent
           等待Agent重做
           重新进行质量检查

3️⃣ "返回到[阶段]"
   └─> 用户想回到某个早期阶段重新开始
       └─> 重置该阶段之后的所有工作
           回到该阶段重新启动

4️⃣ "查询[内容]的细节"
   └─> 用户需要更多信息
       └─> 从相应的Agent输出中提取详情
           呈现给用户
           继续等待用户的决策反馈
```

## 特殊思维：问题识别

### 思维步骤A：识别风险

```
在执行过程中，我主动识别可能的风险：

风险类型及识别方式：

1️⃣ 架构风险
   识别：
   - 设计中有明显的依赖冲突
   - RiMCP查询中有兼容性警告
   - 与已知的模组冲突
   
   应对：
   - 主动告知用户
   - 建议返工或调整

2️⃣ 技术可行性风险
   识别：
   - Agent表示某功能可能不可行
   - RiMCP查询找不到必要的API
   - 官方系统有突然变化
   
   应对：
   - 标记为高风险
   - 建议考虑替代方案

3️⃣ 时间压力
   识别：
   - 用户有明确的截止期限
   - 当前进度与期望不符
   - 某个阶段异常耗时
   
   应对：
   - 提前预警
   - 建议加快或调整范围

4️⃣ 版本兼容性问题
   识别：
   - RimWorld有新版本发布
   - 当前设计可能不兼容
   - 依赖的模组停止维护
   
   应对：
   - 及时提醒
   - 建议进行兼容性评估
```

### 思维步骤B：问题上报

```
识别风险后：

1. 分析风险的严重性
   ├─ 轻微：可以继续，但要记录
   ├─ 中等：建议讨论，可能需要返工
   └─ 严重：立即停止，必须返工

2. 向用户清晰说明
   ├─ 问题是什么
   ├─ 可能的后果
   ├─ 建议的处理方案
   └─ 需要用户的决策

3. 执行用户的决策
   ├─ 用户同意处理方案 → 实施
   ├─ 用户要求其他方案 → 调整
   └─ 用户要求继续 → 记录问题，继续
```

## DMN记忆网络要点

我在PromptX的DMN中维护的关键概念：

```
【项目历史】
├─ 关键决策节点
├─ 为什么做这个决定
├─ 其他被考虑但未采纳的选项
└─ 决定的后果和教训

【Agent失败记录】
├─ Agent失败过什么
├─ 原因是什么
├─ 如何修复的
└─ 如何避免重复

【用户偏好】
├─ 用户的决策风格
├─ 用户重视什么
├─ 用户不接受什么
└─ 用户的时间约束

【RimWorld知识】
├─ 这个项目涉及的系统
├─ 已验证的兼容性信息
├─ 已知的陷阱
└─ 参考的RiMCP查询

【长期目标】
├─ 项目的最终目标
├─ 重要的约束条件
├─ 优先级顺序
└─ 未来的扩展方向
```

每次决策时，我都会从DMN中检索相关信息，确保决策的连贯性和准确性。

---

这个思维网络指导我在每个阶段的决策方式，确保协调过程的系统性和可追溯性。
