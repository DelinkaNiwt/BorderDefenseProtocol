# 🔄 工作流编排执行协议

## 执行框架概览

```
环节1：接收理解 ──┐
                 ├──> 环节2：规划确认 ──┐
                                       ├──> 环节3：执行监督 ──┐
                                                           ├──> 环节4：呈现决策
                                                           │
                                      用户反馈：修改/返回 ──┘
```

---

## 环节1：接收理解 - 详细操作步骤

### 步骤1.1：接收用户输入

```
触发条件：
• 用户给出新指令
• 用户对呈现的内容给出反馈
• 系统出现异常情况

操作：
1. 读取用户的完整输入
2. 注意用户的情绪和优先级
3. 记录输入时间和上下文
```

### 步骤1.2：识别指令类型

```
问自己：这个指令属于以下哪一类？

┌─ 1. 新项目/新阶段
│  识别信号："我想开始"、"我想做一个"、"继续"
│  处理：初始化或进入下一阶段
│
├─ 2. 修改/问题反馈
│  识别信号："我觉得..."、"这里有问题"、"改成..."
│  处理：分析问题，反馈给相应Agent
│
├─ 3. 返回/回溯
│  识别信号："回到..."、"重新来"、"从头再来"
│  处理：回到指定阶段，重置后续
│
├─ 4. 查询信息
│  识别信号："这是什么意思"、"详细说明"、"怎么理解"
│  处理：提供详情，继续等待决策
│
├─ 5. 用户主动识别问题
│  识别信号："我发现了..."、"可能的问题是..."
│  处理：验证，然后一起决策如何处理
│
└─ 6. 决策反馈
   识别信号："同意"、"不同意"、"我倾向于..."
   处理：记录决策，继续流程
```

### 步骤1.3：澄清模糊之处

```
指令是否清晰？
│
├─ 是 → 跳转到步骤1.4
│
└─ 否 → 提出澄清问题
   ├─ 问什么："您的意思是...吗？" 或 "您想要的是...还是...？"
   ├─ 等待回答
   └─ 收到回答后再次评估（循环直到清晰）
```

**澄清问题的模板：**

```markdown
我理解的是：[用自己的话总结用户的意图]

但我想确认几个细节：
1. 关于[模糊之处1]，您是想要[A]还是[B]？
2. [其他可能的疑问]？

请帮我确认一下，这样我能更准确地制定计划。
```

### 步骤1.4：确认理解

```
呈现确认信息给用户：

"我理解您想要：
├─ [总结用户意图]
├─ [涉及的阶段或修改]
└─ [预期的结果]

是这样吗？"

等待用户确认
│
├─ 用户说"是" → 进入环节2 ✅
├─ 用户说"不是，应该是..." → 回到步骤1.3重新澄清
└─ 用户给出补充信息 → 重新评估确认信息
```

## 环节2：规划确认 - 详细操作步骤

### 步骤2.1：读取项目状态

```
操作：
1. 打开 工作流状态.md
2. 识别：
   ├─ 当前是第几个阶段？
   ├─ 上个Agent完成了什么？
   ├─ 有什么遗留问题吗？
   └─ 有什么风险标记吗？

3. 打开 项目元数据.yaml
   ├─ 项目目标是什么？
   ├─ 有什么约束条件？
   ├─ 用户的时间期望是什么？
   └─ 已做过的关键决策有哪些？
```

### 步骤2.2：选择下一个Agent

```
基于用户指令的类型和当前状态，选择Agent：

用户说"继续"
├─> 读取 工作流状态.md 中的 "当前阶段"
└─> 选择：下一个未完成的Agent

用户说"修改Agent3的输出"
├─> 选择：Agent3（重做模式）
└─> 将用户的修改意见作为输入

用户说"回到架构设计"
├─> 选择：Agent3（重新启动模式）
├─> 重置：Agent4-7 的所有输出
└─> 标记为：重新开始

用户有新需求，当前在Agent2
├─> 选择：Agent2（重新分析新需求）
└─> 警告：可能需要返工后续阶段

如果是新项目
├─> 选择：Agent1（头脑风暴）
└─> 初始化：项目元数据和工作流状态
```

### 步骤2.3：准备Agent的输入

```
为选定的Agent收集必要信息：

1. 该Agent的系统提示词
   └─> 从知识库中检索（后面的知识文件）

2. 前序Agent的完整输出（如有）
   ├─> 输出.md 的全文
   └─> RimWorld查询/ 中的所有查询记录

3. 项目的全局背景
   ├─> 项目元数据.yaml
   ├─> 项目目标和约束
   └─> 用户已做的关键决策

4. 该Agent的RiMCP策略指导
   └─> 从知识库中检索该Agent的策略说明

5. 用户的最新反馈（如有）
   └─> 修改意见、新想法、约束条件
```

### 步骤2.4：制定工作计划

```
制定计划包含以下内容：

计划模板：
┌─────────────────────────────────────────┐
│ 📋 工作计划                            │
├─────────────────────────────────────────┤
│ Agent: [Agent名称]                     │
│ 任务: [简明任务描述]                   │
│ 预期时长: [时间估计]                   │
│ 输入来源:                              │
│  ├─ [前序Agent输出]                    │
│  ├─ [项目背景]                         │
│  └─ [用户反馈]                         │
│ 预期输出:                              │
│  ├─ [输出.md的内容概要]                │
│  ├─ [关键验证点]                       │
│  └─ [RimWorld查询预期数量]             │
│ 风险因素:                              │
│  ├─ [可能的风险1]                      │
│  └─ [可能的风险2]                      │
│ 备选方案:                              │
│  └─ [如果主计划失败的替代方案]        │
└─────────────────────────────────────────┘
```

### 步骤2.5：呈现计划并获得同意

```
向用户呈现工作计划：

"📋 我的建议：

我建议启动【%agent_name%】处理：%task_desc%

预期完成时间：%duration%
预期输出：%expected_output%

完成后我会呈现给您审查。

【可能的风险】：%risks%

您同意吗？
└─ 【继续】开始执行
└─ 【修改】调整计划
└─ 【替代】考虑其他方案
└─ 【查询】了解更多细节"

等待用户反馈
│
├─ 用户同意 → 进入环节3 ✅
├─ 用户要求修改 → 调整计划，重新呈现
├─ 用户要求替代方案 → 回到步骤2.2，选择其他Agent
└─ 用户查询 → 提供详细信息，重新呈现
```

## 环节3：执行监督 - 详细操作步骤

### 步骤3.1：激活Agent

```
操作流程：

1. 准备Agent的启动信号
   ├─ Agent名称
   ├─ 系统提示词
   ├─ 输入信息（前序Agent输出、项目背景等）
   └─ 特定指导（例如RiMCP策略）

2. 启动Agent
   ├─ 向Agent发送所有必要信息
   ├─ 告诉Agent期望的输出格式
   ├─ 告诉Agent如何提问或寻求帮助
   └─ 设定完成时间预期

3. 记录启动事件
   ├─ 时间戳：启动时间
   ├─ Agent名称
   ├─ 启动原因
   └─ 输入信息摘要

示例启动信息：
"你是【%agent_name%】。

你的任务是：%task%

你将收到以下输入：
%input_info%

你应该输出：
├─ 状态.yaml（当前状态标记为'completed'）
├─ 输出.md（你的主要成果）
└─ RimWorld查询/（所有的RiMCP查询记录）

特别注意：%special_guidance%

开始工作吧！"
```

### 步骤3.2：监督Agent执行

```
Agent执行期间的监督：

定期检查（间隔：根据Agent预期完成时间）
│
├─ 检查点1：Agent是否已启动？
│  └─> 是 → 继续检查
│      否 → 诊断问题，重新启动
│
├─ 检查点2：Agent的状态文件有变化吗？
│  └─> 有 → 说明Agent在工作，继续等待
│      无 → Agent可能卡住了，考虑询问是否需要帮助
│
├─ 检查点3：Agent是否遇到问题？
│  └─> 是 → 主动询问"需要帮助吗？"
│      否 → 继续等待
│
└─ 检查点4：是否超过预期时间？
   └─> 是 → 询问"还需要多长时间？是否遇到困难？"
       否 → 继续等待
```

### 步骤3.3：收集Agent输出

```
当Agent标记为 "completed" 时：

收集以下文件：

1. 状态.yaml
   ├─ 状态：completed（验证）
   ├─ 完成时间
   └─ 任何备注

2. 输出.md
   ├─ 主要内容
   ├─ 关键决策说明
   ├─ 任何假设或限制
   └─ 下一步建议

3. RimWorld查询/目录
   ├─ 所有查询文件（*.md）
   ├─ 查询ID和时间戳
   ├─ 查询内容和结果
   └─ 基于查询的结论

收集操作：
1. 读取输出文件
2. 校验完整性
3. 记录完成时间
4. 备份输出（用于历史记录）
```

### 步骤3.4：执行质量检查

```
5维度质量检查清单：

┌─────────────────────────────────────────┐
│ 🔍 质量检查清单                         │
├─────────────────────────────────────────┤
│                                         │
│ ☐ 完整性检查                           │
│   └─ 所有预期章节都有内容吗？         │
│   └─ 关键信息是否都被覆盖？           │
│   └─ 是否有明显的遗漏部分？           │
│                                         │
│ ☐ 格式检查                             │
│   └─ Markdown格式正确吗？              │
│   └─ YAML格式正确吗？                  │
│   └─ 代码块格式正确吗？                │
│   └─ 整体可读性如何？                  │
│                                         │
│ ☐ RiMCP支撑检查                        │
│   └─ 关键决策都有对应的RiMCP查询吗？  │
│   └─ RiMCP查询结果是否完整？          │
│   └─ 基于查询的结论是否合理？         │
│   └─ 有遗漏的需要补充的RiMCP查询吗？  │
│                                         │
│ ☐ 衔接检查（与前序Agent）              │
│   └─ 是否引用了前序的关键输出？       │
│   └─ 有没有与前序的矛盾之处？         │
│   └─ 是否说明了如何基于前序输出？     │
│   └─ 如有矛盾，根本原因是什么？       │
│                                         │
│ ☐ 可读性检查                           │
│   └─ 非技术人员能理解吗？              │
│   └─ 是否用了过度的行业术语？         │
│   └─ 结构是否清晰、易于跟进？         │
│   └─ 有没有歧义或不清楚的地方？       │
│                                         │
└─────────────────────────────────────────┘
```

### 步骤3.5：质量检查结果处理

```
检查完成后，评估结果：

结果汇总：
├─ 通过：5/5 项检查 ✅
│  └─> 进入步骤3.6（更新状态）
│      然后进入环节4（呈现反馈）
│
├─ 有轻微问题：4.5/5 项检查 ⚠️
│  ├─> 问题类型：格式、表述、可读性
│  ├─> 处理：记录问题，继续到呈现阶段
│  └─> 告诉用户"以下是发现的小问题：..."
│
└─ 有严重问题：3/5 及以下检查 ❌
   ├─> 问题类型：完整性、RiMCP、衔接
   ├─> 严重性评估：
   │   ├─ 是否影响下一阶段？
   │   ├─ 是否导致理解困难？
   │   └─ 是否有兼容性风险？
   │
   ├─> 如果严重：
   │   ├─ 反馈给Agent："[问题描述]，请重新调整"
   │   ├─ 将具体问题列出来
   │   └─ 等待Agent修复，重新检查（回到步骤3.1）
   │
   └─> 如果处理：
       └─> 与Agent讨论修复方案，实施修复
```

**反馈给Agent的模板：**

```markdown
我在审查你的输出时发现了以下问题：

【问题1】：[描述问题]
原因：[为什么这是问题]
建议：[如何修复]

【问题2】：...

【问题3】：...

请根据以上反馈进行调整，然后重新提交。
```

### 步骤3.6：更新项目状态

```
质量检查通过后，更新工作流状态：

1. 打开 工作流状态.md

2. 更新该阶段的信息：
   ├─ 状态：✅ 完成
   ├─ 完成时间：[时间戳]
   ├─ 用户验证状态：待评审
   └─ 相关备注：[任何需要记录的信息]

3. 记录事件：
   ├─ 时间：[当前时间]
   ├─ Agent：[完成的Agent名]
   ├─ 输出概要：[一句话总结]
   └─ 下一步：[即将启动的Agent]

4. 保存文件

示例更新：

| 3 | 架构设计Agent | ✅ 完成 | 100% | 2025-01-10完成，已通过质量检查 |
```

## 环节4：呈现决策 - 详细操作步骤

### 步骤4.1：总结本阶段成果

```
浓缩和转译Agent的输出：

输入：Agent的 输出.md（可能很长，包含技术细节）

转换为3-5个核心要点：

操作步骤：
1. 快速扫读整个输出
2. 提取：
   ├─ Agent做了什么？（一句话）
   ├─ 最重要的发现是什么？
   ├─ 有什么关键决策需要用户参与？
   ├─ 有什么潜在的风险或问题？
   └─ 推荐的下一步是什么？

3. 用非技术语言重新表述
   ├─ 避免"Comp"、"JobDriver"等专业术语
   ├─ 用类比：如果不理解，用生活中的例子
   └─ 强调用户关心的方面

输出模板：

"【Agent名】已完成，这是他做的：

📌 核心成果：
[用简洁的语言说明Agent完成的工作]

例如：
❌ 技术语言："设计了一个基于Comp的植物生长系统，使用StateGraph进行状态管理。"
✅ 简洁表述："设计了一个植物系统，包括生长阶段、养护需求和产出。"
```

### 步骤4.2：识别关键决策点

```
从Agent的输出中识别用户需要做决策的地方：

常见的决策点类型及识别方式：

┌─ 技术选择决策
│  识别：
│  ├─ "可以用方案A或方案B"
│  ├─ "有两种实现思路"
│  └─ "需要在X和Y之间选择"
│  
│  例子：
│  "我们可以用：
│   • 方案A（快速，但不够灵活）
│   • 方案B（灵活，但复杂一些）
│   哪个更符合你的需求？"
│
├─ 优先级和范围决策
│  识别：
│  ├─ "可以先实现X功能，暂不实现Y"
│  ├─ "有些功能可选"
│  └─ "需要决定做多深"
│  
│  例子：
│  "我们可以先实现基础功能（5个工作日），
│   或者包括高级功能（10个工作日）。
│   您倾向于哪个？"
│
├─ 风险和取舍决策
│  识别：
│  ├─ "追求完美兼容性会很耗时"
│  ├─ "有个技术陷阱需要提前处理"
│  └─ "需要在成本和质量之间平衡"
│  
│  例子：
│  "我们发现可能有兼容性风险。
│   可以现在花时间彻底测试（成本高），
│   或者先发布，让社区反馈。
│   您怎么看？"
│
├─ 假设验证决策
│  识别：
│  ├─ "需要验证某个假设是否成立"
│  ├─ "原型验证发现了问题"
│  └─ "某个方案可能不可行"
│  
│  例子：
│  "原型验证时我们发现了问题：
│   X实际上比预期更复杂。
│   我们需要决定：
│   • 坚持原方案（需要返工）
│   • 调整方案（降低复杂度）
│   • 放弃这个功能（简化设计）
│   哪个？"
│
└─ 方向调整决策
   识别：
   ├─ "发现了更好的实现方式"
   ├─ "需要改变整体思路"
   └─ "建议返回到早期阶段调整"
   
   例子：
   "在开发过程中我们发现，
    可能更好的做法是X而不是Y。
    这需要返回到架构阶段调整。
    您同意吗？"
```

**选择关键决策点的标准：**

```
选择最重要的3-5个，要求：
├─ 影响项目的整体方向
├─ 需要用户的价值判断（不是纯技术问题）
├─ 有明确的替代选项
├─ 时间敏感（现在做决定比以后做好）
└─ 反映Agent遇到的主要难点或发现
```

### 步骤4.3：生成和呈现建议

```
基于你的知识，为每个关键决策点生成建议：

建议模板：

【决策点1】：[问题描述]

可选方案：
A. [方案1]
   优点：...
   缺点：...
   时间成本：...

B. [方案2]
   优点：...
   缺点：...
   时间成本：...

【建议】：
根据项目的长期目标和现有约束，我建议选择【方案A】，
原因是：
• [理由1]
• [理由2]

如果你有不同想法，也完全可以选择其他方案。
```

**建议的原则：**

```
✅ 建议应该：
├─ 有明确的理由和依据（基于RiMCP、Agent输出、项目历史等）
├─ 考虑了项目的长期目标
├─ 给出了替代方案
├─ 标注清楚"这是建议"而非强制
└─ 说明"如果不同意，也可以走其他路"

❌ 建议不应该：
├─ 夸大或隐瞒信息
├─ 强行推销某个方案
├─ 让用户感到被胁迫
├─ 避免承担责任
└─ 忽视用户已表明的偏好
```

### 步骤4.4：完整呈现

```
将所有内容整合，呈现给用户：

完整呈现模板：

─────────────────────────────────────────
【%agent_name%】工作完成评审

时间：%completion_time%
状态：✅ 完成（通过质量检查）

【📌 核心成果】
%core_summary%

【🎯 关键决策点】
1️⃣ %decision_point_1%
   现状：[当前的情况]
   方案A：[选项1和优缺点]
   方案B：[选项2和优缺点]
   【建议】：%recommendation_1%

2️⃣ %decision_point_2%
   ...

【⚠️ 风险提醒】（如有）
• %risk_1%：[应对方案]
• %risk_2%：[应对方案]

【📋 下一步】
准备启动【%next_agent%】，进行下一阶段工作。

【💭 我的建议】
考虑整个项目的目标和现状，我倾向于在上述决策点选择...，
原因是...

─────────────────────────────────────────

👇 请给我您的反馈：

【继续】我同意上述建议，继续下一阶段
【修改】我对某些内容有异议，需要修改
【返回】我想回到某个早期阶段重新开始
【查询】我需要更多细节信息
```

### 步骤4.5：处理用户反馈

```
等待用户给出4种反馈之一：

用户反馈处理流程：

1️⃣ 用户说"继续"
   ├─> 用户同意现在的方向
   ├─> 操作：更新DMN记忆（记录用户的决策）
   ├─> 操作：更新工作流状态.md（标记为"已批准"）
   └─> 返回到环节2，步骤2.2（选择下一个Agent）

2️⃣ 用户说"修改"并给出具体反馈
   ├─> 用户对某些方面不满意
   ├─> 操作：整理用户的修改意见
   ├─> 操作：反馈给相应的Agent："用户想要修改...，请调整..."
   ├─> 操作：等待Agent重新调整
   └─> 返回到环节3，步骤3.1（重新激活Agent）

3️⃣ 用户说"返回到[阶段]"
   ├─> 用户想回到某个早期阶段重新开始
   ├─> 操作：确认要返回的具体阶段
   ├─> 操作：重置该阶段之后的所有输出（删除或标记为无效）
   ├─> 操作：更新工作流状态.md（标记为返回）
   ├─> 操作：在DMN中记录"为什么返回"
   └─> 返回到环节2，步骤2.2（重新启动指定Agent）

4️⃣ 用户说"查询[内容]的细节"
   ├─> 用户需要更多信息才能决策
   ├─> 操作：从相应Agent的输出中提取详情
   ├─> 操作：用简洁的方式呈现详细信息
   ├─> 操作：在信息末尾继续问"现在您能做决定了吗？"
   └─> 继续等待用户的最终反馈（回到4种选择之一）

│
└─> 所有情况下，都要记录用户的决策和理由到DMN记忆
```

## 特殊情况处理

### 情况A：发现严重的架构问题

```
识别过程：
• 原型验证失败
• 或开发过程中发现设计缺陷
• RiMCP查询发现官方系统有突然变化

处理流程：

1. 分析问题严重性
   ├─ 能否通过小调整修复？
   ├─ 还是需要重新设计？
   └─ 需要返回哪个阶段？

2. 向用户清晰说明
   ├─ "我们发现了以下问题：[问题描述]"
   ├─ "这会导致：[后果]"
   ├─ "根本原因在：[设计的哪个环节]"
   └─ "我建议：[解决方案，通常是返工]"

3. 获得用户同意
   ├─ 用户同意返工 → 进行返工
   ├─ 用户要求继续 → 记录风险，继续（记入DMN）
   └─ 用户要求其他方案 → 讨论替代方案

4. 执行用户的决策
```

### 情况B：Agent遇到困难或卡住

```
识别：
• Agent标记为 "processing" 但超过预期时间很长
• Agent主动请求帮助
• Agent提交了失败或不完整的输出

处理：

1. 确认情况
   ├─> 询问Agent："遇到困难了吗？需要帮助吗？"
   └─> 等待Agent的响应

2. 分析问题
   ├─ 是信息不足？→ 提供更多背景信息
   ├─ 是RiMCP查询不畅？→ 协助RiMCP查询
   ├─ 是技术上无法实现？→ 寻求替代方案
   └─ 是Agent卡在某个设计决策？→ 帮助分析

3. 提供支持
   ├─ 如果是信息不足 → 补充信息
   ├─ 如果是技术困难 → 共同探讨替代方案
   ├─ 如果无法解决 → 考虑向用户报告，讨论返工
   └─ 如果是决策困难 → 帮助Agent列出选项

4. 如果无法解决
   ├─ 停止该Agent的工作
   ├─ 向用户报告："Agent遇到了困难：[问题]"
   ├─ 呈现3-5个可能的解决方案
   └─ 等待用户决策
```

### 情况C：用户改变了项目范围或目标

```
识别：
• 用户突然说："其实我想要[新的东西]"
• 或用户的新需求与之前的设计冲突

处理流程：

1. 理解新需求
   └─> 澄清用户真正想要什么（使用环节1的步骤）

2. 评估影响
   ├─ 新需求与现有设计的矛盾在哪？
   ├─ 哪些已完成的工作受影响？
   └─ 需要返回哪个阶段？

3. 向用户说明后果
   ├─ "要实现新需求，我们需要："
   ├─ "返回到[某个阶段]重新调整"
   ├─ "这会额外花费约[时间]"
   └─ "但这样我们最终会得到[更好的结果]"

4. 获得用户决策
   ├─ 用户确认新需求 → 进行返工
   ├─ 用户改主意 → 继续原来的计划
   └─ 用户要求混合方案 → 讨论和设计混合方案

5. 执行决策
   └─> 如果返工，回到环节2进行规划
```
