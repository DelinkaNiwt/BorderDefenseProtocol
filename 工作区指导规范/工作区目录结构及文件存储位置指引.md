# 工作区目录结构及文件存储位置指引

## 元信息

**摘要**：定义RimWorldModStudio工作区的目录结构，指导AI和用户合理存储文件，避免混乱。核心原则：避免重复存储（RiMCP可查询的API不存）、强调经验积累（记录错误而非"正确做法"）、明确知识边界（题材与工程分离）。

**版本**：v1.1
**修改时间**：2026-01-03
**关键词**：目录结构、文件组织、工作流、知识库、模组工程
**标签**：[定稿]

## 各部分说明

### 工作区指导规范/
- **用途**：全局规范文档，约束整个工作区的文件组织和命名
- **文件数量**：控制在 5-10 个，避免过多导致查找困难
- **维护**：由用户或知识提炼者维护，遵循全景规划 v0.4

### 模组知识/
**核心用途**：存储可复用的知识，供多个模组项目参考

**关键原则**：避免重复存储。RiMCP 可查询的 API 文档不需存储，改为记录实践中的错误和陷阱。

**各子目录说明**：
- **技术基础** - Harmony最佳实践、性能优化案例分析（RiMCP查不到的显性知识）
- **工程范例** - 从已完成项目提炼的架构模式（如权限系统、能量管理系统等）
- **题材库** - 创意改编领域的知识（创意改编案例、文案范例、题材陷阱）
- **经验教训**
  - *开发总结* - 已完成项目的全过程总结和经验
  - *问题分析及解决方案* - 常见错误、API陷阱、解决方案（而非API文档本身）
- **游戏百科** - 纯游戏知识（与模组开发无直接关系）
- **头脑风暴** - 创意灵感初稿，未经加工的想法（进入项目后转移到项目docs）

### 模组工程/
**核心用途**：进行中的模组项目

**文件存放规则**：
- `About/`：模组基本信息
- `docs/`：工作流相关文档，按 9 步工作流序号命名（可选）
  - 创意方案、需求分析、设计文档、测试报告等阶段产出
- `版本号目录/`（如 1.5/, 1.6/）：RimWorld 版本对应的代码（Assemblies, Defs, Languages, Textures 等）
- `content/`：版本间共享的内容
- `LoadFolders.xml`：版本加载配置

### 参考资源/
- **用途**：管理外部参考模组的源文件
- **与知识库的区别**：
  - 参考资源 = 原始文件和完整项目
  - 模组知识 = 提炼后的知识和经验

### 临时/
- **用途**：AI 在推理过程中生成的中间文件、背景资料、草稿等
- **清理周期**：项目完成后，有价值的内容搬到对应知识库或档案，其余删除

## 文件存放指导

### 何处存储文件

| 内容类型 | 存放位置 | 说明 |
|---------|---------|------|
| 创意灵感或初步想法 | `模组知识/头脑风暴/` | 未经加工，可能进入项目 |
| API 错误和陷阱 | `模组知识/经验教训/问题分析/API错误与陷阱/` | ✅ 不存储API文档（用RiMCP查询），而是存储常见错误 |
| Harmony最佳实践 | `模组知识/技术基础/` | RiMCP查不到的显性知识 |
| 性能优化案例 | `模组知识/技术基础/性能优化/` | 项目中验证过的优化方案 |
| 架构模式（已验证） | `模组知识/工程范例/` | 从完成项目中提炼，如权限系统、能量管理等 |
| 创意改编方向 | `模组知识/题材库/创意改编案例/` | 创意总监的改编经验（如三体→RimWorld） |
| 文案范例 | `模组知识/题材库/文案范例/` | 具有题材特色的游戏内文案 |
| 题材陷阱记录 | `模组知识/题材库/题材陷阱记录/` | 这个题材常见的改编误区和解决方案 |
| 项目进行中的文档 | `模组工程/具体模组/docs/` | 包括需求、设计、测试报告 |
| 项目的游戏资源 | `模组工程/具体模组/版本号目录/` | Defs, Textures 等 |
| 完成项目的总结 | `模组知识/经验教训/开发总结/` | 整个项目的经验沉淀 |
| 外部参考模组源码 | `参考资源/模组资源/` | ProjectAldon等完整参考项目 |
| AI 推理中间产物 | `临时/` | 过期或不需要的删除 |

## 重要约束

1. **禁止位置**：
   - ❌ 不在 `模组知识/` 中直接放置未来项目的代码
   - ❌ 不在 `模组工程/` 中存储通用知识（应归档到知识库）

2. **文件名规范**：
   - 参考 `工作区指导规范/文件命名规范.md`
   - 模组工程 docs/ 内的文件：`序号_阶段名.md`（如 `2_需求和系统设计.md`）

3. **版本管理**：
   - 所有规范文档需要版本号（如 `_v1.0.md`）
   - 修改文档后，更新版本号并记录于历史表格
